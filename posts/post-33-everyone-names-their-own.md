# Post 33 — Everyone names their own

- post 31 ended on a promise: the userspace-metrics milestone, *designed* to be un-poisonable from the first commit — per-process names, no shared vocabulary, emit authorized by registration. this is the post where it shipped. but the part worth writing down isn't "I built the thing I said I'd build." it's the three things shipping it taught me that the whiteboard hadn't: that the metric everyone leaned on **wasn't a metric**, that the exact same hole was sitting one module over in the **span** path, and that the secure design **leaks more memory than the bug did** — and that's the right trade.

## the total that was never a total

- the old path was `Invoke` → a `TelemetrySink` cap fused to a kernel-registered counter → `snitchos.user.telemetry_total`. every userspace program emitted to that one shared counter. retiring it was supposed to be the boring migration step. then I actually looked at what flowed through it.
- nothing used it as a *total*. `hello` emitted `42`. the heap probe emitted `524288` — the size of an allocation it wanted to prove succeeded. the RPC client emitted its reply value; the IPC receiver emitted the word it received; the badge server emitted the badge it was handed. fifteen scenarios asserted **exact values**, never a rate. `telemetry_total` was a **printf-for-tests bus wearing a counter's name** — a v0.7a vestige from before any of the cap machinery existed. graphing it in Grafana would be nonsense: a line lurching from 42 to 524288 to a badge id.
- so I didn't keep it. every program now registers *its own* marker metric — `snitchos.hello.marker`, `snitchos.fs_client.marker` — through the new per-process path, and emits through a handle the kernel validates against that process's own table. the shared forgeable counter is gone, not migrated. the lesson that keeps recurring: **the legacy thing you're about to carefully preserve may not deserve preservation** — look at what actually flows through it before you build a bridge to it.
- retiring `Invoke` also meant renumbering the syscall ABI — and a small moment of "wait, am I allowed to do that?" the postcard *frame* format is genuinely frozen: the collector decodes captures, old and new, so you append fields and never renumber. but syscall numbers? the kernel and every userspace program are compiled from the **same crate, in the same build** — the user ELFs are embedded into the kernel image. nothing persists a syscall number across a version boundary. so `Invoke = 0` could just *go*, and the rest slide down. **know which of your contracts is actually frozen.** I'd been treating two very different things as equally sacred because they lived near each other.

## the same hole, one module over

- here's the one I didn't see coming. post 31's win was a *method*, not just a fix: ask "who can forge this?" of every authority. so I asked it of the path right next to metrics — **span names**. and the answer was identical.
- a userspace `SpanOpen` copied the span name in and interned it against the **global** name table, content-deduped. which means a process could open a span named `"kernel.heartbeat"`, resolve to the kernel's own string id, and emit a span *under the kernel's name*. the same poisoning, the same shape, the same root cause — a global deduped vocabulary — sitting in a path I'd already shipped three milestones ago and never questioned. the metric hole I caught on a whiteboard before writing it; the span hole had been live for months.
- the fix is the metric fix, copy-pasted in spirit: a per-process span-name table, no cross-process dedup. a process opening `"kernel.heartbeat"` now gets its *own* string id for that name — distinct from the kernel's — so it literally cannot emit under another principal's span identity. (spans were always fine on the *attribution* axis — every `SpanStart` carries the emitting `task_id`, so Tempo always knew *who* opened a span. the hole was the *name*: the task_id said "task 5," but the name said "kernel.heartbeat." you fix the name.)
- the meta-lesson sharpened: a security question you ask once is worth re-asking of **every sibling path**. "who can forge a metric name?" and "who can forge a span name?" are the same question; finding the answer for one should have sent me looking for the other immediately. it didn't — I found the span hole only because I went looking on purpose, after. the discipline isn't asking the question; it's asking it *everywhere the same authority shape repeats*.

## the secure design leaks more

- this is the honest one. the *whole point* of per-process scoping is **no cross-process dedup** — two processes' `"requests"` are two different metrics, because they're two different emitters. correct, and the security property depends on it. but it has a cost I hadn't priced: the old global table interned a name **once** and every process reused it. the new per-process tables intern a fresh copy **per process** — so every time the shell spawns a program, it re-leaks that program's metric and span names. the leak went from O(distinct names) to O(spawns × names-per-program).
- the secure design leaks *more* than the insecure one did. that's not a bug in the fix; it's the fix's shape. dedup was the thing saving memory, and dedup was exactly the thing that made the snitch poisonable. you can't have the sharing and the isolation.
- so I did the un-heroic thing: **named the cost, bounded it, and deferred the reclaim.** each process's name table is quota-capped (16), so a single process can't run away; the total grows only with spawns, which today is a handful. real reclamation — drop a process's leaked names when it exits — wants the address-space teardown hook that the *next* milestone is building anyway, so it rides along there. I wrote the bound into the docs and a plan for the GC, and moved on. the temptation was to solve the leak now, in the same change, to feel "done." but a documented, bounded, deferred cost is a fine place to leave a system — far better than coupling a security fix to a memory-reclaim project because they happened to touch the same field.

## what the collector had to learn

- one consequence reached all the way to the host side. once two processes can name a metric the same, the collector sees two different string ids that resolve to the *same* Prometheus name — and Prometheus exposition with two `# TYPE` lines for one metric name is simply **invalid**. the wire change was tiny (stamp the registering task on the one-per-metric registration frame, not on every sample); the collector groups by name and separates the emitters with a `task="…"` label. two `snitchos.probe.custom` series, one per process, distinguished by who emitted them.
- the satisfying part was *why metrics needed this and spans didn't*. a span is an **event** — every `SpanStart` already carries its full context, the emitter included, because that's what an event *is*. a metric sample is an **aggregate** — deliberately lean, just `(name, value, hart)`, no room for "who" on the hot path. so the emitter had to be reconstructed for metrics (from the registration) and was already free for spans. **events carry their own context; aggregates throw it away to stay cheap, and you pay to get it back exactly when you need to tell two of them apart.**

## what i learned

- **look at what flows through the legacy thing before you preserve it.** `telemetry_total` was a counter in name only — a test bus. the right migration was deletion, not a bridge.
- **know which of your contracts is frozen.** the postcard frame format is (old captures decode forever); syscall numbers aren't (one crate, one build). treating both as sacred is cargo-cult caution.
- **a security question generalizes to every sibling path.** "who can forge this name?" answered for metrics should have immediately been asked of spans. the same authority shape hides the same hole.
- **the secure design can cost more resources than the bug.** no-dedup is the security property *and* the memory cost; you can't keep the sharing. name the cost, bound it, defer the reclaim to where the machinery already lives — don't fuse a security fix to a reclamation project.
- **events carry their context; aggregates pay to recover it.** spans had the emitter for free; metrics had to reconstruct it. that's the difference between a thing that happened and a thing that's been summed.

## what's next

- the lifecycle milestone has one milestone-sized piece left open on purpose: a general **notification primitive** — an async kernel→user signal. child-exit is its first consumer (a parent waiting on a child), but the real reason it's worth its own design is that **device interrupts reuse it**: the same "wake userspace when something happened" path that reaps a zombie will later deliver a keystroke. after that, the milestone everything has been pointing at — the **explicit-authority shell**, where you watch least-authority happen: every program launched with exactly the caps it needs, every delegation a `CapEvent` on the wire. the snitch has been made honest; next it gets a shell to narrate.

## addendum — a stack that names its own overflow

- a different theme, same instinct, so it gets a footnote rather than a section. back in v0.11 a deep syscall path overflowed a 16 KiB kernel stack — `Box::new(BigValue)` materialized a stack-sized temporary on the *caller's* stack before moving it to the heap. it surfaced as the worst kind of bug: unrelated crashes that varied per run (a null-deref one boot, an alloc-error the next), because a stack overflow corrupts whatever heap object sits below it, and the damage shows up at the *victim*, never the cause. the specific trigger was already prevention-fixed (allocate the stack directly on the heap, never via a stack temporary), so nothing was crashing. but there was still **no detection** — the next overflow would be just as mysterious.
- so I made the failure observable, which is the only move this project knows. each kernel stack is filled with a sentinel byte at creation; the bottom 16 are a **canary**. the scheduler checks it on every context switch (prompt, but running on the possibly-damaged stack), and the heartbeat checks every task's canary as a backstop (slower, but running on its *own* healthy stack). a breach **snitches a `Log` — "kernel stack overflow: task N (name)" — then panics.** the overflow stops being a varying mystery and becomes a named line on the telemetry wire. plus a per-task `stack_high_water_bytes` gauge from a bottom-up scan, so you can watch a stack creep toward its limit *before* it blows.
- the testing problem was the interesting part, and it rhymes with post 31's "the test that proves a panic is gone has to try to cause the panic." here the opposite shape: a *real* overflow is exactly the unpredictable-corruption scenario — it might crash anywhere before cleanly hitting my detector, so I can't write "overflow and assert the panic" and trust it. the fix is a **controlled clobber**: a test-only task overwrites its own canary directly, `sp` untouched, so the rest of the stack stays intact and detection fires deterministically. it proves *detect → name → halt*, which is the mechanism I'm actually testing. surviving an *arbitrary* overflow gracefully is a different guarantee — that's **guard pages** (fault at the overflowing store, before any corruption), deferred to ride the same address-space-teardown machinery as the name-GC above.
- the honest framing: this is **post-hoc** detection. the canary tells you the stack overflowed *after* it did, on memory that may already be damaged — it turns a silent corruption into a named fatal, which is a real upgrade, but it is not prevention. naming a failure cheaply is worth a lot when the alternative is hours of bisecting a crash that moves every run; it is not the same as making the failure impossible. tier A names it; tier B will forbid it.

# Stitch 3 — there is no method call

- last post ended on a promise: `on`/methods and `contract`s, the one polymorphism Stitch has — the "how Java's `invokevirtual` actually works" lesson, written by hand rather than generated, on purpose. this is that. by the end of it `reading.toFahrenheit()` works, types carry behavior, contracts hand out default methods, and a `mut` method can change its receiver without quietly changing everything that aliases it.
- the surprise this time wasn't the dispatch. dispatch is a hash lookup. the surprise was that **half the work was deciding what a method call even *is*** — and the other half was mutation, which is the question a value-semantics language spends its whole life avoiding and then can't avoid once methods can write fields.

## the shape — there is no method call

- here's the thing i kept having to re-internalise: **Stitch has no method-call node.** `reading.toFahrenheit()` parses as a plain call whose callee happens to be a field access — `Call { callee: Field { object: reading, name: "toFahrenheit" } }`. the exact same `Field` shape as reading a struct field. the parser doesn't know "method" from "field"; it just builds the tree.
- so dispatch is a decision made at *eval* time, by **intercepting that shape before evaluating the callee**. when the evaluator sees a call whose callee is a field, it doesn't evaluate the field (that would just try to read a struct member and fail) — it pulls the object and the name apart, evaluates the object to a receiver, and looks the name up in a method table keyed by the receiver's type. this interception is the whole trick. everything else is bookkeeping.
- the method table itself rides the exact machinery the globals already used: a write-once, shared `Rc<OnceCell>` table threaded through every scope. same letrec shape as last post's mutual-recursion fix. methods are top-level, mutually visible, filled once after parsing — so they get the same plumbing, for free.

## a vtable, slowly

- this is the `invokevirtual` lesson, and writing it by hand makes the comparison vivid. a Java vtable resolves a method name to a **slot index at compile time**; the call site is `load the object's table, index slot N, jump`. no names at runtime, O(1), cache-friendly. it works because static types told the compiler that `foo` is slot 3.
- Stitch, having no types yet, does the honest dynamic thing: **resolve the name by string, every call** — hash the receiver's type name, scan its methods. it's what Python and Ruby do, not what Java does. slower, flexible, no types required.
- and that's the point of building the slow version first: **a vtable is just this map with the lookup pre-computed.** once static types land, the compiler can turn "method named `foo`" into "slot 3" and the hash lookup becomes an array index. i now know exactly what the fast version optimizes, because i wrote the thing it's an optimization *of*. same bet as the tree-walker-before-bytecode arc — understand the obvious version, then earn the clever one.

## the arguments that mattered

- **the receiver is never implicit.** inside a method, you reach a sibling method as `@speak()`, never bare `speak()`. i was genuinely on the fence — implicit `self` reads nicely (`speak()` instead of `@speak()`). but making bare names sometimes-a-method reintroduces exactly the ambiguity the `@` sigil was invented to kill: is `speak` a local, a global, or a method on me? with the rule "bare = lexical, `@` = receiver," name resolution stays one flat decision. i kept the noise and bought the predictability.
- **mutation is copy-then-write-back, not shared mutation.** this is the one a value-semantics language can't dodge once `@balance = @balance + amount` exists. the easy implementation — make the record's fields a shared mutable cell — would give *reference* semantics: `let a = account; account.deposit(10)` would silently change `a` too. so instead a `mut` method runs against a mutable *copy* of the receiver and, when it returns, writes the mutated copy back to the caller's binding. records stay immutable; "mutation" is rebuild-then-reassign-the-place. `let a = account` keeps its own value. the footgun is designed out, not documented away — same instinct as last post's equality decision.
- **only `mut` fields can be assigned.** `prod Account(owner: Str, mut balance: Int)` says balance is mutable and owner isn't, and the runtime enforces it: assigning `owner` is an error even through a mutable binding. it's dynamically typed today, but this previews the static discipline rather than contradicting it — the same reason `1 + 2.0` is still a type error.
- **conformance is the compiler's job, not the interpreter's.** `on Celsius : Show` declares that Celsius implements the `Show` contract. checking that it *actually* implements all of Show — that's a static, coherence question. a running interpreter assumes the program type-checked. so the dynamic dispatcher ignores contracts almost entirely; the *only* time it consults one is to borrow a **default method** the type didn't override. that realisation deleted a whole layer i'd started to build.

## the parts that fell out for free

- **late binding was already done before i wrote it.** a contract can ship a default method whose body calls another method — `loud() = "{@speak()}!"`. when a Dog that implements `speak` inherits `loud`, calling `dog.loud()` has to dispatch `@speak()` to *Dog's* `speak`, not some contract-level stub. that's open recursion — the template-method pattern, the reason default methods are worth having. and it needed zero new code: `@speak()` is just another `Call { callee: Field }`, so it re-enters the same dispatcher with `@` still bound to the concrete Dog. the receiver carries its real type the whole way down. when dispatch is uniform, virtual-from-within-a-default is not a feature you add — it's a thing you fail to prevent.
- **default methods are just copied in.** rather than a runtime fallback chain, each contract's default methods get folded into every conforming type at registration — unless the type already defines that name, in which case the concrete one wins. resolution stays a single flat lookup; the "use the default if there's no override" semantics happen once, up front.
- **a bug from reusing the channel.** last post i made `?`'s early-return ride the error channel, caught at the function boundary. methods are a function boundary too — and i'd forgotten to catch it there. a method using `?` would have unwound straight past its own call. the fix was one line, but it's the kind of gap you only find by writing a method that uses `?` — which the mutation tests finally did.

## hunting the corners on purpose

- with the system green, i went looking for edge cases adversarially — writing tests i *expected* to fail, to find where dispatch and fields collide. two real gaps fell out.
- **a field that holds a function couldn't be called.** `box.f(10)` where `f` is a field holding a closure errored with "no method `f`" — because the interception assumed every `obj.x()` is a method call. the fix: when no method matches, fall back to reading the field and calling its value. methods take precedence; a callable field is the fallback. now records-of-callbacks work, and `box.n` (field) and `box.n()` (method) can even share a name.
- **`free` methods don't work on sum types yet.** a `free` (no-receiver, called-on-the-type) method like `Counter.zero()` resolves the type through its value — and a sum's type name isn't a value, only its variants are. left it as a documented, `#[ignore]`d test: an honest marker beats a silent hole. the corners you write down are the ones you fix on purpose later.

## what i learned

- **dispatch is a lookup; the design is *which namespace, and when*.** the mechanism took an afternoon. what took thought was deciding that methods are a separate, type-directed namespace from variables — and that the whole vtable story is just moving the "when" of that lookup from runtime to compile time.
- **value semantics survive mutation only if you pay for it.** the copy-write-back dance is the price of "`let a = x` means `a` is its own thing." it would have been so easy to make records share a cell and call it a day, and the language would have been quietly worse forever.
- **uniform plumbing pays compound interest.** late binding, `@`-field access, sibling calls, the `?`-fix — all of them fell out of "a method body is just an expression evaluated with `@` bound." every time the new feature is a special case of the old machinery, you've done something right.
- **find your gaps before your users do.** the most useful tests i wrote this stage were the ones designed to fail. two did. one's fixed; one's a labelled hole. both are better than not knowing.

## what's next

- **the `?` operator becomes an open contract.** it's been hardcoded for the built-in `Maybe`/`Result` since post 2. now that dispatch exists, `?` can desugar to a `Try` contract any type implements — a custom `Permission = Granted | Denied` short-circuits on `Denied` like an `Err`. the payoff of building dispatch is that the hardcoded thing becomes a mechanism.
- then **lazy sequences** — the no-loop-keywords story isn't finished until infinite producers and `takeWhile` work — and after that the two things that were always the point: the **bytecode VM with a real GC** (where this hand-written dispatcher becomes inline caches and slot indices), and the **capability effect system** wired to the real telemetry protocol.
- but types carry behavior now. a contract hands out a default and the right concrete method answers the call. it dispatches — slowly, honestly, and exactly the way the fast version will, once it has types to cheat with.

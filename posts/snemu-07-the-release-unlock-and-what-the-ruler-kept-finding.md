# snemu 07 — the release unlock, and what the ruler kept finding

- snemu-06 ended on a promise and a bug. the promise: a release kernel would cut debug's per-instruction bloat suite-wide. the bug: release codegen killed the timer and spewed UART garbage, so the itests wouldn't even boot. "the ruler found the bug; next post, we hunt it." this is that post. we hunt it, fix it, and the release build unlocks — and then the same ruler keeps pointing at the next thing, four more times: a userspace UB class, an O(n) buried in every context switch, a leak that couldn't be reaped without wedging, and a poster-child optimization that measured out to _nothing_. we go **10.8s → ~4s** on the way, and end by disproving the thing I was most sure would help.

## the bug: a pointer truncated across the trampoline

- the release kernel reached `entering heartbeat` under snemu and then vomited `.rodata` onto the console; under QEMU the timer IRQ just stopped. classic UB signature — works in debug, dies optimized. the first move was a `--release` flag on `snemu-itest` so the whole thing reproduced in-process, deterministically, in seconds instead of a QEMU round-trip.
- the fault handler couldn't even format its own panic — every `{}` came out as `�`. because the fault _reused the faulting stack_, and `core::fmt`'s scratch landed past a blown `sp`? no — that was a red herring. a fresh `const` formatted fine; only `info` deref was garbage. so it wasn't the stack, it was the **pointer**. a raw-hex fault report (no `core::fmt`, a nibble loop straight to the UART) gave clean numbers: `stval=0x00000000_803222d8`, and the same address should have been `0xffffffff_803222d8`. the high `0xffffffff` was gone. **a 64-bit kernel VA truncated to 32 bits.**
- `llvm-objdump` the faulting `sepc`: it was `ld t0, 24(tp)` in `trap_entry` — the new per-hart exception-stack read. `tp` held a truncated `PER_HART_DATA` address. and `tp` is set once, in `percpu::init`, right after `kmain`'s higher-half trampoline. disassemble the two `mv tp` sites and there it is: the `auipc` that materializes `&PER_HART_DATA` got **scheduled before the trampoline `jr`** — where PC is still physical — so it computed a physical address and carried it, in a callee-saved register, across the jump into a higher-half world.
- the trampoline is not a barrier the compiler respects. under `code-model=medium`, `&static` is PC-relative, and the optimizer is free to hoist that computation to wherever, including before the very jump that redefines what "PC-relative" means. debug never hoisted it. and `current_hartid()`'s range check silently swallowed the bad `tp` for _months_ — returning hart 0 — until the exception-stack asm read it raw with no fallback. a dormant bug, exposed by an unrelated feature and a codegen change at the same time.
- the fix is three lines: materialize the base with a **side-effecting `asm!("lla …")`** instead of a plain `&`. a non-`pure` asm block is ordered after the trampoline's asm, so it can't hoist across. release snemu-itest went **0/108 → 104/108**, and QEMU booted clean to heartbeat. the promised UB, dead.

## the second UB, and the pin

- the remaining four were the FS server and the spawn/reap path, and they were a _different_ bug: under `opt-level ≥ 2`, the userspace `talc` allocator's OOM handler looped mapping 68 KiB regions until it hit the per-process heap cap, then hung in the panic handler. the embedded userspace builds `--release` too, so a release kernel dragged it in. an opt-level bisect (per-crate) pinned it to `snitchos-user`, plus at least one more crate — a _class_, not a single site, and heisenbug-y (a `debug_write` in the path made it vanish).
- the release-itest speedup is kernel-dominated — userspace is a tiny instret fraction — so `build.rs` now pins the embedded userspace to **opt-1** while the kernel stays opt-3. sidesteps the whole class for ~free. root-causing the userspace UB is filed, not done. **108/108, ~10.8s.** the unlock is real.

## the ruler, per-function this time

- snemu-06 built a per-_scenario_ instret table. it says _which_ scenario is the pole, never _why_. so: `snemu-profile` — exact per-PC instret counting (every retired instruction attributed to where it ran, deterministic like the clock), rolled up to kernel function names via the ELF symtab. a profiler for the guest, with no sampling error.
- first thing it said, pointed at `spawn-storm`: **`kernel::sched::prepare_switch` is 66%.** two-thirds of that workload's instructions were in the context-switch path. and it's the _shared_ hot path — the same 66% shows up across every switch-heavy workload. the per-scenario ruler could never have told me that; it's cross-cutting.

## the O(n) hiding in every context switch

- `prepare_switch` scanned the entire task table — `for task in tasks { if task.id == current … if task.id == next … }` — **every switch**. O(tasks). and the table _accumulated `Exited` zombies_: `spawn-storm`'s 200 fire-and-forget kernel tasks exited but were never removed (a documented leak: `Box<Task>` + a 16 KiB stack each). so switches were O(200) over 199 dead entries. a real O(n²) in the scheduler, and a real memory leak, in one.
- fix A — a `TaskDirectory` (id→slot map, host-tested in `kernel-core`): the scan becomes two direct lookups, the two tasks being switched between. `prepare_switch` **66% → 19%**. and we made the O(1) _assertable_: a `live-tasks` workload holds 200 genuinely-live tasks, and a scenario asserts probes-per-switch stays constant (**2.0** observed; a scan would be ~200). then proved the test bites — revert the lookup to a scan and it reads **79.9**, that scenario fails, every behavioural test still passes. a killed mutant, not a vibe.
- fix B — reaping the zombies — was where it got interesting. the naive version dropped `Box<Task>`, whose `Drop` unmaps the kernel stack, which does a cross-hart **TLB shootdown per page**. run from the heartbeat sweep, that shootdown _wedged_ — hart 0 spin-waiting on hart 1's ack while hart 1 waited on hart 0. the fix isn't "reap more carefully," it's **don't unmap**: a `MAPPED_STACK_POOL` where `Drop` returns the slot _mapped_ and the next spawn reuses it. no unmap, no shootdown, memory bounded by peak concurrency. `TaskDirectory::slot_of` fell out of the `spawn-storm` profile entirely. (bonus: the userspace reap path is shootdown-free now too.)

## the pole was memory you had to touch

- with the switch cost gone, the pole was `spawn-reclaims` (75M ×2). profiler: **41% `memset`** — 30 children each touching ~4 MiB, sized to exceed the 128 MiB machine so a non-reclaiming kernel OOMs (the test's teeth). that instret floor is ∝ the machine RAM: you _must_ touch more than RAM to prove no-leak.
- so shrink the machine. 48 MiB + 15 children = 60 MiB touched, which cuts the child count 2× (halving the `memset`) _and_ makes the teeth **more** robust — the old `120 < 128` leaned on kernel overhead eating the margin; `60 > 48` clears it outright. **75M → 41M**, makespan **4.6s → 3.7s**, utilization 81% → 90%. (a reviewer caught the fragility of the old margin — the shrink fixed the speed and the teeth in one move.)

## the memset that wasn't

- `memset` + `memcpy` were ~47% of that pole, and they're the classic emulator collapse: the guest spends 1544 instructions doing what the host does in one `memset`. so — tier-0.5 of a JIT: intercept the kernel's `memset`/`memcpy` at their ELF-resolved entry PCs, run the op natively on guest RAM, and charge the instret the loop _would_ have retired (so the deterministic clock, and thus the frame stream, is unchanged). architected as the shared "PC → native block" dispatch a real JIT generalizes.
- it worked. it also **didn't help.** the measurement, honestly:
  - **fidelity needs a gate.** collapsing a memset while another hart runs robs it of interleaved instructions and diverges cross-hart timing — measured, it wedges `viewer-reads-delegated-file`. so fire only when every other hart is idle. but the idle task _flickers_ (`wfi; yield`), so the gate misses the _workload_ memsets — only single-hart boot collapses. total instret −15%, boot −45%, but `spawn-reclaims` unmoved at 41M.
  - **ungate it and the win is huge** — `spawn-reclaims` 41M → **9M**, 4.5× — **but fidelity breaks** (that viewer wedge). the safe version and the win are in direct tension; capturing both needs a lockstep-preserving helper (fire, but advance the other harts by the charged instret), which is more than tier-0.5.
  - **the decisive one: it doesn't move the makespan at all**, because the current pole isn't memset-bound — it's `smp-tlb`, a scheduler/`assert_absent` scenario. the helper can't touch it, and its per-instruction dispatch adds a hair of overhead everywhere, so gated it's a slight _regression_ (4.0 → 4.4s) despite cutting total instret.
- this is snemu-06's idle-skip result again, one level up. the ruler measured the hypothesis I was surest of — "half the pole is memset, collapse it" — and disproved it as a _makespan_ lever. it's a throughput win and a valid JIT foundation; it is not the thing that makes the suite faster right now. the pole is scheduler and spin-wait, so the lever is a full JIT (which speeds _every_ block) or pole-specific trimming — not a memset helper. left it in, off by default, gated, as the dispatch a JIT will plug into.

## what I learned

- **the trampoline is a compiler barrier you have to build.** "PC-relative" silently means "relative to wherever the optimizer put the computation," and a jump that redefines PC needs a side-effecting fence or it'll carry a stale address across. a whole class of higher-half bugs lives here.
- **when the fault handler can't speak, believe the raw bytes.** garbage formatting looked like a blown stack; it was a truncated pointer. a `core::fmt`-free hex dump + `objdump` the `sepc` cut through in one step. don't debug the reporter.
- **profile the shared path, not the slow scenario.** `prepare_switch` was invisible in the per-scenario table and 66% in the per-function one. the biggest wins are cross-cutting, and only a function-level ruler finds them.
- **reaping is a TLB problem, not a bookkeeping one.** the leak was easy; freeing without a cross-hart shootdown was the actual design. pool the mapping, don't tear it down.
- **the ruler disproves your best hypotheses.** twice now — idle-skip, then the memset helper — the thing I was most sure would win measured out to zero-or-negative. building the instrument first is what makes those cheap to discover instead of expensive to ship. the makespan is pole-bound; optimize the pole, or optimize everything (a JIT), but measure which one the pole _is_ before you build for it.

## what's next

- the pole is `smp-tlb` (a `assert_absent` scan cap) and the scheduler. cheap: trim the cap. structural: the full JIT — the tier-0.5 dispatch is already in, and unlike the memset helper it speeds the scheduler and spin-wait blocks that actually bound the makespan. and the deeper-snapshot work (boot is still ~25% of instret) is orthogonal to the JIT by construction — the code cache is a pure function of the immutable kernel binary, so it's shared-immutable and never part of a snapshot. three levers, and this time we know which one the pole is.

## addendum — the viewer wedge was confounded evidence

- a reviewer caught that my "ungated memset wedges `viewer-reads-delegated-file`" is _not_ clean evidence of a fidelity break. `viewer`/`shell` are exactly the two known-flaky supervision-blocked scenarios — they fail in debug and pass at opt=mid only by timing luck. a scenario that flips under a timing perturbation just tells you it's timing-sensitive, which we already knew. so ungated might well be fine _for the suite_ once the underlying supervision race is fixed; the clean test is to exclude those two and see if any _correct_ scenario flips.
- what survives, though, is a fidelity concern independent of viewer: snemu's lockstep (1 instr/hart/round) actually models parallel hardware decently — during hart 0's 1544-instruction memset, hart 1 also advances ~1544 instructions, like real silicon. ungated collapses that to one round, so hart 1 advances _one_ instruction during it — it **starves the other hart of interleaving**. memset writes private memory, so it doesn't reorder shared accesses directly, but it moves _when_ hart 1 reaches its next shared access — i.e. it shifts race windows. faithful cross-hart interleaving is the whole reason snemu caught the TX_STAGING deflake race, so the risk isn't "the suite fails," it's "a _future_ race silently doesn't reproduce." passing the suite and being fidelity-exact for race detection are different bars.
- and the whole debate is moot with the right helper. the **lockstep-preserving** version fires the memset always (no gate), then advances the other running harts by the charged instret before continuing — reproducing the interpreter's interleaving _exactly_ (hart 1 gets its N steps during hart 0's collapsed memset). fidelity-exact, captures the workload win, no `only_running_hart` gate. strictly better than both gated and ungated. so the corrected takeaway: the gate was a symptom-fix, the viewer wedge was confounded evidence for it, and the real answer is to preserve lockstep rather than gate around it.
- note(chloe): maybe in future we can make this a build flag, so that we can choose between "as fast as possible" and "race-preserving".

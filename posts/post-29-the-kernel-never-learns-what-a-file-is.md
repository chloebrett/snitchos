# Post 29 — The kernel never learns what a file is

- post 28 ended pointing here: the IPC was built, two processes could talk and you could watch the whole conversation as one trace, and the next thing was to make that IPC carry a *real* protocol instead of a demo's sentinel. v0.10 is a filesystem — but the storage is the boring part. the deliverable is the *interface*, reached only through capabilities, spoken only over the IPC from last milestone. and the design rule i kept bumping into is a single sentence that decides almost everything below it: **the kernel never learns what a file is.** the two things i'll remember are how literally that rule held — the kernel moves a file's bytes without ever being able to interpret them — and that the capability i needed to make it safe was one i'd already built for something else.

## the interface is the deliverable

- the whole milestone exists to pin down one trait. `Filesystem`: `root`, `lookup`, `stat`, `read`, `write`, `create`, `remove`, `readdir`. it lives in its own crate that imports *no* capability or IPC types — so it compiles and unit-tests on the host like every other piece of pure logic in this project, no QEMU, no kernel. the implementation behind it (`RamFs`) is deliberately trivial: a flat, single-directory, in-memory store; creating a subdirectory returns `Unsupported`. that's not a limitation i'm apologising for — it's the point. the trait is the expensive thing, fixed once; the implementation is cheap and meant to be swapped. CoW, snapshots, content-addressing are all *additive behind this same trait* later. interface before implementation, the project's oldest rule, applied to its most tempting milestone to do backwards.
- one inheritance from how filesystems actually work, worth saying because it shapes the cap model: a file's identity is its **inode**, not its name. a name is just a directory entry pointing at an inode. that decoupling is what lets a capability name a file that survives a rename — you hold a cap to the *inode*, and renaming only touches the *directory entry*. same identity-vs-name split as capabilities themselves, as string interning, as sockets. it keeps showing up.

## the kernel never learns what a file is

- here's the bet stated as mechanism. the kernel provides *badged endpoints* — one IPC endpoint, and every message arrives stamped with an unforgeable 64-bit badge the sender can't choose. the filesystem packs `(inode, file-rights)` into that badge. so a "file capability" is just a badged endpoint cap, and the server demultiplexes every request by the badge: *which inode, which rights*. the kernel carries the badge and never reads it. it carries file-rights like `READ`/`WRITE` and never interprets them — those are the *filesystem's* rights namespace, enforced by the filesystem, per message. the kernel's own rights (`SEND`/`RECV`/`MINT`) govern the *endpoint*, not the file. two rights systems, one carried inside the other, and the kernel only understands the outer one.
- the filesystem is therefore the *sole authority* over what any file cap names. it holds the mint right; handing out a narrower cap is a filesystem mint, not a kernel operation. a client gets its first cap by *connecting*: it calls the server on a bare endpoint cap (badge 0 = "attach"), and the server mints it a root capability and hands it back in the reply. from there the client `lookup`s and `create`s its way to more caps, each minted by the filesystem with exactly the inode and rights the filesystem chooses. the kernel inserts the caps across the process boundary, but it's the filesystem deciding what they *mean*. "the kernel provides badged endpoints; the filesystem provides all file meaning" stopped being a slogan and became the actual call graph.

## don't communicate by sharing memory

- then the question that the whole "real protocol" promise comes down to: a message is four machine words, and a filename or a file's bytes don't fit in four words. how do they cross? two honest answers. **share memory** — map the same physical pages into both processes, zero-copy, the way high-throughput systems move bulk data. or **pass the message** — the data is *copied* from one address space to the other, ownership moves, nothing is shared. it's Go's maxim — *don't communicate by sharing memory; share memory by communicating* — and at the OS level, across a boundary between two mutually distrusting processes, it's *stronger* than it is between goroutines: a shared region exposes the whole buffer, including whatever the sender left next to the part it meant to share, and now two parties have to coordinate who writes when. message-passing has none of that. ownership is unambiguous because the receiver gets a private copy.
- i chose copy. the honest cost is that copying isn't free — but it's exactly the cost of `read()`/`write()` on any normal OS, which copy between the page cache and your buffer on every call. the zero-copy alternative is `mmap`, and it's real, and it wins for large sustained streams where the copy bandwidth starts to bite. so it's not banished — it's *deferred*, the data-plane optimization you reach for when measurement says the copy is the bottleneck, not before. a RAM-backed filesystem for a learning kernel is nowhere near that line. and every copy is already a span, so if i ever cross the line, the traces will say so before i guess.

## the copy the kernel can't `memcpy`

- the part i underestimated: even having chosen "copy," you can't just copy. when the server wants the bytes behind a client's pointer, that pointer is a virtual address in the *client's* address space — which isn't the one loaded in the MMU, because the server is the running process. dereferencing it reads the server's memory at that address, which is garbage or a fault. the trick the kernel uses everywhere else — flip the "supervisor can touch user memory" bit and dereference — only works for the *currently active* process.
- so the cross-address-space copy has to do it the long way: walk the *other* process's page table by hand to translate its virtual address to a physical one, then reach the physical bytes through the kernel's linear map, which is mapped into every address space at once and so is reachable no matter whose page table is live. and because a buffer spanning a page boundary lands in two *non-contiguous* physical frames — with the source and destination at different offsets within their pages — the copy proceeds page by page, advancing each step by the smaller of "bytes left in the source page" and "bytes left in the destination page." that little min-of-three is the only real arithmetic in the whole feature, and it's the part i host-tested hardest, because it's the part that's wrong in a way nothing else would catch.

## the reply cap was already the key

- which left the question that decides whether any of this is *safe*: who is allowed to copy from whose memory? a server reaching into an arbitrary process's address space is exactly the authority you do not want ambient. and the answer turned out to be a capability i already had. v0.9b's **reply capability** — the one-shot cap the kernel mints at a `call` rendezvous so a server can answer *this specific caller, exactly once* — names a *blocked* caller. and that's precisely the authority i needed: a server may touch a caller's memory *only* while it holds that caller's reply cap, which means only a caller that called *it* and is parked awaiting *its* reply. the caller can't mutate its buffer mid-copy because it isn't running. the scope can't be forged because the cap can't be. i didn't add a "memory access" capability; the reply cap already *was* one, viewed from the right angle.
- this is the same lesson post 28 ended on, wearing a different hat. there i'd built a mechanism (`revoke`) before naming the concept (single-use capability). here i nearly built a new authority for cross-process memory access before noticing the authority already existed — the reply cap, which i'd been thinking of as purely "permission to send one reply," is more generally "a handle on a specific blocked counterpart," and *copying to/from its memory* is just another thing that handle legitimately lets you do. the copy syscalls *borrow* the reply cap (they don't consume it — only the final reply does), so a server can pull a request's data, do its work, and answer, all under one grant. reach for the concept before the mechanism; the concept you need is often one you already shipped.

## the test that wanted to be deleted

- a smaller story, because it's the kind that repeats. my first cut of the copy validated the whole range up front — walk every page, check it's mapped with the right permissions — and only then copied, so a bad pointer could never leave a half-written destination. clean, atomic, correct. and nearly impossible to mutation-test: the validation pass walks the pages but its *result* doesn't depend on the exact stepping, so when i mutated the loop arithmetic, the tests didn't notice. a whole pass of logic that no test could pin. that's a smell, not a victory.
- the fix was to delete it. one pass: translate, check, copy, page by page, and bail on the first bad page. the cost is that a refusal can leave leading bytes already copied — but a failed copy is a discarded transfer anyway (the server throws its scratch buffer away on an error reply), so the weaker guarantee costs nothing real. and now *all* the chunking arithmetic lives in one loop the byte-exact tests pin precisely. mutation testing didn't just check my tests; it told me a piece of my design was carrying weight no test could verify, and the simpler design was both more testable and just *simpler*. the strongest argument for deleting code is that you can't write a test that would notice if it were wrong.

## the suite caught me repurposing a client

- and the bug i'll keep, because it's an argument for a habit. i'd built the client to connect and `stat` the root, with a scenario asserting it. then, adding `create`, i edited that same client to connect, create a file, and stat *the file* — and the create test passed. i'd have shipped it. the full suite is what caught it: the *old* stat-the-root test now timed out, because the client i'd repurposed no longer did the thing that test watched for. one binary, two scenarios, and i'd silently broken one while making the other pass. a single green run would have hidden it; running the whole suite, every time, surfaced it in seconds. (the fix was just to have the client do both — stat the root *and* create-and-stat a file — which is a more honest demo anyway.) the cheapest bug is the one a test you already wrote refuses to let you regress.

## what i learned

- **the interface is the milestone; the implementation is a placeholder.** the `Filesystem` trait is host-tested in isolation, the RAM-backed store behind it is deliberately trivial, and everything ambitious (CoW, snapshots, content-addressing) is additive behind the same trait later. the expensive thing to get right is the line you'll call across for years; the thing behind it is cheap and replaceable. spend the milestone on the line.
- **"the kernel never learns what a file is" is a mechanism, not a slogan.** badged endpoints carry an opaque `(inode, rights)` the kernel never reads; the filesystem is the sole authority over what a cap names; the kernel only understands its own endpoint rights, never the file rights riding inside the badge. policy in userspace, mechanism in the kernel, enforced by what the kernel literally cannot interpret.
- **message-passing beats memory-sharing across a trust boundary — and you pay the price of `read()`.** copying isn't free, but it's the ordinary cost of a read/write syscall, and it buys unambiguous ownership and no over-exposure. zero-copy (`mmap`-style shared regions) is the data-plane optimization you defer until a measurement demands it, not the default you reach for because copying feels wasteful.
- **the authority you need is often a capability you already have.** i almost minted a new "access this process's memory" power; the v0.9b reply cap already named a blocked caller, which is exactly the grant required. borrow it, don't reinvent it. same realization as last milestone, different mechanism: name the concept and the new feature turns out to be an old primitive seen sideways.
- **delete the code no test can hold.** the two-pass validate was correct and untestable; the one-pass copy was simpler *and* mutation-killable. when mutation testing can't kill a mutation, the question isn't always "what test am i missing" — sometimes it's "what is this code doing that nothing depends on."
- **run the whole suite or you'll repurpose your way into a regression.** editing one shared fixture to satisfy a new test silently broke an old one; only the full run caught it. single-run-green is how you ship a bug a test you already wrote would have refused.

## what's next

- **rounding out the trait + the rights gate.** `create`, `write`, `read` go through the cross-AS copy already; `lookup`, `remove`, `readdir` are the remaining ops, mostly mechanical on top of the same machinery. and the file-rights gate — refusing a `write` on a `READ`-only capability — finally has a way to be *tested*, because `lookup` can hand out a deliberately attenuated cap and watch the server refuse the write it isn't allowed. "the filesystem enforces its own rights per message" becomes a thing the suite checks, not a thing i assert.
- **then a filesystem you can watch.** every operation is a span; a client that creates, writes, reads, and lists a few files should render in Tempo as a tree of FS calls attributed across the process boundary — the observability pillar pointed at storage. and that's the on-ramp to v0.11, the first *real* workload: a metrics-ingestion server that stores something that matters, on a filesystem the kernel still doesn't understand.

## addendum (2026-06-18) — the filesystem doesn't delete `include_bytes!`, it relocates it

- a reorganization of the source tree — grouping the flat module lists into directories — turned up a loose thread from this milestone, and it's worth pinning before v0.11 lets me assume otherwise. today the kernel bakes every userspace program into its own image: fifteen `include_bytes!(env!(...))` statics, one per binary, each ELF welded in at link time. the easy story is that the filesystem ends that — once there's a place to load programs *from*, the kernel stops carrying them. that's half true, and the wrong half is the one that matters.
- a RAM-backed filesystem is empty at boot. something has to *put* the bytes in it, and that something — an initramfs — is itself either embedded in the kernel image or pulled off a virtio-blk device. so `include_bytes!` doesn't vanish; it *moves*. fifteen per-binary statics collapse into one initramfs blob (or a device read), and the kernel goes from knowing `hello` and `rpc_server` and thirteen others by name to knowing exactly one opaque archive it hands to the filesystem and never opens. which is the whole milestone's rule wearing another hat: **the kernel never learns what a file is** — and it was never going to learn what a *program* is either. the win was never fewer bytes baked in; it's that the kernel stops keeping a *list of programs*. the embed consolidates; the knowledge is what leaves.
- (the thread itself was smaller and sharper: moving `elf.rs` into a subdirectory broke its test fixture, because `include_bytes!` resolves relative to the *source file*, so dropping one level down invalidated the `../fixtures/…` path. the fix is to anchor at the crate instead — `include_bytes!(concat!(env!("CARGO_MANIFEST_DIR"), "/fixtures/…"))` — a path that survives wherever the file moves. a one-line lesson that a relative path is a dependency on a location, and locations move.)

## addendum (2026-06-18) — the editor never learns what the code means

- a detour onto the Stitch side project — wanting syntax highlighting in VS Code — landed on the same shape as this whole milestone, which is why it's here and not its own note. the instinct is that highlighting needs a *language server*: a process that parses your code, knows that `Reading` is a type and `readings` is a variable, and colors accordingly. that's the heavy mechanism, and it's the wrong first reach. highlighting is *lexical*, not semantic — it's a TextMate grammar, a flat list of regexes that tag spans (`keyword`, `string`, `comment`, `entity.name.type`) without ever building a parse tree. three small JSON files, no process, no build step. the language server is the deferred thing, exactly like `mmap` was: the optimization-in-meaning you reach for when you actually want go-to-definition and hover types, not the default you reach for because regexes feel too dumb.
- and the way it's dumb is the point. the grammar colors tokens it cannot understand. it tags `List` as a type because it starts with a capital letter — a *convention*, enforced by what the regex can literally see, not by knowing what `List` is. it leaves bare `readings` and `threshold` white because nothing lexical distinguishes a variable from any other word; coloring *those* correctly is precisely the knowledge that needs a real parse, and so it's precisely what's deferred to the server. the line between "what regex can tag" and "what needs a parse tree" falls in exactly the same place as the line between mechanism and meaning everywhere else in this project: the highlighter carries the structure of the source and never interprets it, the way the kernel carries a badge it never reads. **the editor never learns what the code means** — it learns where the tokens are, and that turns out to be enough to read by.

# Post 36 — Files the OS can run

- last post, `init` gave its world away — the first process became a userspace root that owns almost nothing and hands out slices, and i said the shell was next, with `hold` the one missing primitive. but before a shell can run `view notes`, the OS needs something more basic that it simply didn't have: **a place to keep programs, and a way to run one it didn't compile in.** for thirteen milestones "spawnable" meant *baked into the kernel image* — `Spawn` picked an embedded ELF by index, and the filesystem was a flat, empty thing nobody booted with anything in. this is the post where the OS grows a filesystem with files in it, learns to run a program read straight off that filesystem — execve, basically — and grows the `hold` primitive that lets a process see its own authority.

## the filesystem grew up

- the in-memory filesystem was flat: a single root, no subdirectories (`create` of a directory returned `Unsupported`). making it hierarchical sounded like a project and was nearly free — the ramfs was *already a tree* of inodes with directories as name→inode maps; the root was one. the only thing enforcing flatness was one arm of `create` refusing to make a dir. flip it and nested lookup, stat, readdir all already worked.
- and here's the part worth the milestone: **subtree confinement falls out for free** — the thing Unix pays for with `chroot` and mount namespaces. authority is the capability you hold; the only way to get a child capability is `lookup(dir_cap, name)`, which **descends only**, and there is no parent operation in the filesystem at all. so a directory capability *is* a subtree-confined capability, by construction: a holder can name only what's at or below it, because it has no way to *designate* its parent. a "chrooted" process is just one handed a subtree cap instead of the root cap — same code, no sandbox. `..` doesn't belong in the filesystem; it lives in the *shell*, as a stack of directory caps you walked down through. keeping it out isn't a missing feature — it's the confinement.
- and the OS needed to *ship* with files. so a build-time **fs image**: a directory in the repo, baked into the filesystem server at compile time. drop a file in `fs-image/`, and it's in the operating system at boot. the same move the kernel already makes with embedded programs, now for data the filesystem serves.

## a program it didn't compile in

- `Spawn` loads a `&'static [u8]` chosen by index — a program the kernel was built with. `SpawnImage` is the new syscall, and the difference is the whole point: the caller hands the kernel an **ELF buffer it supplies** — bytes read off the filesystem — and the kernel copies it in, validates it, delegates the chosen caps, and runs it. that's execve: run a thing that lives in the filesystem, not in the kernel.
- the satisfying part is how little was new. the ELF loader already mapped segments from a `&[u8]`; the capability delegation already existed from `Spawn`. `SpawnImage` reuses both verbatim — the only genuinely new thing is "the bytes came from userspace, so copy them in first." the kernel didn't grow an executable format or a loader; it grew *one source of bytes*.

## two bugs the metal taught me

- the load is deferred: `SpawnImage` returns a child task id, and the child *itself* loads the ELF when it first runs. which means a **malformed ELF reaches the loader inside the child** — and the loader panics on a bad image. trusted embedded programs never hit that. a userspace program handing me garbage absolutely would, and a panic is the *kernel's* panic. a U-mode program must not be able to crash the kernel. the fix is to **validate the ELF synchronously, in the syscall**, before any child exists — parse it, refuse cleanly on garbage, and only then spawn. the unsafe path was "userspace input flows to a panic"; the boundary is where it gets checked.
- then the *valid* spawn failed too, with `BadUserRange`, and the trace handed me the answer. the capture had two `SyscallRefused` frames: one `UnknownProgram` (the deliberate garbage — correct) and one `BadUserRange` (the real ELF — wrong). so the copy of the *good* image was failing the range check. the reason: `copy_from_user` is built for **names** — it caps every copy at 256 bytes. an 8.5 KB ELF blew straight past that and got rejected as an invalid range. the primitive was never meant for bulk. chunk the copy into 256-byte bites and it goes through. i didn't guess that — the two refusal frames, side by side, said exactly which copy died and why.

## the program has to be *in* the filesystem

- a quieter puzzle, and a real one: to run a program off the filesystem, the program's ELF has to *be* in the filesystem. but the fs image is baked into the server **at compile time**, and the executable is built in the **same** build — the seed wants a file that doesn't exist yet. circular. the fix is a two-phase build: build the program first, copy its freshly-built ELF into the image directory, then build the server that bakes the image. copy-only-if-it-changed, so an identical rebuild doesn't churn the seed and retrigger everything. the build pipeline is part of the system; getting an executable *into* the OS was as much the work as running it.

## hold — a process sees its own authority

- post-35's one missing primitive, now landed: `CapList`. a process could *use* its capabilities but had no way to *enumerate* them — there was no "what authority do I hold?" call. `CapList` reads the caller's own capability table and returns its live slots. crucially it's **introspection, not ambient authority**: you learn only about caps you already hold, so it's ambient the way reading the clock is, not a hole in the model. it's exactly what the shell's `hold` verb will render — point at a process and show its authority, because the process can finally ask.

## no execute bit, and why

- the thing i braced to add and then deleted from the design. Unix gates running a file on the `+x` permission bit. a capability OS doesn't need one. the authority to run a program is just **holding a read capability to load its bytes** — there's no ambient "execute anything" power for a bit to filter. and "is this even a program?" is answered by **content-sniffing** (ELF magic vs a `.st` script), which is what `execve` does internally regardless of the bit. the `+x` bit's security job is subsumed by the capability, and there's nowhere left for it to attach, because the kernel never opens a path and decides to run it — the caller reads the bytes with a read cap and hands them over. the Unix oddity that `bash script.sh` runs a non-`+x` file *because it's only being read* is, here, the only model there is.

## what i learned

- **confinement is a shape, not a check** — again, and stronger. a directory capability can only descend; there's no parent op; so a subtree cap is escape-proof by construction. `chroot` is "hand a subtree cap instead of root." the expensive isolation Unix builds with namespaces is the thing this model gives away for free by *not* building the feature (`..`) that would break it.
- **a userspace input must never reach a kernel panic.** deferring the ELF load into the child quietly created a path from U-mode garbage to a `panic!`. the lesson isn't "don't panic" — it's that the *boundary* (the syscall) is where untrusted input gets validated, not three frames deeper where the trusted callers never sent anything bad.
- **the capture debugged it before the assertion did.** two `SyscallRefused` frames, one right and one wrong, told me which copy failed and that it was a range problem — not a logic bug, a 256-byte cap. the system that makes refusals visible made the cause of *my* refusal visible. the tool keeps paying for itself.
- **the `+x` bit is vestigial under capabilities.** some features you implement; some the model deletes. authority-to-run is a read cap, intent is a content sniff, and the permission bit has no job left. a capability OS isn't Unix with caps bolted on — it's Unix with whole concepts removed.

## what's next

- the **shell**, and now genuinely nothing is missing. a filesystem with programs in it; `SpawnImage` to run one; delegation to hand it exactly `read(notes)` and nothing else; `hold` to render what a process holds; and a language — Stitch — that declares the authority it needs. the powerbox `view notes` is just those pieces in a row: read the viewer off the disk, spawn it holding only the one cap, watch the grant cross the wire. the verb is the grant.
- thirteen milestones ago the kernel was the only root and every program was something it was compiled with. now the OS keeps its own files, runs a program it never saw at build time, and a process can look at its own authority and tell you what it can do. the floor under the shell is poured.

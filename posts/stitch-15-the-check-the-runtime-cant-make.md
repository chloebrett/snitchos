# Stitch 15 — the check the runtime can't make

- Stitch got a type checker this stretch. That's the headline and also the least interesting way to say it, because "a language got a type checker" is a thing that has happened a great many times. What I actually want on the record is one check the checker makes that I've never gotten from a compiler before — and that, on reflection, *no runtime could ever make*, because the information it needs doesn't exist at runtime. It's the check that tells you you declared authority you never used.
- but to get there I have to walk through what got built, because the whole point is that this one check falls out of the rest almost for free, and only makes sense on *this* platform.

## the thing that was missing

- post 8 was "the language learns to say no." a function declares the authority it needs — `log(msg) uses Telemetry` — and if it doesn't declare `Telemetry`, it can't `emit`. that's the capability model, and for a long time it was a purely *runtime* fact: the check happened the instant the effect fired, in the guts of the `emit` native, against the authority the running program happened to hold. break the rule and you got a refusal *while the program ran* — a `SyscallRefused` on the wire, post 8's whole payoff.
- which is the right backstop and the wrong first line of defense. capabilities are the load-bearing idea of the entire OS; "you find out you over-reached when you reach" is not how you want to learn about a violation. you want to learn before you run. that's a *type* question — does the shape of this program respect the authority it declares? — and Stitch had no one to ask.
- so: a checker. bidirectional (it reads a type off an expression where it can, checks against an expected type where it must), and — this is the part that made everything else possible — **gradual**.

## the load-bearing idea: "I don't know" that never lies

- here's the trick that shows up in every single step, so I'll spend a paragraph on it. the checker has a type called `Dyn` — the gradual unknown. it's the type of anything the checker can't figure out yet: an unannotated parameter, a call into code it doesn't model, a generic it hasn't instantiated. and `Dyn` is *consistent with every other type, in both directions.* checking `Str` against `Dyn` succeeds. checking `Dyn` against `Int` succeeds. `Dyn` is the type-system equivalent of a shrug that's never wrong.
- that one property is what let me build the checker as a *stack of increasingly strict checks* instead of a monolith. I added return-type checking, then parameter types, then constructor arguments, then exhaustive `match`, then contract subtyping, then generic type arguments, then the effect checks — each its own small, test-driven step — and **not one of them ever broke the existing programs**, because everything the new check didn't understand stayed `Dyn`, and `Dyn` never complains.
- I have a test I'm quietly proud of that pins this: it runs the *actual* standard library and the *actual* editor — the real `.st` code, hundreds of lines of generics and contracts and `match` and effects — through the checker after every change, and asserts **zero warnings.** the prelude is the most generic, effectful, hard-to-check code on the platform, and it comes out clean at every step. that's not luck. it's the shrug holding the line: a checker that only speaks up when it's *sure* can get arbitrarily strict without ever crying wolf on correct code. strictness you can add a spoonful at a time, because "I don't know" is always a safe answer.

## the forward check: saying no, earlier

- the first effect check is the obvious one, and it's post 8 moved to compile time. walk a function body; every time it calls an effect native — `emit`, `print`, `readFile`, the eight of them — note the capability that native needs. every time it calls *another function*, note that function's declared `uses`. union it all up: that's what this body actually exercises. then check it's a subset of what the function declares. `f() = emit("x", 1)` with no `uses Telemetry`? error, before it runs, pointing at the `emit`.
- the second half — calls propagate — is where it gets a spine. if `g` calls `f` and `f` uses `Telemetry`, then `g` needs `Telemetry` too. authority flows *up* the call graph: to call something that needs a capability, you must be holding it. that's the discipline the design always wanted — the compiler tracking `uses` across the whole graph — and it's completely decidable, no inference, just set-union walking a tree.
- and here's a wrinkle I only understood by reading the interpreter: **the checker is now stricter than the machine it runs on.** the runtime, today, is *soft*. when you call a function, its authority isn't intersected with yours — it's *replaced* with whatever it declares (`with_authority` at the call boundary, least-privilege-by-replacement). so at runtime a function gets its declared authority granted fresh, and `g` calling `f` doesn't actually require `g` to hold anything. a program the checker rejects might still *run*. that's not a bug in either one — it's the compiler enforcing the strong discipline ahead of a runtime that hasn't reified it yet. the type system is where the capability model gets teeth the interpreter is still growing.

## the check the runtime can't make

- now the one I came here to write about. the forward check has a mirror image, and the mirror image is the interesting one.
- forward: you perform an effect you didn't declare — *unsafe*, an error. reverse: you *declare* a capability you never perform — over-privileged, and I made it a **warning.** `f() uses Telemetry = 1`, a function that asks for the telemetry capability and then never emits anything: "declares `uses Telemetry` but never uses it." it's the same analysis — I already computed the set of capabilities the body exercises for the forward check — just the complementary subtraction: *declared minus used.*
- and the thing that stopped me: **the runtime cannot make this check. ever.** at runtime, holding a capability you never exercise is completely, silently fine. nothing is wrong. the program does exactly what it would do without the extra grant — it just... has more authority than it touches. there is no runtime event that corresponds to "you're carrying power you didn't need." the whole notion of *over*-privilege is invisible from inside a running program. it only exists when you can look at the code and compare what it *claims* to what it *does* — which is to say, statically, and nowhere else.
- that's the payoff of doing this in the type system instead of the interpreter. the forward check is, honestly, a nicer-timed version of something the runtime already does. the reverse check is *new information the runtime has no way to produce.* and the two together do the thing the entire OS is named after: they pin your `uses` to *exactly* what you need. under-declare and it's unsafe; over-declare and it's non-minimal; the sweet spot in the middle is least authority, and now it's checkable. the platform's whole thesis — authority should be minimal and observable — became a lint you can't ignore.

## the two halves shake hands

- last piece, because it closed a loop I opened two posts ago. post 13's addendum built *attenuation*: `without Telemetry { … }` drops a capability for a block, so an effect inside it gets refused even though the function holds the capability everywhere else. that was a runtime construct — the block narrows authority, the effect fires, the gate says no.
- so I made the checker flow-sensitive enough to see it. the effect walk now carries a set of *dropped* capabilities; descending into `without Telemetry` adds `Telemetry` to it, and an effect whose declared capability is currently dropped is an error — *"needs `uses Telemetry`, but it is withheld here by `without`."* `f() uses Telemetry = without Telemetry { emit(…) }` gets caught at compile time, and it's caught for the exact same reason the runtime would refuse it.
- which means the two halves of the effects story — the runtime `without`/`handle` machinery from post 13, and this static checker — now agree on the same program. they were built a stretch apart, for different reasons, and they close on the same errors. that's the tell that the model underneath is actually one model.

## what I'm not pretending

- honest gaps, worst-annoyance first:
- **generic *functions* aren't checked.** generic *types* are — `Maybe<Int>` and `Maybe<Str>` are distinct, and a constructor like `Some(5)` infers `Maybe<Int>` by solving the parameter from its argument. but `id<T>(x: T) -> T` doesn't even *parse* — functions carry no type parameters yet, and the call-site inference behind them (`id(5)` solving `T = Int`) is a real unification engine, the checker's first. that's the gated next chunk, and it's a genuine step up in difficulty from everything above, which was all decidable set-shuffling.
- **the reverse warning is conservative on purpose.** it only sees capabilities used through direct calls, not through *method* calls or a function passed as a value. so a capability exercised *only* that way could get a false "unused." no real code hits it today — the guard is green — but it's exactly why it's a warning and not an error: a warning that's occasionally silent is fine; an error that's occasionally wrong is not.
- **it reports; it doesn't block.** every one of these — the type errors, the effect errors, the unused warnings — prints to stderr and the program runs anyway. the checker is a stream of advice, not a gate. that's deliberate for now (the standard library was written under the old soft rules and I didn't want to hold it hostage while the checker grew up), and a strict mode that turns errors fatal is a one-line switch away. but it's a switch I haven't flipped, and until I do, "the type checker" is really "the type *adviser*."
- **the unused-capability warning points at the function body, not the `uses` clause.** the clause loses its source position when the program is lowered, so I can't yet draw the caret under the word `Telemetry` you should delete. the diagnostics from post 13 know how to point at things; this particular thing just isn't carrying its span far enough yet.

- still. the errand I set out on is done, and then some. `uses` used to be a boolean the runtime enforced the moment you crossed it. now it's a two-sided static contract: perform what you don't declare and the compiler stops you; declare what you don't perform and it tells you to travel lighter. the second half is the one I keep turning over — a compiler warning that no amount of running the program could ever produce, because being over-powered is not an event, it's a *shape*, and you can only see a shape from outside. the whole reason this OS makes authority a first-class, watchable thing was so you could reason about it. the type system is where the reasoning finally got automatic.

# Post 27 — Take the CPU, take a number

- for seven milestones the scheduler has been *polite*. a task runs until it calls `yield_now` and hands over of its own accord. that's cooperative scheduling, and it has a fatal honesty problem: a task that never yields runs forever, and there's nothing the kernel can do about it. v0.8 fixes that — the timer learns to take the CPU back whether the task likes it or not. and then v0.8b asks the next question: once you *can* take the CPU back, *who do you give it to?* the answers came with two surprises i didn't see coming — preemption was easier than i feared, and priority was much weirder.

## the thing i was dreading

- a voluntary context switch is cheap. `yield_now` is a normal function call, so by the time it runs the compiler has already spilled every caller-saved register the task cared about; the cooperative `switch` only has to save the 14 callee-saved ones. that's how every switch in this kernel has worked since v0.5.
- preemption is supposed to be the scary cousin. a timer interrupt lands at an *arbitrary* instruction — mid-computation, registers live — so a naive preemptive switch has to save the *full* GPR set plus `sepc` plus `sstatus`. i'd been telling myself for two posts that this was the hard part, the step that needed a whole new switch path.
- it isn't, and the reason is a thing i already had. when the timer fires, `trap_entry` *already* saves the complete trap frame onto the task's kernel stack — that's just what trap entry does, it has since v0.3. the full state is sitting right there. so preemption doesn't need a bigger switch; it needs to call the *existing* cooperative one from inside the timer handler. the full frame parks on the task's stack, `switch` swaps `sp` and carries it along, and when the scheduler comes back to this task later it unwinds the frame and `sret`s to the exact instruction the timer interrupted. the task never knows it was gone.
- this is the "layered" trick, and it's an xv6 idea: don't build a heavyweight preemptive switch, *stack* a lightweight cooperative one on top of the trap frame the hardware-ish trap path already built for you. the entire v0.8 mechanism is "make `handle_timer` call `reschedule`." the dread was paying interest on a loan i'd already paid off in v0.3.

## the kernel is never preempted

- one gate makes the whole thing safe: `sstatus.SPP`, the bit that says what privilege the trap came from. if the timer interrupted *user* code, `SPP == 0`, and we reschedule. if it interrupted *kernel* code, we don't — we just note the tick and return, exactly as before. v0.8 preempts userspace only; kernel code keeps its cooperative "i have the CPU until i yield" guarantee.
- that's a deliberate scope cut, and it's a good one. the runaway tasks i want to defend against are userspace; the kernel's own threads yield by construction. preempting kernel code would mean auditing every lockless critical section for "what if the timer fires *here*" — a real milestone of its own. the `SPP == User` predicate dodges all of it, and the upgrade later is a one-line change of the gate. all the hard machinery — the trap-frame switch, the timer→reschedule wiring — carries over untouched.
- the gate also bought me a free correctness argument i didn't expect. the rule in this kernel is *never emit a telemetry frame from an interrupt handler* — the virtio TX path takes a lock, and an IRQ that lands on the lock-holder deadlocks. so i assumed the `ContextSwitch { Preempt }` frame would have to be deferred. but `SPP == User` *guarantees* the interrupted code was in userspace, and userspace holds no kernel lock — so on this hart the TX lock is provably free, and i can emit the frame inline, right where the switch happens, like any other. "never emit from a handler" was really "never emit when a lock might be held," and the gate proves it isn't.

## building the bug on purpose

- to test preemption you need a task that *won't* cooperate. so i wrote `user_hog`: a userspace program that's a single `loop {}` with no syscalls, no yield, nothing — the only way the kernel can ever get the CPU back from it is the timer. i put it on a hart next to a cooperative `worker` peer and wrote the scenario `user-hog-starves-peer` — which asserts the peer makes *no* progress. it passes. on the pre-preemption kernel, that's correct: the hog runs forever, the peer never gets a turn, and the test green-lights the bug.
- then v0.8 lands and i *replace* that scenario with `preempt-runaway-user-task`, which asserts the opposite — the peer now climbs, and a `ContextSwitch { reason: Preempt }` shows up on the wire with the hog as the `from`. the hog never yields, so a switch *leaving* it can only have come from the timer. characterise the bug, then flip the test: the same fixture proves the starvation existed and then proves preemption killed it. the git history holds both halves.

## who gets the CPU?

- preemption gives you the *power* to take the CPU back. it says nothing about *who you hand it to*. round-robin — everyone gets an equal slice — is the simplest answer and what v0.8 ships. v0.8b asks for more: some tasks matter more than others.
- so: a `Priority` — `Low`, `Normal`, `High` — on every task, and a scheduler that runs the highest-priority ready one. the whole decision is two pure functions, host-tested off-target the way everything in `kernel-core` is. but a naive "always run the highest" starves the low tasks the instant a high one stays busy, and starvation is just the cooperative honesty problem wearing a different hat. the fix is **aging**: the longer a task waits, the higher its *effective* priority climbs, so a starved `Low` task eventually out-bids a steady stream of `High` work, runs, and resets. ordered, but fair.
- and then i fell in a hole i'd dug myself. the pick has to find the highest-effective-priority ready task — inherently a scan of the ready set, fine, O(n). except the runqueue only stored task *ids*, and the priority + wait-time lived on the `Task` in a separate table. so for each ready task i did a linear *lookup* to fetch its priority. O(n) candidates × O(n) lookup = O(n²) per context switch, and the spawn-storm stress test — 200 tasks — promptly timed out. the fix is the boring database move: stop normalising. store the priority and the wait-clock *in the runqueue entry* next to the id, so the pick reads everything it needs in one pass. the lookup wasn't algorithmically necessary; it was an artefact of splitting one logical record across two structures. back to O(n), storm green.

## two things priority taught me the hard way

- i built the demo the obvious way — a `High` worker and a `Low` worker, both cooperative, and asserted the High one runs more. it ran *3× as much*, not the 10× i expected, and chasing that gap surfaced two facts i should have seen coming.
- **cooperative tasks alternate 1:1 no matter their priority.** with exactly two workers, when the High one yields, the Low one is the *only* ready task — there's no *choice* for priority to express. and a voluntary yield isn't priority-gated anyway: a High task that chooses to give up the CPU is allowed to. priority only bites under *contention* — when several ready tasks compete and the scheduler has to pick. with two polite tasks taking turns, there's never a contest. you cannot demonstrate priority with a workload that has no scarcity.
- so the demo needs a task that *hogs* — back to `user_hog`, now as a High-priority CPU-bound greedy task that never yields, next to a Low cooperative worker. and you measure the right thing: not run-*count* (a task that runs continuously is *picked* rarely) but CPU-*time* (it holds the CPU for long stretches). greedy dominates the clock; the Low worker only gets the brief, aged-in turns aging rescues for it. ordered *and* fair, finally visible.
- **the preemption i'd just built was secretly inverting priorities.** v0.8's timer preempted the running task in favour of *whatever* was ready — correct for round-robin, but with priorities it means the timer would yank a High task off the CPU and hand it to a *Low* one. that's textbook priority inversion, caused by the very mechanism meant to enforce fairness. the fix: make preemption priority-aware. on quantum expiry, only reschedule if a ready task is of *equal-or-higher* effective priority — time-slice *within* a level, yield to a higher arrival, but never *demote* across levels. and because aging lifts a starved task's effective priority, a `Low` task that's waited long enough can legitimately reach the running level and preempt even a `High` hog — which is exactly how aging rescues it. the "fair" half of "ordered but fair" is one `>=` in a gate that almost wasn't there.

## and you can watch it

- the whole point of this box is that you can see everything, so priority shouldn't be something you *infer* from run counts — it should be on the wire. so `ThreadRegister`, the frame emitted once per spawn, grew a `priority` field (appended at the end of the struct, because postcard encodes positionally and the wire format is a contract). the kernel stamps each task's level; the collector resolves it onto every span as a `thread.priority` OTLP attribute — `Low` / `Normal` / `High`. now Tempo can colour the timeline by priority, and you can watch the High lane stay saturated while the Low one runs in aged-in bursts. the scheduler's choices are no longer a thing you reason about; they're a thing you look at.

## what i learned

- **the scary step had already been paid for.** preemption's "save the full register set from an arbitrary PC" is real work — done by `trap_entry`, three milestones ago. layering the cheap cooperative switch on top of the trap frame the kernel already builds turned the dreaded milestone into "call `reschedule` from the timer handler." the lesson keeps recurring: before building the heavyweight thing, check whether the substrate you already have does the heavy part for free.
- **a gate that constrains behaviour can also hand you a proof.** `SPP == User` was there to keep the kernel non-preemptible. but it *also* guarantees the interrupted code held no kernel lock — which is exactly the precondition that makes emitting a frame inline safe. one predicate, two jobs: a scope cut and a correctness argument for free.
- **priority needs scarcity to exist.** i spent real time confused about a weak demo before realising cooperative tasks taking turns have *no contention*, so there's nothing for priority to decide. you can't observe a preference in a system where every choice is forced. the demo had to manufacture scarcity — a CPU hog — before priority had anything to say.
- **the fairness mechanism inverted fairness.** the timer-preemption i built to *stop* one task hogging the CPU would, with priorities, hand a high task's CPU to a low one. the tool and the bug were the same line of code, distinguished only by whether you'd added priorities yet. building B forced me to go back and make A correct — and "make preemption priority-aware" wasn't a feature, it was a latent bug A had been carrying since the moment it shipped.
- **O(n²) hid in a data-layout decision, not an algorithm.** the pick was always going to be a scan. the quadratic came from looking each task's priority up in a second table — a normalisation i never decided to make, it just fell out of "the runqueue holds ids." colocating the data the hot path reads is the whole fix, and it's a reminder that "what's stored next to what" is a performance decision even when it doesn't look like one.

## what's next

- **v0.9: IPC over capabilities.** two processes that actually talk — a synchronous, capability-mediated channel, the first workload where a span crosses a *process* boundary. and it lands on exactly the substrate these two posts built: blocking on an empty channel is a `yield_now`, waking is a reschedule, and now both happen on a scheduler that preempts and ranks. the piece i deliberately *didn't* build — **priority inheritance**, where a high task blocked on a low task's resource lends it priority so it doesn't get stuck behind a medium one — has no home until there's a resource to block on. v0.9 is that resource. the scheduler learned to take the CPU and to choose who gets it; next it learns to let go of it on purpose, and wake back up when someone calls.

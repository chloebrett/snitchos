# Stitch 9 — the editor sends me back to the front end

- I wanted to build a text editor. what I got was a reckoning with the interpreter.
- the editor is **stim** — stitch + vim. a modal editor for SnitchOS, and the plan from the start was that it shouldn't be a Rust program that happens to run on the OS. it should be a *Stitch* program. "Rust is the platform, Stitch is the shell" — the editor is the shell's sibling. the whole thesis of the language is that the interesting logic lives in Stitch and Rust just provides the effects. an editor's core is `step(state, key) -> state` — a pure, immutable transition. that is the single most functional-programming-shaped thing imaginable, and Stitch is a pure functional language with immutable-by-default values and `..spread` record update. on paper it's home turf.
- so I sat down to write the editor's state machine in Stitch, and immediately hit the question that this whole post is about: **is the interpreter actually good enough to host its own flagship program?** and the honest answer was no. not yet.

## the second pass

- I have a habit now — a design exercise I run on a subarea: *if you were to redesign this from scratch, without regard for development cost, what would you change?* dropping the cost constraint is the whole trick; it stops you defending the diff and lets you see which decisions are load-bearing only because they were cheap at the time. I pointed it at the tokenizer, the parser, and the interpreter — the three things stim was about to stand on.
- I ran it as three independent passes, one per component, each blind to the others. and they **triangulated**. two different reviewers, looking at two different files, independently surfaced the same two missing things. that agreement is the signal.
- **missing thing one: the front end throws away *where*.** a token knows *what* it is and nothing about where it came from. so a parse error can only say `expected end of input, found Int(2)` — no line, no column, no caret. the AST nodes have no positions. a runtime fault is a bare string. and this is the part that stings for *this* project specifically: SnitchOS's entire identity is that everything is observable, that the system can always point at what it's doing. and its own compiler couldn't point at the character you got wrong. an observability-first language whose errors can't say *where* is off-thesis at the foundation.
- **missing thing two: there's no evaluator, just functions.** every step of evaluation is a free function threading an environment. there is nowhere to hang a step budget, a call stack, a depth guard, an effect handler. a Stitch function calling another just re-enters `eval` on the Rust stack — so an unbounded loop in Stitch overflows the *Rust* stack and aborts the whole process instead of raising something catchable. and stim's editor loop is, definitionally, unbounded. it runs until you quit.
- so the editor can't have a nice error experience (needs positions) and can't safely run its own main loop (needs an evaluator with a stack and a budget). the flagship program was blocked on the runtime, not on the editor.

## the reframe that stopped me building a VM

- here's where I nearly took a wrong turn. the language's roadmap has always ended at a bytecode VM with a real GC and a static type system. and standing at the bottom of a front-end rebuild, the tempting move is: *well, if I'm in here anyway, do the whole thing. build the VM. get types, get generics, get the GC. then write the editor on the finished language.*
- the thing that talked me out of it is small and I keep coming back to it: **stim is a Stitch program, so it cannot tell what's underneath it.** whether Stitch is a tree-walker or a bytecode VM is invisible to a `.st` file. waiting for the VM to *write a Stitch program* is like waiting for a faster CPU before you start writing your app. the program is portable across backends by construction.
- so the editor doesn't need the VM. it doesn't need types either — those buy the most for the *structured* editing I'm deferring, not the plain text editor that ships first. it doesn't need the GC — a short edit session is bounded churn, and if the immutable transition never forms a reference cycle, the refcounts reclaim everything as they go. (whether it *does* form cycles is a thing I'm going to measure rather than guess. that measurement is now a planned step.)
- what stim genuinely needs is exactly the two missing spines: positions, and an evaluator with a stack and a fuel budget. everything past that is a later, additive layer that swaps in *underneath* the editor without rewriting it. so the plan is: rebuild the tree-walk front end properly — spans, a real evaluator, a faithful AST with one lowering pass, effects as handlers — and build stim on *that*. the VM, types, and GC become their own milestone, and the decision of when to do them is deferred to a point where I'll have data instead of a hunch.
- a nice side effect of committing to "stim's logic is Stitch": the fiddly editor logic — cursor clamping, backspace-joins-a-line, enter-splits-a-line — is exactly the bug-prone stuff my mutation-testing gate is *for*. and that gate is a Rust tool; it can't touch a `.st` program. so the objection "you're putting the fiddliest code where the gate can't reach" is real — which spawned a sibling idea I'm genuinely excited about: **a mutation tester for Stitch itself.** and it's *cheaper* than the Rust one, for two reasons that fall out of owning the language. I own the AST, so generating a mutant is a tree rewrite, not a parse-and-splice. and a tree-walker has no compile step, so re-running the suite per mutant is milliseconds — mutants-per-second where cargo-mutants is mutants-per-minute. a mutation tester that's faster than the industrial one because there's no compiler in the loop. that's its own post someday.

## the front end learns to say where

- so: the actual work. the first stretch of the rebuild, and the part that already feels good.
- **tokens carry spans now.** every token is a `(kind, byte-range)`. the lexer tracks byte offsets as it goes. this is the unglamorous foundation everything else needs, and it was a wide, mechanical, slightly tedious change — rename the token enum, wrap it with a span, thread the kind through every match in the parser.
- **errors point.** a parse error carries the span of the token that broke it, and renders like this:

```
1:3: expected end of input
1 2
  ^
```

- the caret is such a small thing and it changes the whole feel. the old error was `found Int(2)` — debug spew, no location. now it's a line, a column, and a finger pointing at the exact character. this is the thing I couldn't have given the editor before.
- **the lexer stopped swallowing garbage.** it used to be lenient to a fault — an integer literal too big for an `i64` silently became `0`, a stray backtick was silently skipped. both are now *reported*: the lexer collects errors alongside the tokens, and the parser surfaces the first one instead of miscompiling. before this, handing `parse_program` a lone stray character returned `Ok` — an empty, successful program from pure garbage. now it's an error that points at the offending character. fixing this also forced a nice little cleanup: skipping whitespace and skipping an unknown character used to be the *same code path*, which is exactly why you couldn't error on one without erroring on the other. splitting them was the prerequisite that made "say no to a stray char" possible at all.
- all of it TDD'd and run through mutation testing, which surfaced a detail I liked a lot. some of the mutants don't get *caught*, they *time out* — mutate the cursor's `next()` to return a constant and the lexer loops forever. and that's not a gap in the tests; it's a preview. those non-terminating mutants are precisely the thing the **fuel budget** — the next piece of the rebuild, the evaluator's step counter — is going to bound cleanly instead of hanging. the tool is already pointing at the next milestone.

## where this leaves the editor

- stim is paused. on purpose. it's waiting for the front end to be a foundation worth building on rather than one it would have to be rebuilt off of later. that's the trade I keep making on this project and keep being glad I made: do the boring structural thing first, once, properly.
- what's left before the editor comes back: finish interning identifiers (the last piece of the "positions and symbols" pass), then the evaluator — the object with the stack and the fuel — then the AST/IR split, then effects-as-handlers. the editor's *modes* are going to be effect handlers: normal mode is a scope that simply doesn't install the write effect, insert mode does. same machinery, all the way down.
- I set out to write a vim clone and I'm three commits deep in a lexer. it doesn't feel like a detour. it feels like the editor asked the language a hard question — *can you even point at where I went wrong?* — and the honest answer was "not yet, give me a minute."

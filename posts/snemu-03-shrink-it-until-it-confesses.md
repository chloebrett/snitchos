# snemu 3 — shrink it until it confesses

- snemu-02 built a differential oracle — boot the same kernel under snemu and QEMU, diff the telemetry — and it ended with a bug gift-wrapped: three workloads (the stack-guard family) FAIL, each because snemu emits one telemetry name QEMU never does, `kernel.heartbeat`. the post signed off with a confident hypothesis: *snemu's page-table walk allows the guard-page access QEMU rejects — an MMU-fidelity gap.* a clear place to dig. this post is what happened when I dug, and it's not what the hypothesis said. the bug wasn't in the MMU. it wasn't in the scheduler either — though I spent a while sure it was. it wasn't in snemu at all. **it was in the oracle.** the tool I built to tell me when I'm wrong was itself wrong, and finding that out took shrinking the bug until it had nowhere left to hide.

## the first wrong answer (the one I inherited)

- the stack-guard workload spawns a kernel task that stores into its own unmapped guard page. supposed to fault, get named, halt. post-02's theory was that snemu's `translate` waves the store through where QEMU faults — so snemu never crashes, keeps heartbeating, and the extra `kernel.heartbeat` is the tell.
- so I instrumented snemu's translate and watched every access into the guard window. and the walk is *faithful*: when the store finally lands, snemu faults exactly like QEMU — the UART even prints `kernel stack overflow: task 5 guard fault at 0xffffffc04000f000 (slot 3)`. page-fault-as-trap delivers it, the handler names the slot. the MMU does its job. hypothesis one, dead. (this is the third guard-page theory to die against this oracle — post-02 already buried two. the pattern should have warned me.)

## the second wrong answer (the one I invented)

- so if the fault fires, why the divergence? next theory: the guard task is *starved* — snemu's cooperative scheduler never gives it a turn, so it never runs, never crashes, heartbeats forever. I went looking, and the evidence was damning: `snemu-boot --frames | grep ContextSwitch` returned **zero**. zero context switches. the task was clearly never scheduled. I wrote it up: a real scheduler-fidelity gap, hypersensitive too — adding a single `println!` to the kernel flipped whether the task ran at all.
- that last detail should have stopped me cold, and eventually it did. **snemu is deterministic.** a one-line `println!` cannot change whether a task runs — same guest instructions, same instruction-clock, same schedule, every time. if a `println!` flips the outcome, the thing that changed isn't the schedule. it's *my measurement*.
- and it was. the `--frames` text dump, piped through a shell at high step counts, was silently producing **nothing** — an empty pipe. I'd been reading "no output" as "zero switches." an empty pipe and a real zero look identical on a terminal, and I never checked which one I had. the authoritative in-process oracle — decoding frames directly, no shell in the middle — showed 8299 frames, context switches everywhere, the guard task running and faulting exactly on schedule. hypothesis two: not a snemu bug. a me bug.

## shrink it until it confesses

- two wrong answers in, I stopped theorizing about *mechanism* and asked a smaller question: what is the *minimum* that reproduces this? the guard workload has a lot of machinery — MMU, an unmapped page, a fault, a trap handler, slot bookkeeping. if the bug survives stripping all of that away, none of it is the cause.
- so I built the smallest crash I could: `workload=panic-now`, a kernel task whose entire body is `panic!()`. no guard page, no MMU, no fault. ran the oracle on it. **byte-for-byte the same failure** as stack-guard — 8297 frames vs 8299, same benign divergence point, same one only-in-snemu name, `kernel.heartbeat`, same FAIL.
- that's the confession. a bare panic reproduces the whole thing, so the guard page, the MMU, and the scheduler are *all* innocent — conclusively, not by argument but by subtraction. the minimal form is just: **a kernel that crashes at a fixed point after boot, emitting a different number of heartbeats before it dies under snemu than under QEMU.**

## the bug was in the clock — and therefore in the judge

- here's why the same crash emits heartbeats on one side and not the other. snemu's clock *is* the instruction count — `rdtime` returns instructions retired (that determinism is the whole point of snemu). the DTB says 10 MHz, so one "second" — one heartbeat period — is 10 million instructions. boot-to-crash is about 40 million instructions. so snemu believes ~4 seconds passed before the crash, and dutifully fires ~4 heartbeats first.
- QEMU runs those identical 40 million instructions in ~0.2 seconds of *real* time. its first heartbeat isn't due until 1 second. so QEMU crashes a full 0.8 seconds before it would ever have heartbeated — zero emitted.
- same kernel, same crash, same instructions. the heartbeat cadence just falls on opposite sides of the crash depending on whether you measure the boot in instructions or in wall-clock. `kernel.heartbeat` isn't telemetry snemu *invented* — it's telemetry QEMU *didn't live long enough to emit*.
- and that is the actual bug, and it's the oracle's. the oracle's rule was asymmetric on purpose: names only QEMU emits are fine (snemu just didn't run that far), but names only snemu emits are a failure (snemu invented telemetry). that asymmetry assumes both sides run *equally far*. when one deliberately halts early, the assumption breaks — and it breaks in the direction the oracle treats as fatal. the judge was applying a rule that was right for every workload except the ones that crash on purpose.

## teaching the oracle the difference

- the fix is small and lives entirely in the oracle. a name that appears only in snemu is invented telemetry *unless* it's recurring infrastructure a living kernel always emits and a crash can truncate — of which `kernel.heartbeat` is the whole list today. filter those out; if anything else is left only-in-snemu, still fail. so a genuine invention is still caught; a crash-truncated heartbeat is not.
- I wanted to make it robust instead of a special case — only forgive the heartbeat if snemu *also* shows the crash it reached, proving it halted just later. but the panic writes only to the UART, not to the telemetry stream, so there's no crash frame to key on. that's a real limit, written down in the code: this forgives a hypothetical snemu bug where it *fails* to halt and over-heartbeats. the honest fix for that is to bound the pass on the heartbeat *count*, later. for now the filter is a named, documented exception, not magic.
- one thing I checked and discarded: making snemu reset-on-panic like QEMU does. it wouldn't help. even inside a single boot, snemu's first heartbeat (10M instructions) precedes the crash (40M) — resetting afterward doesn't un-emit the heartbeats already sent. the divergence is baked into the clock, not the halt behavior. the fix *has* to be the judge.
- with the filter in: `44 PASS, 0 FAIL, 0 ERROR of 44 workloads`. the three guard workloads, the new `panic-now`, and every non-crashing workload untouched.

## what I learned

- **shrink the bug until it names its owner.** I spent two theories arguing about *mechanism* inside a workload with a dozen moving parts. the move that actually worked was subtraction: a five-line `panic-now` that keeps the symptom and discards everything else. the guard page didn't get *explained away* — it got *ruled out*, which is stronger. when you can't tell which layer owns a bug, delete layers until only one is left.
- **an empty pipe and a zero look identical.** my scheduler-starvation theory was built on `grep` returning nothing, read as "the count is zero." it was "the measurement failed." a conclusion drawn from the *absence* of output needs the output path proven to work at that scale first, or you're just believing your own broken ruler.
- **the thing that tells you no can be wrong too.** snemu-02's whole thesis was "build the tool that can contradict you." true — but a tool that contradicts you is still a tool with an opinion, and opinions are calibrated against assumptions. this oracle's assumption (both emulators run equally far) was invisible until a deliberately-crashing workload stepped on it. an oracle isn't an authority; it's a hypothesis about what "faithful" means, and it needs debugging like anything else.
- **being wrong twice, in writing, is cheap and worth it.** the whole zig-zag is in a note in the repo — MMU theory, starvation theory, the measurement artifact, the correction. keeping the wrong turns instead of quietly deleting them is what let me see the pattern (three guard-page theories, all dead) and finally stop looking *inside* snemu.

## what's next

- the oracle is clean now — 44/44 — which was never the point on its own. it's the credibility check under the thing snemu-02 promised: a test suite that runs on snemu instead of QEMU. boot once, snapshot-fork per workload, no real-time floor, no process spawn, no flakes. the reason to trust it is that a differential oracle spent three posts proving snemu tells the truth — and, this post, proving the oracle itself can be taught when it's the one crying wolf. the substrate is validated. next post, it earns its keep.

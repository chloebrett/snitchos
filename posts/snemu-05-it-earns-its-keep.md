# snemu 5 — it earns its keep

- four posts of throat-clearing. post-01 built snemu to let the kernel tell me what to implement next; 02 built a differential oracle so QEMU could tell me when snemu was wrong; 03 debugged the oracle when *it* was the one crying wolf; 04 made snemu ~3× faster so this post wouldn't crawl. all of it was in service of one promise, made at the end of 03 and designed toward since 02: **stop running the kernel's integration-test suite on QEMU, and run it on snemu instead.** this is that post. the suite has 108 scenarios. snemu passes 106 of them — and the two it doesn't are the most interesting result in here.

## the trick: don't rewrite the tests

- the itest suite is 108 scenarios, each a `fn(&mut View)` that waits for telemetry frames and asserts on them — "boot reaches the heartbeat," "the frame allocator OOMs cleanly," "a userspace program can't read a kernel page." they were written against QEMU: spawn a process, read frames off a socket, assert.
- the move that made this tractable was refusing to touch them. the scenario bodies don't care *where* the frames come from — a QEMU socket or a snemu buffer, it's the same `OwnedFrame`s. so I gave snemu a `View` that produces those frames by stepping the machine in-process, and pointed the *exact same 108 assertion functions* at it. no scenario rewrites, no snemu-specific fork of the suite. the audit is `cargo xtask snemu-itest`, and its first honest number was **74/108.**
- 74 isn't the story. *which* 34 failed, and why, is — because the failures sorted themselves into piles, and each pile taught something different.

## the pile that was a real bug

- one scenario asserts the deepest guarantee an OS makes: **a userspace program cannot read kernel memory.** it faults a U-mode task at a kernel address and checks that the fault fires. under snemu, it didn't fault. the read went through.
- this is the entire reason the audit exists. snemu's Sv39 page-table walk checked the read/write/execute permission bits — and completely ignored the **U bit**, the one that says "user-accessible." so snemu happily let U-mode code read a supervisor page, which on real hardware is the fault that keeps processes from reading the kernel. a genuine soundness hole, and the audit walked right into it.
- the fix threads the privilege level and the `SUM` flag into `translate` and checks the U bit properly: U-mode may only touch user pages; supervisor may reach user pages only for data, only with `SUM` set, never to fetch. and here's the quiet confirmation it was right — it caused **zero regressions**. every userspace scenario that passed before still passed, because QEMU *already* enforces this, so the kernel was already compliant; snemu had just been forgiving a rule the kernel never actually broke. the audit didn't just measure fidelity. it found the place snemu lied about the one thing an OS can't lie about.

## the pile that needed a real device

- eleven scenarios failed identically: "inject input: no stdin." these are the interactive ones — a console echo, the Stitch REPL — that type a command and check what comes back. they can't work against a *recording*, because the reply frames don't exist until you send the input, and you can't send input to a buffer.
- so the batch model had to become a live one. two pieces. first, snemu had a UART that could only transmit; I gave it a receive path — a byte queue, the data-ready bit in the line-status register the kernel's driver polls, a way for the host to push console input. second, the `View` got a *live* mode: instead of replaying a finished buffer, it owns a running machine and steps it on demand. `wait_for` steps until the next frame appears; `send_input` pushes bytes straight into the modelled UART; and the reactive loop — wait for a prompt, type, wait for the answer — finally closes. still no scenario rewrites: the same `fn(&mut View)` bodies, now driving a real machine. console echo passed. nine of ten Stitch REPL scenarios passed.
- a bonus fell out of the live model: it's *faster* than the recording for the passing majority. a scenario that finds what it's looking for stops stepping; only a failing one runs to the budget. and it's the foundation for something bigger, which is the next post's problem.

## the pile that made the guest confess

- one Stitch scenario kept failing with "no `primes.compute` span" — the on-target interpreter loads `primes.st` off the filesystem and computes the first ten primes, and the span that wraps that work never appeared. that error tells you *nothing*. did the load fail? the parse? the eval? you'd have to re-run by hand and squint at a console you couldn't see.
- so I made the failure show its work. an interactive scenario that fails now prints **the guest's own console output** — what the program actually said. re-ran it, and the guest confessed in one line: `runtime error: Stmt::Use reached eval — lowering pass not applied`. not a snemu bug at all — a bug in the *Stitch interpreter*, whose effect-lowering pass wasn't wired up for that construct. it would have failed under QEMU too (it did). the console dump turned a blind "something didn't happen" into a named defect with an owner, and the interpreter fix followed immediately.
- and then the loop closed the satisfying way: with the interpreter fixed, snemu ran `primes(10)` **faithfully** — the lowering error gone, the same computation QEMU does — it just needed a big instruction budget, because a tree-walking interpreter computing primes by naive trial division is genuinely heavy work. correct, not fast; exactly snemu's shape. (this is the same lesson as post-03's `2>/dev/null`: a test that fails silently is a ruler with no markings. put the guest's words in the failure and you stop guessing.)

## the pile that was asking the wrong question

- some tests didn't have bugs — they had *units*. `heartbeat-cadence` asserted the interval between heartbeats falls between 10 ms and 10 s. under snemu that came out to ~22 seconds and "failed." but snemu's clock is instructions (post-03's whole punchline), so "seconds" is a fiction: the kernel does about 220 million instructions between heartbeats, and dividing that by a 10 MHz timebase gives 22, and there's nothing wrong. the test was asserting a wall-clock magnitude that only means something under a wall-clock.
- so I rewrote it to the *behavioral* truth underneath the magnitude: two heartbeats arrive, with strictly advancing timestamps. that's what "the timer keeps firing and the clock progresses" actually is — and it's true under both a real clock and an instruction clock. same for the priority scheduler test: it demanded the high-priority task hog CPU by "10×," and snemu's deterministic round-robin gives a clean-but-lower ~7×. the exact ratio is a scheduler artifact; the thing worth testing is *dominance versus parity*, so the threshold became "clearly dominates" (≥5×), which both engines satisfy and which still catches a scheduler that ignored priority (that's ~1×). the rest of the failures were just budget — assertions calibrated for QEMU's wall-clock seconds, which are hundreds of millions of snemu instructions, so a handful of compute-heavy scenarios get a bigger per-scenario step budget and pass.

## the two it doesn't pass — and why I left them

- 106/108. the two holdouts are the *same* bug wearing two names: a shell revokes a file capability while a viewer is mid-read, and the test asserts the viewer got at least one byte before the cap died. under QEMU's timer-driven interleaving, the viewer wins the race and reads a byte. under snemu's deterministic round-robin, the revoke wins, and the viewer reads zero.
- neither ordering is *wrong* — it's a race, and real hardware could resolve it either way. the test is over-specified for one particular outcome. I could paper over it, but the honest fix isn't in the emulator or the test — it's in the kernel: this is exactly the kind of ordering that a real supervision model (init as a service supervisor, deterministic teardown) is supposed to make well-defined. so I'm leaving these two red on purpose. they're not a snemu fidelity gap; they're a design smell the audit surfaced, and the fix belongs upstream. a failing test that's pointing at the right thing is more useful than a green one that isn't.

## the number that matters isn't MIPS

- so is it worth it? the full 108-scenario suite: QEMU runs it in **57 seconds**, snemu in **~355** — single-threaded, and ~6× slower on the raw wall-clock. if that were the whole story, snemu would be a curiosity.
- it isn't the whole story, for two reasons. first, **that 355 is one thread, and it's ~90% spent in a handful of compute-heavy scenarios** — the OOM leaks and the Stitch tree-walker, billions of instructions each. the other ~100 scenarios run in about 30 seconds, and snemu is *faster* on those, because it has no per-scenario firmware boot to pay. `priorities` is 1.6 s on snemu against 5.2 on QEMU; the crawl is entirely the compute tail, which is exactly what post-04's JIT tiers exist to attack.
- second, and this is the one that actually flips it: **QEMU flakes, so the commit gate isn't one run — it's ten.** `itest --repeat 10`, because a single green pass has hidden real race bugs before. that's ~570 seconds. snemu is *deterministic* — a seeded run reproduces exactly, so one run replaces ten. **355 versus 570: snemu already wins the gate that matters,** and it wins it without the flake anxiety that made the gate ten runs in the first place. QEMU's 57 seconds is one roll of dice you have to roll ten times; snemu's 355 is a proof you run once.

## what I learned

- **a fidelity audit is a bug finder pointed at yourself.** I built `snemu-itest` to measure how faithful snemu is, and its first real dividend was a *snemu* bug — the missing U-bit check — that no amount of "does the demo boot" would have caught. testing snemu against the kernel's own assertions is testing the emulator with the most demanding program you have. the measurement was the discovery.
- **don't rewrite the test; rewrite the assumption.** the interactive scenarios worked untouched once the frames came from a live machine. the wall-clock scenarios worked once I asserted the behavior instead of the magnitude. every time I was tempted to write a snemu-specific test, the better move was to find the thing the original test *actually* meant and assert that — which usually made the QEMU test better too.
- **put the guest's words in the failure.** the single highest-leverage change in the whole audit was making a failing interactive scenario print the console. it converted "something didn't happen" into "the interpreter said its lowering pass isn't applied," and the fix was immediate. the second time in three posts that the fix was *make the failure legible* rather than *guess the mechanism*.
- **leave the right tests red.** the two revoke-race holdouts could be green with a one-line threshold change. they're more valuable failing, because they're pointing at a real kernel design gap that a supervision model should close. green is not the goal; *green for the right reason* is, and sometimes the honest score is 106.

## what's next

- snemu is a validated, deterministic substrate that runs the real test suite and wins the real gate — and it does it on **one core.** it's an in-process library with no process, no socket, no shared mutable state between scenarios; each scenario already owns its own machine. that is the exact shape that parallelizes trivially. fan the audit across host threads and the ~355 seconds of single-threaded CPU becomes ~44 seconds of wall-clock on eight cores — under QEMU's *single* run, never mind its ten. next post: snemu goes multi-core, and the emulator that started as a way to let the kernel tell me what to build becomes the fastest, most reliable way to know the kernel still works.

# Post 31 — A poisonable snitch

- between milestones I went after debt — the architectural and "elegance" kind, not bugs. most of it was the ordinary satisfying work: collapse six parallel lists into one table, hoist a copy-pasted pattern into an abstraction, label a task's exit distinctly on the wire instead of pretending it was a yield. worth doing, not worth a post. the two things that *were* worth writing down are the ones I didn't go looking for: how often this project's oldest rule — *mechanism in the kernel, meaning in userspace* — turned out to be a **debt detector** and not just a design principle, and that a single design *question*, asked before a line of code, surfaced a security hole that would have quietly made the whole kernel's reason for existing a lie.

## the scheduler never learns what a program is

- the biggest cleanup: the kernel enumerated its userspace programs in **six** places — a build-script embed list, a wall of `include_bytes!` statics, eighteen near-identical entry functions, a `WorkloadKind` enum, a 130-line spawn match, and a heartbeat match that listed every workload again just to do nothing for most of them. adding one program meant touching all six. the smell isn't six lists; it's one missing abstraction: a *registry*. one table of `{ name, elf, rights, layout }`, and the six lists collapse.
- the catch is the scheduler. `spawn` takes a bare function pointer — no argument slot; the context switch only rigs the return address and the stack. so the eighteen entry functions *can't* just become "spawn this generic entry with that descriptor," because there's nowhere to put the descriptor. the fix was a generic per-task **arg word**: the scheduler carries one opaque machine word it never interprets, and a single `program_entry` reads it back as the address of its program spec. the scheduler doesn't learn what a program is — it carries a word, and userland-land code decides what the word means. the same move as the badge the kernel can't read and the file bytes it can't interpret, applied to the spawn path. mechanism-not-meaning wasn't the goal of the refactor; it was the *shape the refactor wanted to take* once I stopped fighting it.

## a counter abstraction, and where it should *not* live

- the next one: "bump an atomic in a hot path, drain it in the heartbeat" was copy-pasted across eight subsystems feeding nine separate drain functions. a `DeferredCounter` that bundles the atomic with its name and drains itself collapses the nine into one loop. straightforward.
- the interesting part was a question I got asked mid-design: *is the heartbeat even the right place for this — should the drain be its own timer?* and the honest answer was to **separate two questions that were tangled**: the *abstraction* (a self-registering counter) is independent of *where the drain runs*. the real smell wasn't the cadence — it was that the heartbeat module *owned the list of every counter in the kernel*. the registry inverts that: each subsystem owns its counters, the heartbeat just calls `drain_all`. it becomes the *scheduler* of the drain, not the *owner* of the data. and a second timer? premature — there's no evidence metrics need a different rate than the heartbeat, and the registry makes splitting one out later trivial if a reason ever shows up. (one counter stayed hand-written: the cross-hart oracle whose drain needs an `Acquire` load, not the registry's `Relaxed`. the abstraction is for the common case; the load-bearing exception earns its bespoke code.)

## the panic that called itself a "deferred refinement"

- a smaller one with a sharp edge. when a userspace program hands the kernel a pointer for a syscall, the kernel validated that the pointer was *in the user half* — and then dereferenced it. if the address was in range but **unmapped**, the deref faulted in supervisor mode, which the kernel treats as its own bug: a panic. the comment at the copy site admitted it outright — "fault-graceful copy is a deferred refinement; today such a pointer faults to S-mode." userspace could crash the kernel with a bad pointer to a syscall.
- the fix was already half-built: the v0.10 cross-address-space copy walks a page table by hand to validate every page before touching it. I lifted that same walk to pre-check the single-address-space copy — translate each page, confirm it's mapped and readable, *then* set the supervisor-access bit and copy. an in-range-but-unmapped pointer is now a refusal (`BadUserRange`), not a kernel fault. and I wrote the probe I'd been missing: a tiny program that deliberately passes an unmapped address to a syscall and emits a survival marker if it's still running afterward. the test that proves a panic is gone has to *try to cause the panic*.

## a poisonable snitch

- here's the one that mattered. I was breaking down the big remaining debt — letting userspace **name its own metrics** instead of the kernel pre-registering every name (the layering inversion that bit me in v0.10, when a userspace filesystem's gauge had to be registered *by the kernel*). the natural design: reuse the kernel's existing name table — userspace passes a name, the kernel interns it, dedups it against names it's already seen, hands back an id. clean, reuses everything.
- then a single question, asked before I'd written a line: *isn't letting any process check which names are already taken a problem?* and pulling that thread unravelled the whole design. the name table is **global** — shared across every process *and the kernel's own metrics*. dedup means a name resolves to a *shared* id. so a process could register `"snitchos.frames.allocated_total"` — the kernel's own frame-allocator metric — receive the kernel's id back, and **emit into the kernel's telemetry stream**. userspace forging the kernel's own snitch. or quietly polluting another process's gauge by registering its name. for an operating system whose entire pitch is a *trustworthy* observer — the snitch you can believe — a **poisonable snitch is the one thing it cannot be**. the disclosure angle (probe which names exist by watching id ordering) was the minor cousin; the poisoning was the real hole.
- the fix is the project's own rule, again: a global deduped name table is ambient shared state, exactly what capabilities exist to abolish. scope names **per process** — no cross-process dedup (two processes' `"requests"` are simply two different metrics, which is *correct*; they're different emitters), and emit is authorized by a per-process handle, so you can only write to a metric you yourself registered. the kernel stops running one shared vocabulary and starts carrying a name it can't read, per authority.
- the meta-lesson is the part I want to keep. this wasn't found by a fuzzer or an exploit or a post-mortem. it was found by **asking a security question during design, before the code existed** — the cheapest possible place to find it. a capability kernel's threat model isn't a review you do at the end; it's a question you ask of every new authority as you draw it. "who can observe this? who can forge it?" — asked of `RegisterMetric` on a whiteboard, it cost one conversation. asked after shipping, it costs a CVE.

## prefactor, or build?

- last thread, because it's a judgment I keep having to make. the instinct, looking at the crufty old telemetry-sink path before adding the new one, was to *prefactor first* — clean up the existing shape so the new feature lands on something tidy. and it's a good instinct, often right. here I argued against it. the new mechanism is genuinely **additive** — a separate per-process table, separate syscalls, reusing the now-fault-safe copy. prefactoring first would mean rewriting the *working* emit path that every program depends on, toward a target the new design hadn't yet *validated*. the clean shape I wanted the old `TelemetrySink` to become was sitting right there — its sibling `SpanSink` was already authority-only, no fused payload, name-per-call — but "I can see the target" isn't the same as "the target is proven." so: build the new path additively, migrate one real consumer to prove it, and *then* retire the legacy path against a shape I've watched work. **refactor toward a validated target, not a hoped-for one.** prefactor last, when "last" is cheaper than "first."

## what i learned

- **mechanism-not-meaning is a debt detector, not just a design rule.** every place the refactors got stuck was a place the kernel *knew* something it had no business knowing — what a program is, which metrics exist, what a name means. the cleanup and the principle were the same edit.
- **a poisonable snitch defeats the premise.** an observability-first kernel that lets userspace forge or pollute its telemetry has given up the one property it's selling. least-authority has to reach the metric path: per-process names, per-process emit.
- **ask the security question in design, not after.** the hole was found by a whiteboard question about a syscall that didn't exist yet. for every new authority: who can read it, who can forge it? it's a one-conversation cost up front and a catastrophe down the line.
- **refactor toward a validated target.** when the new thing is additive, build it, prove it, *then* retire the old — don't rewrite working code toward a shape you haven't watched succeed.
- **separate the abstraction from its cadence.** "the heartbeat drains the counters" and "the heartbeat owns the list of counters" are different claims; only the second was the debt.

## what's next

- the userspace-metrics milestone — now designed to be un-poisonable from the first commit: per-process handles, no shared vocabulary, emit authorized by registration. when it ships, the kernel will carry a metric's name without being able to read it, the way it already carries a badge it can't interpret and a file's bytes it can't read. one more thing the kernel never learns — and the snitch stays honest because nobody but its owner can write to it.

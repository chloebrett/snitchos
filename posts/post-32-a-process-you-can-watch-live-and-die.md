# Post 32 — A process you can watch live and die

- the north star is an explicit-authority shell: type a command, the kernel launches it holding exactly the capabilities it needs and nothing more, and you watch least-authority happen on the wire. a shell is really three verbs — **read** a command, **launch** it with delegated authority, **reap** it when it's done — and this milestone built all three. the through-line is the kernel's oldest rule wearing another hat, plus something new: a process now has a whole *life* you can follow in a trace. born clutching delegated caps, running under exactly those, exiting with a status, and — the part that took the most care — every frame it ever touched handed back.

## the kernel learns to read

- until now telemetry was a firehose pointing out. the kernel emitted; nothing came back in. a shell needs the other direction, so: Tier-0 polled UART RX — no interrupt, the kernel reads the ns16550a receive register into a buffer when userspace asks for it via a `ConsoleRead` syscall.
- the satisfying part was a symmetry I didn't expect. I'd built TX as a *kernel* thing — MMIO, the virtio path — and RX *felt* like it'd be the same shape, bare-metal by nature. but the actual logic is a ring buffer: head, tail, len, drop-on-full. that has nothing to do with MMIO; it's pure data structure. so `ConsoleRing` lives in `kernel-core`, host-tested with ordinary `cargo test`, and the kernel side is a dozen lines polling a register into it. rx and tx are symmetric in the way that matters — the *meaning* is host-testable, the *mechanism* (which register, which DMA ring) is the thin bare-metal part. TX already worked this way, it turns out; the UART wouldn't have booted otherwise. I just hadn't named it.
- the copy back to userspace rides the fault-safe `copy_to_user` from last milestone, so a bad buffer pointer to `ConsoleRead` is a refusal, not a kernel fault. one detail I'm glad I got right: the handler validates the pointer *before* it drains the ring, so a program that passes garbage doesn't silently lose the bytes it asked for.

## handing authority down

- spawn-with-caps is the brick the whole shell rests on. `spawn(program, &[handles])`: the parent names some of its own capabilities, and the child boots holding exactly those, plus the ambient bootstrap caps every program gets. the demo: a `spawner` hands its child *only* its span cap, the child opens a span through it, and the trace shows that span attributed to the child — authority that flowed across a process boundary, made visible.
- the rule is **all-or-nothing**. you list the handles to hand down; hold them all and the child gets copies of all; if even one isn't yours, the whole spawn is refused. there's a small design pleasure to record here. the delegation step started life returning a `Vec<Result<Cap>>` — resolve each handle, collect the outcomes — and that's the wrong shape. a *partial* delegation should not be a thing that can exist; a child holding three of the four caps its parent meant to give it is a security bug waiting to be written. collapsing it to `Result<Vec<Cap>>` — the whole batch succeeds, or the whole batch refuses — makes the all-or-nothing invariant a *type* instead of a discipline. and `collect()` does it for free: a `Vec<Result<T, E>>` collects straight into a `Result<Vec<T>, E>`, short-circuiting on the first error. the standard library already knew the shape I wanted.
- delegation is copy, not move — the parent keeps its caps — and a process can only ever hand down authority it already holds; there's no `spawn` that mints authority from nothing. the kernel mediates the transfer but, as ever, doesn't interpret it: it checks you hold the handles and copies the capability records. it has no opinion about what a span cap or a telemetry cap actually *is*.

## the bug that crashed differently every time

- a war story, because the diagnosis is worth more than the bug. with spawn wired up, the spawn-a-child integration test crashed — ten runs, ten crashes, never the *same* crash. one run, a null dereference at one line; the next, an allocation error in a completely unrelated subsystem; the next, something else again.
- that variance is itself the diagnosis. a logic bug crashes the same way every time: same bad pointer, same line. a crash that *wanders* is **memory corruption** — something is scribbling where it shouldn't, and *what* breaks depends on whatever happened to be sitting there that run. so before instrumenting any single crash site — which would have been chasing ghosts — the shape of the failures said: stop looking at the symptoms, go find the scribble.
- it was the kernel stack. spawning a task allocates its 16 KiB stack as a `Box<Stack>`, and the code built the value and *then* boxed it: `Box::new(Stack::new_zeroed())`. that constructs a 16 KiB temporary **on the current stack** before moving it to the heap — and the current stack was the spawning task's own 16 KiB kernel stack, already several syscall frames deep. a 16 KiB temporary on a 16 KiB stack overflows it, silently, into whatever lives below. hence the wandering. the fix is one line — `Box::<Stack>::new_zeroed()` allocates the heap slot and zeroes it *there*, no stack temporary — but the lesson I wrote down is the signature: **varying crash modes across runs means corruption; suspect it before you instrument.** with a footnote: `Box::new(big_value)` is a stack-temp footgun in exactly the place you can least afford one. it also seeded a plan — guard pages for kernel stacks, so the next overflow faults loudly instead of scribbling quietly.

## ending a process, and the status that comes back

- a program that can start must be able to stop. `Exit` now carries a status code; `Wait(child)` blocks the parent until that child exits and hands back its code. the plumbing is small and lives where it should: the *decision* — is this child already a zombie, or must the waiter block? — is a pure `ReapTable` in `kernel-core`, host-tested and mutation-checked, that knows two maps (exited-but-unwaited; who waits on whom) and nothing about traps or scheduling. the kernel side is just the mechanism: block the parent, wake it on the child's exit, return the status.
- the demo closes the loop the spawn demo opened — parent spawns a child, child exits 42, parent `Wait`s and re-emits 42 as telemetry — and on the wire you watch the whole handshake: child registered, child's delegated span, child exit, parent reporting the reaped status. a process's life, start to finish, in one trace.

## the part that took the most care: giving the memory back

- here's where "it works" and "it's done" diverge, and I didn't want to defer the gap. `Exit` returned a status — and *leaked everything else*. the child's entire address space (its page table, every mapped frame: code, heap, stack), its 16 KiB kernel stack, its `Process` record with the cap table and per-process metrics. a shell spawning a few hundred short-lived commands would march straight into OOM. the lifecycle wasn't done; it was half-done in the way that looks finished in a demo and falls over in use.
- the crux is *who* frees it, and *when*. a process can't free its own address space on the way out — it's still running on it; you can't saw off the branch you're standing on, and the page table *is* the live `satp`. and the `Process` record turned out to live as a local on the exiting task's own kernel stack, so nothing ran its destructor. the answer is **reclaim on reap**: the *parent*, in its own address space, does the freeing when it `Wait`s — by which point the dead child has already switched away for the last time. walk the user half of the child's page table, hand every frame back to the allocator, run the `Process` destructor in place to release its heap, drop the kernel stack. carefully *not* the shared kernel half — those tables are mapped into every process; freeing them would be freeing the kernel out from under everyone.
- and because this is SnitchOS, the test isn't "it doesn't crash." the test *watches the memory come back.* I wrote a workload — a `reaper` that spawns a 4 MiB-hungry child thirty times and `Wait`s each — and asserted on telemetry. **without** reclaim, thirty times four megabytes blows past RAM: the free-frame count hits zero and stays there (I watched it — `frames.free` → 0, the loop wedged at child 29). **with** reclaim, the freed-frame counter climbs into the tens of thousands and the whole run finishes in under a second. the integration test fails if the memory leaks and passes if it's reclaimed — the cleanup is a property I can see on a graph, not a claim in a comment. the page-table walk underneath it is a pure function tested against a mock: it frees the leaves and the tables and the root, never follows the kernel half, skips huge pages. the unsafe, frame-allocator-touching part is the thin shell around it.
- one honest edge left open, written down rather than hidden: an *orphan* exit — a process nobody `Wait`s on — still leaks, because reclaim is the reaper's job and an orphan has no reaper. the shell always waits on what it launches, so it doesn't bite yet; the fix (a sweep, or re-parenting the dead to init) is a known follow-on, not a surprise.

## what it adds up to

- read, launch, reap — the three verbs are in. none of them is a shell yet; that's the next milestone, and the v1.0 north star (a shell plus a tiny editor — something demoable). but the foundation has the property I care about most: a process's entire life is observable. you can watch authority handed down to it at birth, watch it run under exactly those caps, watch it exit with a status, and — now — watch the memory it held flow back to the allocator. born, lived, reaped, reclaimed, all on the wire.
- the meta-lesson I'm keeping is the one about "done." three of the four pieces here were easy to make *work*; the real work was making them *complete* — the all-or-nothing delegation type instead of the partial-success vec, the fault-safe copy instead of the panic, and above all reclaim instead of the leak that demos perfectly. the distance between a feature that works in a demo and one that survives a shell spawning commands all day is exactly the distance a learning project is tempted to skip. writing the test that fails on the leak was the thing that turned "exit works" into "exit is done."

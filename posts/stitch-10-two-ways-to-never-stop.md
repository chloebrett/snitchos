# Stitch 10 — two ways to never stop

- last post the front end learned to say *where* — tokens got positions, errors got carets, the lexer stopped swallowing garbage. this post the evaluator learns to *stop*. or more precisely: to notice when a program won't, and give up in a way you can catch instead of a way that takes the whole process down with it.
- the motivation is still stim. an editor's main loop runs until you quit — it is, by design, a program that doesn't terminate on its own. and the interpreter had no way to survive hosting something like that. worse: it had no way to survive a program that *accidentally* doesn't terminate, which is most of them, sometimes.
- the review I ran on the interpreter called the missing piece an "evaluator context" — there was nowhere to hang a budget, a depth limit, a handler stack, because evaluation was just free functions passing an environment around. the grand version is to reify the whole thing into an object. I didn't do the grand version. I noticed the environment *already* carries a bunch of run-wide shared state — the telemetry sink, the platform, the capability set, all shared across every scope of a run via a refcount. so a **budget is just one more of those**. that's the whole trick of this post: the place to put execution limits already existed, it was just holding other things.

## fuel

- the first limit is fuel. the environment carries a counter; every step of evaluation spends one; when it hits zero, the program faults with `evaluation fuel exhausted` instead of running forever. the default budget is effectively infinite, so nothing that already worked changed — you opt into a finite budget when you want one.
- this is the smallest possible change that makes non-termination *survivable*. and it's the thing a whole other project was waiting on: a mutation tester for Stitch (mutate the program, re-run the tests, see which mutations the tests catch) needs exactly this. a mutation can turn a terminating program into a non-terminating one — flip a base case, and recursion never bottoms out. without a budget, that mutant hangs the whole test run. with one, it faults, gets marked as "caught," and the run moves on. the fuel cap *is* the mutation tester's timeout, and now it exists.
- there's a loop-closing detail I enjoyed. building the *front end* last post, I ran the Rust mutation tester over the new lexer, and a few mutants didn't get caught — they *timed out*. mutate the cursor to return a constant and the lexer spins forever. at the time I noted those non-terminating mutants were a preview of exactly what fuel would bound. they were. the tool that motivated the budget had already shown me the shape of the thing the budget catches.

## the other way to never stop

- here's what I didn't fully appreciate until the tests started arguing with each other: **there are two different ways for a program to not terminate, and they need two different guards.**
- one is looping forever — doing an unbounded number of steps while staying flat. `1..` is an infinite sequence; counting it never finishes, but it never gets *deeper*, it just keeps going. fuel catches this: unbounded steps, bounded shape.
- the other is descending forever — recursion that never bottoms out. `f(n) = f(n) + 1` calls itself, and because the `+ 1` is still pending, each call sits on top of the last. this doesn't spin flat; it *grows*, one stack frame at a time, until the Rust stack underneath the interpreter runs out and the process is killed. fuel doesn't save you here — you can run out of stack long before you run out of a generous step budget. you overflow, and overflow isn't a catchable error, it's a `SIGABRT`.
- so the second guard is a depth guard: count how deep the calls are nested, and when it gets too deep, fault with `call stack too deep` — *before* the real stack runs out. same mechanism as fuel (a shared counter on the environment), with one wrinkle: it has to go back *down* when a call returns, so it reflects current nesting, not total calls ever. that's a little RAII guard whose destructor decrements the count — so the depth drops on every way out of a call, including the error paths.
- the two guards caught each other in the tests, which is how I learned the distinction properly. I first wrote the fuel test with a *recursive* non-terminator — and it failed, because the depth guard tripped first. the recursion got deep before it got expensive. so the fuel test had to switch to an *iterative* non-terminator (the infinite sequence) to actually test fuel and not depth. two ways to never stop; you have to pick which one you're testing.

## the humbling number

- the depth limit is where the interpreter told on itself. I set the maximum nesting to something generous, ran the test, and it *still* overflowed — the guard tripped too late, the stack was already gone. so I lowered it. and lowered it again. it came to rest at **48**.
- forty-eight. a program can nest function calls forty-eight deep before Stitch taps out. that's embarrassingly shallow, and the reason is a thing the redesign already has in its sights: `eval` is one enormous function — a single match over every kind of expression the language has — and a giant function has a giant stack frame. every level of interpreter recursion parks a big frame, so the real stack fills up fast. the guard has to trip well before that, so the limit is low.
- and it's worse on the metal, where a userspace process gets a 16 KiB stack, not the host's couple of megabytes. so 48 is a host number; the on-metal number is smaller still. this isn't a limit I'm happy with — it's a *backstop*, the difference between a clean fault and a corpse. the honest fix is two-fold: shrink that `eval` frame (a later, structural job), and — for the specific case of recursion that *is* tail-shaped — stop growing the stack at all.

## what's next, and why the number gets to grow

- that last point is the next piece: a **trampoline** for tail calls. `f(n) = f(n)` is a tail call — the recursive call is the last thing the function does, there's nothing pending on top of it. a call like that doesn't actually *need* a new stack frame; you can throw the old one away and reuse it. the current interpreter doesn't know that, so it stacks them up and hits the depth guard. teaching `eval` to recognize a tail call and loop instead of recurse means tail recursion runs in constant stack — and a program written as `loop(state) = … loop(nextState)` runs forever without descending. which is, not coincidentally, exactly the shape of an editor's main loop. the trampoline is what lets stim's loop be *Stitch* instead of a Rust harness. it also lets the depth number climb back up, because tail recursion stops counting against it.
- and one investigation I'm saving for last: the interpreter is suspected of leaking a little on every run — reference cycles that the refcounting can't collect. stim, being the first genuinely long-running Stitch program, is the thing that would feel it. so before I decide whether stim can live on the tree-walker or needs to wait for the bytecode VM and its garbage collector, I want to *measure* it — run a stim-shaped immutable update loop thousands of times and watch whether memory grows or just churns. that measurement is the difference between a hunch and a decision, and I'd rather have the decision.
- so: the evaluator can now be given a budget, and it fails deep recursion cleanly instead of dying. two ways to never stop, two guards. it is not lost on me that I set out to write a text editor and my last two posts have been about a lexer and a step counter. but the editor keeps asking the language good questions — first "can you tell me where I went wrong," now "what happens when I run forever" — and the answers are making the language into something I'd actually want to write an editor in.

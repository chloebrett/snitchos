# snemu 09 — the fast part wasn't the native part

The snemu-itest gate boots the kernel and replays every scenario's assertions
against snemu's frame stream instead of QEMU's. It runs on every change, so its
wall-clock is a tax I pay constantly. This stretch was about making it cheap — and
it turned into a long, humbling exercise in measuring instead of guessing.

I went in with a hypothesis I was pretty sure of: the way to make an emulator fast
is a JIT that lowers guest instructions to native host code. I ended up building
exactly that — a from-scratch AArch64 JIT, register allocator and all — and the
punchline is that it made things *slower*, and the thing that actually delivered a
3× was code I'd have called boring.

## The build: Backend B, a real native JIT

snemu's block JIT already had the shape: a frontend that discovers a basic block,
lowers it to a reified `Vec<Op>` IR, caches it by PC, and (Backend A) *interprets*
that IR with block-level amortisation — check interrupts once per block instead of
per instruction, advance PC once, keep the register file in a host local. Backend B
was supposed to be the payoff: lower the same IR to native AArch64 and run *that*.

It's the deepest systems code in the project, and it works:

- **Executable memory on Apple Silicon.** The gating risk, cleared first. macOS
  enforces W^X in hardware, so the code buffer is `MAP_JIT` memory you flip
  writable→executable per-thread with `pthread_jit_write_protect_np`, then flush the
  I-cache (ARM's I/D caches aren't coherent). Proving `movz x0,#42; ret` returns 42
  from generated bytes was the first real milestone.
- **A hand-written AArch64 emitter** — a little assembler in Rust, ~40 encoders. No
  LLVM, no build-time stencils; from-scratch like the rest of snemu.
- **The whole ISA-ish surface** the hot code needs: the ALU families, `SetImm`,
  conditional branches (with the fiddly signed-vs-unsigned condition-code mapping —
  a `-1 < 1` test earns its keep there), jumps, jump-registers.
- **A code arena** — bump-allocate blocks into 1 MiB `MAP_JIT` chunks instead of one
  `mmap` per block.
- **Cross-op register allocation** — guest registers live in host registers across
  the whole block, loaded once at entry and stored once at exit, so the body is pure
  register ALU instead of a memory round-trip per op.
- **A direct-mapped native cache** — because a `HashMap` (SipHash) probe per block
  execution is real overhead the interpreter doesn't pay.

Every step was oracle-clean: with Backend B on, the full suite is *byte-identical*
in guest instret to Backend A and to the pure interpreter. Same discipline the whole
emulator runs on — a run with the fast path on is provably the same as one with it
off, or it isn't shipped.

And it's slower than the interpreter-over-IR it was supposed to beat. `--speedup hi`
(Backend A) is ~3.1s; `--speedup extra` (Backend B) is ~3.6s. Consistently.

## Why: the frontend was already the JIT

The reason is obvious in hindsight and I should have seen it before writing a line of
codegen. **A and B share the entire frontend.** Block discovery, the PC→block cache,
the interrupt hoist, the once-per-block PC advance — all of that is Backend A, and
it's where the win lives. B only differs from A in how it runs the ops *inside* a
block: native register ALU vs an interpreted `match` over register-cached array
slots. Those are both fast. The difference per op is a handful of nanoseconds.

Meanwhile B *adds* costs A doesn't: an extra cache lookup per execution, and — the
real killer — an `extern "C"` call boundary into the generated code per block. And
the genuinely hot blocks (the scheduler and spin-wait loops the profiler keeps
naming) all contain loads/stores, which B can't emit natively, so they fall back to
A anyway — while still paying B's per-block overhead to *discover* they fall back.

So Backend B is a correct, complete native JIT sitting at parity-minus-a-lookup with
an interpreter, because the interpreter already stole its lunch: block-level
amortisation is 90% of what "a JIT" buys you, and native codegen is the last 10% that
this workload doesn't cash. The only thing that genuinely pulls ahead is *block
chaining* — patching a compiled block's exit to jump straight into the next one so a
hot loop never returns to the dispatcher — and that's a big, delicate build (it
breaks the interrupt-delivery oracle) with a payoff capped by how hot-loop-bound the
workload is.

## The ladder that told the truth

I bundled the pile of speed flags into a `--speedup low|med|hi|extra` preset. Making
it *monotonic* forced the honesty, because it printed the whole story as a table:

| `--speedup` | adds | full suite |
|---|---|---|
| low | idle-skip only | 11.7s |
| med | + native memops + software TLB | 9.3s |
| hi | + the block JIT (Backend A) | **3.1s** |
| extra | + Backend B native codegen | ~3.6s |

There it is. The block JIT frontend is a ~3× lever. Native memops + a TLB are a
~20% rounding error. Native codegen is a regression. `hi` is the fastest *portable*
config; `extra` is honestly labelled "experimental, and slower."

`med`→`hi` being the whole game is the finding. Everything I found *impressive* to
build lived below the fold.

## The TLB, and measuring instead of theorising (again)

The `extra` result wasn't the first time this session that the obvious lever turned
out flat. I built a software TLB — cache the Sv39 walk, invalidate on `satp`/`sfence`
— dead certain it was *the* fix for the "load-store pole" the profiler named. It
bought ~8%. The page walk is three L1-hot `Vec<u8>` reads; caching cheap reads saves
cheap time. Translation was never the pole. (And the biggest memory consumer,
frame-zeroing `memset`, goes through a different fast path entirely.)

Earlier still, I'd sketched copy-on-write machine forks to kill the per-scenario
clone — until I measured the clone at 0.19s across the whole suite (the right-sizing
work had already shrunk the machines to 16 MiB). Deferred, unbuilt. The premise
didn't survive contact with a number.

Three times in one stretch, the same shape: build the plausible optimisation, measure
it, watch it come in modest or negative, and find that the lever that mattered was
something I'd already shipped and stopped thinking about. I *know* this lesson —
snemu 04 is literally titled "the bottleneck wasn't where the map said" — and I still
had to relearn it, because a native JIT is a much better story than "your interpreter
was already amortising the expensive part."

## What I'm keeping

- The TLB and the direct-mapped native cache — small, sound, real improvements. Kept,
  off by default, oracle-proven.
- The `--speedup` ladder — the most useful artefact of the whole stretch, because it
  makes the cost/benefit legible at a glance and refuses to let a preset lie about
  which knob does the work.
- Backend B — correct, complete, and honestly parked in `extra`. It's not dead code;
  it's the substrate block chaining would build on, if a workload ever justifies it.

The headline optimisation of the month was a wash. The month's actual output was a
table that says so, and the discipline that let me trust it. On a project whose whole
premise is a ruler that measures itself, a native JIT that measured to nothing is
exactly the kind of result the ruler is for.

(Also this stretch: the itest snapshot tree — collapsing observe-only scenarios onto
a shared forward run, sharing interactive prefixes with state-hash-verified fork
nodes, and a critical-path scheduler that took the suite from four phase barriers to
one dependency graph, 85%→99% worker utilisation. That one actually paid — a
different post. And a one-line `fw_cfg` stub in the bus, because the kernel's new
ramfb bring-up touched a device snemu didn't model and faulted the boot. The ruler
caught that too.)

## Epilogue: the lever was switched off

A couple of days after writing this, I went back to shave the snemu-itest wall-clock
again, sure the remaining cost was cross-hart spin-waiting. I built the profile to
prove it — and the debug build obliged, spin functions right near the top. Then I
re-ran it against the *release* kernel, the one the gate actually runs. The spin
functions vanished. A third of the debug profile had been iterator and UB-check
machinery that doesn't exist in an optimised build. I'd relearned this post's lesson
one level down: not just measure, but measure the thing you ship.

The real pole was telemetry emission, and its worst line was the span path
re-interning each span's name with an O(n) table scan on every open — every
heartbeat, every tick. The metrics had cached their ids at registration ages ago;
spans never got the same treatment. One per-call-site cache, byte-identical on the
wire, ~15% of guest instret gone.

And then the part that stings. This post crowned `--speedup hi` — the boring
block-JIT frontend — the 3× lever. When I went to check the gate's numbers, the
everyday `cargo xtask snemu-itest` was taking ~13 seconds. `hi` should have made it a
few. It turned out `--speedup` defaulted to `low`: idle-skip only, block JIT off. The
3× lever this very post had identified had been sitting switched off the entire time.
Changing the default from `low` to `hi` — one line — took the suite from ~13s to
3.6s. Bigger than the span fix. Far bigger than the native JIT the month went into.

So the ledger for the whole stretch: a from-scratch AArch64 JIT delivered nothing,
and the two things that actually made the ruler faster were a small kernel caching fix
and *turning on a speedup I'd already built, measured, and then failed to make the
default.* The bottleneck wasn't where the map said. It never is.

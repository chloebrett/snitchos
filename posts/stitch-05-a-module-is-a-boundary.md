# Stitch 5 — a module is a boundary

- Stitch had one flat namespace. every function, type, and constructor a program defined landed in a single shared table, mutually visible, no walls. that's fine for a toy and fatal for a standard library — you can't grow `map`/`filter`/`fold` into an organised `Seq.`/`List.`/`Str.` if there's nowhere to put the dot, and you can't have encapsulation if everything is public by definition. so this milestone is modules: files that are namespaces, a visibility marker, imports, and — the part i didn't see coming — the boundary that finally gives the orphan rule somewhere to stand.
- the surprise was how little *machinery* it took. i braced for a new subsystem — a resolver, a path-expression node, a symbol-table layer. it turned out modules add almost no runtime: a module is a value, the `.` you already have resolves its members, and closures already scope themselves. what modules actually add isn't machinery. it's **lines you're not allowed to cross.**

## the shape — a module is a value

- a module is a file. `math.st` is the module `math`; its name is the filename. inside, declarations are private by default — the marked form, `ext`, exports one: `ext double(x) = x * 2`. another module reaches it with `use math` and a dotted path, `math.double(21)`, or pulls names in directly with `use math.{double}`.
- the implementation that made me happy: there's no new "module path" expression and no separate resolver. a module is just another runtime value — a `Value::Module` holding its export table — and `math.double` is the *same* `.`-access the language already does for record fields and methods. one new case: if the thing left of the dot is a module, look the name up in its exports; otherwise it's a field or a method, exactly as before. the dot didn't learn a new trick; it just met a new kind of value.
- and each module evaluates in its own namespace for free, because of a decision made three posts ago. a function is a closure, and a closure captures the environment it was defined in. give each module its own environment — prelude, its own items, whatever it imported — and a function resolves its names in its own module without anyone threading a "current module" through anything. the method table is shared (dispatch is by the value's type, program-wide); the namespaces are per-file; and the split falls out of closures already working the way they do.

## the argument that mattered — naming is design

- the longest part of this milestone wasn't code. it was deciding what to *call* exporting. that sounds like bikeshedding, and it half is, but the constraint that settled it is the spine of the whole language: **no token does two jobs.** `->` always means "maps to," `|` always "alternation." so a sigil for visibility was out the moment i reached for one — `+` and `*` are already arithmetic, and overloading them is precisely the Frankenstein move the rule exists to forbid. a *word* it had to be.
- which word matters, because visibility is a modifier and has to sit in a row with the modifiers already there: `mut`, `free`, `uses`. `pub mut insert` (the Rust word) works but is a tell; `+insert` mixes a sigil and a word on one declaration and reads wrong. i went through `out`, `in`, `push`/`pull`, and landed on `ext` to export and `use` to import — two unrelated words, each the best fit for its slot, the way Rust pairs `pub` and `use` rather than forcing a cute `in`/`out` symmetry that doesn't survive contact with grammar (`in foo` reads as "this code is *in* foo," and steals the obvious word for a future membership test).
- the deeper rule underneath the choice: **the default is invisible, the deviation is marked.** `let` is immutable until you write `mut`; a function has no authority until it declares `uses`; and now a declaration is private until you write `ext`. private-by-default isn't a safety nag — it's the same grammar of "the safe thing is the quiet thing" running through the whole language. public-by-default would have inverted it, and a capabilities language that makes *exposure* the frictionless default is telling on itself.

## the part that was actually tricky — except it wasn't

- two modules that import each other. `a` uses `b`, `b` uses `a`. i sat down to write cycle detection — a visited set, a topological sort, an error for the unbreakable case — and then realised the language had already made it a non-problem.
- the trick is *when* names resolve. a Stitch function doesn't look up its free variables until it's called; names resolve lazily, at call time, against the shared globals. so the load is two phases with no recursion between them: first **declare** every module's own items (no imports yet), then **link** every module's imports (every export table now exists). by the time `a` looks for `b.pong`, both tables are fully built — it doesn't matter that they reference each other, because nothing was *evaluated* during loading, only registered. cycles need detection only if you load eagerly and recursively; discover the whole set first and the cycle just... resolves. the thing i braced for evaporated because of a decision (lazy name lookup) made for an entirely different reason.

## the line that needed a *where*

- here's what modules are actually for, beyond namespacing. Stitch's one form of polymorphism is the `contract` — a trait — and traits have a coherence problem: if any module could write `on SomeType : SomeContract`, two modules could implement the same contract for the same type differently, and dispatch wouldn't know which to pick. the fix is Rust's orphan rule: an implementation may only live where the type or the contract does.
- but "where the type lives" is a sentence you can't even *say* without modules. in one flat namespace there's no "where." so coherence was un-enforceable until this milestone — and the moment modules existed, the rule had a home: an inherent `on Type` must be in the type's module; a conformance `on Type : Contract` must be in the module that defines one side or the other. behaviour stays findable with the thing it's attached to; you never have to wonder which far-off file reached over and bolted a method onto your type. the boundary isn't a side effect of modules. for the type system, it's the *point* of them.

## the stdlib, split by a constraint

- the reason this milestone exists is the standard library, and the first cut raised a question i thought would be taste and turned out to be forced. the namespaced modules are `Seq` and `Str` — `Seq.iterate`, `Str.join` — views onto the existing functions, grouped. but `map`/`filter`/`fold` are *polymorphic*: one name, eager on a `List`, lazy on a `Seq` (post 4's bet). so do they become `Seq.map` and `List.map`?
- no — and not because i decided so. a single-file program with no imports doesn't get the namespaced modules at all; they're a multi-file feature. so the universal combinators *have* to stay flat and unqualified, or `[1, 2, 3] |> map(f)` breaks in the simplest possible program. the constraint chose for me, and it chose correctly: the polymorphic core stays one flat vocabulary, and the namespaces hold only what's genuinely type-specific — the lazy producers, the string ops. "one name each" survived — not by conviction this time, but because the alternative didn't fit through the door.

## the soul — boundaries, not machinery

- i keep coming back to how little got *added*. the runtime grew one value variant and a new case on the dot. the loader is a walk over imports with a set to avoid repeats. there's no symbol-table subsystem, no resolver pass, no cycle algorithm. and yet the program can now do things it fundamentally couldn't: hide an implementation detail, organise a library behind dotted names, refuse a method bolted on from afar.
- that's the lesson. a module system reads like *more* — more structure, more machinery — but what it actually is, is *less*: a set of lines the program isn't allowed to cross. private-by-default is a line. the orphan rule is a line. an unimported module being invisible is a line. the machinery was mostly already there; the milestone was deciding where the walls go. a language gets more expressive not only by gaining powers but by gaining the ability to *withhold* them.

## what i learned

- **naming isn't bikeshedding when the name has to fit a grammar.** `ext`/`use` took longer than the loader, and that was time well spent: a keyword that breaks "one token, one job" or sits wrong in the modifier row is a permanent papercut. the constraint that the word has to live next to `mut`/`free`/`uses` did most of the deciding.
- **lazy evaluation pays dividends in places that aren't evaluation.** import cycles are free for the same reason recursion and mutual recursion were free three posts ago — names resolve at use, not at definition. a decision made for one feature keeps quietly solving unrelated problems. that's the sign it was the right one.
- **a boundary is a feature.** the most useful thing modules added is *restriction* — what you can't see, can't import, can't implement. encapsulation and coherence aren't powers; they're refusals, and they're exactly what a flat namespace could never express.

## what's next

- the **organised stdlib** itself, now that there's somewhere to put it — `Seq.`/`List.`/`Str.` with the lazy vocabulary filled in (zip, scan, chunk), and the prelude trimmed to the genuinely-universal core.
- **opaque types** — the one piece of encapsulation i deferred: a type that's public while its constructor and fields stay private, so a module can hand out a value you can hold but can't take apart. visibility on items shipped; visibility on a type's *insides* is the next line to draw.
- and still ahead, the two that were always the point: the **bytecode VM** with a real garbage collector, and the **capability effect system** wired to the telemetry protocol. modules drew the lines inside a program; capabilities draw them around it.
- a flat namespace is a room with no walls. it turns out the walls were the architecture.

# Post 30 — A filesystem you can watch

- post 29 built the bet — the kernel never learns what a file is — and ended pointing two places: round out the trait (the `lookup`/`remove`/`readdir` ops, and the rights gate that until then i could only *assert* held), and then make the whole thing observable, a filesystem you watch happen in a trace. this post is both. and the through-line, again, is the one from last milestone wearing a third hat: twice over, the thing i needed to build a new feature turned out to be a primitive i already shipped. the rights gate snitched through a capability the FS server was already holding and not using; the trace crossed the process boundary through a mechanism the kernel already ran for IPC two milestones ago. i keep reaching for a new authority and finding the old one already fits.

## the gate i could finally test

- post 29 made a claim i couldn't yet check: "the filesystem enforces its own file rights per message." the kernel carries `READ`/`WRITE` in the badge and never reads them; the FS is supposed to refuse a `write` that rides a `READ`-only cap. but to *test* a refusal you need a `READ`-only cap to an actual file — and i didn't have one. the root cap was read-only, but root is a directory; refusing a write to a directory proves nothing about the *rights* gate, because you can't write a directory anyway. every cap to a real file was `READ|WRITE`. the claim was untestable for want of an attenuated cap.
- `lookup` is what mints one. a client looks a file up and asks for the rights it wants; the server mints a child cap badged `parent ∩ requested`. ask for `READ` only, get a `READ`-only file cap, try to write through it — and now the gate has something to refuse that *isn't* refused for any other reason. "the filesystem enforces its own rights" stopped being a sentence in a post and became a thing the suite watches: write through the attenuated cap → refused; write through a `READ|WRITE` lookup of the same file → succeeds. the positive control matters as much as the refusal: a gate that refuses *everything* passes the refusal test and fails the point. you prove the lock works by also proving the key does.

## the intersection, and the cap that had to grow

- `parent ∩ requested` is the whole rule, and it has a consequence i didn't see until i wrote it: the directory cap is a *ceiling*. you can never `lookup` your way to more authority than the cap you looked up *through*. which meant the root cap couldn't stay read-only — if root grants at most `READ`, no writable file is reachable beneath it, and `create`→`write`→`lookup`→`read` falls apart at the first write. so root had to carry `WRITE`.
- and that's more honest, not less. the old root cap was *pretending* to be read-only while `create` cheerfully mutated the directory through it — the gate just wasn't looking. making root `READ|WRITE` stopped the pretense: the cap now says what it actually permits. the test that broke when i changed it (`fs-connect-mints-root` asserted the old read-only badge) wasn't a regression to paper over — it was the suite correctly noticing the authority had changed, and the fix was to assert the truer thing.

## refusals snitch — and the sink was already in my hand

- a refusal that's silent is a bug you can't see, so the gate doesn't just reply `Denied` — it *snitches*: it emits the inode it refused and the right the cap was missing, structured, the way every other authority decision in this kernel is observable. the satisfying part is how it emits. i assumed a structured snitch meant new machinery — a second capability granted at spawn, threaded through a new startup register, captured in crt0, surfaced by a runtime accessor. i started sketching it.
- then i noticed the FS server *already* holds a telemetry sink. every process is born with one. and the FS server had never used it — it emits no metrics, the cap was dead weight. a telemetry capability has the metric's name fused into it at grant time (that's why emitting needs no string from userspace — the authority already names where it writes). so i didn't grant a new authority; i re-aimed the one it already had, binding its sole sink to a denial-named gauge. no new cap, no new register, no crt0. the server's one telemetry authority now names exactly its one telemetry purpose. same shape as post 29's reply cap, a third time: the power you're about to invent is usually one you're already carrying, pointed at the wrong thing.

## the kernel still names the metric (and shouldn't)

- one place the milestone's own rule isn't applied yet, worth naming so i don't pretend otherwise. "the kernel never learns what a file is" — but the kernel *does* still learn what a metric is. that denial gauge's name, `snitchos.fs.denied`, is registered in the *kernel*. the userspace FS can't name its own metric; it can only emit to a name the kernel pre-baked into its cap. the gate worked only because i registered the name kernel-side and aimed the sink at it.
- which is exactly the wrong layer, by this project's own lights. mechanism in the kernel, meaning in userspace — the kernel should transport an opaque metric frame and know nothing of its name or kind, the same way it carries a badge it can't interpret and a denial value (a packed inode-and-right) whose meaning it can't read. the fix is a register-a-metric syscall that copies a name from user memory and interns it — and the kernel already does precisely that for *span* names, so the pattern's sitting right there to copy. i wrote the note and left the wiring for its own milestone. naming the gap is cheaper than pretending the rule is fully kept.

## a filesystem you can watch

- the headline. every FS operation is now a span — `fs.create`, `fs.read`, `fs.lookup` — opened on the client and held open across the call. and in the trace view they don't sit in a flat client-side list: the *server's* handling of each request nests *underneath* the client's op, one tree spanning two processes. you watch a `create` on the client and see, nested inside it, the server doing the work — attributed across the boundary, the observability pillar finally pointed at storage.
- and here's the part i'd budgeted time for and didn't spend: making the trace cross the boundary. i expected to thread a trace context through the message — stamp the parent span into the request, have the server adopt it. i didn't write any of that, because the kernel already does it. when a client `call`s, the kernel grabs the caller's innermost open span and stashes it with the delivered message; when the server `receive`s, the kernel *seeds the server's span cursor* with it before handing control back. so the server's very next `span()` is a child of the client's op, for free. that machinery was built for the v0.9 IPC trace — "watch the whole conversation as one trace" — and the FS spans rode it without a line of new propagation code. the third time this milestone that the new feature was an old primitive seen sideways: the rights snitch was a sink i already held, the cross-process parent was a cursor the kernel already seeded.

## what i learned

- **a claim you can't test isn't enforced, it's hoped.** "the filesystem checks its own rights" was true in the code and unprovable in the suite until `lookup` could mint an attenuated cap to refuse. the feature that made the gate real wasn't the gate — it was the thing that produces a weakened capability to point at it. and prove the key works too, or a gate that refuses everything passes.
- **attenuation makes the directory cap a ceiling — so the ceiling has to be honest.** `parent ∩ requested` means root can't grant more than it holds, which forced root to carry `WRITE` instead of pretending to be read-only while `create` mutated through it. the test that broke was the suite noticing the authority changed; the fix was to assert the truer badge.
- **the authority you need is usually one you already hold.** the structured refusal snitch didn't need a new cap and a new register — the FS server's bootstrap telemetry sink was dead weight, and a telemetry cap already names where it writes, so i re-aimed it. third time this milestone (reply-cap, dead sink, seeded cursor). reach for the concept before the mechanism; the mechanism is often already in your hand.
- **name the place your own rule isn't kept.** the kernel still registers the FS's metric name — meaning in the kernel, exactly the layer this project spends its milestones pushing *out*. it works, it's wrong, and writing that down (the syscall is a copy of the span-name path) beats letting "the kernel never learns what a file is" quietly imply it never learns anything.
- **the trace crossed the boundary because i'd already built the bridge.** i budgeted time to propagate trace context through IPC and spent none: the kernel seeds the receiver's span cursor with the caller's span on every `receive`, built for v0.9. the FS spans nested under client ops with no new code. the cheapest cross-cutting feature is the one a primitive from two milestones ago already carries.

## what's next

- **the kernel stops keeping a list of programs.** post 29's addendum found the thread and it's the natural next pull: today the kernel bakes fifteen userspace binaries in by name via `include_bytes!`. with a filesystem to load from, those collapse into one initramfs blob the kernel hands over and never opens — it stops knowing `hello` and `rpc_server` and the rest by name. not fewer bytes baked in; *the knowledge leaves*. the kernel was never going to learn what a program is either.
- **then the first workload that stores something that matters.** v0.11: a metrics-ingestion server that writes real data to a filesystem the kernel still can't read. and somewhere in there, the metric-naming wiring from above — so the day the FS names its own telemetry, the kernel finally never learns what a metric is, the same sentence one more layer down.

# Post 34 — A tap on the shoulder

- every syscall the kernel has answered so far has been a *question*: a process asks, the kernel replies, the process continues. open this span, send this message, give me memory. the control flow is a call stack — the program is in charge, the kernel is a function it calls. but the physical world doesn't ask permission. a key gets pressed, a timer fires, a disk finishes, a child process dies — these *happen*, whether or not anyone is waiting. an OS needs a way to go the other direction: to **tap a process on the shoulder** and say "something happened." this is the post where SnitchOS grew that — the async kernel→user **notification** — and the interesting part wasn't building it. it was noticing it already existed, badly, in one place, and that the general version is *smaller* than the special case.
- this is also the primitive the Stitch convergence post ended on. [Stitch 7](stitch-07-it-runs-on-the-metal.md) got the language running as a userspace process on the OS — the two side-projects became one — and closed on "the OS just grew an async notification primitive, the gateway to interrupt-driven input and to processes waking each other." this is that gateway, opened. the shell waits on its children through it; the keyboard will deliver keystrokes through it; one day a REPL on the metal blocks on it. same tap, many shoulders.

## the primitive is what it refuses to carry

- a `Notification` is **one machine word of bits and at most one parked waiter.** that's the whole object. `signal` OR-s a mask into the word and wakes the waiter; it never blocks. `wait` returns-and-clears the word if it's nonzero, otherwise the caller parks off the run queue until someone signals. there is no message, no queue, no buffer, no payload beyond that single word of bits whose *meaning the kernel never learns*.
- the temptation, building an async primitive, is to make it a mailbox — let the producer attach data, let messages pile up until the consumer drains them. i built the opposite, and the reason is the most important sentence in the design: **the thing that makes it cheap is the thing that makes it safe.** because signals *coalesce* — three signals before anyone waits collapse into one wake carrying the OR of their bits — there is nothing to buffer, so no kernel memory grows, ever, no matter how hard you hammer it. you cannot spam a notification to exhaust the kernel, because it physically cannot remember more than "it happened (and roughly which bits)." the resource-exhaustion attack surface is zero, and it's zero *for the same reason* the object is semantically poor.
- so coalescing isn't a limitation you apologize for — it's the feature. a notification answers exactly one question, *"did it happen?"*, and refuses every richer one — how many times, in what order, with what data. if you need those, you need a real channel (a synchronous endpoint), and you pay for it. the notification is the floor: the cheapest possible "wake up." (this is seL4's Notification object, more or less; i'm not claiming the idea, just the discipline of keeping it this bare when every instinct says to add a field.)
- one more subtraction with teeth: **one waiter per notification.** a second task trying to wait on an already-waited notification isn't silently queued or silently dropped — it's *refused*, loudly, with a `NotificationBusy` snitch on the wire. that's a scar from the reaping table two milestones back, which had a single-slot waiter that a second writer would silently overwrite — stranding the first waiter to block forever. the lesson stuck: if your invariant is "one waiter," then violating it is an *event*, not a no-op. snitch your own internal contract violations, not just userspace's.

## the thing already existed, as a special case

- here's the part i didn't expect. i thought i was building something new, and i was actually *generalizing something i'd already built last milestone* — and the general version turned out to be less code than the special one.
- v0.12's first half was exit/wait: a parent `Wait`s, a child `Exit`s, the parent wakes with the child's status. under the hood that's a task parking itself off the run queue (`block_current`) and another task putting it back (`wake`), brokered by a little table that remembers who's waiting on whom. i wrote all of it. and when i went to build notifications, i realized: *that is the notification.* "a task parks; something happens elsewhere; the task is made runnable and told it happened" — that's the exact control flow, and child-exit is just one meaning bolted onto it.
- a `Notification` is that control flow **with the meaning removed.** the kernel carries a bit; userspace decides the bit means "your child exited" or "the UART has a byte" or "the timer fired." the kernel stays ignorant — which is the move this project keeps making (badges the kernel never interprets, span names it doesn't dedup across processes, metric names it doesn't know ahead of time). and as before, *the general thing is smaller than the special case*, because the special case has to know what it's about and the general one gets to not care. less mechanism, more reach.
- which raised an obvious, tempting question: should child-exit *become* a notification? rewrite the reap path on top of the new primitive, one wake mechanism instead of two, the clean unification. i decided **no, not yet** — and writing down why was the more useful exercise than doing it. the reap path is shipped, tested, and works; folding it onto notifications would reopen verified code to convey a status (an `i32` exit code) that the notification's bit-word can't hold, for no capability i don't already have. and the consumer that will actually *force* the generic shape to be correct isn't child-exit at all — it's a **device interrupt**, which is genuinely async and genuinely payload-free, where there's nothing to convey but "it happened." build it generic now so that future fold is cheap; do the fold now and you buy nothing and risk something. it's the same shape as last post's deferral: don't couple a thing that works to a unification that can wait for its real motivating case.

## proving a wake you can't see

- the observability problem here is the most interesting one i've hit, because it cuts at the project's whole premise. SnitchOS exists to make the invisible visible — and an async wake is the single most invisible thing in an operating system.
- think about why a *synchronous* call is easy to trace: its causality **is** the call stack. when A calls B, the fact that B runs *because of* A is written into the control flow — the span opens when A calls, closes when B returns, and the parent/child relationship is read straight off the stack. you don't reconstruct it; it's just there. a notification deliberately destroys that. the signaller and the waiter are **not** in a call/return relationship — the signal arrives out of band, from a task that isn't on the waiter's stack and may have already exited. so the question "did this task wake *because of* that signal?" has no answer in the control flow. the causal edge isn't recorded anywhere, because the whole point of async is that there's no stack linking the two.
- so you have to *put it on the wire.* two new frames — `NotifySignal` (a producer signalled, at time t₀) and `NotifyWait` (a waiter woke carrying these bits, at time t₁) — and now Tempo can draw the one arrow a call stack can't: a dependency edge between two tasks that were never on each other's stack. the snitch makes visible precisely the control-transfer that is, by construction, invisible. that's the post angle, and it's the strongest one the project has had: not "here's a metric of a thing," but "here's a causal link that doesn't exist in the code, reconstructed from two events because we chose to emit them."
- and then the test taught me something i'll keep. the integration test boots two userspace processes — a parent that creates a notification, spawns a child handing it the cap, then blocks in `WaitNotify`; a child that signals a known bit pattern (`0b101`) and exits. i assert the `NotifySignal` and `NotifyWait` frames appear. but the assertion that pins it down hardest **isn't a frame at all** — it's a metric: the parent emits the bits its `WaitNotify` syscall *returned*, and i check that value is 5. because a frame on the wire proves *the kernel emitted an event*; the returned value proves *the bits reached the program*. those are different claims. an event can fire while the delivery is broken; the value a syscall hands back to userspace cannot lie about whether userspace got it. the frame is the story; the returned value is the receipt. when something has to be *true* and not just *observed*, assert the effect, not the announcement of the effect.

## what i learned

- **a primitive can be defined by what it won't carry.** the notification's no-queue, no-payload, coalescing shape isn't a set of limitations — it's the design. the property that makes it impossible to weaponize for resource exhaustion is the *same* property that makes it semantically poor. cheap and safe were the same decision.
- **the general thing is often smaller than the special case.** child-exit→wake was already the notification, with a meaning welded on. removing the meaning — letting the kernel carry a bit it doesn't understand — was less code, not more. the recurring win in this project is the kernel knowing *less*.
- **don't fold the special case onto the general one until its real consumer exists.** unifying child-exit onto notifications now would reopen working code for an aesthetic win and a status field that doesn't fit. the device-IRQ consumer is what will prove the generic shape; build for it, wait for it, don't pre-couple.
- **synchronous causality is free; async causality must be reconstructed.** a call stack records "B ran because of A" for nothing. an async wake records it nowhere — so if you want to see it, you emit it. the two notification frames exist purely to draw an arrow the control flow refuses to.
- **assert the effect, not the announcement.** the wire frame says the kernel fired an event; the value the syscall returned says the program actually received it. test the receipt, not the story.

## what's next

- the **shell** — and now it has every piece. a parent that can `spawn` a child with exactly the caps it chooses (v0.11), `Wait` for it to finish and reap it (v0.12a), and be woken asynchronously when things happen (v0.12b, this post). that's the whole skeleton of a command interpreter: read a line, delegate least authority, launch, wait, repeat — and every delegation a `CapEvent` you can watch. the [explicit-authority shell](../docs/roadmap-and-milestones.md) is the next milestone, and it's the one this whole arc has been walking toward.
- and the convergence keeps tightening. Stitch 7 put a real REPL on the metal; this post built the async organ that interrupt-driven input will use, so the keyboard stops being a 1 Hz polled afterthought and becomes a notification a task blocks on. the language runs on the OS, the OS can wake the language, and the shell that ties them together is a Stitch program over real syscalls where `grant` and `watch` and `hold` are the vocabulary. two projects, one tap on the shoulder.
- further out, the primitive collects its real consumer: **device interrupts**, where the kernel's trap handler signals a notification a driver waits on — IRQ→userspace over the exact object built here, unchanged. that's the consumer that earns the generality, and it's the moment child-exit might finally fold on top too. the tap on the shoulder was built to reap a zombie; it'll end up delivering a keystroke.

# 🧑‍💻 v0.7 userspace — concepts

*Mental model for the userspace milestone: privilege modes, the trap
stack-switch (`sscratch`), how a program becomes running memory (ELF
loading), and the three address spaces. Companion to
`plans/v0.7a-first-userspace.md`. Written for the `learning/` track —
context-switch registers and privilege transitions are flagged weak
areas.*

---

# 1. Privilege modes — the one idea underneath everything

RISC-V has three privilege levels. SnitchOS uses all three, but has only
ever *executed* in two so far:

| Mode | Bits | Who | SnitchOS |
|---|---|---|---|
| **M** machine | `11` | firmware | OpenSBI, below us — we never run here |
| **S** supervisor | `01` | the kernel | **everything through v0.6 runs here** |
| **U** user | `00` | unprivileged programs | **new in v0.7** |

A higher mode can do everything a lower mode can, plus more: touch
privileged CSRs, run privileged instructions (`sret`, `sfence.vma`),
ignore the `U` page-permission bit. The hardware enforces the wall; the
kernel decides where the wall sits.

Everything SnitchOS calls a "task" today (the four demo tasks, idle,
main) is a **kernel thread**: S-mode, shared address space, cooperative
switching. A userspace **process** is the genuinely new thing —
**lower privilege *and* (eventually) its own address space**. Those two
properties are separable, which is exactly why v0.7a splits into "drop
privilege" (Step A) and "separate the address space" (Step B).

## The transition is two instructions and one CSR

**Going down (S → U):** there is no "enter user mode" instruction. You
*return* into it, reusing the trap-return machinery:

1. set `sstatus.SPP = 0` — "the Previous Privilege was User"
2. set `sepc = <user entry PC>`
3. `sret`

`sret` jumps to `sepc` and sets the current privilege to `SPP`. It is
the *same instruction* `trap.S` already runs on every trap return —
today `SPP` happens to be `1` (we always trapped from S), so it returns
to S. Flip `SPP` to `0` and the same instruction lands in userspace.

**Coming back up (U → S):** any trap. A timer interrupt, a page fault,
or — deliberately — an `ecall`. The CPU vectors through `stvec` into
`trap_entry`, records the cause in `scause`, the faulting/calling PC in
`sepc`, and the pre-trap privilege in `sstatus.SPP`. **A syscall is just
a trap whose cause is "environment call from U-mode"** (`scause = 8`,
already decoded as `TrapCause::EnvCallFromUMode`).

So the privilege machinery is nearly free. What costs effort is the
*consequence* of a trap coming from a less-trusted, differently-mapped
context — section 2.

## `sstatus` bits that matter for the drop

Before the `sret` into U, set up `sstatus`:

- **`SPP = 0`** — return to U (not S).
- **`SPIE = 1`** — interrupts *enabled* after the `sret`. If you leave
  this 0, the timer stops firing while userspace runs and the whole
  kernel's heartbeat stalls until the next trap.
- **`SUM = 0`** — Supervisor User Memory access *off*. With `SUM=0` the
  kernel faults if it dereferences a `U` page. That's a feature: it
  catches accidental user-pointer dereferences. Set `SUM=1` only briefly
  around a deliberate copy from a user buffer (no such copy in v0.7a —
  the syscall argument arrives in a register).
- **`FS = Off`** — floating-point unit disabled. The kernel is
  integer-only; v0.7a's user program is too. Lazy FP save (v0.9) is when
  this grows up.

# 2. The trap stack-switch: why `sscratch` exists

Look at today's trap entry (`kernel/src/trap.S`):

```asm
trap_entry:
    addi sp, sp, -288    # allocate the trap frame on the CURRENT stack
```

This is safe **only because every trap so far comes from S-mode**, where
`sp` already points at a kernel stack. The instant a trap can arrive
**from U-mode**, `sp` points at the *user* stack — memory the kernel
must not trust and must not spill its registers onto. We have to get
onto a known-good kernel stack *before the first memory write*.

The chicken-and-egg problem: to switch stacks you need a spare register
holding the kernel stack pointer, but every register currently holds the
user's value and you can't spill them yet (no kernel stack). RISC-V
solves this with a dedicated scratch CSR: **`sscratch`**.

## The convention (software, not hardware)

- While running **in the kernel**: `sscratch = 0`.
- While running **in user**: `sscratch =` this thread's *kernel* stack
  pointer (parked there before we `sret`'d into user).

Trap entry's first move is an **atomic swap**:

```asm
csrrw sp, sscratch, sp   # sp <-> sscratch, in one instruction
```

After the swap:
- **Came from user:** `sscratch` held the kernel sp → now `sp` *is* the
  kernel sp, and `sscratch` holds the user's sp (saved, recover it into
  the frame later). 
- **Came from kernel:** `sscratch` held 0 → now `sp` is 0 and `sscratch`
  holds the real kernel sp. Detect the 0, swap back, carry on as today.

```asm
trap_entry:
    csrrw sp, sscratch, sp     # swap
    bnez  sp, .Lfrom_user      # nonzero => we got the kernel sp => from user
    csrrw sp, sscratch, sp     # zero => undo the swap; sp is kernel sp again
.Lfrom_user:
    # sp is a good kernel stack in BOTH cases now
    addi sp, sp, -288
    ...
```

The branch keys off **the contents of `sscratch`**, not the hardware
privilege bit. That is why the v0.7a plan sets `sscratch = 0` at boot
and rewrites entry *first* (Step 1): every existing S-mode trap then
flows through the new code, hits the zero path, and must behave exactly
as before — so the **whole existing test suite becomes the regression
gate** for the rewrite, with no userspace in sight.

### Why not "test it with a fake user context"?

You *could* set `sscratch` nonzero in S-mode and trigger a trap to drive
the `.Lfrom_user` branch. Rejected, because:

1. It tests the cheap half. `SPP` is still 1, so the closing `sret`
   returns to S — you never exercise the real privilege drop or the
   instruction fetch from a `U`-mapped page, which is the part that
   silently triple-faults.
2. It's a footgun. A nonzero `sscratch` in S-mode means a *real* timer
   IRQ that sneaks in first will believe it came from user and "switch"
   to your fake stack pointer.

The honest test of the user path is the real privilege drop (Step 4).
Step 1 only owes "didn't break kernel traps."

# 3. ELF loading: how a program becomes running memory

A program is **not** a flat lump you jump to. It is a set of *segments*,
each wanting a specific virtual address and specific permissions:

| Segment | Contents | Perms | Stored in the file? |
|---|---|---|---|
| `.text` | code | `R-X` | yes |
| `.rodata` | constants, strings | `R--` | yes |
| `.data` | initialised globals | `RW-` | yes |
| `.bss` | zero-init globals | `RW-` | **no — only a size** |

And an **entry point** that is generally *not* the first byte.

ELF (Executable and Linkable Format) is the standard container that
records exactly this. For *loading* (as opposed to linking) you need
only two parts of it:

- the **ELF header**'s `e_entry` — the entry-point virtual address; and
- the **program header table** — an array of segment descriptors. Each
  `PT_LOAD` entry says:

| Field | Meaning |
|---|---|
| `p_offset` | where the bytes live in the file |
| `p_vaddr` | what virtual address to map them at |
| `p_filesz` | how many bytes to copy from the file |
| `p_memsz` | how many bytes the segment occupies in RAM (`≥ filesz`) |
| `p_flags` | R / W / X permissions |

The gap `p_memsz − p_filesz` **is the bss**: memory the program expects
to exist and be zero, but which costs no bytes in the file.

## The loader is four steps per segment

```
for each PT_LOAD segment:
    1. allocate frames covering [p_vaddr, p_vaddr + p_memsz)
    2. map them at p_vaddr with U + the segment's R/W/X
    3. copy p_filesz bytes from the file image into those pages
    4. zero the remaining (p_memsz - p_filesz)   # the bss
then: set the user's initial PC = e_entry
```

That's the whole thing. No symbol resolution, no relocations, no dynamic
linking — those are *linker* concerns. Because v0.7a links the user
program **position-dependent at a fixed virtual address**, the loader
maps each segment at `p_vaddr` verbatim and is done. (Position-
*independent* executables would need relocation processing; explicitly
out of scope.)

## Why ELF and not a flat blob

`objcopy -O binary` can flatten a program into raw bytes you copy and
jump into. It is less code *today* but you throw away per-segment
permissions (everything ends up `RWX` — a W^X violation), the entry
point, and bss handling. More importantly: **the loader is the durable
artifact.** When the filesystem arrives (v0.10), loading a program from
a file runs the *same* four steps — only the source of the bytes changes
from `include_bytes!` to a file read. The flat-blob shortcut is the part
you would delete; the ELF loader is the part you keep.

## Where it lands in the codebase — the project seam

ELF parsing is **pure data transformation**: bytes in, a list of
`{vaddr, file_offset, file_size, mem_size, flags}` + entry out. No
`unsafe`, no MMIO, no CSRs. So it lives in **`kernel-core` and is
host-tested** like everything else there. The `kernel`-side loader is
the thin shell that walks the parsed plan and calls `mmu::map` + copies
bytes. Same pure-core / hardware-shell split as the rest of SnitchOS —
the flat-blob option would give nothing host-testable.

**Trust boundary:** even though v0.7a's input is embedded and trusted,
the parser validates everything and returns `Result` — bad magic, wrong
class/machine, out-of-bounds offsets, `filesz > memsz`. When v0.10 loads
untrusted files, the same parser already does the right thing.

# 4. Three address spaces, one process

After Step B each process sees a single Sv39 virtual address space split
in two:

```
0xFFFFFFFF_........  ┌───────────────────────────┐
 high half           │  KERNEL — shared into      │   root PTE slots ≥ 256
                     │  every process. image,     │   (image, linear map,
                     │  linear map, heap.          │    heap)
                     │  Mapped, but NO U bit →     │
                     │  U-mode faults if it tries. │
                     ├───────────────────────────┤
                     │            ...              │
                     ├───────────────────────────┤
 low half            │  THIS PROCESS — user code,  │   root PTE slots 0–255
                     │  data, stack. U bit set.    │   (empty today → no
0x00000000_........  └───────────────────────────┘    collision)
```

Two design decisions made concrete here:

- **Kernel mapped into every address space** (the Q27a decision,
  `docs/parked-memory-model-and-hal.md`). A trap therefore needs **no
  page-table switch** — kernel code and data are already visible. seL4
  does the same. The cost is the Meltdown attack surface, explicitly
  accepted for a learning OS. Mechanically: Step B copies the kernel's
  *root-level* high-half entries into each new process root; the deeper
  tables are shared, not duplicated.

- **The `U` bit is the firewall.** The kernel's high-half pages are
  mapped in the process's table but *without* `U`, so U-mode code page-
  faults if it touches them — even though the translation exists. The
  page table enforces *reachability*; in v0.7b *capabilities* will
  enforce *authority* on top. Step B's isolation test ("user load from a
  kernel VA faults") is the seed of that story.

For contrast, recall the kernel's own three address ranges (CLAUDE.md
memory-layout section): higher-half kernel VAs, the linear-map window,
and heap VAs. Userspace adds a fourth concern — the **low half**, per
process — and the rule that anything a *device* sees is still a physical
address.

# 5. The deliberate wrongness (why v0.7a ≠ v0.7b)

v0.7a's one syscall — "emit a telemetry metric" — is **ambient**: any
U-mode code can invoke it, the kernel runs no authority check. That is
the Unix model, built on purpose so the absence of capabilities is
*felt*. v0.7b reframes the exact same operation as **invoking a
`TelemetrySink` capability**: the process must hold an unforgeable handle
granting the right to snitch, validated against its own capability table.

The rule the capability design settles
(`docs/capability-system-design.md`): *userspace needs a capability to
snitch; the kernel snitches freely.* v0.7a is the "before"; v0.7b is the
"after," and the diff between them is the milestone's essay.

# Glossary

- **`sret`** — return from supervisor trap; jumps to `sepc`, sets
  privilege to `sstatus.SPP`. The instruction that enters U-mode.
- **`sscratch`** — supervisor scratch CSR. Holds the kernel stack
  pointer while in user, `0` while in kernel; the trap-entry swap pivot.
- **`sepc`** — supervisor exception PC. The PC a trap will return to.
  After an `ecall`, points *at* the ecall → handler must add 4.
- **`scause`** — why the trap happened. `8` = environment call from
  U-mode = a syscall.
- **`sstatus.{SPP,SPIE,SUM,FS}`** — previous privilege / prior
  interrupt-enable / supervisor-user-memory-access / FP state.
- **`PT_LOAD`** — an ELF program-header segment that should be loaded
  into memory.
- **bss** — `p_memsz − p_filesz`; zero-initialised memory not stored in
  the file.
- **`U` bit** — Sv39 PTE permission bit (bit 4); without it, U-mode
  cannot access the page even when mapped.
- **ambient authority** — the ability to act without presenting a
  capability; the thing v0.7a has and v0.7b removes.

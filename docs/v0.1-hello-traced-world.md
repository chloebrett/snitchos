# 🐣 v0.1 — Hello, traced world

*The killer feature exists in v0.1 even if nothing else does.*

# Goal
Prove the observability story works end-to-end on the smallest possible kernel. When this milestone is done, you can boot the kernel in QEMU on RISC-V, watch it emit a nested span tree for its own boot phases plus a heartbeat loop, and see those spans printed in real time as a text tree by a host-side reader.

No userspace. No scheduler. No allocator beyond what boot needs. No interrupts. No capabilities yet. No Grafana yet. Just: kernel boots, kernel talks in structured binary frames, you can see what it's doing.

# Why this shape
The project's identity is observability. If the first thing built is observability infrastructure, every subsequent milestone gets it for free — no retrofit. If built last, the rest of the project is spent regretting it.

This also produces the first piece of public content: a screen recording of the kernel booting and the host-side reader printing the span tree live in the terminal. "Hello world" but the world snitches.

# Architectural decisions for v0.1

## Target architecture: RISC-V (rv64)
QEMU's `virt` machine. 64-bit. S-mode entry via OpenSBI — OpenSBI does the M-mode initialization, kernel starts in S-mode with a clean slate.

- Cleanest boot of any modern architecture.
- Simplest privilege model (M / S / U).
- Documentation tone fits the project (academic, not industrial).
- Emulated only on M1 Mac — fine for a kernel, who cares about a few ms of boot time.

aarch64 is a planned later milestone. By the time it gets added, the HAL will be real, and "SnitchOS now runs on the Raspberry Pi" is its own blog post.

## Tracing protocol: postcard over a dedicated virtio-console
- Postcard (serde-based, no-std friendly, tiny, schema-evolution friendly).
- Two channels: one for normal serial (panic messages, debug shell later), one dedicated to binary telemetry.
- The kernel doesn't speak OTLP. The host-side reader decodes postcard frames.
- Lossy by design under pressure: if the channel backs up, drop frames rather than block the kernel.

## Host-side reader: a small Rust binary
- Reads postcard frames from the QEMU-exposed serial socket (Unix domain socket via `-chardev socket`).
- Decodes and pretty-prints spans as a live text tree to stdout.
- That's the whole tool for v0.1.
- No collector daemon, no docker-compose, no Grafana, no Tempo, no Prometheus. All of that arrives in v0.2.

The protocol shape on the wire is the expensive thing to get right; the consumer is cheap to evolve. Getting the protocol correct now means swapping the stdout printer for an OTLP-emitting collector later is a small, contained change.

## Starting point: from scratch, with a backstop
Write the linker script, boot stub, and entry code by hand the first time. Reference existing crates (`riscv-rt`, xv6-riscv, Phil Oppermann's blog) when stuck.

**Hard time budget on the from-scratch attempt:** one weekend on the bootstrap. If still fighting the linker or CSR setup after that, switch to `riscv-rt` for the lowest level and circle back to the from-scratch version as a separate later exercise. The point of v0.1 is to prove the observability story, not to prove a linker script can be written.

## Workspace layout: monorepo, cargo workspace, multiple crates
Single git repo. Cargo workspace with crates:

- `kernel` — the kernel itself, no_std, RISC-V target
- `protocol` — shared types for the telemetry wire format (no_std-compatible)
- `host-reader` — the host-side stdout pretty-printer (std)
- `xtask` — orchestration: build kernel, launch QEMU with sockets wired up, spawn host-reader

Userspace crates and other tooling get added to this same workspace as the project grows.

## Observability API shape: minimal, but the shape that scales
For v0.1, the kernel-side API is minimal — just enough to emit boot spans and heartbeats. Hardcoded span names are fine. No attributes are fine. No context propagation across context switches because there are no context switches yet.

But the *protocol on the wire* is designed for v1. The wire format includes:

- Span IDs (so spans can have children even if v0.1 doesn't use this much)
- Parent span IDs (ditto)
- Timestamps (monotonic counter; wall clock is meaningless in a freestanding kernel)
- Span name as an interned string ID (with a separate "string registration" frame type) — premature for v0.1 but right for v1
- An open spot for attributes (empty in v0.1)

The wire format is the contract. The kernel-side ergonomics can grow into it.

## What the kernel actually does
1. Boot phase spans: `kernel.boot` with nested children for `serial_init`, `telemetry_init`, etc.
2. Once boot completes, enter a heartbeat loop. Every N cycles, emit a `kernel.heartbeat` span with an incrementing counter.
3. Forever. (No scheduler, no interrupts — just a busy loop.)

The first piece of public content is a screen recording of the boot waterfall arriving live in the terminal, then heartbeats ticking forever.

# Explicit non-deliverables for v0.1
- No userspace.
- No memory allocator (use static buffers).
- No interrupts (busy loop only).
- No threading.
- No capabilities.
- No syscalls.
- No filesystem.
- No real workload — the heartbeat is a placeholder for "something is happening."
- No Grafana, Tempo, Prometheus, Loki, docker-compose.
- No collector daemon. The host reader is the whole consumer side.

# Success criteria
Running `cargo xtask up` does the following, end to end:

1. Builds kernel.
2. Starts QEMU with the telemetry channel wired to a Unix socket.
3. Spawns the host-reader connected to that socket.
4. Terminal shows boot phase spans arriving as a tree, then heartbeats ticking forever.

A screen recording of this is the v0.1 artifact. If it can be put in a blog post, v0.1 is done.

# Time budget
Effort-bounded, not calendar-bounded. Rough budget:

- Bootstrap (linker, entry, serial): 1 weekend, hard cap before switching to `riscv-rt`.
- Protocol + host-reader: 1 weekend.
- Wiring + integration test + xtask: 1 weekend.
- Polish + screen recording + blog post draft: 1 weekend.

Total: ~4 weekends of focused work. Scope cuts are acceptable. If the protocol design takes longer than expected, drop the kernel integration test for v0.1 and re-add in v0.2.

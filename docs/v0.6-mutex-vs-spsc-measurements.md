## Method

Replicable via the `measure` xtask:

```
cargo xtask measure --workload smp --seconds 30 --warmup 6 [--markdown]
```

It boots the `itest-workloads` kernel with `-append workload=<name>`,
connects as a telemetry client, captures the workload metrics
(`snitchos.workload.{samples_produced_total, samples_consumed_total,
lock_wait_ticks_total, queue_depth}`) for the window, skips `--warmup`
seconds of boot transient, and prints steady-state stats (the
stats math is pure + host-tested in `xtask/src/measure.rs`). Timebase is
**10 MHz** (QEMU `virt`, DTB `timebase-frequency`; 1 tick = 100 ns) вЂ”
override with `--timebase-hz`. Ratios (lock-wait fraction, ns/sample)
are timebase-independent; throughput uses the 10 MHz conversion.

Workload: producer on hart 0, consumer on hart 1, `Mutex<VecDeque<u64>>`
queue, batch = 64 samples.

## What the numbers say

**The workload is scheduling-cadence-bound, not lock-bound.** Each hart
wakes on the ~1 s timer IRQ, runs exactly **one** batch, then the idle
task `wfi`s and the hart sleeps until the next timer tick. So:

- Throughput is capped at ~64 samples/heartbeat by the idle-`wfi` sleep,
  **not** by the queue.
- The producer (hart 0) and consumer (hart 1) are awake in tiny,
  rarely-overlapping windows, so they almost never hold/contend the lock
  at the same time в†’ lock-wait is ~0.001 % of wall time.
- `queue_depth` pinned at 64 confirms it: produce-64, consume-64, once
  per second, in lockstep. No backpressure, no oscillation.

## Implication for the SPSC swap (v0.6 step 12)

**Swapping `Mutex<VecDeque>` в†’ `heapless::spsc::Queue` now would move
nothing** вЂ” there is no lock contention to remove (0.001 %). The v0.6
post arc ("here's what the chokepoint cost me" в†’ "here's what removing
it bought") is **not demonstrable at this workload shape**.

To make the comparison meaningful, the workload's run loop must actually
**stress the lock**: the producer and consumer need to run a *burst* of
many batches per scheduling quantum (instead of one-batch-then-yield),
so that during a burst the two harts are concurrently hammering the
`QUEUE` lock cacheline. Only then will:

- `lock_wait` rise to a measurable fraction of wall time (the Mutex
  "cost"), and
- the SPSC swap show that fraction collapse (the "what it bought").

## Mutex under contention вЂ” the `burst=N` sweep

To make the lock contend, the producer/consumer run `burst=N` batches
per `yield_now` (instead of one-batch-then-yield), set via the `burst=N`
kernel bootarg вЂ” so each hart stays in its loop long enough to overlap
the other on `QUEUE`. The queue is bounded at `QUEUE_CAP=4096` (matching
the planned SPSC ring), so a full queue backpressures the producer
rather than growing memory. `cargo xtask measure --workload smp
--burst N` sweeps it with no rebuild:

| burst | throughput | **lock-wait fraction** | queue depth (min/max/avg) | note |
|---|---|---|---|---|
| 1 | 64 /s | 0.0007 % | 64 / 64 / 64 | cadence-bound (baseline above) |
| 256 | 4,079 /s | 0.042 % | 4096 / 4096 / 4096 | queue saturates, still timer-gated |
| 4,096 | 72,701 /s | 1.94 % | 0 / 960 / 285 | harts start overlapping |
| 65,536 | 1.22 M/s | **29.4 %** | 0 / 640 / 155 | continuous bursts, heartbeat stretches to ~3 s |

As `burst` grows, each burst's wall-clock time approaches then exceeds
the ~1 s timer period вЂ” so the harts stop sleeping in idle-`wfi` between
bursts and run their loops continuously, overlapping heavily. At
`burst=65536` the two tasks spend **~29 %** of wall time waiting on the
`QUEUE` lock. **That** is the chokepoint v0.6 is about.

## Next: the SPSC swap (v0.6 step 12)

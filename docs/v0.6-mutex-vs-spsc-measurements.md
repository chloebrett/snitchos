# 📏 v0.6 — producer/consumer queue: Mutex vs SPSC measurements

*Grounding the "chokepoint" claim in numbers before swapping the queue.
The headline finding: at the current workload shape the `Mutex` is
**not** the bottleneck, so the Mutex→SPSC comparison isn't meaningful
yet — the workload's run loop has to change first.*

## Method

Replicable via the `measure` xtask:

```
cargo xtask measure --workload smp --seconds 30 --warmup 6 [--markdown]
```

It boots the `itest-workloads` kernel with `-append workload=<name>`,
connects as a telemetry client, captures the workload metrics
(`snitchos.workload.{samples_produced_total, samples_consumed_total,
lock_wait_ticks_total, queue_depth}`) for the window, skips `--warmup`
seconds of boot transient, and prints steady-state stats (the
stats math is pure + host-tested in `xtask/src/measure.rs`). Timebase is
**10 MHz** (QEMU `virt`, DTB `timebase-frequency`; 1 tick = 100 ns) —
override with `--timebase-hz`. Ratios (lock-wait fraction, ns/sample)
are timebase-independent; throughput uses the 10 MHz conversion.

Workload: producer on hart 0, consumer on hart 1, `Mutex<VecDeque<u64>>`
queue, batch = 64 samples.

## Mutex baseline (`workload=smp`)

| metric | value |
|---|---|
| heartbeat period | **1004 ms** |
| throughput (consumed) | **64 samples/s** (≈ one 64-batch per heartbeat) |
| throughput (produced) | ~61 samples/s |
| `lock_wait_ticks` Δ | **2130 ticks = 213 µs** over 22 s |
| lock-wait fraction of wall time | **0.0010 %** |
| lock-wait per sample | 1.5 ticks = ~150 ns |
| queue_depth (min/max/avg) | **64 / 64 / 64** (flat) |

## What the numbers say

**The workload is scheduling-cadence-bound, not lock-bound.** Each hart
wakes on the ~1 s timer IRQ, runs exactly **one** batch, then the idle
task `wfi`s and the hart sleeps until the next timer tick. So:

- Throughput is capped at ~64 samples/heartbeat by the idle-`wfi` sleep,
  **not** by the queue.
- The producer (hart 0) and consumer (hart 1) are awake in tiny,
  rarely-overlapping windows, so they almost never hold/contend the lock
  at the same time → lock-wait is ~0.001 % of wall time.
- `queue_depth` pinned at 64 confirms it: produce-64, consume-64, once
  per second, in lockstep. No backpressure, no oscillation.

```
timer IRQ        timer IRQ        timer IRQ
  │  ~1 batch       │  ~1 batch       │
  ▼  then wfi       ▼  then wfi       ▼
  ├───────sleep─────┼───────sleep─────┤   hart 0 (producer)
  ├───────sleep─────┼───────sleep─────┤   hart 1 (consumer)
       ▲ overlap window is a sliver → lock barely contended
```

## Implication for the SPSC swap (v0.6 step 12)

**Swapping `Mutex<VecDeque>` → `heapless::spsc::Queue` now would move
nothing** — there is no lock contention to remove (0.001 %). The v0.6
post arc ("here's what the chokepoint cost me" → "here's what removing
it bought") is **not demonstrable at this workload shape**.

To make the comparison meaningful, the workload's run loop must actually
**stress the lock**: the producer and consumer need to run a *burst* of
many batches per scheduling quantum (instead of one-batch-then-yield),
so that during a burst the two harts are concurrently hammering the
`QUEUE` lock cacheline. Only then will:

- `lock_wait` rise to a measurable fraction of wall time (the Mutex
  "cost"), and
- the SPSC swap show that fraction collapse (the "what it bought").

## Mutex under contention — the `burst=N` sweep

To make the lock contend, the producer/consumer run `burst=N` batches
per `yield_now` (instead of one-batch-then-yield), set via the `burst=N`
kernel bootarg — so each hart stays in its loop long enough to overlap
the other on `QUEUE`. The queue is bounded at `QUEUE_CAP=4096` (matching
the planned SPSC ring), so a full queue backpressures the producer
rather than growing memory. `cargo xtask measure --workload smp
--burst N` sweeps it with no rebuild:

| burst | throughput | **lock-wait fraction** | queue depth (min/max/avg) | note |
|---|---|---|---|---|
| 1 | 64 /s | 0.0007 % | 64 / 64 / 64 | cadence-bound (baseline above) |
| 256 | 4,079 /s | 0.042 % | 4096 / 4096 / 4096 | queue saturates, still timer-gated |
| 4,096 | 72,701 /s | 1.94 % | 0 / 960 / 285 | harts start overlapping |
| 65,536 | 1.22 M/s | **29.4 %** | 0 / 640 / 155 | continuous bursts, heartbeat stretches to ~3 s |

As `burst` grows, each burst's wall-clock time approaches then exceeds
the ~1 s timer period — so the harts stop sleeping in idle-`wfi` between
bursts and run their loops continuously, overlapping heavily. At
`burst=65536` the two tasks spend **~29 %** of wall time waiting on the
`QUEUE` lock. **That** is the chokepoint v0.6 is about.

### Canonical contended baseline (`workload=smp burst=65536`, 30 s)

| metric | value |
|---|---|
| throughput (consumed) | **1,218,164 samples/s** |
| lock-wait fraction of wall time | **29.35 %** |
| lock-wait per sample | 241 ns |
| queue_depth (min/max/avg) | 0 / 640 / 155.4 |
| heartbeat period | 3098 ms |

## The SPSC swap (v0.6 step 12) — and the twist

Three variants, same producer/consumer/histogram and the same `burst=N`,
differing only in the queue:

| `workload=` | queue | cross-hart fence | A/B with `cargo xtask measure --burst 65536` |
|---|---|---|---|
| `smp` | `Mutex<VecDeque>` | once **per 64-batch** (the lock) | the contended baseline |
| `smp-spsc` | `heapless::spsc` | **per item** (Release store of `tail` each `enqueue`) | the naive lock-free swap |
| `smp-spsc-batch` | `kernel_core::batch_ring` | once **per 64-batch** (batched publish), lock-free | the controlled fix |

### Result at burst=65536 (3 runs each, 15 s)

| variant | throughput (samples/s) | lock-wait | spread |
|---|---|---|---|
| `smp` (Mutex) | 1.92M / 1.29M / 1.30M | ~30 % | **~48 %** (erratic) |
| `smp-spsc` (per-item) | 0.91M / 0.86M / 0.88M† | **0 %** | tight, slowest |
| `smp-spsc-batch` (per-batch) | 1.54M / 1.52M / 1.56M | **0 %** | **~2 %** (steady) |

† `smp-spsc`'s heartbeat period stretches to ~4 s (its per-item bursts
are the slowest, so heartbeats are sparse); the 3rd run's 0.88M is a
re-measure — the original landed <2 samples inside the 15 s window and
`summarize` reported "not enough samples." Corroborated by the 25 s run
(0.80M) and an 18 s run (0.88M) elsewhere in this doc's sweep. Use a
longer `--seconds` for the slow variants.

### What it says

1. **Naive SPSC is *slower* than the Mutex.** Lock-free removed the
   lock-wait (29 % → 0 %) but throughput *dropped* (~1.3M → ~0.88M).
   `heapless::spsc` fences (Release on `tail`) **per item**; the Mutex
   amortizes one cross-hart sync (the lock) over a **64-item batch**.
   So naive SPSC pays 64× the cross-hart fence traffic — and under QEMU
   TCG, cross-vCPU fences are expensive. (Confirmed: `heapless::spsc`
   uses only atomic load/store + Acquire/Release — **no `compare_exchange`**,
   so it's *not* RISC-V LR/SC / QEMU's exclusive-monitor; it's the
   fences.)
2. **Batching the fence recovers it.** `smp-spsc-batch` writes the 64
   slots `Relaxed` and does **one** `Release` store of `tail` per batch
   — same sync granularity as the Mutex, but lock-free. Throughput
   jumps back to ~1.54M at 0 % lock-wait.
3. **The Mutex wasn't actually faster — it was *erratic*.** Its
   throughput swings 1.29M–1.92M run to run (~48 % spread); the lock
   *couples* the two harts' timing, so throughput is hostage to per-boot
   phase luck. Its *median* (~1.3M) is **below** the batched ring's
   rock-steady ~1.54M (~2 % spread). The genuine lock-free win here is
   **predictability**, not peak — the tail-latency argument, made visible.

**Caveat:** measured under QEMU TCG `thread=multi`, where cross-vCPU
fence cost may not match real silicon — so the *magnitude* of the
per-item penalty is emulation-flavoured, but the *ranking* (per-item
worst; batched ≈ lock amortization; lock-free = predictable) is a
property of how often you pay cross-hart synchronization.

The lesson isn't "lock vs lock-free" — it's **how often you pay the
cross-hart sync**. The Mutex batched it by accident (one lock per
batch); naive SPSC didn't; the batched ring does it deliberately *and*
without the wait.

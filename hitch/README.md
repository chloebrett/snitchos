# Hitch

**The self-describing algebraic value model for SnitchOS** — the lingua franca
data takes when it crosses a boundary: an IPC endpoint, the telemetry channel, a
file's metadata, or the cross-process `~>` pipe.

A serialized value is **a hitch**. The verbs are **`hitch`** (serialize) and
**`unhitch`** (deserialize). The name is triple-loaded, and each meaning is load-
bearing:

- it **hitches a ride** on every channel — it's the payload, not the transport;
- a *hitch* is a **knot/fastening** — it ties data to a wire (the sewing theme of
  its sibling, *Stitch*, the language);
- "**hitched**" = *married* — a self-describing hitch is married to its schema:
  the shape rides inline, inseparable from the data.

Design docs: [`docs/typed-processes-and-the-data-model-design.md`](../docs/typed-processes-and-the-data-model-design.md)
(supersedes §7 of [`docs/userland-text-streams-and-the-actor-model-design.md`](../docs/userland-text-streams-and-the-actor-model-design.md)).

## The model

One algebraic shape, a superset of everything SnitchOS already moves around:

```
Value ::= Scalar(Bool | I64 | U64 | F64 | Str | Bytes)
        | Seq([Value])
        | Product { type, fields: [(name?, Value)] }   # structs / records / `prod`
        | Sum     { type, variant, payload: Value }     # enums / variants / `sum`
```

**`Value` is 64-bit; the schema carries the width.** A `Value` integer is always
`I64`/`U64` (the dynamic carrier, like Stitch's `Int`) — but a `TypeSchema` has the
full fixed-width set (`I8`/`I16`/`I32`/`I64`, `U8`…`U64`, `F32`/`F64`). The packed
codec narrows on the way out and widens on the way in, so a `U64(7)` packs to 1/2/4/8
bytes against a `U8`/`U16`/`U32`/`U64` schema. That width is what lets a packed value
byte-match a real C-ABI struct (`CapDesc`'s `u32` fields), and it keeps the
runtime/value side simple (and the Stitch bridge trivial: `Int → I64`, width is the
schema's business). Conformance range-checks the value against the schema's width.

`protocol::Frame` is a `Sum`; a Rust `struct` is a `Product`; Stitch's
`DataValue { type_name, variant, fields: [(Option<String>, Value)] }` is already
this shape. One model; **two encodings with different jobs and different
mechanisms:**

- **self-describing** (`hitch`/`unhitch`) — postcard the `Value` itself (it carries
  names) → any consumer can `unhitch` it into a named record with no prior schema.
  *Married to its schema.* The dynamic / cross-process / renderer path. Bigger.
- **packed** (`hitch_packed`/`unhitch_packed`) — positional, **fixed-width
  little-endian** bytes against a *known* schema (the schema lives once, in the
  program's interface manifest). Byte-identical to the `repr(C)` in-memory image of
  the equivalent Rust type, so a no-padding **POD** struct can be *transmuted*, not
  serialized — the syscall-return / same-machine path. Small; needs the schema.

Self-describing rides postcard (convenience, never transmuted); packed is
hand-rolled fixed-width C-ABI (transmutable, predictable size, language-neutral — a
C or asm consumer reads a struct, where nobody outside Rust speaks postcard). See
`docs/typed-processes-and-the-data-model-design.md` §2.

## Why Hitch is its own thing

There are many serialization formats. Hitch is not a general-purpose competitor to
any of them — it is the interchange for **one** unusual OS, and three properties
that come together nowhere else justify it:

1. **Schema derived from native types, stored in the program, not a separate
   IDL.** A program's input/output types and its required capabilities are its
   *interface manifest*, generated by `#[derive(Schema)]` + the `#[entry]` macro
   into an ELF note, surfaced as the `user.iface` xattr. The schema lives with the
   executable, so a generic consumer reads it to decode packed payloads — no
   `.proto`, no codegen step, no schema registry.
2. **Capability- and observability-native.** Hitch is the value layer of a
   *capability microkernel* where "a process is a function with its signature in
   the filesystem." The same model carries telemetry frames, IPC messages, and a
   process's typed I/O — and capability handles / span context are first-class (or
   reserved to become so). The format and the authority/trace model are one
   substrate.
3. **One model, two encodings, picked by job.** Self-describing (postcard) for
   generic/dynamic consumers (the table renderer, `frames`, a `.st` peer); packed
   (fixed-width C-ABI) for statically-paired stages and syscall returns, where a
   no-padding POD type transmutes with zero copy and a caller can size its buffer
   by element count.

## Similar to / different from

| | Similar to Hitch | Different |
|---|---|---|
| **serde** | A Rust data model + a derive; Hitch's `#[derive(Schema)]` is serde_derive-shaped, and Hitch *builds on* serde + postcard. | serde is the *machinery* of (de)serialization; it carries **no runtime schema** and erases names in compact formats. Hitch makes the **schema first-class data** (a `TypeSchema` you can read, ship, and store in an xattr) and defines a concrete self-describing value. |
| **Cap'n Proto** | The closest relative: a schema-based canonical format from the **object-capability** lineage (promise pipelining). Same spirit — caps + typed messages. | Cap'n Proto is zero-copy/arena with its **own schema language + codegen**. Hitch derives schemas from the *native* Rust/Stitch types, targets `no_std`-tiny, and its capabilities are **kernel caps + the `uses` row**, not interface-embedded caps. |
| **Protobuf** | Schema-driven; the *packed* encoding is positional/compact like proto; schema kept apart from data. | proto needs a separate **`.proto` IDL + codegen toolchain** and isn't self-describing on the wire (field numbers). Hitch has **no IDL** (schema = the program's own types) and a self-describing mode too. |
| **Avro** | "Schema travels with the data" — Hitch's packed-payload + schema-in-the-manifest is Avro-shaped. | Avro ships the schema in the **data file header**. Hitch puts it in the **executable's interface manifest** (the typed process signature), binding the format to the process/capability model. |
| **CBOR / MessagePack** | Self-describing binary; an algebraic-ish scalar/seq/map model — Hitch's self-describing mode is CBOR-like in spirit. | CBOR has no first-class **named products/sums with a declared schema**, no capability/telemetry scalar kinds, and no "schema in the program" story. Hitch is narrower and OS-specific (its self-describing mode rides postcard, not CBOR; packed is fixed-width C-ABI). |
| **WASM component model / WIT** | A component declares typed imports/exports as an **interface in a file** — Hitch's typed-process manifest is WIT-shaped, and SnitchOS has a WASM direction it converges with. | WIT is an IDL + ABI for WASM components. Hitch is the **value model under SnitchOS's native processes** (Stitch programs *and* Rust ELFs) and adds **capabilities** to the interface (the `uses` row) — the thing WIT lacks. |
| **JSON** | Self-describing; the mental model of "values carry their shape." | JSON loses type distinctions (no sums, no bytes, int/float ambiguity), is text/bloaty, and schema-less. Hitch is typed, binary, and schema-carrying. |

In one line: **Hitch is what you get when the interchange format is designed for a
capability-secure, observability-first OS where typed pipes, cap-mediated
processes, and self-describing telemetry are the same substrate** — schemas come
from the code, live in the filesystem, and authority rides along.

## Where it sits in the workspace

Hitch is a **leaf**: `no_std` + `alloc`, host-buildable, depending only on
`serde` + `postcard`. Everything depends on Hitch, never the reverse — so the
interpreter never leaks into `protocol`/`kernel`.

```
hitch          (this crate — leaf)
hitch-derive   (proc-macro; `#[derive(Schema)]`, re-exported via a `derive` feature)

stitch    → hitch          DataValue ↔ Value bridge lives in `stitch` (orphan rule)
user/*    → hitch (+derive) `#[entry]` hitches the manifest; programs hitch/unhitch I/O
collector → hitch          decode hitches for rendering
```

`protocol` and `kernel` deliberately **do not** depend on Hitch (yet): `Frame`
stays its own postcard enum; a `Frame ↔ Value` bridge is a host-side convenience
for when a consumer wants it, not a kernel dependency.

# v0.10 — Minimal RAMfs behind a stable `Filesystem` trait

**Work lands on:** `main` (no feature branches — see CLAUDE.md)
**Status:** Design complete; cap-agnostic core already built. Sits on the
v0.9c capability IPC substrate (badged endpoints, `MintBadged`,
badge-on-receive, `reply_with_cap`).

A RAM-backed filesystem running as a **userspace** component, reached only
through capabilities, where every operation is a span. The headline:
**"a filesystem in userspace — and the interface that lets it grow."**

Full philosophy + locked decisions predating this plan:
[docs/filesystem-design.md](../docs/filesystem-design.md). This plan is the
v0.10 *implementation cut* of that design.

## Where it stands (already built, host-tested, mutation-clean)

| Piece | Crate | What |
|---|---|---|
| The trait (the deliverable) | `fs-core` | `Filesystem` + `InodeId`/`NodeKind`/`Stat`/`DirEntry`/`FsError`. No cap/IPC types. |
| First impl | `ramfs` | `RamFs` — flat single root, subdirs `Unsupported`. |
| Front-end protocol | `fs-proto` | `FileRights` (READ/WRITE, EXEC reserved), `Badge` (Q4 packing), `Op` (one per method, stable discriminants). |
| Cap IPC substrate | kernel / `user/runtime` | v0.9c: badges, `MintBadged`, badge in `a6` on both receive paths, `reply_with_cap`, one-shot reply caps. |

**The one mechanism still missing:** the option-D cross-address-space copy
(Step 3). Everything else is wiring.

## Scope decision: flat RAMfs, single endpoint, option-D payloads

The v0.10 *core* is the smallest honest demonstration of the headline.
Everything richer is a numbered follow-on, each its own plan.

| in the v0.10 core | deferred (own follow-on plan) |
|---|---|
| Flat single-root FS (`RamFs`), READ/WRITE file rights | Hierarchical dirs + dir rights (`LOOKUP`/`LIST`/`CREATE`/`REMOVE`); subtree-scoped caps (container isolation) |
| One FS endpoint, **badge → (inode, rights)** demux | Multiple FS instances / mount points |
| `stat`/`lookup`/`create`/`remove`/`read`/`write`/`readdir` over IPC | `EXEC` enforcement (~v0.11, loading executables) |
| Payload via **option D** (kernel copies `(ptr,len)` across AS) | **B** (shared `MemoryRegion`, zero-copy bulk data plane) — only if measured |
| `lookup`/`create` mint a child File cap (`reply_with_cap`) | Client cap re-delegation; FS-side revocation (`Stale`) |
| Persistence: **none** (RAM only) | A real block device; CoW, snapshots, content-addressing, Merkle (post-v1.0) |

## Decisions locked in (from the design doc)

| decision | choice | ref |
|---|---|---|
| Trait surface | `fs-core::Filesystem`, inode-addressed, no `open`/`close` | built |
| First impl scope | flat single root; `create(_, _, Dir)` → `Unsupported` | Q3 |
| File rights | `READ`/`WRITE` only; `EXEC` reserved (bit exists, unenforced) | Q1/Q2 |
| Badge packing | `inode:u32 \| rights:u16 \| spare:u16` in `u64` | Q4, `fs-proto::Badge` |
| Message framing | **option D** — cross-AS copy (message passing, not shared mem) | Q5 |
| File cap | badged endpoint cap to the FS endpoint, `badge = pack(inode, rights)` | design §File caps |
| `lookup`/`create` | the cap-minting ops: mint child `badge=(child, parent∩requested)`, transfer via `reply_with_cap` (authority = possession of the reply cap) | design §Connecting layer |
| Names under D | passed as `(ptr,len)`, kernel-copied → **no inline length limit**; `NameTooLong` is an FS policy bound, not a wire bound | design §Message framing |

## Steps

Every step is RED-GREEN-MUTATE-KILL MUTANTS-REFACTOR. No production code
without a failing test. *Present acceptance criteria and confirm before
coding each step.* Natural PR boundaries marked. Bare-metal (`user/fs`,
kernel) layers have no host `#[test]`s — they are pinned by **itest**
scenarios; pure logic lives host-side (`fs-proto`, `kernel-core`).

### Step 1: `fs-proto` request/response codec (scalar fields) ✅ DONE

**Acceptance criteria**: `fs-proto` gains `Request`/`Response` framing that
encodes an `Op` + its **scalar** args (offset, len, kind, and on the
response side a status code + `Stat` fields) into/out of `[u64; MSG_WORDS]`.
Pointer/length refs for name & data are *reserved word slots* defined now,
exercised at Step 3/4. Pure, host-tested, no kernel/IPC types.
**RED**: round-trip every `Op`'s scalar request; decode of a malformed
opcode → error; response round-trip for `Stat` (kind+size) and for the
error path (`FsError` → status code → `FsError`).
**GREEN**: the encode/decode match arms.
**MUTATE**: `cargo mutants -p fs-proto` — verify field-position and
status-mapping mutants have killers.
**KILL MUTANTS**: add cases for survivors (document genuine equivalents).
**REFACTOR**: only if it adds value.
**Done when**: host tests + mutation report green, human approves commit.

> **PR boundary** — pure codec, independently mergeable.

### Step 2: `user/fs` serving `Stat` end-to-end (no D yet) ✅ DONE

**Acceptance criteria**: a new `user/fs` binary holds `RECV | MINT` on one
endpoint; a client holds a **root File cap** (`badge = pack(root, READ)`);
the client `call`s `Stat`; the server unpacks the badge → `RamFs::stat` →
replies kind+size. `Stat` carries no payload, so this runs on plain v0.9c
IPC. Proves the whole front-end: cap wiring, badge demux, trait
integration, reply. An itest asserts the client observes the root as an
empty `Dir`.
**RED**: itest scenario `fs-stat-root` (`workload=fs`): boot FS + client,
assert the decoded wire shows a successful `Stat` reply describing the root
dir (and a `SyscallRefused`/`Denied` when a `WRITE`-attempt rides a
`READ`-only badge — the rights gate).
**GREEN**: `user/fs` receive loop (`reply_recv` → unpack badge → match
`Op::Stat` → `fs.stat` → reply); client bin; spawn/cap wiring in init.
**MUTATE**: n/a (bare-metal); covered by the itest + `fs-proto`/`ramfs`
host tests.
**KILL MUTANTS**: n/a.
**REFACTOR**: assess the loop shape (it is the template the rest extends).
**Done when**: `fs-stat-root` green under `--repeat 10`, human approves.

> **PR boundary(s)** — likely two: (a) `user/fs` + client skeleton; (b)
> cap/spawn wiring + the itest. Resolve the **endpoint-bootstrap question**
> here (see Open questions).

### Step 3: the option-D cross-address-space copy primitive

**Design (decided):** a **reply-cap-authorized cross-AS copy** — a general RPC
bulk-transfer primitive. Two server-initiated syscalls,
`CopyFromCaller { reply, caller_src_va, len, my_dst_va }` (pull, for write) and
`CopyToCaller { reply, my_src_va, len, caller_dst_va }` (push, for read). The
kernel resolves the one-shot reply cap → the blocked caller's `root_pa`, walks
**that** process's page table (the active-AS `SUM` deref only reaches the
current process), and copies through the linear map (`pa_to_kernel_va`). The
reply cap *is* the authority (borrowed, not consumed); the parked caller can't
mutate mid-copy; the kernel stays object-ignorant (opaque bytes). Full mechanism
+ rationale: [docs/ipc-design.md](../docs/ipc-design.md) → *Bulk transfer*.
*Teach-first — novel systems code (two page tables, user-pointer validation on
both sides).* Two sub-steps / PRs:

**Step 3a — pure kernel-core walk + chunking (host-tested).**
**Acceptance criteria**: `kernel-core` gains `translate(root_pa, va, &dyn PtMem)
-> Option<Leaf{ pa, perms }>` (walk to the 4 KiB leaf) and a `copy_across`
orchestrator that, given two translators + a byte-copy callback, copies `len`
bytes page-by-page, advancing by `min(src-to-page-end, dst-to-page-end,
remaining)`; refuses on `ptr+len` overflow, out-of-user-range (`user_range_ok`),
unmapped page, or missing `R`/`W`/`U` leaf perm. No kernel/QEMU.
**RED**: host tests — single-page copy; copy spanning two pages with **mismatched
src/dst page offsets** (scattered frames, the chunking proof); unmapped page →
refuse; `ptr+len` overflow → refuse; source-not-`R` / dest-not-`W` → refuse.
**GREEN**: `translate` + `copy_across` over a `PtMem` mock + a mock byte store.
**MUTATE**: `kernel-core` — the `min`/chunk-advance + the boundary comparisons.
**KILL MUTANTS**: address survivors (esp. `<`/`<=` and the `min` arms).
**REFACTOR**: only if valuable.
**Done when**: host tests + mutation report green, human approves.

> **PR boundary** — pure logic, independently mergeable.

**Step 3b — the syscalls + itest.**
**Acceptance criteria**: `abi::Syscall` gains `CopyFromCaller`/`CopyToCaller`
(appended); the kernel resolves the reply cap → caller `root_pa`, runs
`copy_across` with `KernelPtMem` translation + linear-map copy, refuses + snitches
(`SyscallRefused`) on any validation miss; the runtime exposes both. An itest
`fs-copy-smoke` proves a userspace→userspace round-trip and that an
out-of-bounds/unmapped pointer is refused, not faulted.
**RED**: itest `fs-copy-smoke` (a tiny copy workload, or fold into the FS once
Step 4 lands) — assert the bytes arrive + the refusal path snitches.
**GREEN**: the two syscalls threaded through the trap/IPC path + runtime bindings.
**MUTATE**: n/a (bare-metal); covered by 3a host tests + the itest.
**Done when**: itest green under `--repeat 10`, human approves.

> **PR boundary** — the kernel mechanism, independently mergeable; no FS op
> depends on it until Step 4.

### Step 4: full FS ops on D (`lookup`/`create`/`remove`/`read`/`write`/`readdir`)

**Acceptance criteria**: each op works over IPC: names & data ride as
`(ptr,len)` copied by Step 3; `lookup`/`create` mint a child File cap
(`badge=(child_inode, parent_rights ∩ requested)`) and transfer it via
`reply_with_cap`; rights gates enforce READ/WRITE; refusals snitch
`MissingRight` + inode + attempted right. `readdir`'s list-output framing
decided here (indexed `readdir(n)` vs one buffer copy — see Open
questions).
**RED**: one itest per op (or grouped) asserting the decoded behaviour —
e.g. `create` then `lookup` returns a usable child cap; `write` then
`read` round-trips bytes across the AS boundary; a `WRITE` on a
`READ`-only cap is refused.
**GREEN**: the per-op match arms in `user/fs` + the `fs-proto` pointer-field
codec; child-cap mint on `lookup`/`create`.
**MUTATE**: `fs-proto` (the pointer-field codec).
**KILL MUTANTS**: address survivors.
**REFACTOR**: dedup the op-dispatch if a pattern emerges.
**Done when**: per-op itests green under `--repeat 10`, human approves.

> **PR boundary(s)** — group by op family; each independently mergeable on
> top of the running `Stat` server.

### Step 5: client workload + trace coverage + the post

**Acceptance criteria**: a demo program drives a realistic sequence
(`create` → `write` → `lookup` → `read` → `readdir`), each op a span; the
trace view shows the operations nested under the client's calls and
attributed across the process boundary. The post is drafted.
**RED**: itest `fs-workload` asserting the full sequence's wire frames +
span structure.
**GREEN**: the demo program + workload registration.
**MUTATE**: n/a.
**KILL MUTANTS**: n/a.
**REFACTOR**: n/a.
**Done when**: `fs-workload` green; post drafted in `posts/`; human
approves.

> **PR boundary** — workload + itest; post is separate.

## Pre-PR Quality Gate

Before each PR:
1. `cargo mutants` on touched host crates (`fs-proto`, `kernel-core`).
2. `cargo xtask itest --repeat 10` on touched scenarios (IPC/FS changes
   hid flakes at single-run before — see CLAUDE.md commit gate).
3. `cargo xtask clippy` (whole workspace, riscv + host).
4. Trait/protocol stability: no renumbered `Op` discriminants, no reordered
   `protocol::Frame` / `CapObject` variants.

## Open questions

- **Endpoint bootstrap (Step 2).** How does the client get a cap to the FS
  endpoint *and* how does the FS get its `RECV | MINT` cap? v0.9 was
  kernel-brokered at boot (`Process::bootstrap` with a fixed cap pair);
  badge-handout used the same. Decide: extend that bootstrap, or introduce
  a minimal spawn-with-caps. Lean: reuse the v0.9c bootstrap path for the
  core, defer spawn-with-caps.
- **Exact per-op word layout (Step 1).** Finalize which words carry
  `op`/`off`/`len` and which carry the `(ptr,len)` refs D consumes.
- **`readdir` output shape (Step 4).** Indexed `readdir(n)` returning one
  entry (inline, simple, chatty) vs one D copy of a serialized entry buffer
  (one round-trip, needs a serialization format). Lean: indexed first,
  buffer later — additive.
- **`read` destination (Step 4).** The client passes a `(dst_ptr, len)`;
  the FS copies file bytes *into* the client via D (FS→client direction).
  Confirm the copy primitive supports both directions at Step 3.

## Follow-ons (each its own plan, after the core)

- Hierarchical directories + dir rights (`LOOKUP`/`LIST`/`CREATE`/`REMOVE`)
  → subtree-scoped caps → **container isolation** falls out (design doc).
- Option **B** shared-memory data plane — *only if* a workload proves the
  copy is the bottleneck (every copy is already a span, so the traces will
  say so first).
- FS-side revocation (`Stale`), client cap re-delegation.
- Persistence: a real block device, then CoW + snapshots + content-address
  + Merkle (post-v1.0, "the day SnitchOS learned to remember things").

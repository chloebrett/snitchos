# v0.4 higher-half: findings from the failed attempt

A scratch document so a future attempt at higher-half doesn't
rediscover what we already learned. Read this BEFORE starting any
linker change.

## What we tried

Merge step 2 ("higher-half + identity unmap") with step 6 ("early
satp"): change the linker to higher-half ORIGIN, dual-map identity
+ higher-half in `mmu::enable`, move `mmu::enable` to top of
`kmain`, trampoline + unmap as part of the same step.

## What worked and was kept

These survived the stash. Both are net wins regardless of whether
we eventually do higher-half:

- **`FrameSink` monomorphized.** Changed `&mut dyn FrameSink` →
  `&mut S: FrameSink` in `kernel_core::intern::InternTable::{register_or_lookup,
  register_metric}`. Eliminates the `KernelSink` vtable. Required
  for any higher-half attempt; useful even without one.
- **`code-model=medium` in `.cargo/config.toml`.** PC-relative
  addressing for symbols within ±2 GiB of PC. The default
  (`medlow`) assumes symbol addresses fit in sign-extended low
  32 bits, which silently breaks higher-half link. `medium` is
  the correct setting forever; switching back would be a step
  backwards.

## What broke and why

### A. Static absolute fn-pointer references break pre-MMU

After changing the linker to higher-half ORIGIN, every absolute
function-pointer value embedded in the linked binary points at a
higher-half VA. With MMU off, jumping/loading through those values
faults silently.

Three classes of code do this and all need handling:

1. **`dyn Trait` vtables.** Solved by monomorphizing. Mostly under
   our control.
2. **`fmt::Arguments` formatters.** Every `println!("{}", x)`
   constructs a stack-local `Arguments` whose `formatter` field is
   `&<T as Trait>::fmt as fn(...)` — the linker fills this with the
   symbol's absolute VA. **Cannot be eliminated by refactoring**:
   it's how `format_args!` is structured. Must run with higher-half
   mapped before any formatted print.
3. **Function-pointer storage in user code.** Anywhere we write
   `fn() -> T` into a struct or static stores an absolute VA. The
   trap vector setup (`trap_entry as *const () as usize`) is one
   instance.

### B. The DTB walk in `mmu::enable` died mid-iteration

When dual-map was being built, the `for node in dtb.all_nodes()`
loop crashed somewhere between starting iteration and reaching the
satp write. **In isolation, `dtb.all_nodes()` works fine pre-MMU**
(we proved this with a separate probe in `kmain`). What's specific
to `mmu::enable` is unclear — possibly an interaction with the
borrow shape of three concurrent `&raw mut` to `static mut`
page tables. Did not chase to ground.

**Workaround that worked**: hardcode the MMIO region as
`(0x10000000, 2 MiB)` rather than discovering from the DTB. Boot
recovered.

### C. UART output via raw poke is unreliable

`(0x10000000 as *mut u8).write_volatile(byte)` skips the
NS16550A's THRE-ready check. Writing in a tight loop drops
characters — confirmed by dumping a known constant and getting
garbage. Don't trust hex dumps from this pattern.

**For future debugging**: use the `Uart16550` driver
(`Uart16550::new(0x10000000)` then `putchar` in a loop), or insert
a busy spin after each poke.

### D. `&raw const __kernel_start as usize` value confirmed unclear

With `code-model=medium`, the value loaded for an `extern "C"
static` symbol *should* be computed PC-relative (giving the
physical address at runtime, since the code is loaded at
physical). Our debug attempt got bogus hex (see C); we didn't
verify what the value actually is. **Disassemble before assuming.**

### E. Manual QEMU runs are misleading

`qemu-system-riscv64 -machine virt -kernel kernel` without
`-chardev/-device virtconsole` exposes only 0 virtio,mmio slots
configured for console. `virtio_console::init` returns NotFound
and looks like a paging regression. **Match the harness's QEMU
invocation in `xtask/src/itest/harness.rs` exactly** when running
manually, or look at what `cargo xtask test` actually runs.

## What we should have done differently

We picked option 1 from "Step 2 v2" (merge with step 6 / early
satp). It was the "conceptually clean" path. It was not the
"minimum risk" path. The "smaller still" option (2a + 2b split)
was previously declined; in retrospect even smaller increments are
warranted.

## The minimum-step path (next time)

Treat higher-half as **four separate landings**, each
independently verifiable, in this order:

1. **Identity dual-map, linker stays at identity.** Just add a
   higher-half mapping for the kernel image to the boot table
   alongside identity. Kernel still runs at identity VAs; the
   higher-half mapping is unused. Verifies that we can build a
   page table with two entries for the same physical region.

2. **Add `KERNEL_OFFSET` constant + a host test that a higher-half
   VA correctly walks via the dual-mapped table.** Pure
   kernel-core test (no QEMU).

3. **Linker change to higher-half ORIGIN.** With everything from
   (1) already in place, this might *just work* once paging is on.
   Critical landmines:
   - Move `mmu::enable` to top of `kmain` before any formatted
     `println!` or other absolute-fn-pointer code. Test by booting
     and confirming `cargo xtask test`.
   - Prepare for surprises: the `dtb.all_nodes()` problem in
     `mmu::enable`, the trap_vector pointer, anything else that
     stores absolute symbol VAs.
   - **Do NOT do the trampoline or identity unmap in the same
     landing.** Kernel still RUNS at identity addresses; the
     higher-half mapping is just resident so absolute fn pointers
     resolve.

4. **Trampoline + identity unmap.** Kernel now actually executes
   at higher-half VAs. `sp += KERNEL_OFFSET`, absolute jump to
   higher-half symbol, walk root table and zero identity entries.

This is the same path as the rejected 2a/2b split, but extended to
2c and 2d. The point is: never combine "linker layout change" with
"trampoline" in the same landing — that doubles the bug surface.

## Specific gotchas to investigate up-front

When attempting (3) above:

- **What does `(&raw const __kernel_start) as usize` actually
  return** under medium code model with higher-half link? Either
  the physical address (PC-relative, what we want) or the
  higher-half VA (absolute, what we don't). Disassemble before
  assuming.
- **Why does `dtb.all_nodes()` work standalone but crash inside
  `mmu::enable`?** Possibly stack pressure, possibly aliasing of
  `&raw mut BOOT_PT_*`. Worth understanding before relying on it.
- **What's the right place to translate stored physical addresses
  to higher-half VAs?** `CONSOLE` (in `virtio_console`) and `UART`
  (in `console`) hold physical addresses cast to `usize`. After
  identity unmap, these are unreachable.

## Update (after step 2c lands)

Step 2c succeeded — kernel now linked at higher-half, boots via early
satp, runs at identity. Net state: 39 kernel-core tests + 4
integration scenarios passing.

Things we learned this round that the original findings missed or
got wrong:

- **`dtb::timebase_hz`'s for-loop form also crashes pre-MMU.** The
  original findings claimed the for-loop "worked once during
  debugging." It does not. Both forms crash. The actual workaround
  was to **move `timebase_hz` to after `mmu::enable`** — once MMU
  is on (with DTB mapped), it works fine.
- **Anything that walks the DTB pre-MMU crashes.** Confirmed across
  `timebase_hz` AND `collect_mmio_regions`. Suspect: the fdt
  crate's iteration uses some form of indirect call that the medium
  code model doesn't make PC-relative, but we haven't isolated it.
  **Mitigation: do not use the DTB pre-MMU.** Hardcode the MMIO
  region during boot, use the DTB freely afterward. This is what
  step 2c does.
- **`span!` macros pre-MMU also crash** when wrapped around real
  code. An empty span block worked once but the consistent pattern
  is: `span!` pre-MMU is unreliable. Mitigation: don't span around
  pre-MMU code. The `kernel.mmu.enable` span is gone for now; the
  `mmu-higher-half-verify` scenario covers MMU-up assertion.
- **The DTB region MUST be in the boot page table.** If `&dtb` is
  reused after `mmu::enable` and the DTB region isn't mapped, the
  first deref post-MMU faults. Step 2c maps it explicitly.
- **`set_trap_vector` can move to after `mmu::enable`.** The
  earlier worry that pre-MMU traps would deadlock is moot — we
  don't actually take traps during the small DTB-parse + satp-on
  window.

## Update (after step 2d-1 attempt — reverted)

Tried trampoline + sp fix-up. Boots through the trampoline fine
(`X` and `Y` markers printed) but `virtio_console::init` then hangs
silently. The root cause is fundamental, not a bug:

**With PC at higher-half, `&static as usize` is PC-relative →
higher-half VA.** But virtio device drivers pass static addresses to
the device as DMA targets. Devices have no MMU and treat those
values as physical addresses. Higher-half VAs don't resolve in the
physical address space, so DMA fails.

Affected code (every place we pass a static / stack address to
a device):
- `virtio_console::transmit` — `&raw const TX_QUEUE.desc as u64`
  and `bytes.as_ptr() as u64` (the postcard-encoded frame on the
  stack).
- Any future driver that does DMA.

**To make the trampoline work** we'd need to convert every device
address through `KERNEL_OFFSET` subtraction (`va_to_pa`) in the
same step. That's significantly more code change than a "minimum
viable trampoline."

**Decision: stop step 2 at 2c.** The kernel is linked at
higher-half, runs at identity, with a dual-map that supports
absolute-fn-pointer references on the higher-half side. This is
enough for v0.4 step 3 (frame allocator) and beyond. Actually
moving PC + sp to higher-half (with all the DMA-address fix-ups
that requires) is its own milestone, deferred to whenever userspace
preparation forces it.

Things we also learned this round:

- **`verify_higher_half_mapping`'s semantics break post-trampoline.**
  Once PC is at higher-half, `&__kernel_start` resolves to
  higher-half (not physical), and adding `KERNEL_OFFSET` wraps into
  non-canonical Sv39 territory. The verify check is only meaningful
  when PC is at identity. Removed it as part of the step 2d-1
  revert.
- **`stval` + `sepc` in the trap panic message is invaluable.** The
  original panic just said `unhandled trap: ...`; we added stval
  and sepc to surface the bad VA and the faulting PC. Keep this.

## Update (after step 2d-1 second attempt — succeeded)

Pushed through. Kernel now actually runs at higher-half PC. Two
checkpoints, each leaving tests green:

**Checkpoint 1: `va_to_pa` at device-address boundaries.** Added a
`pub const fn va_to_pa(va: usize) -> usize` in `kernel::mmu` that
strips `KERNEL_OFFSET` from higher-half VAs and passes identity
VAs through. Used it at four sites in `virtio_console.rs`:
`setup_queue` (desc, avail, used queue addresses) and `transmit`
(buffer pointer). Pre-trampoline it's a no-op; tests stayed 4/4.

**Checkpoint 2: trampoline + sp fix-up.** Inline `asm!` in `kmain`
right after `mmu::enable`:

```asm
lla  t0, 1f         # PC-relative addr of label 1 = identity VA
add  t0, t0, t1     # t1 = KERNEL_OFFSET; t0 now = higher-half VA
add  sp, sp, t1     # sp from identity to higher-half
jr   t0
1:
```

Removed `verify_higher_half_mapping` and its scenario (semantics
break post-trampoline — `&__kernel_start` resolves to higher-half,
and adding `KERNEL_OFFSET` wraps to non-canonical). Tests now 3/3.

The earlier 2d-1 attempt failed because **DMA addresses weren't
translated**. With `va_to_pa` doing the translation at the
device-driver boundary, `virtio_console::transmit` hands the device
a physical buffer pointer, the device reads from physical memory,
the frame arrives on the wire. **Checkpoint 1 must come before
the trampoline** — landing the trampoline first means DMA breaks
the moment the kernel hits any `&static as u64`.

What's left for the full higher-half story (still deferred):
- **Checkpoint 3: identity unmap.** Zero the identity-half root
  entries + `sfence.vma`. Requires patching `CONSOLE` and `UART`
  statics to hold higher-half VAs (they currently store physical
  addresses, which would stop resolving once identity-MMIO is
  unmapped), and adding higher-half mappings for MMIO regions.
- That's its own session.

Things we got right second-time-around:
- **Found all device-DMA sites before doing the trampoline.**
  `grep -n "as u64\|as_ptr() as" virtio_console.rs` enumerated
  them in one pass.
- **Made `va_to_pa` a no-op-on-identity, so it can land before
  the trampoline.** Cheap to use, doesn't require reasoning about
  "is the kernel at higher-half right now."
- **Kept it inline in `kmain` rather than as a function.** A
  `pub unsafe fn jump_to_higher_half()` would return through `ret`,
  which would jump back to caller's identity-VA `ra` — defeating
  the purpose. Inline asm + label, no return.

## Update (after checkpoint 3 — partial identity unmap)

Step 2d is functionally complete. Kernel now runs at higher-half PC
+ sp with the identity-kernel mapping torn down.

**Checkpoint 3: unmap identity-kernel gigapage.** Added
`PageTable::set_entry(idx, value)` to `kernel_core::mmu` (one new
host test) and `mmu::unmap_identity_kernel()` to `kernel::mmu` —
clears root entry 2 + `sfence.vma`. Called from `kmain` after
`trap::init_timer`. After this returns, any access to a kernel-image
identity VA faults. Tests: 3/3 still pass.

What's NOT unmapped: identity MMIO (root entry 0). `CONSOLE` and
`UART` statics still hold physical addresses; the panic handler's
fallback path still pokes physical `0x10000000`. Tearing those down
needs more setup:
- Add higher-half mappings for MMIO regions (new `BOOT_PT_MID_HIGHER_MMIO`).
- Patch `console::init` / `virtio_console::init` to store higher-half
  VAs.
- Patch `console::QEMU_VIRT_UART_BASE` (or the panic handler's use of it)
  to higher-half.

That's a real piece of work and a separate session. Current
half-state ("kernel image only, identity MMIO kept") proves the
kernel doesn't accidentally use identity for any *code or static*
access while leaving the diagnostic UART path intact.

Final test count for the full step 2: **40 kernel-core (was 38) +
3 integration**. The integration count went from 5 (at step 2a) →
3 (now) because the two "verify"/"mmu-enabled" scenarios depended
on machinery (a span around `mmu::enable`, the verify function
reading via identity then higher-half) that became invalid as the
kernel actually moved to higher-half PC.

## Update (after step 2 full completion — MMIO identity unmap)

Five small increments finished what was left of step 2. Each landed
green:

1. **Add `BOOT_PT_MID_HIGHER_MMIO` + dual-map MMIO** in `mmu::enable`.
   Mapping exists, nothing reads it yet.
2. **`console::init` stores higher-half VA** in `UART`. Macro-emitted
   prints now route through higher-half MMIO.
3. **`virtio_console::init` stores higher-half VA** in `CONSOLE`.
   Frame DMA writes go through higher-half MMIO; buffer pointer still
   goes through `va_to_pa`.
4. **`console::emergency_uart_base()` reads satp** and picks the right
   base. Used by `_pre_init_uart` and the panic handler so they work
   pre-`mmu::enable` (physical), between `enable` and unmap (either),
   and post-unmap (higher-half only).
5. **`mmu::unmap_identity`** (renamed from `unmap_identity_kernel`)
   clears root entries 0 (MMIO) and 2 (kernel image), then
   `sfence.vma`. Kernel runs purely in higher-half.

End-state test count: 42 kernel-core (was 40 — two `va_to_pa` tests
moved to host) + 3 integration.

### Latent landmines (audit findings — not bugs today, worth knowing)

After landing the unmap, audited every place that touches an MMIO
address. Things that work today but would bite if assumptions shift:

- **`init_handshake` runs against physical `base`.**
  `virtio_console::init` does `init_handshake(base)` before adding
  `KERNEL_OFFSET` to store in `CONSOLE`. Today this is fine — identity
  MMIO is mapped when `init_handshake` runs. If anyone moves
  `unmap_identity` earlier than the virtio init phase, `init_handshake`
  faults. Robust fix: hand `init_handshake` the higher-half VA.

- **DTB region unreachability post-unmap is implicit.** Nothing reads
  through `&dtb` after `init_timer`, so we're fine, but the borrow
  checker doesn't enforce it. An explicit `drop(dtb)` before
  `unmap_identity` would make the intent visible.

- **MMIO region is still hardcoded** to `QEMU_VIRT_MMIO_BASE`.
  `collect_mmio_regions` (the would-be DTB-driven version) is parked
  behind `#[expect(dead_code)]`. Porting to another machine means
  editing the constant.

- **`fmt::Arguments` pre-MMU constraint is still latent.** No current
  code triggers it (DTB parse and `mmu::enable` don't use formatted
  println), but if anyone adds a formatted println before `mmu::enable`,
  it crashes silently. Worth defending against in code review.

- **No explicit "kernel runs at higher-half" assertion in tests.**
  The three integration scenarios passing implicitly proves it (the
  trampoline + unmap + DMA-with-`va_to_pa` path is exercised end-to-end),
  but there's no single-line check. A scenario reading `sepc` or
  `auipc` and asserting it's >= `KERNEL_OFFSET` would make it explicit.

## Don't repeat

If you find yourself doing any of these, stop and check if it's
because of one of the above:

- **Adding a marker, removing a marker, adding a marker again.**
  The `(0x10000000 as *mut u8).write_volatile` pattern doesn't
  wait for THRE. Use a proper UART or accept garbled output.
- **Running QEMU manually and panicking that virtio_console fails.**
  Check the harness invocation first.
- **Reasoning about "this MUST be PC-relative under medium" without
  disassembling.** Just look at the generated code.
- **Trying to land higher-half in one step.** It's at least three.

## Edges to close (the "do it right" list)

### 1. Address-space / resource teardown on Exit — IN PROGRESS
`Exit` still **leaks** the child's user address space (page table + all mapped
frames: ELF, heap, user stack), kernel stack (16 KiB `Box<Stack>`), `Box<Task>`,
`Process` (and its `CapTable` `Vec` + metrics `BTreeMap`). A shell spawning many
short-lived commands OOMs without this. **The crux:** `Process` is currently a
local on the *exiting task's own kernel stack* (raw pointer in `CURRENT_PROCESS` /
`Task.process`), so nothing runs its `Drop`. And the user AS can't be freed by the
exiting task itself (it's the active `satp`). → **reclaim on reap**, in the
reaper's context. Design in the next section.

### 2. General notification primitive — TODO
v0.12 delivered child-exit→parent-wake via the existing `block_current`/`wake`
(the narrow case). The *general* kernel→user Notification object (async,
payload-free; what device drivers will reuse for IRQ→userspace) is not built. Its
own small slice. (Roadmap framed notifications as part of v0.12.)

### 3. One waiter per child — TODO
`ReapTable.waiters` is `child → parent` (single slot); a second `Wait` on the same
child silently overwrites the first → the first parent blocks forever. Enforce
(refuse a second waiter) or document the invariant. v0.12 assumes one parent.

### 4. No parent/child relationship enforcement — TODO
Any task may `Wait` on any id; waiting on a non-child or never-exiting id blocks
forever (no `ECHILD`-style error). A real OS scopes `Wait` to the caller's
children. Decide: track the spawner as parent + refuse non-children, or document.

### 5. Scenario naming — TODO
`spawn-delegates-to-child` now also asserts the reap; rename or split out a
`spawn-waits-for-child` scenario so the name matches what it proves.

`roadmap-and-milestones.md` v0.12 entry should reflect what shipped vs the above.

## Teardown design (edge #1) — *to be filled after scoping the Process lifetime*
Reclaim-on-reap: when the parent's `Wait` reaps a child (`WaitStep::Ready`), free
the child's user AS (walk user-half of its root, free leaf frames + intermediate
tables + root — never the shared kernel high-half), drop the `Process` members,
and remove its `Box<Task>` from `sched.tasks` (dropping the 16 KiB kernel stack).
Open question being scoped: own the `Process` in a `Box` on the `Task` (clean
`Drop`) vs `drop_in_place` the stack-resident `Process`. Orphan exits (no waiter)
still leak until a reaper sweep — track if it matters (the shell always `Wait`s).

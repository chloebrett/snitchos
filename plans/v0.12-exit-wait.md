## Edges to close (the "do it right" list)

### 1. Address-space / resource teardown on Exit — SHIPPED ✅
Reclaim-on-reap: when a parent's `Wait` reaps an `Exited` child, the kernel frees
the child's user address space (page table + every mapped frame: ELF, heap, user
stack), drops its `Process` (running `Drop` to release the cap-table `Vec` +
per-process metric `BTreeMap` heap), and removes its `Box<Task>` (freeing the
16 KiB kernel stack). The shared kernel high half (root slots `256..512`) is never
touched. Runs in the *reaper's* address space (the child has already switched away
via `exit_now`), so freeing the child's root `satp` is safe.

Pieces:
- `kernel_core::mmu::free_user_tree(root_pa, mem, free)` — pure walk of user-half
  root slots `0..256`, freeing leaves → L0 → L1 → root; skips huge leaves + the
  kernel half. Host-tested (3 tests: frees-everything, never-follows-kernel-half,
  skips-huge-leaves).
- `kernel::mmu::free_user_root(root_pa)` — binds the walk to `KernelPtMem` +
  `frame::free`; `frame::PhysFrame::from_addr` rebuilds frames from walked PAs.
- `kernel::sched::reap_task(child)` — swap-removes the `Box<Task>` under the
  SCHEDULER lock, drops it before touching the frame allocator (lock discipline),
  `free_user_root` → `drop_in_place(process)` → drop the `Box<Task>`.
- `handle_wait` calls `reap_task(child)` on `WaitStep::Ready`.

Proven by the **`spawn-reclaims-memory`** itest (`workload=spawn-reap`): a
`reaper` spawns + `Wait`s a `memhog` child (~4 MiB each) 30×. Without reclaim,
120 MiB > RAM → OOM stall (RED, confirmed: `frames.free`→0). With reclaim,
`frames.freed_total` → ~31 700 and `reaper.done` fires in <1 s (GREEN). Full
suite 72/72; `--repeat 10` clean.

**Still leaks (documented, not yet closed):** orphan exits — a process that
`Exit`s with no parent `Wait`ing on it is never reaped (its zombie `Box<Task>`
+ address space linger). The shell always `Wait`s, so this doesn't bite v0.12;
a reaper sweep / re-parent-to-init is the follow-on if it ever matters.

### 2. General notification primitive — SHIPPED ✅
The general kernel→user Notification object (async, coalescing, payload-but-one-
mask; what device drivers will reuse for IRQ→userspace) is built and verified.
`kernel_core::notify::{Notification, NotifyTable}` (host-tested, mutation-clean),
`Object::Notification` gated by `SIGNAL`/`WAIT`, syscalls `NotifyCreate`/`Signal`/
`WaitNotify` (21/22/23), `NotifySignal`/`NotifyWait` wire frames, runtime
`Notification` bindings. Proven by the `notify-signal-wakes-waiter` itest
(`workload=notify-smoke`), 0/10 flakes on `--repeat 10`. Full plan +
what-shipped-vs-planned: [v0.12-notifications.md](v0.12-notifications.md); design:
[../docs/notification-design.md](../docs/notification-design.md).

Child-exit was **not** folded onto it (Option A): the existing `block_current`/
`wake` reap path stays, and the notification primitive ships proven by a *fresh*
consumer. Folding child-exit on (Option B) waits for the device-IRQ consumer to
validate the generic shape under real async load.

### 3. One waiter per child — TODO
`ReapTable.waiters` is `child → parent` (single slot); a second `Wait` on the same
child silently overwrites the first → the first parent blocks forever. Enforce
(refuse a second waiter) or document the invariant. v0.12 assumes one parent.

### 4. No parent/child relationship enforcement — TODO
Any task may `Wait` on any id; waiting on a non-child or never-exiting id blocks
forever (no `ECHILD`-style error). A real OS scopes `Wait` to the caller's
children. Decide: track the spawner as parent + refuse non-children, or document.

## Teardown design (edge #1) — RESOLVED (see "SHIPPED" above)
The open question — own the `Process` in a `Box` on the `Task` vs `drop_in_place`
the stack-resident one — resolved to **`drop_in_place`**: `Process` stays a local
in the `run_with_caps`/`spawned_entry` frame (owned by `task._stack`), and
`reap_task` runs its `Drop` in place before freeing the backing stack. No struct
change, no extra `Box`. Orphan exits (no waiter) still leak — tracked above.

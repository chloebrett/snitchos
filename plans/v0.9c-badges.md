# v0.9c — Badged endpoints (the generalized reply cap)

**Work lands on:** `main` (no feature branches — see CLAUDE.md)
**Status:** Design complete, not started. Follows v0.9b (`call`/`reply` +
single-use caps, `plans/v0.9b-call-reply.md`) and builds directly on its
cap-transfer precedent, `Multiplicity`, and the reply-cap minting path.

One endpoint serves many clients and many server objects, each distinguished
by an **unforgeable, server-chosen badge** the kernel delivers to the receiver.
This is the substrate the filesystem (v0.10) needs to hand out per-file
capabilities — but the headline of *this* milestone is the minimal proof:
**"one endpoint, two clients, told apart by capability — not by who they are."**

> **Numbering note:** notifications (the payload-free async primitive) were
> pencilled as v0.9c in earlier plans; they shift to **v0.9d**. Badges jump the
> queue because they unblock v0.10 and retire a design fork, whereas
> notifications are additive and independent.

## Why this milestone, and the framing that makes it small

The naïve read is "import seL4 badges" — a foreign mechanism, a lot of new
machinery. The real read: **you already built a badge in v0.9b and called it
`Object::Reply { caller }`.** A reply cap is a capability the kernel *stamps
with an unforgeable value* (`caller`), *transfers* into a process's table, and
*delivers* to the holder on invoke so it knows which thing this is about. A
general badge is that exact mechanism freed along three axes already present in
the codebase:

| | reply cap (v0.9b, shipped) | general badged cap (this milestone) |
|---|---|---|
| who mints | kernel, at `call` | a **server**, via a syscall |
| lifetime | `Once` | `Once` **or** `Persistent` (the `Multiplicity` axis exists) |
| stamped value | `caller: TaskId` | arbitrary **`badge: u64`** the server chose |

So the work is not "add a subsystem" — it is "let a server do what the kernel
already does for reply caps, with a value it picks." The general thing is
*simpler* than the special case already shipped.

**What it earns beyond the FS.** A badge is "a server-chosen value the kernel
delivers unforgeably." That is the generic **capability-extension** primitive
(the long-deferred "#4"): every future server — net, display, the FS — defines
its own object zoo behind one endpoint, with the kernel attaching meaning to
nothing. One mechanism, all servers. (We stop short of kernel-enforced
narrowing of *object* rights; those live in the badge, server-interpreted —
see "deliberate restraint.")

**The capability-discipline payoff.** Without badges, a single server endpoint
can only tell two clients apart by sender `TaskId` — identifying by *who you
are*, which is ACL thinking and against the whole thesis. A badge is the
capability answer: the server demuxes by *what cap was used*, never by identity.

## Scope decision: badge + mint + badge-on-receive + reply-carries-cap

| in v0.9c | deferred (own follow-on) |
|---|---|
| `badge: u64` on the endpoint cap (badge `0` = the bare owner/RECV cap) | General `send`-carries-caps + a `GRANT` right (v0.9c ships **reply**-carries-cap only) |
| `Rights::MINT` + `mint_badged` (pure derive, then the syscall) | Client-side **re-delegation** (a client minting narrower sub-caps — needs the FS to grant `MINT` onward) |
| Badge delivered to the receiver on `receive` (one endpoint, demux by badge) | Kernel-enforced narrowing of *object* rights (full #4) — file rights stay **in the badge**, FS-enforced |
| Reply-carries-an-arbitrary-cap (generalize the reply transfer) | `MemoryRegion` bulk transfer; the channel library |
| A minimal **badge-demux demo** (one endpoint, two badges) + itest | Notifications (now **v0.9d**); the filesystem itself (**v0.10**) |

**The FS is the first *real* consumer, not this milestone.** v0.9c proves the
mechanism with a tiny `badge-server`/`badge-client` demo. `open`/`lookup`
returning a `badge=(inode,rights)` cap is v0.10, built on exactly this.

## Decisions locked in

| decision | choice |
|---|---|
| Badge on the cap | `Object::Endpoint { id: EndpointId, badge: u64 }`. `badge == 0` is the **bare** endpoint (owner/RECV, as today); a nonzero badge is a derived `SEND` cap. Existing `bootstrap`/`DEMO_ENDPOINT` sites construct `badge: 0` — mechanically additive. |
| Kernel treats badge as **opaque** | The kernel never reads, compares, or interprets a badge beyond *carrying* it (mint → store on the cap; receive → deliver). All meaning is the server's. Mirrors "the kernel never learns what a file is." |
| New right | `Rights::MINT` (next free bit, `0b1000`). Authority to derive badged `SEND` caps naming an endpoint you hold. The FS owner cap is `RECV \| MINT`. |
| Mint rights model | A `MINT`-holder **owns the object** and sets the derived cap's rights freely (it is granting authority to *its* endpoint, not attenuating its own). Monotonic narrowing bites only on re-derivation by non-owners — and a client holds no `MINT`, so it cannot mint at all (re-delegation is deferred). For the FS, client caps are simply `SEND`; the *file* rights ride in the badge, FS-interpreted. |
| Badge delivery | `receive` additionally returns the **badge of the sender's cap** in `a6` (next to v0.9b's reply handle in `a5`). A bare-endpoint sender yields `a6 = 0`. Backward-compatible with v0.9/v0.9b receivers that ignore it. |
| Cap-transfer vehicle | v0.9c generalizes **only the reply path**: a `reply` may carry one capability handle from the replier's table; the kernel moves it into the caller's table and returns a fresh handle, emitting `CapEvent::Transferred` (reusing v0.9b's path). General `send`-carries-caps + `GRANT` is the follow-on. (The FS replies to `open` with a minted badged cap — reply is exactly the right vehicle.) |
| Syscall | Append `MintBadged = 11` to `abi::Syscall` (positional — append only). `a0`=endpoint handle (needs `MINT`), `a1`=`badge:u64` (one register — riscv64 GPRs are 64-bit), `a2`=rights bits, `a3`=multiplicity (0=Persistent,1=Once) → returns the new handle in `a0` (or `u64::MAX` refused). |
| Multiplicity composes | A badged cap may be `Persistent` (a durable file handle) or `Once` (a one-shot write token / single-use claim) — the v0.9b axis applies unchanged, orthogonal to the badge. |
| Revocation | Two reaches, as the FS design states. **FS-side (per-object):** drop the badge→object mapping; later invokes resolve to a server-level not-found. **Kernel-side (whole-cap):** bump the slot generation (v0.9b's consume path) — coarse, kills the cap. |
| Wire | `CapEvent` gains an appended `badge: u64` field (observability — the collector can show "client got a cap to endpoint E with badge X"; it stays opaque, uninterpreted). Append-only; `OwnedFrame` + consumers updated. |
| Telemetry | `snitchos.cap.mints_total`; badged-receive count a stretch. Deferred-emission, drained in the heartbeat, as v0.9. |
| Lock discipline | As v0.9b: compute under the endpoint lock, drop before block/wake; cap insert/mint takes the target's `caps` lock, never held across a switch. |

## Steps

Every step is RED-GREEN-MUTATE-KILL MUTANTS-REFACTOR. No production code without
a failing test. *Present acceptance criteria and confirm before coding each
step.* PR boundaries marked.

### Step 1: `badge` on the endpoint cap (pure)

**Acceptance criteria**: `Object::Endpoint { id, badge: u64 }`; `badge: 0` is
the bare endpoint. `invoke_send`/`invoke_recv` resolve **regardless** of badge
(badge is carried, not gated). All existing construction sites
(`bootstrap`, demo endpoints) set `badge: 0`; behaviour unchanged.
**RED**: a badged and a bare endpoint cap both `invoke_send`/`invoke_recv`
successfully; the resolved `Capability` carries the badge verbatim; badge does
not affect the `SEND`/`RECV` rights check.
**MUTATE**: the badge is carried through resolve (not zeroed/dropped); the
rights check ignores the badge.
> **PR boundary** — additive field, no behaviour depends on it yet.

### Step 2: `Rights::MINT` + `mint_badged` (pure derive)

**Acceptance criteria**: `Rights::MINT` (`0b1000`); a pure
`mint_badged(parent: Capability, badge, rights, mult) -> Result<Capability,
Denied>` that, given a `MINT`-righted `Endpoint` cap, returns an `Endpoint {
id: parent.id, badge }` carrying the requested rights + multiplicity; refuses a
parent lacking `MINT` (`MissingRight`) or naming a non-endpoint (`WrongObject`).
**RED**: a `RECV|MINT` parent yields a `SEND`-badged child for the same `id`;
no `MINT` → `MissingRight`; non-endpoint parent → `WrongObject`; the child's
badge/rights/multiplicity are exactly as requested.
**MUTATE**: the `MINT` check; `id` propagation; rights/badge/mult plumbing.
> **PR boundary** — pure attenuation primitive.

### Step 3: Badge threaded through the rendezvous (pure)

**Acceptance criteria**: `kernel-core::ipc`'s delivered-message record gains a
`badge: u64`; the rendezvous carries the **sender's cap badge** to the
receiver. State-machine tests assert badge transport across `send`→`receive`
and `call`→`receive` (bare sender → `0`).
**RED**: a badged sender's badge surfaces in the delivered record; a bare
sender yields `0`; badge is independent of the message words and reply path.
**MUTATE**: badge capture at send; badge carried to deliver.
> **PR boundary** — pure transport, no syscall surface yet.

### Step 4: `badge` on the `CapEvent` wire frame

**Acceptance criteria**: `Frame::CapEvent` gains an appended `badge: u64`; a
`CapEvent` roundtrips with a nonzero badge; `OwnedFrame::from_borrowed` arm
updated; `collector` + `xtask` harness handle it (badge shown raw, uninterpreted).
**RED**: protocol roundtrip with a badge; exhaustive matches fail to compile
until handled.
> **PR boundary** — wire addition, append-only, emitter follows.

### Step 5: `MintBadged` syscall — a server mints a badged cap

**Acceptance criteria**: `abi::Syscall::MintBadged = 11` + `handle_mint_badged`:
resolve the endpoint cap (needs `MINT`), `mint_badged` (Step 2), insert the
child into the **caller's** `CapTable`, return its handle; snitch
`CapEvent::Transferred` carrying the badge (parent = the minting cap).
A non-`MINT` endpoint handle → `SyscallRefused`. Runtime wrapper
`Endpoint::mint(badge, rights, mult) -> Handle`.
**RED**: under a workload, a process holding a `RECV|MINT` endpoint mints a
badged `SEND` cap and observes a valid handle; a `SEND`-only holder is refused
(`SyscallRefused` on the wire).
**MUTATE**: the `MINT` gate; the insert-into-own-table path; the 0-vs-handle return.
> **PR boundary** — a server can manufacture badged caps (into its own table).

### Step 6: Reply-carries-a-cap — transfer the badged cap to the client

**Acceptance criteria**: generalize v0.9b's reply path so `reply` may carry one
capability handle from the replier's table; the kernel moves it into the blocked
caller's table and returns a fresh handle in the caller's `call` result;
`CapEvent::Transferred` records the cross-process edge. Reuses the reply-cap
insertion mechanism — now with a *server-supplied* cap, not the kernel's
auto-minted reply cap.
**RED**: itest — a server `mint_badged`s a cap and returns it via `reply`; the
client's `call` yields a working handle; a `reply` with no cap still works
(plain v0.9b response).
**GREEN**: thread an optional cap handle through `Reply`; move-on-reply into the
caller's table.
**MUTATE**: the present-vs-absent cap gate; the move/insert; the returned handle.
> **PR boundary** — the keystone: servers can hand capabilities back to clients.

### Step 7: Badge delivered to userspace + the demux headline

**Acceptance criteria**: `receive` returns the sender-cap badge in `a6`; the
runtime threads it (`Received { words, reply, badge }`). Demo: `badge-server`
holds one bare endpoint (`RECV|MINT`); `init`/broker arranges that two clients
each hold a **distinct-badged** `SEND` cap to that one endpoint (via Step 6
transfer or boot grant). The server `receive`s from both in **one loop** and
reads two different badges. `WorkloadKind::BadgeDemo` (bootargs, host-tested) +
spawn layout.
**RED**: itest `badged-endpoint-demux` — the server re-emits each received badge
via telemetry; the wire shows **two distinct badge values arriving on one
endpoint**, demuxed without any reference to sender identity. The assertion must
be impossible to satisfy on a bare (badge-less) endpoint.
**GREEN**: `a6` plumbing; runtime `Received.badge`.
**MUTATE**: badge → `a6`; bare → `0`; the runtime field.
**Done when**: `cargo xtask itest --repeat 10` green (scheduler/trap path).
> **PR boundary** — one endpoint, demux by capability. Shippable v0.9c.

### Step 8: Mint telemetry

`snitchos.cap.mints_total`, deferred-emission, drained in the heartbeat; itest
asserts ≥1.
> **PR boundary** — observability for the new behaviour.

## Open questions

- **Badge register.** `a6` for `receive`'s badge (alongside `a5` reply handle)
  is proposed; confirm no collision and clean runtime threading. Resolve at Step 7.
- **Mint rights model.** Locked as "owner sets child rights freely"; revisit only
  if client-side re-delegation lands (a client would then need `MINT` onward, and
  *then* monotonic narrowing of the kernel rights mask matters).
- **Transfer authority.** v0.9c allows reply-carries-cap with no extra gate (the
  reply cap is already scoped one-shot authority). A `GRANT` right gating
  *general* `send`-carries-caps is the follow-on — decide when that milestone lands.
- **Badge width / encoding.** `u64`, one register. The FS will pack
  `inode:u32 | rights:u16 | spare:u16`; the kernel stays fully opaque, so the
  layout is a v0.10 (FS) decision, not a kernel one. Recorded here only so the
  register width is sized for it.
- **Revocation reach.** Per-object revocation is FS-side (drop the badge mapping);
  whole-cap revocation is the kernel generation bump. No new kernel mechanism in
  v0.9c; confirm the FS design's split still holds when v0.10 builds on this.
- **Reconcile the design docs.** `docs/ipc-design.md` and
  `docs/filesystem-design.md` describe badges in the abstract (seL4 framing) and
  list them as *deferred*. On landing this, re-frame both onto "badge = the
  generalized reply cap," move badges from deferred → shipped, and point the FS's
  access model at `Object::Endpoint { id, badge }` rather than an unbuilt
  primitive.

# v0.4 step 5 — page-table mutation API

Add `map(va, pa, perms)` — the primitive for installing a leaf PTE
at a given VA, allocating intermediate tables on demand. After this
step, the kernel can change the live page table after boot. Heap is
the first eventual customer (P2, see follow-up below); userspace
page tables in v0.6 will be the second.

Conceptual walkthrough was done interactively. Recap in §"Why these
choices."

Scope: **P1 only** in this plan — the primitive + host tests + a
wired-but-unused kernel-side function. P2 (migrate the heap to a
dedicated VA range and use `map` to grow it) is a follow-up; flagged
at the end.

## Why this step

Boot built a static page table with three huge-page leaves
(kernel, linear, MMIO). Nothing in the kernel mutates the table
after boot. To grow the heap, build userspace page tables, or
`ioremap` a late-arriving device, we need a function that:

- walks the table from `BOOT_PT_ROOT` to a leaf slot for an
  arbitrary VA,
- allocates intermediate 4 KiB tables (from `frame::alloc_zeroed`)
  if the path doesn't exist yet,
- writes a leaf PTE with the requested permissions,
- `sfence.vma`s the VA.

It's a tree walk that may allocate up to two frames per call. The
math (Sv39 VPN decomposition, PTE encoding, leaf vs non-leaf
detection) is pure and host-testable; the CSR-touching parts
(`sfence.vma`, reading/writing the root pointer) stay in `kernel/`.

## Decisions locked in

| decision | choice |
|---|---|
| Behaviour on already-mapped VA | Return `Err(MapError::AlreadyMapped)` |
| Permission encoding | `Perms(u8)` newtype with `R`/`W`/`X`/`U`/`G` flag constants |
| Frame allocator dependency | Trait `FrameAlloc` in `kernel-core` so the walk is host-testable with a mock |
| Cleanup on partial failure | Leak intermediate table on level-0 alloc failure; document; revisit if/when long-running mutation churn appears |
| TLB flush | `sfence.vma vaddr` per `map` call (single-hart; SMP-broadcast is a v0.7+ concern) |
| Where it lives | Walk logic + `Perms` + `MapError` in `kernel-core::mmu`; `kernel::mmu::map` is a thin wrapper supplying root + `FrameAlloc` impl |
| P1 vs P2 | P1 here. P2 = heap migration (dedicated VA, grow via `map`) is a follow-up plan |

## Why these choices

- **`Err(AlreadyMapped)`** — silent overwrite is dangerous (could
  alias a userspace mapping with kernel writes), assert is hostile
  to callers that want to test idempotence. `Result` lets the caller
  decide.
- **`Perms(u8)`** — Sv39 PTE permission bits already sit at bit
  positions 1..=4 / 5 (R/W/X/U) and 5 (G). A newtype that wraps the
  raw bit value keeps construction self-documenting (`Perms::R |
  Perms::W`) and serialisation trivial — just OR into the PTE.
- **`FrameAlloc` trait** — the walk allocates intermediate tables.
  In the kernel, that's `frame::alloc_zeroed`; in host tests, a
  `Vec<u64; 512>`-backed mock that hands out indices. This is the
  only seam needed to make the walk fully testable without
  `riscv64gc-unknown-none-elf`.
- **Leak on failure** — alternative is two-phase commit (pre-allocate
  both tables, install only if both succeed). Worth ~30 LOC and a
  bit of bookkeeping. Not worth it now: the only caller in v0.4 is
  heap grow, and a single leaked intermediate table on OOM is
  recoverable noise. Document the choice.
- **Single-hart `sfence.vma`** — `sfence.vma vaddr, asid` invalidates
  the TLB entry for that VA on this hart. SMP needs IPIs to other
  harts; not in scope for v0.4 (single hart).
- **P1-only deliverable** — `map` with no consumer is testable host-
  side and review-able in isolation. Risk of a Sv39 walk bug is
  highest; better to land it small and isolated than as part of a
  big heap-migration patch.

## Architectural details

### `Perms` newtype

```rust
// kernel-core/src/mmu.rs
#[derive(Clone, Copy, Debug, PartialEq, Eq)]
pub struct Perms(u8);

impl Perms {
    pub const R: Perms = Perms(1 << 1);
    pub const W: Perms = Perms(1 << 2);
    pub const X: Perms = Perms(1 << 3);
    pub const U: Perms = Perms(1 << 4);
    pub const G: Perms = Perms(1 << 5);

    pub const fn bits(self) -> u8 { self.0 }
}

impl core::ops::BitOr for Perms {
    type Output = Perms;
    fn bitor(self, rhs: Perms) -> Perms { Perms(self.0 | rhs.0) }
}
```

Host tests:
- `perms_or_combines_flags`
- `perms_r_alone_is_just_r_bit`
- `perms_bits_match_pte_layout` — explicit assert that `Perms::R.bits() == 0b10`, etc. Catches anyone misremembering the Sv39 bit positions.

### `MapError`

```rust
pub enum MapError {
    /// The VA already has a leaf PTE installed (at any level — could
    /// be a huge-page leaf in the root that prevents finer-grained
    /// mapping, or a 4 KiB leaf in the level-0 table).
    AlreadyMapped,
    /// `FrameAlloc::alloc_zeroed` returned `None` while trying to
    /// install an intermediate table.
    OutOfFrames,
}
```

### `FrameAlloc` trait

```rust
pub trait FrameAlloc {
    /// Allocate a zeroed 4 KiB frame, return its physical address.
    fn alloc_zeroed(&mut self) -> Option<usize>;
}
```

Two implementations:
- **Kernel-side**: a unit struct wrapping `crate::frame::alloc_zeroed()`.
- **Test-side**: a `Vec<[u64; 512]>`-backed mock that tracks
  handed-out frame indices; the test passes a "physical address" =
  the index, the mock dereferences back to the vec when the walk
  reads/writes. Mocks the linear map implicitly.

### `map` walk

```rust
// kernel-core/src/mmu.rs
pub fn map(
    root: &mut [u64; 512],          // VPN[2] table
    va: usize,
    pa: usize,
    perms: Perms,
    fa: &mut impl FrameAlloc,
    // for test-side: also need a way to deref a frame PA into the
    // intermediate table. In kernel, that's pa_to_kernel_va. In tests,
    // it's the mock's index → &mut [u64; 512]. Possible shapes:
    //   - second trait method on FrameAlloc: `table_mut(pa) -> &mut [u64; 512]`
    //   - separate trait `PtView`
    // Lean: fold into FrameAlloc to keep the surface small.
) -> Result<(), MapError>
```

Algorithm:
1. Compute `vpn2 = (va >> 30) & 0x1ff`, `vpn1 = (va >> 21) & 0x1ff`, `vpn0 = (va >> 12) & 0x1ff`.
2. Look at `root[vpn2]`. If it's a leaf → `AlreadyMapped`. If it's a non-leaf → walk in. If invalid → allocate, install non-leaf PTE, walk in.
3. Same at level 1 (table at `root[vpn2]`'s PPN, indexed by `vpn1`).
4. At level 0: if `table[vpn0]` is V=1 (leaf or non-leaf) → `AlreadyMapped`. Else write leaf PTE: `((pa >> 12) << 10) | A | D | perms.bits() | V`.
5. Return `Ok(())`. (TLB flush is the kernel-side wrapper's job; pure walk doesn't touch CSRs.)

PTE layout helpers (`leaf_pte`, `nonleaf_pte`, `is_leaf`, `is_valid`, `pte_ppn`) already exist or are easy adds — mirror `mmu::enable`'s helpers.

### Host test plan

- `map_into_empty_table_installs_leaf` — root empty; one `map` call; assert leaf PTE at expected indices.
- `map_allocates_two_intermediate_tables` — root empty; one `map`; mock reports 2 frame allocations.
- `map_reuses_existing_intermediate_table` — first map at va A; second map at va B in same level-1 table; mock reports only 1 additional alloc.
- `map_returns_already_mapped_if_leaf_at_level_0` — two `map` calls to same VA; second returns `AlreadyMapped`.
- `map_returns_already_mapped_if_root_has_huge_leaf` — root[vpn2] is a 1 GiB leaf; map into that range returns `AlreadyMapped`.
- `map_returns_out_of_frames_when_allocator_exhausted` — mock returns `None` on first call; `map` returns `OutOfFrames`.
- `map_propagates_perms_to_leaf_pte` — passes `Perms::R | Perms::W`; reads leaf PTE; asserts R+W bits set.
- `map_sets_a_and_d_bits` — leaf PTE always has A=1, D=1 (per project convention to dodge unset-bit traps).
- `map_encodes_pa_at_correct_pte_bits` — PPN at bits 53:10; tests via specific PA.

### Kernel-side wrapper

```rust
// kernel/src/mmu.rs (add)
pub fn map(va: usize, pa: usize, perms: Perms) -> Result<(), MapError> {
    let mut fa = KernelFrameAlloc;
    let root = unsafe { &mut *(&raw mut BOOT_PT_ROOT) };
    // SAFETY: BOOT_PT_ROOT is the live root table; we have exclusive
    // mutable access via the kernel's single-thread guarantee at
    // boot, and at heartbeat-time stays single-threaded under spin
    // locks held by callers. (Document expected lock discipline as
    // call sites land in P2.)
    let result = kernel_core::mmu::map(root, va, pa, perms, &mut fa);
    if result.is_ok() {
        // SAFETY: sfence.vma vaddr invalidates the TLB entry for `va`
        // on this hart only — fine for v0.4 single-hart.
        unsafe { asm!("sfence.vma {0}, zero", in(reg) va) };
    }
    result
}

struct KernelFrameAlloc;
impl kernel_core::mmu::FrameAlloc for KernelFrameAlloc {
    fn alloc_zeroed(&mut self) -> Option<usize> {
        crate::frame::alloc_zeroed().map(|f| f.addr())
    }
    // ... table_mut deref via pa_to_kernel_va
}
```

The wrapper is dead-code in P1 (`#[expect(dead_code)]`); it gets called from `heap::extend` in P2.

## What this doesn't do

- No call sites in P1. Heap is unchanged. The kernel binary builds
  with `map` present but inert.
- No two-phase commit on partial allocation failure. Documented.
- No SMP TLB broadcast. Single-hart only.
- No `unmap` / `protect` (permission change) yet. `unmap` is the v0.6
  per-process-table-teardown shape; `protect` is W^X enforcement
  later.
- No huge-page mapping support. `map` always installs 4 KiB leaves.
  Huge-page support is a separate API once we need it (DMA buffers,
  v0.6+).

## Risk and known unknowns

- **Sv39 PTE encoding off-by-one.** PPN at bits 53:10, not 0. The
  walk's host tests should explicitly pin the bit layout (one test
  uses a specific PA whose PPN bits have a recognisable pattern, asserts
  the PTE matches).
- **`sfence.vma` operand syntax.** `sfence.vma vaddr, asid` — both
  operands are register-encoded. Wrong operand order is silent on
  single hart (TLB flush still happens for *something*). Worth
  triple-checking against the RISC-V Privileged spec before commit.
- **Mock-vs-real divergence.** Host tests cover the *walk*; they
  don't cover "post-map, an actual MMU access translates correctly."
  P2's integration test (heap grow → write → read back) closes that
  gap. Until then, P1 carries some trust.

## Done means

- `cargo test -p kernel-core mmu::` covers the 9 walk tests.
- `cargo xtask build` builds the kernel with `map` present (dead-code).
- All 8 existing integration scenarios still pass.
- CLAUDE.md memory section mentions `map` is available; the (P1 then P2)
  shape is documented so the next session knows P2 is the actual

## P2 follow-up (separate plan)

Migrate the heap to a dedicated VA range and use `map` to grow it.
Shape:

- Reserve root PTE slot 256 for the heap; `HEAP_VA_BASE =
  0xffffffc0_00000000`.
- `heap::init(initial_frames)`: instead of `alloc_contiguous`, loop
  N times calling `frame::alloc_zeroed` + `mmu::map(HEAP_VA_BASE +
  i*4096, frame_pa, Perms::R | Perms::W | Perms::G)`. Hand the VA
  range to `linked_list_allocator::init`.
- `heap::extend(extra_frames)`: same loop starting at the next free
  heap VA; call `linked_list_allocator::extend` after.
- Heartbeat watermark: if `stats().free < THRESHOLD` and the heap
  hasn't hit its 1 GiB ceiling, call `heap::extend(N)`. Extend
  failure → no-op (next `alloc` will return null, `alloc_failed_total`
  ticks as today).
- New integration scenario `heap-grows-on-pressure`: a feature-gated
  smoke that allocates past the initial 4 MiB; assert
  `heap.bytes_capacity` rises and the kernel keeps heartbeating.

P2 is where `map` first actually shapes a translation. Risk lives
there, not here.

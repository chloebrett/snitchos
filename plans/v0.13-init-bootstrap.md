# Plan: `init` — the first-process bootstrap (root of the delegation graph)

**Branch**: feat/init-bootstrap
**Status**: Active
**Scope decision (2026-06-27)**: build **`init` only**; the Rust-vs-Stitch shell
choice is deferred until `init` exists and the cap-handoff ABI is concrete (per
user). This plan ends with a working delegation-graph root that *manufactures its
own world* (creates an endpoint, spawns + supervises the FS server and a client) —
*not* an interactive shell.

## Goal

A real userspace `init` process that boots as the single first userspace program
holding **only** telemetry/span, **creates its own IPC endpoint**, and delegates
authority downward — FS server with `RECV | MINT`, a client with a minted `SEND` on
the *same* endpoint — supervising both as a true reaper. The kernel stops knowing
the IPC topology; the capability derivation tree gets a userspace root; every grant
is a visible `CapEvent::Transferred`.

## Why now / context

Everything below `init` has shipped (`spawn-shell-and-console.md`): `Spawn` (15)
with all-or-nothing cap delegation, `Exit`/`Wait`/`reap`, the notification
primitive, `ConsoleRead`/`ConsoleWrite`, the FS-over-IPC stack, `MintBadged`. What's
missing is the **root**: today `kmain` spawns each userspace program directly and the
IPC endpoint is a kernel static (`DEMO_ENDPOINT`) — so the *kernel* is the root of
every delegation and knows the FS topology. `init` moves that root into userspace.

## Decisions (resolved 2026-06-27)

- **D1 — Endpoint origin: `init` creates its own via a new `EndpointCreate`
  syscall.** The kernel no longer pre-creates an endpoint for the init path; `init`
  manufactures it and is its owner. The kernel forgets IPC topology entirely (the
  project's "kernel knows less" thesis, taken to its conclusion). *(More ambitious
  than the stepping-stone lean; chosen deliberately.)* The existing
  `DEMO_ENDPOINT`-based `fs`/`ipc` itest workloads stay as-is (additive).
- **D2 — Kernel grants `init` only the bootstrap caps.** Because `init` creates its
  own endpoint, the kernel hands it **nothing but telemetry+span** (the
  `Process::bootstrap` pair); `Spawn` and `EndpointCreate` are ambient. The kernel
  spawns `init` via the existing `spawn_process_with_caps` path with an **empty**
  delegated set — no new code path, no special "init layout".
- **D3 — `init` fully supervises (true reaper).** Needs a new **wait-for-any-child**
  primitive (today `Wait` blocks on a specific child id). `init` parks in
  wait-for-any and reaps whichever child exits.

### Caveats this chain introduces (document; do not block)

- **`init` over-holds the endpoint.** Copy-semantics delegation means after `init`
  grants `RECV|MINT` to the FS server, `init` *retains* `RECV` on an endpoint it
  never serves (two receivers; only the server calls `Recv`). Real fix = move
  semantics / revocation via the cap `generation` hook (dead-weight at 0 today).
  Deferred; noted.
- **`EndpointCreate` ambient = the resource axis (Q-D).** Unbounded userspace
  endpoint creation is an exhaustion surface, same as ambient `Spawn`. Ambient for
  now; quota deferred. Snitch refusals stay loud.

## Grounding (verified via code map; cite before editing)

- **Spawn (15)**: `kernel/src/syscall/process.rs:62` — `spawnable_program` →
  `copy_from_user` `[u32;N]` (max `MAX_DELEGATE=16`) → `kernel_core::cap::delegate`
  (all-or-nothing, copy) → `spawn_process_with_caps(hart, name, image, delegated, prio)`.
- **SPAWNABLE registry**: `kernel/src/trap/user.rs:405`. Add program =
  `include_bytes!(env!("SNITCHOS_<NAME>_ELF"))` + a row + add bin to `USER_PROGRAMS`
  in `kernel/build.rs`.
- **Startup-cap ABI (already correct for Spawn — finding 2026-06-27)**: crt0
  `user/runtime/src/lib.rs:84` reads `a0`=telemetry, `a1`=span, `a2`=endpoint;
  `delegated_handle(i)=2+i`. The `endpoint()`(a2)-vs-`delegated[0]`(handle 2)
  aliasing exists **only across two disjoint launch paths**: the Spawn path
  (`run_with_caps`, `trap/user.rs:585`) lays out telemetry@0/span@1/delegated@2+ and
  passes `a2=0` (no endpoint); the IPC path (`run_ipc`, `:664`) puts the endpoint at
  handle 2 / `a2`. **Within the Spawn path there is no collision** — `spawnee.rs` /
  `notify_signaller.rs` already read `delegated_handle(0)` and work. So an
  init-spawned child's ABI is **already collision-free and exercised**; there is no
  ABI fix to make. Folding `run_ipc` onto the general path (dropping the special `a2`
  endpoint slot) is **deferred-until-forced** — reopening working code for an
  aesthetic win is exactly the anti-pattern post 34 names. The only ABI-adjacent work
  is that an init-spawned FS server reads its endpoint via `delegated_handle(0)`
  instead of `endpoint()` — folded into Step 6 (init spawns FS server), not a step.
- **Bootstrap caps**: `Process::bootstrap` (`kernel/src/sched/process.rs:112`) →
  telemetry+span (`EMIT`).
- **Endpoint create (kernel-internal today)**: `crate::ipc::create` →
  `DEMO_ENDPOINT` static, called by `kmain` (`kernel/src/main.rs:467`) when
  `needs_endpoint`. `EndpointCreate` exposes this to userspace + returns an owning cap.
- **MintBadged** (shipped): mints a badged `Endpoint{id,badge}` cap with chosen
  rights from a `MINT`-holding parent — how `init` makes the client's `SEND` cap.
  `kernel_core::user::cap::mint_badged` (`:375`), requires `Rights::MINT`.
- **Wait (18) + reap**: `kernel/src/syscall/process.rs:39`; `block_current`/`wake`
  + the reaping table; `reap_task` frees AS/Process/stack. Today keyed on a specific
  child id — Step 3 adds wait-for-any.
- **CapEvent frame**: `protocol/src/lib.rs:151` — `CapEvent { kind, cap_id,
  parent_cap_id, holder, object, rights, badge, t, hart_id }`. `Transferred` exists
  (reply-cap mints). **Verified 2026-06-27:** `parent_cap_id` is hardcoded `0` at
  *every* emit site (`obs/tracing.rs:285,304`) and `Capability {object, rights}`
  stores no id → linkage = Step 1's spine. `run_with_caps` (`trap/user.rs:620`) emits
  delegated caps as `Granted`, not `Transferred` → Step 2 is real new behavior.

## Acceptance Criteria

- [x] The startup-cap layout for a `Spawn`'d child is unambiguous and host-tested:
      each role (telemetry, span, delegated[i]) maps to a distinct handle, no aliasing.
      *(Met 2026-06-27: the Spawn path already lays out telemetry@0/span@1/delegated@2+
      with no endpoint in `a2`; `spawnee`/`notify_signaller` read `delegated_handle(0)`
      end-to-end. The `endpoint()`/`delegated[0]` aliasing exists only across the
      disjoint `run_ipc` path — see grounding. No work needed.)*
- [ ] Every capability holding carries a stable global `cap_id`; a transferred cap
      records its source holding's id as `parent_cap_id` (the derivation-tree spine).
- [ ] Each cap `Spawn` delegates emits `CapEvent { kind: Transferred, holder=child,
      parent_cap_id=source }`.
- [ ] A process can call `EndpointCreate` and receives a `RECV | MINT` cap naming a
      fresh endpoint; two creates yield distinct endpoints.
- [ ] A parent blocked in wait-for-any wakes with the id+status of whichever child
      exits first, and reaps it.
- [ ] On default boot the kernel starts exactly one userspace program — `init`,
      holding only telemetry+span — and `init` is the parent of every other
      userspace process.
- [ ] `init` creates an endpoint, spawns the FS server with `RECV|MINT`, and the
      existing FS round-trip works with the server parented to `init`.
- [ ] `init` mints a `SEND` cap and spawns a client with it; an itest shows the two
      children share one endpoint with different rights, two `CapEvent::Transferred`
      frames both rooted at `init`'s endpoint.

## Steps

Every step follows RED-GREEN-MUTATE-KILL MUTANTS-REFACTOR. No production code without
a failing test. Host-test pure logic in `kernel-core`/`abi`/`protocol`; cover
kernel-only wiring with a QEMU itest scenario. Run `cargo xtask itest --repeat 10`
on new/changed scenarios before commit.

> **Dropped (finding 2026-06-27):** the original "pin the startup-cap ABI" step is
> removed — the Spawn ABI is already collision-free and exercised (`spawnee` /
> `notify_signaller` read `delegated_handle(0)`), so there was no failing test to
> write. Folding `run_ipc`/`a2` is deferred-until-forced (see grounding). Steps
> renumbered.
>
> **Added (finding 2026-06-27):** linkable `parent_cap_id` (decided D-parent) is
> unimplemented infra today — `parent_cap_id` is hardcoded `0` at *every* emit site
> and `Capability` carries no stored id. So a new **Step 1 (cap-id spine)** lands
> before the Spawn trace; it retires the `parent_cap_id: 0` debt at all transfer
> sites at once.

### Step 1: Stable global cap-id on each holding — the derivation-tree spine

**Acceptance criteria**: every capability *holding* (a `CapTable` slot) carries a
stable global `cap_id`, assigned when it enters a table; a transferred/delegated cap
records the **source holding's** `cap_id` as its `parent_cap_id`. The existing
transfer sites (reply-cap `ipc.rs`, notify, `cap.rs`) and grants stop hardcoding
`parent_cap_id: 0` where a real parent exists. Host-tested in `kernel-core`.
*(Present the slot-id model + minting seam to human before code.)*
**RED**: `kernel-core` host test — insert a cap (gets `cap_id` X), then derive/insert
a child from it and assert the child slot's recorded parent id == X and its own id ≠ X.
(Today `Slot` has no `cap_id` field → fails to compile / fails.)
**GREEN**: add `cap_id: u64` to `Slot`, set on insert; `CapTable` exposes
handle→`cap_id` lookup so the kernel can read a parent's id at delegate/transfer.
`cap_id` is **minted kernel-side** (`next_cap_id()`) and passed into insert —
`kernel-core` stays pure (no kernel dep); the assignment + parent-link logic is the
host-tested part. Wire `parent_cap_id` at the existing transfer sites.
**MUTATE / KILL MUTANTS / REFACTOR**: per skill (watch the parent-link wiring + the
fresh-id-per-holding invariant).
**Done when**: holdings carry stable ids, transfers link to their source, `parent_cap_id:
0` survives only where the cap genuinely is a root; human approves.

### Step 2: `Spawn` emits `CapEvent::Transferred` per delegated cap (the payoff trace)

**Acceptance criteria**: a `Spawn` delegating K caps produces K `Transferred` frames
(`holder`=child, `parent_cap_id`=the caller's **source holding** cap_id from Step 1).
Today `run_with_caps` (`trap/user.rs:620`) snitches delegated caps as `Granted` — so
this is real new behavior. Verified via the existing `spawnee` workload.
**RED**: itest asserting `Transferred` frames for the `spawnee` delegation, each with
`parent_cap_id` == the parent's source cap_id (fails today: emitted as `Granted`,
parent 0).
**GREEN**: in the delegated-cap loop emit `Transferred` (parent_cap_id from the Step-1
spine) instead of `Granted`; bootstrap caps stay `Granted`/root.
**MUTATE / KILL MUTANTS / REFACTOR**: per skill; share the emit helper with the
reply-cap-mint site if diverged.
**Done when**: any Spawn's delegation chain is visible + linked in Tempo; human approves.

### Step 3: wait-for-any-child primitive

**Acceptance criteria**: a parent blocked waiting for *any* child wakes with the
id+status of whichever exits first, then reaps it; a specific-child `Wait` still
works unchanged. *(Present the shape to human: sentinel child-id on `Wait` vs a new
`WaitAny` syscall.)*
**RED**: host-test the wait-table "any" matching where it's pure (which parent a
given child-exit should wake); itest: parent spawns two children, the second exits
first, parent wakes with the second's id+status.
**GREEN**: extend the wait/reap table + exit-wake path to support an "any" waiter;
add the syscall surface (`abi` variant + `from_usize`, host-tested).
**MUTATE / KILL MUTANTS / REFACTOR**: per skill (watch the any-vs-specific match).
**Done when**: wait-for-any reaps the first-exiting child; specific Wait unregressed;
human approves.

### Step 4: `init` boots as the sole first userspace program and supervises a child

**Acceptance criteria**: default boot starts exactly one userspace program (`init`,
holding only telemetry+span); `init` `Spawn`s `spawnee` and reaps it via
wait-for-any. itest asserts init→child parentage on the wire and the reap.
*(Present to human: which child for the first increment.)*
**RED**: itest `init-supervises-child` — child's spawn/ThreadRegister attributable
to init's task id; reap observed.
**GREEN**: add `init` to SPAWNABLE + `USER_PROGRAMS`; write `user/init` (crt0 →
`Spawn(spawnee, [])` → wait-for-any → reap, loop); change the **default** boot path
in `kmain` to spawn only `init` with an empty delegated set.
**MUTATE / KILL MUTANTS / REFACTOR**: per skill; fold per-workload kmain spawns
toward "kmain spawns init; init spawns the rest" without regressing
`itest-workloads` (keep their direct spawns — additive).
**Done when**: default boot has a supervising userspace root; human approves.

### Step 5: `EndpointCreate` syscall (userspace manufactures an endpoint)

**Acceptance criteria**: `EndpointCreate` allocates a fresh endpoint and inserts a
`RECV|MINT` cap into the caller's table, returning its handle; two calls yield
distinct `EndpointId`s; ambient (any process). Refusals snitch.
**RED**: host-test the cap-insert + distinct-id logic where pure (kernel-core);
itest: a program calls `EndpointCreate` and successfully `Recv`s on the returned cap
(proves the cap is real and owning).
**GREEN**: `abi` variant + `from_usize`; `kernel/src/syscall/...` handler over
`ipc::create` that builds `Capability{ Endpoint{id,badge:0}, RECV|MINT }` and inserts
it; runtime binding in `user/runtime`.
**MUTATE / KILL MUTANTS / REFACTOR**: per skill.
**Done when**: userspace can create an endpoint it owns; human approves.

### Step 6: `init` creates the endpoint and spawns the FS server with `RECV|MINT`

**Acceptance criteria**: `init` `EndpointCreate`s, then `Spawn`s `fs-server`
delegating `RECV|MINT`; the existing FS round-trip works with the server parented to
`init` (its caps trace to init's endpoint, not `DEMO_ENDPOINT`).
**RED**: itest where init spawns the FS server; assert the round-trip works and the
server's endpoint cap's `parent_cap_id` is init's created endpoint.
**GREEN**: `init` resolves its created-endpoint handle and `Spawn`s `fs-server` with
`[endpoint(RECV|MINT)]`; wire the workload so init is the FS parent.
**MUTATE / KILL MUTANTS / REFACTOR**: per skill.
**Done when**: FS works under init, endpoint manufactured by init; human approves.

### Step 7: `init` mints `SEND` and spawns a client — two-rights delegation graph

**Acceptance criteria**: `init` `MintBadged`s a `SEND` cap on its endpoint and
`Spawn`s a client with it; an itest shows server(`RECV|MINT`) + client(`SEND`)
talking over the one endpoint, and two `CapEvent::Transferred` frames both rooted at
init's endpoint with *different rights*. The least-authority demo in miniature.
**RED**: itest `init-delegates-two-rights` — both grants trace to init's endpoint;
rights differ.
**GREEN**: `init` `MintBadged(endpoint, badge, SEND)` → handle → `Spawn(client,
[handle])`; client `Send`s to the server.
**MUTATE / KILL MUTANTS / REFACTOR**: per skill.
**Done when**: the two-child, two-rights graph is visible and rooted at init; human approves.

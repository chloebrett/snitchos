# v0.4 step 2c — relink at higher-half, satp moves early

The risky one. Change the linker so every symbol's VA is at
higher-half, and move `mmu::enable()` to the top of `kmain` so the
higher-half mapping is live before any code that stores absolute
function-pointer values to higher-half addresses runs.

**Kernel still runs at identity addresses after this lands.** The
linker change doesn't move the running PC; it only changes what
values absolute references compile to. The trampoline + sp fix-up +
identity unmap (the "kernel actually executes at higher-half" stuff)
is step 2d.

Read first:
- `plans/v0.4-memory-findings.md` — what already went wrong.
- `plans/v0.4-memory-step-2a-dual-map.md` — the prereq.
- `plans/v0.4-memory-concepts.md` § 3-4 — Sv39 + higher-half background.

## Why this is risky

This is the step where last time's attempt broke. We're changing
the link-time addresses of every symbol in the binary while the
kernel keeps running at the physical addresses where OpenSBI loaded
it. That introduces a mismatch that requires the page table to
bridge — and any code that runs before `mmu::enable()` is operating
without that bridge.

Two classes of code stop working pre-MMU after the linker change:

1. **Anything that reads an absolute symbol VA into a register and
   uses it as an address.** Function pointers in vtables (already
   eliminated via monomorphization), function pointers in
   `fmt::Arguments` (NOT eliminable — every formatted `println!`
   builds an Arguments with type-formatter fn pointers), trap
   vector address (we currently write `trap_entry` symbol's value
   to stvec).
2. **Things relying on iteration / closure chains that might emit
   absolute references for trait dispatch.** Last time
   `dtb::timebase_hz`'s `.and_then(|n| n.properties().find(...))`
   chain crashed pre-MMU under higher-half link. Cause never fully
   isolated. We mitigate this step by minimising what runs before
   `mmu::enable()`.

## The minimum-risk path

Four small landings inside this step, each individually testable:

1. **Refactor: collect MMIO regions before `mmu::enable()`.**
   Removes DTB iteration from inside `mmu::enable`. Pure refactor
   under identity-link baseline. Verify all 5 scenarios still pass.
2. **Linker change + boot reorder.** ORIGIN → higher-half,
   `AT(0x80200000)`, `mmu::enable` moves to top of `kmain`,
   `set_trap_vector` moves to after `mmu::enable`. Verify 5
   scenarios pass.
3. **Remove the higher-half-verify scenario.** With identity still
   live, the verify is still meaningful technically, but it's about
   to become tautological in step 2d. Keep it for now; remove in
   step 2d cleanup.
4. **(Skipped — moved to 2d.)** Identity unmap. Trampoline. sp
   fix-up.

## What the boot order looks like after 2c

```rust
pub extern "C" fn kmain(_hart_id: usize, dtb_phys: usize) -> ! {
    // (1) DTB parse. No formatted output, no absolute fn pointers
    //     involved. Pre-MMU.
    let dtb = unsafe { Fdt::from_ptr(dtb_phys as *const u8) }.unwrap();
    let timebase_hz = dtb::timebase_hz(&dtb).expect("...") as u64;

    // (2) Collect MMIO + DTB regions while MMU is still off but
    //     before we hand them to mmu::enable. Decouples DTB
    //     traversal from page-table construction; sidesteps the
    //     dtb.all_nodes()-inside-mmu::enable crash we saw before.
    let mmio_regions = mmu::collect_mmio_regions(&dtb);

    // (3) Turn the MMU on. Identity + higher-half mapped for kernel
    //     image, identity for MMIO and DTB. Code runs at the SAME
    //     identity addresses it did before; higher-half mapping is
    //     resident only so absolute fn pointers resolve.
    unsafe { mmu::enable(mmio_regions, dtb_phys) };

    // (4) Now safe to install trap vector. `trap_entry` symbol's
    //     value is higher-half VA; with MMU on + higher-half mapped,
    //     it resolves on trap.
    unsafe { trap::set_trap_vector() };

    // (5) Everything past this is identical to today.
    {
        span!("kernel.boot");
        ...
    }
    ...
}
```

## Architectural details

### Linker change

```ld
ENTRY(_start)
MEMORY {
  ram (rwx) : ORIGIN = 0xffffffff80200000, LENGTH = 128M
}
SECTIONS {
  __kernel_start = ORIGIN(ram);
  .text : AT(0x80200000) {
    KEEP(*(.text.boot))
    *(.text .text*)
  } > ram
  .rodata : { *(.rodata .rodata*) *(.srodata .srodata*) } > ram
  .data : { *(.data .data*) *(.sdata .sdata*) } > ram
  .bss : {
    __bss_start = .;
    *(.bss .bss*)
    *(.sbss .sbss*)
    __bss_end = .;
  } > ram
  . = ALIGN(16);
  __stack_bottom = .;
  . += 0x10000;
  __stack_top = .;
  __kernel_end = .;
  /DISCARD/ : { *(.eh_frame .eh_frame.*) *(.eh_frame_hdr) *(.note .note.*) *(.comment) }
}
```

`AT(0x80200000)` on the first section anchors LMAs at the
physical address OpenSBI loads us at. Subsequent sections inherit
contiguous LMAs.

### MMIO region collection — new `kernel-core::mmu` helper

`kernel-core` shouldn't depend on `fdt`. So:

- New module on the kernel side: a small `regions: ArrayVec<(usize, usize), 8>` or fixed `[Option<(usize, usize)>; 8]` to hold up to 8 MMIO regions. Each entry is `(base, size)`.
- `mmu::collect_mmio_regions(dtb: &Fdt) -> MmioRegions` — walks the DTB once, returns the list. Lives in `kernel/src/mmu.rs`.
- `mmu::enable(regions: MmioRegions, dtb_phys: usize)` — uses the list to identity-map each region's 2 MiB page. No DTB borrow during page-table construction.

### Page-table contents after 2c

- Identity gigapage 2 (`[0x80000000, 0xC0000000)`):
  - Mid-table KERNEL: entry 1 = 2 MiB leaf to PA `0x80200000` (kernel image).
  - Possibly entry 63 = DTB at PA `0x87e00000` (depends on QEMU's
    DTB placement).
- Identity gigapage 0 (`[0x00000000, 0x40000000)`):
  - Mid-table MMIO: entry 128 = 2 MiB leaf to PA `0x10000000` (UART + virtio-mmio).
- Higher-half gigapage 510 (`[0xffffffff_80000000, 0xffffffff_C0000000)`):
  - Mid-table HIGHER-KERNEL: entry 1 = 2 MiB leaf to PA `0x80200000` (kernel image, mirror of identity).
- Everything else: invalid.

Note: MMIO does NOT need a higher-half mapping in this step. The
kernel reads/writes MMIO via the CONSOLE / UART statics, which
hold physical addresses. Identity-mapped, those work as-is.

### What changes in `set_trap_vector`'s call site

Currently first thing in `kmain`. Moves to AFTER `mmu::enable`.
Justification: `trap_entry` symbol's link-time address is
higher-half after the linker change. Writing it to `stvec` while
MMU is off would mean a trap goes to an unmapped higher-half VA
and double-faults. After `mmu::enable` the higher-half mapping is
live and stvec resolves correctly.

Between OpenSBI handoff and `set_trap_vector`, stvec holds an
undefined value. If a trap fires in that window, the kernel hangs
or jumps to garbage. Mitigation: nothing between OpenSBI handoff
and `mmu::enable` should be able to trap (DTB parse + memory-only
ops). Same situation as today's kernel, just shifted later.

## Tests

No new kernel-core tests this step.

Integration: all 5 existing scenarios should pass unchanged. The
`mmu-higher-half-verify` scenario still meaningfully tests the
dual-map (the higher-half read still succeeds because the kernel
still runs at identity and the mapping resolves the higher-half VA
on demand).

## Decisions to confirm before execution

1. **MMIO regions API.** `MmioRegions` as a fixed-size array of
   `Option<(usize, usize)>`, capacity 8. (QEMU `virt` has UART +
   8 virtio-mmio slots = 9, but adjacent slots collapse into the
   same 2 MiB page so we dedupe to 1-2 entries.) Capacity 8 leaves
   headroom.
2. **Does `dtb::timebase_hz` need refactoring?** Last time it
   crashed pre-MMU with the linker change. Hypotheses unclear.
   First try: leave it as-is, see if it still crashes now that we
   do `mmu::enable` very early (which would suggest the crash was
   about something else entirely). Second try: rewrite as an
   explicit for-loop (we did this once already during debugging;
   it worked).
3. **Identity unmap deferred?** Yes. Identity stays mapped post-2c.
   Step 2d handles the unmap as part of moving the kernel's PC to
   higher-half.
4. **Trampoline deferred?** Yes. PC stays at identity. We're
   building infrastructure; behavioural transition is 2d.

## Risks and known weaknesses

- **`dtb::timebase_hz` chain might still crash pre-MMU.** If it
  does, rewrite as a for-loop (it's 4 lines).
- **`set_trap_vector` move means the window between OpenSBI and
  satp has no installed trap handler.** Same as today — we don't
  expect traps in that window. Flagged.
- **`(&raw const __kernel_start) as usize` value unverified.** The
  findings doc flags this. Under medium code model it *should* be
  PC-relative → physical. Disassemble after the linker change to
  confirm; if it's higher-half, the satp PPN computation breaks
  and we need to subtract `KERNEL_OFFSET` explicitly.
- **`(&raw const BOOT_PT_ROOT) as usize` same concern.** Same
  remedy if needed.
- **Unknown — what specifically crashed inside `mmu::enable`'s
  DTB iteration last time.** Mitigated by pre-collecting regions.
  We don't fully understand the bug; we just avoid re-encountering
  it.

## Done state

- `linker.ld` ORIGIN at higher-half, `AT(0x80200000)` on `.text`.
- `mmu::enable()` called at top of `kmain`, before
  `set_trap_vector`, before any `span!` or `println!`.
- `mmu::enable` takes a pre-collected `MmioRegions` instead of
  borrowing `&Fdt`.
- ELF program headers: `vaddr=0xffffffff80200000+, paddr=0x80200000+`.
- All 5 integration scenarios still pass.
- Kernel binary boots, reaches heartbeat, emits frames identically
  to before.

## What's not in this plan

- **Step 2d**: Trampoline + sp fix-up + identity unmap. The kernel
  finally runs at higher-half VAs. CONSOLE + UART statics get
  patched to higher-half addresses. Identity mapping is torn down
  from the boot table.

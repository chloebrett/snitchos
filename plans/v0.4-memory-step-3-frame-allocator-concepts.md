# v0.4 step 3 — frame allocator (concepts)

Read this before the implementation plan. Concepts and trade-offs;
no code commitments yet. The actual plan (`plans/v0.4-memory-step-3-frame-allocator.md`)
will be written after we lock the decisions at the bottom.

## 1. What a frame allocator is and why we need one now

A frame is a fixed-size chunk of physical memory — for us, **4 KiB**
(the Sv39 base page size). The frame allocator's job is to hand out
free physical frames on request and accept them back when they're
no longer needed. That's the whole interface:

```rust
fn alloc() -> Option<PhysFrame>;
fn free(frame: PhysFrame);
```

Why now? Every higher-level piece of v0.4 and beyond wants it:

- **Step 4 (kernel heap)**: the heap needs backing pages. Without a
  frame allocator, the heap is stuck at whatever fixed-size
  statically-reserved region we give it.
- **Future page tables**: per-process page tables in v0.6 are made
  from fresh frames. The boot table we use today is `static mut` —
  fine for a one-off, not scalable.
- **`ioremap`-style mappings**: any new device mapping wants a fresh
  page table leaf, which needs a frame.
- **Drivers that DMA**: need physical-contiguous buffers. v0.4 step 3
  only does single-frame allocation, but it's the substrate.

It's also the simplest "real new capability" thing in v0.4. Steps 1
and 2 were paging infrastructure; step 3 is when the kernel can
actually *do something it couldn't do before*.

## 2. What goes in the free pool

Physical RAM minus everything that's already in use:

- **Below `0x80000000`**: MMIO, ROM, reserved by the platform.
  Not RAM.
- **`0x80000000` to `0x80200000`**: OpenSBI firmware. Reserved.
- **Kernel image**: `__kernel_start` to `__kernel_end` (linker symbols).
  Includes `.text`, `.rodata`, `.data`, `.bss`, stack, the four
  `BOOT_PT_*` page tables.
- **DTB**: at `dtb_phys` (typically around `0x87e00000`), some small
  size (< 64 KiB).
- **The bitmap itself** (or whatever metadata structure we pick).

Everything else in the DTB's `/memory` node is fair game.

## 3. Tracking strategies — three real options

### Bitmap

One bit per frame; 1 = free, 0 = in-use (or the opposite — pick one).
Lookup is a linear scan for the first set bit, which is fast on
modern CPUs (`ctz` / `cttz`). Free is O(1).

**Size**: 1 bit / 4 KiB = `RAM / 32768` bytes.
- 128 MiB RAM → 4 KiB bitmap.
- 1 GiB RAM → 32 KiB bitmap.
- 16 GiB RAM → 512 KiB bitmap.

**Pros**: simple, low overhead per frame, easy to reason about,
trivial to host-test.

**Cons**: scan time is O(n/64) per alloc — for 16 GiB worth of
frames that's 64K words scanned worst case. Fine until v0.7+
workloads, then revisit.

### Linked free list

Each free frame holds a pointer to the next free frame, embedded in
the page itself. Allocator pops the head, frees push onto the head.
O(1) for both.

**Size**: zero metadata cost (the pointers live in unused frames).

**Pros**: O(1) operations, no extra memory.

**Cons**: each alloc/free dirties the page being touched (cache
behavior), reads pointer from possibly-unmapped page (need physical
access via linear map), can't easily ask "is frame N free?", harder
to host-test.

### Buddy

Tree of power-of-two-sized blocks; supports allocations of varying
contiguous sizes (1 frame, 2 frames, 4 frames, ...).

**Pros**: handles contiguous allocations cleanly (DMA buffers,
huge pages).

**Cons**: significantly more code, more state, overkill for v0.4
where we only allocate single 4 KiB frames.

### Recommendation

**Bitmap.** Pure logic in `kernel-core`, easy host tests, fits the
"simple thing that works" rule. If we ever need contiguous
allocations we can layer a buddy on top later, or just write a
small "scan for N consecutive set bits" routine — that's a 20-line
addition.

## 4. The bootstrap problem

The bitmap needs storage. We don't have a heap. Three patterns:

### (a) Static array sized for max RAM

```rust
const MAX_FRAMES: usize = 16 * 1024 * 1024 * 1024 / 4096; // 16 GiB
static mut BITMAP: [u64; MAX_FRAMES / 64] = [0; MAX_FRAMES / 64];
```

For 16 GiB capacity, that's 32 KiB of `.bss`. Always allocated even
if we only have 128 MiB of RAM. Wasteful at the small end.

### (b) Bumper into a known location

`mmu::enable` is followed by frame-allocator init. The init code
treats some specific physical region (say, just after the kernel
image) as the bitmap location. The bitmap is sized to match actual
RAM (from DTB).

Trickier: have to map this region in the boot page table so the
kernel can write the bitmap. And we don't want it to overlap with
anything else.

### (c) Static array sized for **expected** RAM

Pick a generous max (say 4 GiB or 8 GiB), declare a static. Anything
more, we'd need to revisit. Simpler than (b), more honest than (a).

### Recommendation

**(c) with 4 GiB max** — `4 GiB / 4 KiB / 8 = 128 KiB` of `.bss`.
That's a lot relatively (the kernel image is currently ~70 KiB), but
absolute it's nothing. If RAM exceeds 4 GiB we either raise the
constant or do (b). For QEMU `virt` we typically run 128 MiB to
1 GiB.

Could halve to 64 KiB if we cap at 2 GiB. Probably fine for now.
Let's discuss.

## 5. Memory map from the DTB

The DTB's `/memory` node lists the RAM regions:

```
memory@80000000 {
    device_type = "memory";
    reg = <0x00000000 0x80000000 0x00000000 0x08000000>;  // base, size
};
```

That's `base = 0x80000000`, `size = 0x08000000` (128 MiB) on QEMU
`virt` with `-m 128M`. The fdt crate exposes this via
`dtb.memory().regions()`.

But — there's a catch we already hit: **DTB iteration crashes
pre-MMU under higher-half link** in a way we haven't isolated. The
frame allocator init runs *post-MMU*, so this isn't a problem here.
Just flagging it.

After identity-MMIO unmap, the DTB region is also unmapped — so
frame-allocator init has to happen *before* `unmap_identity`. Or
we map the DTB at higher-half. Either works.

## 6. API shape

```rust
/// Physical frame, 4 KiB aligned.
#[repr(transparent)]
pub struct PhysFrame(usize);

impl PhysFrame {
    pub const fn from_addr(addr: usize) -> Option<Self>;
    pub const fn addr(self) -> usize;
}

pub fn alloc() -> Option<PhysFrame>;
pub fn alloc_zeroed() -> Option<PhysFrame>;  // useful for new PTs
pub fn free(frame: PhysFrame);
pub fn stats() -> Stats;

pub struct Stats {
    pub total: usize,
    pub in_use: usize,
    pub free: usize,
}
```

Open question: is `alloc()` free function (uses a static allocator)
or method on a struct (passed around)? Method-on-struct is cleaner
for testing but pollutes every caller. Free function with a static
matches what the rest of the kernel already does (`INTERN_TABLE`,
`UART`, etc.).

**Recommendation**: free function backed by a static `Mutex<Bitmap>`,
matching the pattern. Bitmap struct + tests live in `kernel-core`.

## 7. The linear map question — big one

The frame allocator returns physical addresses. The caller often
wants to *write to* the frame (zero it, install a PTE, etc.). With
the MMU on and identity unmapped, the kernel can't access raw
physical addresses.

Two options:

### (a) Linear map all of physical RAM at a higher-half VA range

Pick a base, say `0xffff_ffd0_00000000`, and map all RAM there with
1 GiB Sv39 huge pages. Anything at PA `p` is accessible at VA
`p + LINEAR_OFFSET`. The kernel just shifts the address when it
wants to touch frame contents.

Pros: simple, one-time setup, every frame is reachable without
per-frame mapping work.

Cons: uses up some VA space (negligible — we have 256 GiB of
canonical-high). Every PT walk goes through the linear map's
top-level PTE, but that's cheap.

### (b) Per-frame `kmap`-style transient mapping

Borrow a VA, map a frame to it, access, unmap. Linux did this on
32-bit i386 because the VA space was tiny. Sv39 has no such
constraint — we have a quarter-terabyte of higher-half. Linear map
wins.

### (c) Use 1 GiB huge-page linear map for "all physical RAM"

Specifically: on QEMU `virt`, RAM is at `[0x80000000, 0x80000000 + RAM)`.
We can map `[LINEAR_OFFSET + 0x80000000, ...)` to this range with
one 1 GiB leaf (covers up to 1 GiB) or several. A single 1 GiB leaf
for 128 MiB to 1 GiB of RAM is one entry in the root table.

### Recommendation

**(c)**. One 1 GiB leaf in the root page table mapping the kernel's
view of all physical RAM at higher-half. `LINEAR_OFFSET = 0xffff_ffd0_00000000`
or similar. We need to add it to the boot table.

This unlocks frame allocator + heap + future per-process page tables
basically for free.

## 8. Instrumentation

Every alloc/free is observable. Metrics:

- `snitchos.frames.allocated_total` (Counter): every successful
  `alloc`.
- `snitchos.frames.freed_total` (Counter): every `free`.
- `snitchos.frames.in_use` (Gauge): current count.
- `snitchos.frames.free` (Gauge): current free count.
- `snitchos.frames.alloc_failed_total` (Counter): every `alloc`
  that returned `None`.

**Important constraint** (same as the trap handler): the allocator
can't call back into `register_counter` / `emit_metric` from inside
its own lock — those would lock the intern table and then the
virtio-console mutex, and any code holding the allocator lock that
also wants to emit a frame would deadlock. Same fix as the IRQ
handler: maintain atomic counters inside the allocator, drain from
the heartbeat thread.

## 9. Where things live

- **`kernel-core::frame`** (new module): `Bitmap` type, bit-twiddling,
  reservation logic. Pure data — host-tested.
- **`kernel/src/frame.rs`** (new module): the static `Mutex<Bitmap>`,
  the global `alloc`/`free` API, init from DTB, instrumentation.

Same pattern as `kernel-core::intern` + `kernel/src/tracing.rs`.

## 10. Decisions to lock in before the plan

1. **Tracking strategy**: bitmap, linked list, or buddy.
   (My take: bitmap.)
2. **Bitmap storage**: static array (which cap?), bumper, or
   something else.
   (My take: static array sized for 4 GiB cap, ~128 KiB of bss.)
3. **Linear map**: yes/no, and at what VA base.
   (My take: yes, 1 GiB huge-page leaf at `0xffff_ffd0_00000000`.)
4. **API shape**: free functions on a static, or struct passed around.
   (My take: free functions, matches existing patterns.)
5. **Instrumentation**: atomic counters drained by heartbeat, or
   sync emit at every alloc/free?
   (My take: deferred / atomic counters, matches IRQ handler.)
6. **Init order**: before or after `unmap_identity`? (We need DTB
   access at init time. Either map DTB at higher-half too, or do
   frame-allocator init before `unmap_identity`.)
   (My take: init before `unmap_identity`. Simplest.)
7. **What counts as "in use" at boot?** Kernel image bounds, the
   four `BOOT_PT_*` tables (currently inside the image — verify),
   DTB region, bitmap itself. Anything else?

## 11. Things we're explicitly not doing in step 3

- **Per-frame metadata** (Linux's `struct page` thing). Just a bitmap.
- **NUMA / memory zones**. Single flat region.
- **Contiguous multi-frame allocation**. Single frame only.
- **Frame reference counting**. Caller owns the frame until free.
- **Lazy / on-demand bitmap init**. Init it all up front.

These are real concerns once we have userspace + drivers; punting
them keeps step 3 small.

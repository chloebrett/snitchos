  - **Priority-aware preemption** (`should_preempt`, pure, **3/3 mutants**):
    `maybe_preempt` now only reschedules when a ready task is of *equal-or-higher*
    effective priority. The timer time-slices *within* a level and yields to a
    higher arrival, but never demotes High→Low (that was priority inversion in
    v0.8's plain time-slicing). Aging lets a starved low task reach the running
    level and so preempt even a higher-priority hog.
  - **Demo** (`workload=priorities`, host-tested bootargs arm, 1/1 mutant): a
    High CPU-bound `greedy` (the `user_hog` loop) + a Low cooperative `worker_b`.
    Scenario `priorities-ordered-but-fair`: `greedy` dominates CPU time by 10x+
    (priority respected) **and** `worker_b.runs ≥ 2` (aging rescues it from
    starvation). 6 s, green.

> **Why CPU-bound High + measure CPU time (not run-count):** two *cooperative*
> workers alternate ~1:1 regardless of priority — when one yields, the other is
> the only ready peer, so there's no *choice* for priority to express, and
> voluntary yields aren't priority-gated. Priority only bites under contention/
> saturation. A CPU-bound High task (never yields) + priority-aware preemption is
> the minimal setup where priority and aging both show, measured by CPU time
> (the High task holds the CPU for long stretches; the Low one only gets brief
> aged-in turns). The hart's idle/`hart_1_main` task still naps and halves
> throughput, but CPU-time dominance holds regardless — so no global idle-priority
> change was needed.

- **Step 5 (priority on the wire) — optional, not done.** Append a `priority`
  field to `ThreadRegister` (postcard-positional) + `OwnedFrame::from_borrowed`,
  and surface it as an OTLP attribute so Grafana/Tempo can group/colour by
  priority. Pure observability upgrade; the per-task metrics already tell the

The scheduler stops treating all ready tasks equally: each task has a priority,
and the scheduler runs the highest-priority ready task. **Aging** keeps low
tasks from starving — the longer a task waits, the higher its *effective*
priority climbs, so it eventually runs even against a steady stream of
higher-priority work. The headline: **"watch the scheduler choose — and watch it
be fair anyway."**

## Why now (before v0.9 IPC), and why *without* inheritance

- **Now:** the preemption substrate (v0.8) is fresh and proven; priorities only
  change *which* ready task `reschedule` picks — the timer gate, the layered
  switch, and the `SPP==User` reasoning are all untouched. Small, well-scoped,
  with a strong observability demo.
- **Without inheritance:** priority *inversion* (a low task holding a resource a
  high task needs) only exists once there's a resource to block on — i.e. **IPC
  (v0.9)**. Building inheritance now would be solving a problem we don't have
  yet. v0.9 introduces block/wake and is the right place to make endpoints
  priority-aware (inheritance/donation) from birth. v0.8b deliberately stops at
  *which task runs*, not *who is blocked on whom*.

## The crux: pick-by-effective-priority, not FIFO

Today the per-hart runqueue is a single FIFO; `reschedule` does `pop_front`.
v0.8b replaces that single pick with **"run the ready task of highest *effective*
priority, ties broken by longest wait."** Effective priority = base priority
boosted by how long the task has waited (aging). Two pure, host-tested functions
carry the policy; the kernel only builds the candidate list and applies the
choice.

**Why a single scanned list, not N FIFO queues.** Per-level FIFO queues give
clean round-robin *within* a level but make aging awkward (a task sits in its
base-level queue while its effective priority climbs above that level —
promotion means moving it between queues on a timer). A single ready list scanned
for max effective priority makes aging intrinsic: a long-waiting low task simply
*computes* a higher effective priority and wins the next scan. The scan is O(ready
tasks) per switch; with a handful of tasks that's nothing (contrast the OOM
lesson, which was an O(words) bitmap scan *per heartbeat* — a different scale).
Ties broken by longest wait give round-robin-within-level for free.

## Decisions (proposed — confirm before coding)

| decision | choice |
|---|---|
| Levels | `enum Priority { Low, Normal, High }` (3); default `Normal`. Higher runs first. |
| Pick policy | highest **effective** priority ready task; tie → longest wait (FIFO-fair) |
| Aging | effective priority rises +1 level per `AGING_STEP_TICKS` waited, capped at `High` |
| Runqueue shape | **single ready list per hart**, scanned for max effective priority (not N FIFOs) |
| Decision logic | two pure host-tested fns in `kernel_core::sched`: `aged_priority` + `pick_next` |
| Per-task state | `Task.priority: Priority` + `enqueued_tick: u64` (when it last entered the ready set) |
| Anti-starvation | aging only (no inheritance, no separate sweep) |
| Wire format | none new — priorities ride existing frames; `ThreadRegister` *may* gain a `priority` field (append-only) if the demo wants it on the trace |
| Telemetry | per-task `runs_total` already exists; add `snitchos.sched.priority_inversions_avoided`? **No** — defer; the demo reads per-task runs + the aging visibly |
| Quanta | **single `QUANTUM_TICKS`** for v0.8 carries over; per-priority quanta deferred (the plan's original Step 6 idea — not needed for the static-priority demo) |

## Steps

### Step 1: `Priority` + `aged_priority` (pure, host-tested)

**Acceptance criteria**: a `Priority` enum (`Low`/`Normal`/`High`) and a pure
`aged_priority(base: Priority, waited_ticks: u64, step: u64) -> u8` that returns
the base level with `+1` per `step` ticks waited, saturating at `High`. Boundary
(`waited == step` → one boost), `step == 0` guard (no div-by-zero / treat as no
aging), and the cap are all covered. No kernel change. *Confirm before coding.*
**RED**: `kernel-core` unit tests: no boost below `step`; one boost at `step`;
saturates at `High`; a `Low` task out-ages a `Normal` one only past the cap
boundary; `step==0` doesn't panic.
**GREEN**: the arithmetic (mirror `quantum_expired`'s `checked_*` discipline).
**MUTATE**: the `>=` boundary, the saturating cap, the `step==0` guard.

> **PR boundary** — pure aging policy.

### Step 2: `pick_next` (pure, host-tested)

**Acceptance criteria**: a pure
`pick_next(candidates: &[Candidate], now: u64, step: u64) -> Option<TaskId>`
where `Candidate { id, base: Priority, enqueued_tick }`. Returns the id of the
highest `aged_priority`; ties → smallest `enqueued_tick` (longest wait); empty →
`None`. *Confirm before coding.*
**RED**: highest base wins; an aged low task overtakes a fresh normal one; equal
effective → longest waiter; empty → `None`; single candidate → itself.
**GREEN**: the scan + comparison reusing `aged_priority`.
**MUTATE**: the comparison direction, the tie-break, the empty case.

> **PR boundary** — pure pick policy. The whole scheduling decision is now
> host-tested off-target.

### Step 3: Kernel wiring — `Task.priority` + `enqueued_tick`, `reschedule` uses `pick_next`

**Acceptance criteria**: `Task` gains `priority` (set at spawn; default `Normal`)
and an `enqueued_tick` updated whenever the task (re-)enters the ready set.
`reschedule` builds the candidate list from the hart's ready tasks and switches
into `pick_next`'s choice instead of `pop_front`. Existing cooperative + preempt
behaviour is **unchanged when all tasks share the default priority** (the full
suite is the regression gate). `spawn`/`spawn_on` gain a priority (or a
`spawn_with_priority` variant; default-priority wrappers keep call sites stable).
*Confirm before coding — touches the hot path.*
**RED**: the existing suite must stay green with everything at `Normal`
(regression). New behaviour is proven in Step 4.
**GREEN**: thread priority + enqueued_tick through `Task`/`spawn`; replace the
`pop_front` pick with the candidate-scan + `pick_next`.
**MUTATE**: kernel-side covered transitively by itest (bare-metal); the pure
pick is already mutated.
**Done when**: `itest --repeat 10` green with all-Normal tasks.

> **PR boundary** — priorities wired, behaviourally inert until tasks differ.

## Anti-starvation is the whole risk

Pure static priority is trivial; the *only* thing that makes v0.8b more than a
toy is that **aging keeps it fair**. The `priorities-ordered-but-fair` scenario
is the load-bearing test: it asserts both halves (ordering *and* eventual
progress). Tune `AGING_STEP_TICKS` so a `Low` task reaches `High` after a
visible-but-bounded wait. Given the 1 s timer granularity (v0.8), pick a step of
a few seconds so the demo shows the low task waiting, then running.

## Explicitly out of scope (deferred)

- **Priority inheritance / donation** — v0.9-era (needs IPC's blocking primitive).
- **Per-priority quanta** — the v0.8 single quantum suffices for static levels.
- **Borg-style two-tier SLO scheduler** — post-v1.0 (the *real* priority story:
  latency-sensitive + batch tiers, counterfactual "why X not Y" analysis).
- **Dynamic priority APIs** (a task changing its own priority at runtime) — add
  if a workload needs it.

## Open questions to settle during impl

- **3 levels vs a numeric range.** 3 named levels reads well and bounds the test
  matrix; a numeric `u8` is more flexible but over-general for the demo. Lean 3.
- **`enqueued_tick` reset semantics.** Reset on every (re-)enqueue (aging counts
  *current* wait), or accumulate? Lean reset-on-enqueue — simpler, and matches
  "how long since this task last got a turn."
- **Where the candidate scan reads priority.** From the `Task` table under the
  scheduler lock (already held in `reschedule`). No new lock.
- **Demo: userspace or kernel workers?** Userspace `worker_a`-style at different
  priorities is the honest demo (and reuses the v0.8 fixtures); a kernel peer is
  simpler. Lean userspace.

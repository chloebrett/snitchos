# v0.3 — interrupts & clock

Wire up RISC-V S-mode trap handling, drive a periodic timer interrupt
via SSTC, refactor the heartbeat loop from busy-spin to wfi+IRQ, and
get the first histogram metric on the wire.

## Why this milestone

v0.2 made the kernel observable. v0.3 makes it **responsive** — the
kernel sleeps until something happens, instead of burning a CPU
spinning on the cycle counter. First real kernel primitive: trap
entry/exit. Foundation for everything subsequent (scheduler, syscalls,
capability invocation, IPC — they all live on this machinery).

Per the roadmap: "Teaching the kernel what time it is."

## Architectural decisions

### A. SSTC for timer

RISC-V has two ways to program a timer interrupt from S-mode:

- **SBI `set_timer`**: `ecall` to M-mode firmware, which writes
  `mtimecmp`. Universal, but every arm is an M-mode round trip.
- **SSTC extension**: S-mode writes `stimecmp` directly via CSR.
  No trap to M-mode. Cleaner, faster.

The DTB's ISA-extensions list already includes `sstc`, and QEMU's
`virt` machine supports it. Use SSTC.

Fallback path if a future board lacks SSTC: detect via `misa` / DTB
and fall through to SBI. Not v0.3 — single-platform for now.

### B. Deferred work, not in-IRQ

The IRQ handler does the **minimum** — ack the interrupt by arming
the next timer, set a tick-pending flag, return. The actual work
(emit heartbeat span + metrics) runs in the main thread after `wfi`
returns.

Why not run heartbeat in the IRQ?

- Heartbeat touches `INTERN_TABLE`, `PRE_INIT_BUFFER`, and
  `virtio_console::CONSOLE`. All `spin::Mutex`. If main is holding
  any of them when timer fires, IRQ handler trying to relock →
  deadlock (spin::Mutex isn't reentrant).
- This is the corner already documented in `plans/scaling-corners.md`
  under "Locks and interrupts (deadlock at v0.3)."

Deferred work keeps the lock graph trivial. IRQ handler is purely
control-plane.

### C. Single-vector handler

RISC-V supports two trap-vector modes:

- **Direct** (`stvec.mode = 0`): all traps land at `stvec.base`.
  Handler reads `scause` to dispatch.
- **Vectored** (`mode = 1`): each interrupt cause has its own entry
  at `base + cause*4`. Faster dispatch for many interrupt sources.

Vectored mode is for kernels that handle dozens of IRQs and care
about per-IRQ latency. v0.3 has one IRQ (timer). Direct mode it is.

### D. No nested interrupts

When we enter the trap handler, the CPU automatically clears
`sstatus.SIE`. We **leave it cleared** for the handler's body —
interrupts stay disabled while we're servicing one. `sret` restores
the pre-trap `SIE` state from `sstatus.SPIE`.

Pros: no re-entrancy worry in the handler, simpler stack management,
no nested-IRQ stack overflow.

Cons: long handlers delay other interrupts. v0.3 handlers are tiny.

### E. Shared stack

Single hart, no preemption-during-IRQ, IRQ handler runs short. Use
the main kernel stack. No per-IRQ stack, no `sscratch` swap dance.

When SMP arrives we'll revisit — each hart needs its own kernel
stack, and the IRQ handler needs to find its own without colliding
with another hart's.

### F. `Clock` trait surface

```rust
pub trait Clock {
    /// Monotonic ticks since boot (raw cycle counter).
    fn now(&self) -> u64;
    /// Schedule a timer interrupt for `deadline` ticks since boot.
    /// Calling again before the deadline reprograms (and acks any
    /// previous pending timer).
    fn arm(&self, deadline: u64);
}
```

Just those two. `subscribe(callback)` could come later but isn't v0.3
— the IRQ dispatcher is hand-written.

Concrete impl: `SstcClock` using `rdtime` + write-to-`stimecmp`.
`SbiClock` as a fallback we don't implement yet.

## Trap frame layout

Saved on entry:

```
offset  field
  0     ra   (x1)
  8     sp   (x2)  — caller's sp, restored later
 16     gp   (x3)
 24     tp   (x4)
 32     t0   (x5)
 40     t1   (x6)
 48     t2   (x7)
 56     s0   (x8)
 64     s1   (x9)
 72     a0   (x10)
 80     a1   (x11)
 88     a2   (x12)
 96     a3   (x13)
104     a4   (x14)
112     a5   (x15)
120     a6   (x16)
128     a7   (x17)
136     s2   (x18)
144     s3   (x19)
152     s4   (x20)
160     s5   (x21)
168     s6   (x22)
176     s7   (x23)
184     s8   (x24)
192     s9   (x25)
200     s10  (x26)
208     s11  (x27)
216     t3   (x28)
224     t4   (x29)
232     t5   (x30)
240     t6   (x31)
248     sepc
256     sstatus
```

30 GP regs × 8 bytes + 2 CSRs × 8 bytes = 256 bytes. Round up to
288 for alignment headroom. `x0` is zero, no save.

Entry assembly:

1. Allocate trap frame on the current stack (`addi sp, sp, -288`).
2. Store each register to its offset.
3. Read `sepc` and `sstatus` into temps; store them.
4. `mv a0, sp` — pass the trap frame pointer as the first argument
   to the Rust dispatcher.
5. `call trap_handler` (Rust).
6. Restore `sepc` and `sstatus` from the frame.
7. Restore each register (reverse order).
8. `addi sp, sp, 288` — release the frame.
9. `sret` — return from S-mode trap.

Rust dispatcher signature:

```rust
#[no_mangle]
pub extern "C" fn trap_handler(frame: *mut TrapFrame) {
    let cause = scause_read();
    if cause.is_interrupt() && cause.code() == TIMER_INTERRUPT {
        handle_timer();
        return;
    }
    panic!("unhandled trap: cause = {:?}", cause);
}
```

## SSTC timer code

```rust
fn arm_timer(deadline: u64) {
    unsafe {
        core::arch::asm!("csrw stimecmp, {}", in(reg) deadline);
    }
}
```

That's it. One CSR write. The pending bit clears automatically once
`time < stimecmp`. Compare to SBI's `ecall` sequence — strictly
worse on a board that supports SSTC.

Note: the `stimecmp` CSR address is `0x14d`. Some assemblers may
not know the symbolic name yet; might need `csrw 0x14d, {}` as a
fallback. We'll know on first compile.

## Heartbeat refactor

Before:
```rust
let mut next = tracing::timestamp() + timebase_hz;
loop {
    while tracing::timestamp() < next {}
    { span!("kernel.heartbeat"); ... }
    next += timebase_hz;
}
```

After:
```rust
clock.arm(clock.now() + timebase_hz);
loop {
    while !TICK_PENDING.swap(false, Ordering::Relaxed) {
        unsafe { asm!("wfi") };
    }
    { span!("kernel.heartbeat"); ... }
    // arm_timer is re-armed inside the IRQ handler; nothing to do here.
}
```

`TICK_PENDING: AtomicBool`. IRQ handler sets it after arming the
next deadline.

The `swap(false)` is the atomic read-and-clear — handles the case
where wfi spuriously returned without an interrupt.

## IRQ handler body (Rust)

```rust
static TICK_PENDING: AtomicBool = AtomicBool::new(false);

const TIMER_INTERVAL_TICKS: u64 = 10_000_000; // 1 sec at 10 MHz

fn handle_timer() {
    let span = tracing::span_start("irq.timer");
    let t_start = tracing::timestamp();

    // Arm the next deadline (also acks the current interrupt).
    let next = t_start + TIMER_INTERVAL_TICKS;
    unsafe { core::arch::asm!("csrw stimecmp, {}", in(reg) next); }

    // Wake the main thread.
    TICK_PENDING.store(true, Ordering::Relaxed);

    let elapsed = tracing::timestamp() - t_start;
    tracing::emit_metric(IRQ_TIMER_DURATION, elapsed as i64);

    drop(span); // SpanEnd emitted here
}
```

Note: `span_start` and `emit_metric` both call `emit_frame`, which
locks `virtio_console::CONSOLE`. **This is precisely the IRQ↔lock
deadlock corner.** Two ways to handle it:

1. **Don't emit telemetry from the IRQ handler.** Set the flag,
   return. Main thread does both the heartbeat *and* the IRQ tracing
   when it wakes. Loses precision on `irq.duration_ticks` (we'd
   measure "wake to processing" rather than "trap entry to exit").
2. **Disable interrupts around the lock-taking code paths in main.**
   When main is holding a console-shared lock, IRQ can't fire. The
   IRQ handler can then take the lock freely. Pattern is well-known.

For v0.3 first cut: **option 1** (defer the metric to main thread).
Simpler, no new interrupt-disable machinery, gets us the milestone.
Revisit when we need accurate IRQ timing — probably v0.5+ when we
have a scheduler.

Revised handler:
```rust
fn handle_timer() {
    let t_start = tracing::timestamp();
    let next = t_start + TIMER_INTERVAL_TICKS;
    unsafe { core::arch::asm!("csrw stimecmp, {}", in(reg) next); }
    LAST_IRQ_DURATION.store(/* measured below */, Ordering::Relaxed);
    TICK_PENDING.store(true, Ordering::Relaxed);
    let elapsed = tracing::timestamp() - t_start;
    LAST_IRQ_DURATION.store(elapsed, Ordering::Relaxed);
}
```

Main thread reads `LAST_IRQ_DURATION` after wfi and emits the
histogram sample then.

## Histogram on the wire

v0.2 left `MetricKind::Histogram` as a placeholder. v0.3 actually
emits one.

Open question: what does a `Frame::Metric` with `name_id` pointing
at a histogram-registered name mean? Options:

a) **Send each observation as a Metric**. Host accumulates into
   buckets. Simple, wastes wire bytes for high-frequency
   observations.
b) **Add a `Frame::HistogramObservation` variant** with the value.
c) **Kernel buckets locally; emits a batched HistogramSnapshot
   periodically**. Most efficient. Most code.

For v0.3: **(a)**. Each timer IRQ emits one `Frame::Metric` with
the duration. Host (collector) accumulates into Prometheus histogram
buckets in `prom.rs`. Wire format unchanged.

Once we have many timer-like sources (v0.7 IPC latency, audio xruns
later), revisit toward (c).

## Plan for the IRQ-disable infrastructure (deferred)

When v0.3-revisit or v0.4 needs IRQ-safe locks:

```rust
pub struct IrqGuard { prev_sie: u64 }
impl IrqGuard {
    pub fn new() -> Self {
        let prev: u64;
        unsafe { core::arch::asm!("csrrci {}, sstatus, 2", out(reg) prev); }
        // bit 1 of sstatus is SIE; csrrci clears + returns previous
        Self { prev_sie: prev }
    }
}
impl Drop for IrqGuard {
    fn drop(&mut self) {
        if self.prev_sie & 2 != 0 {
            unsafe { core::arch::asm!("csrsi sstatus, 2"); }
        }
    }
}
```

Then critical sections become `let _g = IrqGuard::new(); /* lock + work */`.

Not needed for v0.3 because we deferred the IRQ telemetry.

## Open questions / deferred

- **Vectored mode.** Not v0.3. Useful when we have many IRQs.
- **External interrupts** (PLIC). The Platform-Level Interrupt
  Controller is how devices like UART RX or virtio queues raise
  interrupts. Not v0.3 (we have no devices generating IRQs other
  than the timer).
- **Software interrupts** (IPI). For cross-hart signaling. v0.5+
  when SMP arrives.
- **Save FP registers?** Kernel is integer-only per the policy in
  `docs/concepts-and-findings.md`. Don't save FP. If we ever break
  the no-FP policy in kernel code, the trap path must save them.
- **Trap from M-mode / U-mode.** v0.3 only handles S-mode interrupts
  (timer). Exception handling and userspace traps come at v0.6a.

## Success criteria

- Kernel boots, `cargo xtask up + collect` flow works.
- After "I am alive — entering heartbeat", main thread sleeps in
  `wfi` (verify with `top` showing low CPU on QEMU, or by adding
  a "wfi entered" event).
- Heartbeat fires once per second via timer, not by busy-spinning.
- Grafana dashboard shows:
  - `snitchos.heartbeat.count` ticking up as before.
  - **New panel**: `snitchos.irq.timer.duration_ticks` as a
    histogram (or initially a stat showing latest value).
- IRQ handler measurably takes ≪ 1 ms (in cycles, well under
  `timebase_hz / 1000`).

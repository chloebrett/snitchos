# v0.2 — Grafana arrives

Replace the text-only host-reader with a real observability stack:
Tempo (traces) + Prometheus (metrics) + Grafana (dashboards). Same
kernel workload (boot tree + heartbeat); the same frames now flow into
a stack that looks like a product.

## Why this milestone

v0.1 proved the wire format and the channel work end-to-end. v0.2 makes
it *useful* — span trees in a real trace viewer, metrics in dashboards.
First public-content moment: a Grafana screenshot that visibly looks
like "SnitchOS is a real OS doing real work."

Per the roadmap: "From printf to a real observability stack."

## Architectural decisions

### A. Single binary `collector`

`host-reader` gets renamed `collector` and gains output-mode flags. Same
decode loop, multiple output sinks running concurrently:

- `--text` — print decoded frames to stdout (current host-reader
  behavior; for ad-hoc debugging).
- `--otlp <endpoint>` — POST spans to Tempo's OTLP/HTTP receiver.
- `--prometheus <port>` — serve `/metrics` text on `0.0.0.0:<port>`.

Flags compose: `--text --otlp http://localhost:4318 --prometheus 9091`
runs all three.

xtask exposes two convenience subcommands over the same binary:
- `cargo xtask collect` — full output (otlp + prometheus, maybe text).
- `cargo xtask reader` — equivalent to `--text` only, for muscle-memory.

### B. Protocol change — typed metrics

Add `MetricKind` and `MetricRegister` to the wire format. Counters,
gauges, and histograms are *semantically* different; faking counters
as gauges and synthesizing via Prometheus `rate()` is wrong (counter
resets get hidden, monotonic guarantees lost).

```rust
#[derive(Serialize, Deserialize, PartialEq, Eq, Debug, Clone, Copy)]
pub enum MetricKind {
    Counter,    // monotonically increasing
    Gauge,      // snapshot value
    Histogram,  // distribution (placeholder; bucket encoding TBD)
}

pub enum Frame<'a> {
    // ... existing variants ...
    MetricRegister {
        name_id: StringId,
        kind: MetricKind,
    },
}
```

Same "describe once, use many" pattern as `StringRegister`. The
`Frame::Metric` variant stays unchanged — collector looks up the kind
by `name_id`.

Kernel adds three ergonomic helpers:
```rust
tracing::register_counter(name)   -> StringId
tracing::register_gauge(name)     -> StringId
tracing::register_histogram(name) -> StringId
```
Each registers the string (intern table) + emits the `MetricRegister`
frame the first time. Subsequent `emit_metric(name_id, value)` calls
just send the `Frame::Metric`.

### C. OTLP/HTTP, not gRPC

Use `opentelemetry-otlp` with `http-proto` feature. HTTP is curlable for
debugging, smaller dep surface than `tonic`, and `opentelemetry-otlp`
handles both transports under the same Rust API. Effort is the same
either way; bias toward debuggability.

Tempo accepts OTLP/HTTP at `:4318/v1/traces`.

### D. Wall-clock anchor at session start

The kernel emits raw u64 cycle counts (`t` field). Tempo wants wall-clock
timestamps. Conversion:

- **On `Hello` arrival** (first frame of a session): record
  `session_start_wallclock = Instant::now()` and `t_anchor =
  hello_frame.t` (or, more precisely, the smallest `t` from the
  pre-init burst — see quirk below).
- **For every subsequent frame:** `wall_time = session_start_wallclock +
  (frame_t - t_anchor) * 1_000_000_000 / timebase_hz`.

This means each session is anchored to the *moment the collector
connected*. Tempo retains both sessions' spans at correct (non-overlapping)
wall-clock times. Drift across a single session is microseconds; we
don't care.

**Quirk: pre-init spans land slightly before `session_start_wallclock`.**
Frames emitted before `Hello` (kernel.boot's SpanStart, etc.) have
smaller `t` values than Hello. After conversion they appear in Tempo
at timestamps a few microseconds *before* the anchor. Accepted — it's
the truth of when those frames were created.

### E. Kernel disconnect → collector exits

When the kernel disconnects (QEMU shutdown, kernel panic, etc.):

- `decode_stream` returns `Ok` at EOF.
- Collector flushes any in-flight spans, prints a message
  ("kernel disconnected; restart with `cargo xtask collect`"), exits.
- **The docker stack stays up.** Tempo keeps the session's traces;
  Prometheus retains the last scrape; Grafana still shows everything.
- User starts the collector again to pick up a new session.

This is the right shape for "many small kernel sessions, one persistent
dashboard."

### F. OTLP export cadence

Per-frame export for v0.2. SpanEnd → build OTLP span → POST. Easy to
swap for batched export later (the export call site is one place).

Worth batching if/when:
- Heartbeat frequency grows past ~10 Hz.
- The kernel emits in bursts (e.g., post-IPC spans flooding).

Until then, simplicity wins.

### G. Span reassembly state

```rust
struct State {
    timebase_hz: u64,                          // from Hello
    session_anchor: Option<SessionAnchor>,     // wall-clock anchor
    strings: HashMap<u32, String>,             // StringRegister
    metric_kinds: HashMap<u32, MetricKind>,    // MetricRegister
    open_spans: HashMap<u64, SpanStart>,       // SpanStart → wait for End
    counters: HashMap<u32, i64>,               // last value per counter
    gauges: HashMap<u32, i64>,                 // last value per gauge
}
```

- `SpanStart` → insert into `open_spans`.
- `SpanEnd` → remove from `open_spans`, build OTLP span with
  `start = wall_time(start_t)`, `end = wall_time(end_t)`,
  `name = strings[name_id]`, parent linkage via `parent`. POST.
- Spans never matched by an End (kernel crashed mid-span) leak in the
  map. Accept the leak for v0.2.
- `Metric` → bump counter / overwrite gauge by `name_id`.

## docker-compose stack sketch

`stack/docker-compose.yml` with three services + provisioning configs:

```yaml
services:
  tempo:
    image: grafana/tempo:latest
    command: -config.file=/etc/tempo.yaml
    volumes:
      - ./tempo.yaml:/etc/tempo.yaml
    ports:
      - "4318:4318"  # OTLP/HTTP
      - "3200:3200"  # Tempo HTTP API

  prometheus:
    image: prom/prometheus:latest
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml
    ports:
      - "9090:9090"

  grafana:
    image: grafana/grafana:latest
    environment:
      - GF_AUTH_ANONYMOUS_ENABLED=true
      - GF_AUTH_ANONYMOUS_ORG_ROLE=Admin
    volumes:
      - ./grafana/provisioning:/etc/grafana/provisioning
    ports:
      - "3000:3000"
```

- `tempo.yaml` configures Tempo for local storage + the HTTP OTLP receiver.
- `prometheus.yml` scrapes `host.docker.internal:9091/metrics` every 5s.
- `grafana/provisioning/datasources.yml` points at both backends.
- `grafana/provisioning/dashboards/snitchos.json` is the live dashboard.

Stack files live under `stack/` at the repo root.

## Collector crate sketch

```
collector/
├── Cargo.toml          deps: protocol (std), opentelemetry-otlp,
│                              tiny_http or hyper, clap, anyhow
└── src/
    ├── main.rs         arg parsing → connect → decode loop
    ├── state.rs        State struct, frame handlers
    ├── otlp.rs         OTLP/HTTP exporter
    └── prom.rs         /metrics text formatter + HTTP server
```

Main loop:

```rust
fn main() {
    let args = Args::parse();
    let state = Arc::new(Mutex::new(State::new()));

    if args.prometheus.is_some() {
        spawn_metrics_server(state.clone(), args.prometheus.unwrap());
    }
    let exporter = args.otlp.as_ref().map(otlp::Exporter::new);

    let mut stream = UnixStream::connect(SOCKET_PATH)?;
    protocol::stream::decode_stream(&mut stream, |frame| {
        let mut state = state.lock().unwrap();
        if args.text {
            println!("{frame:?}");
        }
        if let Some(span) = state.handle(frame) {
            if let Some(exporter) = &exporter {
                exporter.export(span);
            }
        }
    })?;

    println!("kernel disconnected; restart with `cargo xtask collect`");
    Ok(())
}
```

`state.handle(frame)` returns `Option<CompletedSpan>` for SpanEnd
frames; other frames update state and return None.

## Kernel-side changes

New tracing APIs:

```rust
tracing::register_counter(name)   -> StringId
tracing::register_gauge(name)     -> StringId
tracing::register_histogram(name) -> StringId
tracing::emit_metric(name_id, value)
```

`register_*` emits a `Frame::StringRegister` on first sight and a
`Frame::MetricRegister` declaring the kind. Subsequent `emit_metric`
just sends `Frame::Metric { name_id, value, t: timestamp() }`.

Heartbeat-loop updates:

```rust
let heartbeat_count = tracing::register_counter("snitchos.heartbeat.count");
let intern_used = tracing::register_gauge("snitchos.intern.strings_used");
let time_ticks = tracing::register_gauge("snitchos.time.ticks");
let dropped = tracing::register_counter("snitchos.dropped_frames");

let mut count: i64 = 0;
loop {
    while tracing::timestamp() < next {}
    {
        span!("kernel.heartbeat");
        count += 1;
        tracing::emit_metric(heartbeat_count, count);
        tracing::emit_metric(intern_used, current_intern_count() as i64);
        tracing::emit_metric(time_ticks, tracing::timestamp() as i64);
    }
    next += timebase_hz;
}
```

## xtask additions

```
cargo xtask build                  # build the kernel ELF
cargo xtask up                     # build kernel + run in QEMU (just the kernel)
cargo xtask collect [-- <flags>]   # build + run collector (full output)
cargo xtask reader                 # build + run collector --text (text-only debug)
cargo xtask stack up               # docker compose up -d the stack
cargo xtask stack down             # docker compose down
cargo xtask stack logs             # docker compose logs -f
```

`up` stays kernel-only — the unix-y "each command does one thing" feel.

## Open questions / deferred

- **Histogram bucket encoding.** v0.2 reserves the `Histogram` variant
  but doesn't define how buckets travel on the wire. First histogram
  use is v0.3 (IRQ duration); pin down the encoding then.
- **Span attributes.** OTLP supports attribute maps per span. Our
  `Frame::SpanStart` has "an open spot for attributes." When we add it,
  collector translates. Not v0.2.
- **OTLP/gRPC.** Stay on HTTP for v0.2. Revisit if HTTP proves wonky.
- **Trace context across IPC.** Spans don't yet cross process boundaries
  — no userspace processes yet. Real at v0.7.
- **Production-grade error handling in collector.** v0.2 can `unwrap` /
  exit on errors.
- **Logs.** Loki could handle log lines, but kernel `println!` text
  isn't structured enough to be useful there. Skip for v0.2.
- **Multiple kernel sessions in Grafana time-range view.** Each session
  gets its own anchor; sessions don't overlap. Looks correct but
  doesn't visually highlight session boundaries. Could emit a marker
  event at each session start — defer until the need is felt.

## Success criteria

Running `cargo xtask stack up`, then `cargo xtask up` (kernel + QEMU),
then `cargo xtask collect`:

- Tempo shows the `kernel.boot` span with `console_init` /
  `telemetry_init` children.
- Tempo shows a stream of `kernel.heartbeat` spans, one per second.
- Prometheus / Grafana shows `snitchos_heartbeat_count` ticking up.
- A Grafana dashboard with: spans panel (Tempo), metric panels
  (Prometheus), both refreshing live.
- Stop QEMU → collector exits with "kernel disconnected" message;
  stack stays up; previous session's traces still visible in Grafana.
- Start QEMU again → run `cargo xtask collect` again → new session
  appears, doesn't overlap with the previous one.

Screenshot of that dashboard is the v0.2 artifact.

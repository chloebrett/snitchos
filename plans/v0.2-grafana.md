# v0.2 — Grafana arrives

Replace the text-only host-reader with a real observability stack:
Tempo (traces) + Prometheus (metrics) + Grafana (dashboards). Same
kernel workload (boot tree + heartbeat); the same frames now flow into
a stack that looks like a product.

## Why this milestone

v0.1 proved the wire format and the channel work end-to-end. v0.2 makes
it *useful* — span trees in a real trace viewer, metrics in dashboards.
First public-content moment: a Grafana screenshot that visibly looks
like "SnitchOS is a real OS doing real work."

Per the roadmap: "From printf to a real observability stack."

## Architectural decisions

### A. host-reader vs new collector

host-reader is a debug/text tool. Don't load it with OTLP export logic.

**New crate `collector`** is the production-grade consumer:

- Connects to `/tmp/snitch-telemetry.sock` (same as host-reader).
- Decodes via `protocol::decode_stream`.
- Maintains the string intern table from `StringRegister` frames.
- Reassembles span pairs and exports OTLP to Tempo.
- Tracks `Metric` frames and serves them as Prometheus text on
  `http://localhost:9090/metrics` (or similar).

host-reader stays for ad-hoc inspection; collector is what you run
when you want a real dashboard.

### B. Factor decode_stream into `protocol`

Currently in `host-reader/src/main.rs`. Both host-reader and collector
need the read-bytes-and-decode-frames loop. Move it into `protocol`
with a `std` feature flag (kernel stays no_std).

```toml
# protocol/Cargo.toml
[features]
default = []
std = []
```

```rust
// protocol/src/lib.rs
#![cfg_attr(not(feature = "std"), no_std)]

#[cfg(feature = "std")]
pub mod stream {
    // try_decode_frame + decode_stream live here
}
```

host-reader and collector both enable `std`.

### C. OTLP transport: HTTP, not gRPC

OTLP has two transports — HTTP+protobuf and gRPC. Pick HTTP:

- One fewer dependency (no `tonic`, no `prost-build` codegen ceremony).
- Easier to debug (curl-able).
- Tempo accepts HTTP at `:4318/v1/traces`.

Use `opentelemetry-otlp` with the `http-proto` feature, or roll our own
minimal protobuf+POST. The crate is the v0.2-appropriate amount of
engineering; rolling it ourselves is gold-plating.

### D. Metric semantics

Open question: how does the host know whether a `Frame::Metric` is a
counter (monotonic) or a gauge (snapshot)? Three options:

1. **Naming convention** — names ending in `_total` are counters; else
   gauges. Standard Prometheus convention. Simple, no protocol change.
2. **Add a `MetricRegister` frame** declaring `(name_id, kind: Counter|Gauge|Histogram)` once per metric. Parallel to
   `StringRegister`. Cleaner but a protocol addition.
3. **Always-gauge** — every Metric is a gauge. Counters synthesized by
   `rate()`/`increase()` in Grafana. Honest.

**Pick (3)** for v0.2. Refine to (2) when metric semantics actually
matter. Naming convention is fragile.

### E. Span reassembly state

The collector holds:

```rust
struct State {
    timebase_hz: u64,             // from Hello
    strings: HashMap<u32, String>,  // from StringRegister
    open_spans: HashMap<u64, SpanStart>,  // id -> start info
    metrics: HashMap<u32, (i64, Instant)>, // name_id -> latest value
}
```

On `SpanStart`: insert into `open_spans`.
On `SpanEnd`: lookup → compute duration → build OTLP span → export → remove.

What about spans that never get a SpanEnd (kernel crashed mid-span)?
Tempo's OTLP doesn't tolerate open-ended spans; we'd just leak them in
the map. Add a sweep that closes stale spans after some timeout, or
accept the memory leak for v0.2. **Accept the leak.**

### F. Tick → time conversion

Kernel sends raw u64 cycle counts (`t` field). Collector converts to
real time using the timebase from the `Hello` frame:

```rust
let nanos = t.checked_mul(1_000_000_000)
              .and_then(|x| x.checked_div(timebase_hz))
              .unwrap_or(0);
```

Boot reference point is "cycle 0 at kernel boot." For Tempo spans we
need wall-clock-ish timestamps; use collector's `Instant::now()` at
Hello receipt as the anchor:

```rust
absolute_ns = hello_received_at + (t - hello_first_t) * 10^9 / timebase_hz
```

This drifts if the kernel and host clocks disagree, but v0.2 doesn't
need precision better than "spans look right in Grafana."

## docker-compose stack sketch

`stack/docker-compose.yml` with three services + provisioning configs:

```yaml
services:
  tempo:
    image: grafana/tempo:latest
    command: -config.file=/etc/tempo.yaml
    volumes:
      - ./tempo.yaml:/etc/tempo.yaml
    ports:
      - "4318:4318"  # OTLP/HTTP
      - "3200:3200"  # Tempo HTTP API

  prometheus:
    image: prom/prometheus:latest
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml
    ports:
      - "9090:9090"

  grafana:
    image: grafana/grafana:latest
    environment:
      - GF_AUTH_ANONYMOUS_ENABLED=true
      - GF_AUTH_ANONYMOUS_ORG_ROLE=Admin
    volumes:
      - ./grafana/provisioning:/etc/grafana/provisioning
    ports:
      - "3000:3000"
```

Grafana provisioning: a `datasources.yml` pointing at Tempo (3200) and
Prometheus (9090), plus a dashboard JSON.

`prometheus.yml` scrapes `host.docker.internal:9091/metrics` (the
collector running on the host) every 5 seconds.

Configs live under `stack/` in the repo.

## Collector crate sketch

```
collector/
├── Cargo.toml         deps: protocol (std), opentelemetry-otlp,
│                            tiny_http or hyper for /metrics, anyhow
└── src/
    └── main.rs        connect socket → decode_stream → handle each frame
```

Main loop:

```rust
fn main() {
    let stream = UnixStream::connect("/tmp/snitch-telemetry.sock")?;
    let state = Arc::new(Mutex::new(State::new()));

    // /metrics HTTP server on a thread.
    spawn_metrics_server(state.clone(), 9091);

    // Decode loop on main thread.
    protocol::stream::decode_stream(&mut stream, |frame| {
        let mut state = state.lock().unwrap();
        state.handle(frame);  // updates spans/strings/metrics, calls otlp exporter
    })?;
}
```

Three concerns:

1. **OTLP export.** Build `ExportTraceServiceRequest`, POST as protobuf.
2. **Prometheus exposition.** Implement `/metrics` text format. Format
   the gauges from `state.metrics`.
3. **Shutdown.** When kernel disconnects, decode_stream returns Ok(EOF).
   Drain any in-flight spans and exit.

## Kernel-side changes

Emit metrics in the heartbeat loop:

```rust
loop {
    while tracing::timestamp() < next {}
    {
        span!("kernel.heartbeat");
        tracing::emit_metric("snitchos.heartbeat.count", heartbeat_count);
        heartbeat_count += 1;
    }
    next += timebase_hz;
}
```

Add `tracing::emit_metric(name: &'static str, value: i64)` — registers
the string, then emits `Frame::Metric { name_id, value, t: timestamp() }`.

Other metrics worth emitting:

- `snitchos.boot.dropped_frames` (from the pre-init buffer)
- `snitchos.time.ticks` (every heartbeat — lets the host plot
  "ticks vs wall-clock" to see drift)
- `snitchos.intern.strings` (count of registered strings — visibility
  into the table fill rate)

## Open questions / deferred

- **Logs.** Tempo handles traces. Prometheus handles metrics. Loki
  could handle log lines. Skip for v0.2; kernel `println!` text isn't
  structured enough to be useful in Loki anyway.
- **OTLP/gRPC.** HTTP is fine for v0.2. Revisit if Tempo's HTTP path
  proves wonky.
- **Span attributes.** OTLP spans support attribute maps. Our protocol
  has "an open spot for attributes (empty in v0.1)." When we add them,
  collector translates. Not v0.2.
- **Trace context across IPC.** Spans don't yet cross process
  boundaries — there are no other processes. Becomes real at v0.7.
- **Production-grade error handling.** v0.2 collector can `unwrap` /
  exit on errors. Real productionization is a later concern.
- **Histograms.** v0.2 is gauge-only. Histograms come with
  metric-type frames (the deferred option B above).

## Success criteria

Running `cargo xtask stack up`, then `cargo xtask up` (kernel + QEMU),
then `cargo xtask collector`:

- Tempo shows the `kernel.boot` span with `console_init` /
  `telemetry_init` children.
- Tempo shows a stream of `kernel.heartbeat` spans, one per second.
- Prometheus / Grafana shows `snitchos_heartbeat_count` ticking up.
- A Grafana dashboard with: spans panel (Tempo), metric panel
  (Prometheus), both refreshing live.

Screenshot of that dashboard is the v0.2 artifact.

# v0.9b — `call`/`reply` over capabilities (RPC + single-use capabilities)

**Work lands on:** `main` (no feature branches — see CLAUDE.md)
**Status:** Design complete, not started. Follows the v0.9 core (synchronous
`send`/`receive`, `plans/v0.9-ipc.md`) and builds directly on its endpoint
table, block/wake, and trace-context machinery.

Request/response over the synchronous endpoint: a client `call`s and blocks for
the answer; the server `receive`s, does work, and `reply`s. The kernel mints a
**single-use reply capability** into the server at the rendezvous — the
project's **first cap-transfer** (the kernel inserting a capability into
*another* process's table) and the first **single-use capability**. The
headline: **"RPC needs a reply capability"** — and the trace finally **nests**
(the caller's span stays open across the callee's work).

## Why this milestone, and what it earns

- **The observability payoff.** v0.9's one-way `send` produces a *linked-but-
  disjoint* trace — `ipc.recv`'s parent is `ipc.send`, but the parent span has
  already closed by the time the child opens (the sender doesn't wait). `call`
  blocks the caller across the whole round-trip, so the caller's span
  *temporally contains* the callee's — the classic RPC flame-graph shape. Same
  trace-context mechanism as v0.9; what changes the *shape* is the blocking
  discipline.
- **The capability payoff: single-use capabilities.** A capability gains a
  **multiplicity** axis, orthogonal to its rights: `Persistent` (invoke
  repeatedly — `TelemetrySink`, `Endpoint`, `SpanSink`) vs. `Once` (invoke
  exactly once, then **consumed**). This is the affine/linear-capability idea.
  Reply is its first instance. It gives the `Handle` **generation** field —
  reserved-but-unused since v0.7b — a real, general purpose: **consuming a
  `Once` cap is a revocation**, and the generation bump stops a stale handle
  from aliasing whatever reuses the freed slot. Two long-reserved mechanisms
  (cap insertion, generation) come alive, in service of a *named concept* rather
  than a reply special-case.

**Deliberate restraint:** `Multiplicity` stays a **two-variant marker** with one
consumer (reply). No lifetime-policy framework, attenuation, or multiplicity
arithmetic until a second consumer earns it — the likeliest being the deferred
*cap-transfer-in-messages* (a moved cap is `consume-on-send`, the same step).

## Scope decision: `call` + `reply`, single-use reply caps

| in v0.9b | deferred (own follow-on) |
|---|---|
| `call` (send request + block for reply) + `reply` (answer + wake + consume) | `reply_recv` fusion (the one-syscall server loop) — small additive step once `call`/`reply` work |
| Single-use (`Once`) reply capability, kernel-minted into the server, consumed on `reply` | Userspace-directed cap-transfer (caps travelling *in* normal messages) — the second `Once` consumer |
| `receive` returns a reply handle (0 for a plain `send`, nonzero for a `call`) | `MemoryRegion` bulk transfer; the channel library |
| Nested-span RPC trace, asserted | Notifications (v0.9c) |
| Two processes, one hart (client + server time-sliced) | Cross-hart `call` (needs the cross-hart IPI wake) |

**`send`/`receive` stay.** A one-way `send` is still valid; `call` is the
request/response operation layered on the same endpoint. A server's single
`receive` handles both — the **reply handle it gets back** (0 vs nonzero) tells
it whether the message wants an answer. One server-loop shape, two client
disciplines (L4-style).

## Decisions locked in

| decision | choice |
|---|---|
| Capability multiplicity | New `Multiplicity::{Persistent, Once}` on `Capability` (default `Persistent`; all v0.7b/v0.9 caps unchanged). Invoking a `Once` cap performs the op **and consumes** the cap. Two variants only — see "deliberate restraint." |
| Consume = revoke | Consuming a `Once` cap frees its slot and **bumps the slot generation** (so every handle to it now resolves `Stale`). This is the consume step *and* the long-reserved revocation path. |
| Reply capability | `Object::Reply { caller: TaskId }`, granted `Once`. Names the specific blocked caller. A second `reply` on the same handle is `Stale` → `Denied` — one-shot falls out of the multiplicity, not a bespoke flag. |
| Cap-transfer mechanism | The kernel inserts the reply cap into the **server's** `CapTable` at the `call` rendezvous (authority to answer exactly this caller, exactly once). Userspace never names another process's table — only the kernel mints reply caps. |
| Syscalls | Append `Call = 9`, `Reply = 10` to `abi::Syscall` (positional — append only). `Call`: `a0`=endpoint handle, `a1..a4`=request words → blocks → returns reply words in `a1..a4`. `Reply`: `a0`=reply handle, `a1..a4`=response words. |
| `receive` return | `receive` additionally returns a **reply handle** in `a5`: `0` from a one-way `send`, a live `Once` reply-cap handle from a `call`. Backward-compatible with v0.9 receivers that ignore it. |
| Where the caller parks | `call` delivers the request (v0.9 `send_begin`) then `block_current`s awaiting reply; `reply` resolves+consumes the `Once` cap, delivers the response into the caller's pending slot, and `wake`s it — reusing v0.9's `pending` mailbox + block/wake. |
| Trace | Caller opens a span and `call`s inside it; it stays open across the round-trip (blocked), so the server's handling span nests under it. No new tracing mechanism. |
| Wire | Append `CapObject::Reply`; add `CapEventKind::Transferred` for the reply-cap grant (derivation-tree edge: reply *derived from* the call). A `Revoked`/`Consumed` kind for the consume is a stretch (the `reply` itself is already observable). Append-only. |
| Telemetry | `snitchos.ipc.calls_total`, `snitchos.ipc.replies_total`; in-flight gauge a stretch. Deferred-emission, as v0.9. |
| Lock discipline | Same as v0.9: compute under the endpoint lock, drop before block/wake. Cap insert/consume takes the target's `caps` lock — never held across a switch. |

## Steps

Every step is RED-GREEN-MUTATE-KILL MUTANTS-REFACTOR. No production code without
a failing test. *Present acceptance criteria and confirm before coding each
step.* PR boundaries marked.

### Step 1: Single-use (`Once`) capabilities — consume-on-invoke

**Acceptance criteria**: `kernel-core::cap` gains `Multiplicity::{Persistent,
Once}` on `Capability` (default `Persistent`), and `CapTable::consume(handle)`
— frees the slot and bumps its generation so the handle (and any copy) now
resolves `Stale`; a fresh insert reuses the slot at the new generation. The
generic invoke contract: a `Once` cap is consumed after a successful invoke.
Pure, host-tested — the first exercise of the `generation` field.
**RED**: a `Persistent` cap resolves repeatedly; a `Once` cap resolves, then
after `consume` the same handle is `Stale`; a fresh insert into the freed slot
resolves while the *old* handle stays `Stale`; bounds-then-generation ordering.
**MUTATE**: the generation bump; the multiplicity discriminant; stale ordering.
> **PR boundary** — pure single-use mechanism, no behaviour depends on it yet.

### Step 2: `Object::Reply` (`Once`) + `invoke_reply` (pure)

**Acceptance criteria**: `Object::Reply { caller: TaskId }`, and
`invoke_reply(table, handle) -> Result<TaskId, Denied>` (resolve → object match
→ return the caller to wake). The reply cap is granted with `Multiplicity::Once`
so a successful invoke consumes it (Step 1). Mirrors `invoke_send`/`invoke_recv`.
**RED**: accepts a live reply cap → caller id; refuses wrong-object / unknown /
already-consumed (`Stale`).
**MUTATE**: object discriminant + the resolve/consume path.
> **PR boundary** — additive cap object.

### Step 3: `CapObject::Reply` + `CapEventKind::Transferred` on the wire

**Acceptance criteria**: both appended; a `CapEvent` carrying `Transferred` +
`CapObject::Reply` roundtrips; `OwnedFrame` arm exists; consumers (`collector`,
`xtask` harness) handle the new variants.
**RED**: protocol roundtrip; the exhaustive matches fail to compile until handled.
> **PR boundary** — wire addition, append-only, no emitter yet.

### Step 4: Kernel mints + inserts a single-use reply cap at the `call` rendezvous

**Acceptance criteria**: a kernel path that, given a blocked caller, inserts an
`Object::Reply { caller }` (`Once`) into the *server's* `CapTable` and returns
the handle — the first cross-process cap insertion. Snitched as
`CapEvent::Transferred` (parent = the call). Itest confirms the server observes
a nonzero reply handle under a `call` workload; a one-way `send` yields `0`.
**GREEN**: extend `kernel::ipc` so request-delivery mints the reply cap into the
receiver and threads its handle out through `receive`'s `a5`.
**MUTATE**: the mint/insert decision; the 0-vs-nonzero handle gate.
> **PR boundary** — cap-transfer works; reply not consumed yet.

### Step 5: `Call`/`Reply` syscalls — the round-trip

**Acceptance criteria**: under `workload=ipc-rpc`, the client `call`s a request,
the server `receive`s + `reply`s a response, and the client's `call` returns it.
A second `reply` on the same handle is refused — **because the `Once` cap was
consumed** (Step 1), not via a bespoke check. ABI `Call`/`Reply` + runtime
`Endpoint::call`/`reply_to` + two user binaries (`rpc-client`, `rpc-server`) +
`WorkloadKind::IpcRpc` (bootargs, host-tested) + spawn layout.
**RED**: itest `rpc-round-trips` — client observes the server's response value
(re-emitted via telemetry); a `SyscallRefused` on a double `reply`.
**GREEN**: `Call` = deliver request + block awaiting reply; `Reply` =
`invoke_reply` (consumes the `Once` cap) → deliver into caller's pending slot +
`wake`.
**MUTATE**: reply resolution + consume; bootargs arm.
**Done when**: `cargo xtask itest --repeat 10` green (scheduler/trap path).
> **PR boundary** — RPC works end-to-end. Shippable v0.9b.

### Step 6: The nested-span trace (the headline)

**Acceptance criteria**: the client opens `rpc.call` and `call`s inside it; the
server opens `rpc.handle`. Assert on the wire that `rpc.handle` is a child of
`rpc.call` **and** temporally nested — `rpc.handle`'s `[start,end]` lies within
`rpc.call`'s — the caller's span stays open across the callee's work. The
assertion must *fail* on v0.9's one-way `send` shape (disjoint timeline), so it
genuinely proves the difference.
**RED**: itest `rpc-trace-nests` — parent linkage + `t` containment.
> **PR boundary** — the observability headline, asserted.

### Step 7: RPC telemetry

`snitchos.ipc.calls_total` / `replies_total`, deferred-emission, drained in the
heartbeat. Itest asserts both reach ≥1.
> **PR boundary** — observability for the new behaviour.

### Step 8 (follow-on within v0.9b): `reply_recv` fusion

The canonical one-syscall server loop (`reply` to previous client, then
`receive` next). Small additive step once `call`/`reply` are proven; its own PR.

## Open questions

- **Reply-handle delivery register.** `a5` for `receive`'s reply handle is
  proposed; confirm it doesn't collide and the runtime threads it cleanly.
- **`reply` to a vanished caller.** No process teardown in v0.9, but if the
  caller is gone (`Exited`/missing) `reply` must refuse gracefully and not wake a
  dead task — a `caller`-still-valid check distinct from the cap consume. Decide
  at Step 5.
- **Multiplicity surface.** Keep `Once` inferred-or-marked minimally; resist any
  multiplicity machinery until cap-transfer-in-messages is the second consumer.
- **Reuse vs. parallel rendezvous path.** Does `call` ride `send_begin` (request
  delivery is identical to a `send`, only the await-reply parking differs) or
  warrant its own path? Decide at Step 5 (lean reuse).
- **Caller-facing error granularity (deferred, v0.9c+/policy).** The caller gets
  a binary `Err(Denied)` (`a0 = u64::MAX`); the *rich* `RefusalReason` goes only
  to the observer via `SyscallRefused`. That split is deliberate — least
  *information* to an untrusted caller (the 403-vs-404 anti-probing principle),
  rich detail to the trusted observer — and synchronous IPC sidesteps the most
  important missing distinction (`WouldBlock`: `send`/`call` block rather than
  return try-again). But the binary contract is under-specified for *recovery*:
  a caller eventually needs transient-vs-permanent (e.g. a consumed reply cap vs
  a never-valid handle). When that day comes, expose a small **curated** caller
  enum (merge the probe-leaky `NotFound`/`WrongRights` cases; split out only
  what's safe + operationally necessary), not the full internal `RefusalReason`.
  Not built now — speculative until a caller actually recovers.

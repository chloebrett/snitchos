# v0.5 ‚Äî Threading & round-robin scheduler

**Status: SHIPPED.** The scheduler landed as designed (`kernel/src/sched/`:
`yield_now`, `spawn`, `register_bare_task`, `current_task_id`,
`task_snapshots`; `kernel_core::sched::Runqueue`) and is documented as the
project's scheduler in CLAUDE.md. The `sched-*` itests pin it.

The v0.5 "no `unspawn`/`exit`/`join`" limitation was lifted later ‚Äî task exit
landed in [legacy/v0.5.x-task-exit-minimal.md](v0.5.x-task-exit-minimal.md),
and cooperative yield gave way to timer preemption in
[v0.8-preemption.md](v0.8-preemption.md).

Kernel-mode threads, cooperative round-robin scheduling, span context
that survives context switches. After this milestone the kernel is
*organised around tasks* ‚Äî every periodic kernel job lives in a
thread, the scheduler decides who runs, and the wire format makes
those decisions traceable.

Concepts walkthrough was done interactively. Recap in ¬ß"Why these
choices."

## Why this step

Today the kernel is one execution context. Boot ‚Üí `kmain` ‚Üí heartbeat
loop ‚Üí `wfi`. Every span, alloc, IRQ handler is tangled into the
single thread that started in `_start`. That's fine for a kernel that
only emits telemetry; it stops scaling the moment we want:

- **Multiple periodic jobs.** Heartbeat is one. Future driver workers
  (virtio-net RX, virtio-block completion) need their own scheduling.
  The "deferred-work pattern" (atomic flag + heartbeat drains) is a
  thread-shaped abstraction with heartbeat playing every role.
- **Userspace (v0.6).** A userspace process is "a kernel thread that
  also has its own page table and runs in U-mode." Threading is the
  prerequisite, not the deliverable, but it's the necessary first
  step.
- **The first hard observability problem.** A span starts on thread
  A, gets descheduled, A is resumed, the span ends. Meanwhile thread
  B opened its own spans in between. Whose spans go where in the
  trace? This is the question every distributed system asks; v0.5 is
  the first time SnitchOS asks it inside a single binary.

What we deliberately don't do in v0.5: userspace, syscalls,
capabilities, per-process page tables, preemption. Each queues up
behind threading; threading must land cleanly first.

## Decisions locked in

| decision | choice |
|---|---|
| Threading model | Kernel-mode threads, shared address space, single CPU |
| Stack strategy | `Box<[u8; STACK_SIZE]>` from the v0.4 heap |
| Stack size | 16 KiB per thread |
| Preemption | **No** for v0.5 ‚Äî cooperative only via explicit `yield_now()` |
| Scheduler | Round-robin over a single `VecDeque<TaskId>` |
| API surface | `spawn(name, entry: fn() -> !)`, `yield_now()`, `current_task_id()` |
| Boot thread fate | `kmain` converts itself into "task 0 / main" ‚Äî runs heartbeat work, yields to others |
| Idle thread | Always spawned; runs `loop { wfi() }` |
| Span context | Per-task span stack inside the `Task`; swapped on context switch |
| Wire format additions | `ThreadRegister { id, name }`, `ContextSwitch { from, to, t, reason }`, `task_id: u32` on `SpanStart` |
| Telemetry | Scheduler metrics (`context_switches_total`, switch-duration histogram, decision breakdown, runqueue depth) + per-task `cpu_time_ticks{task}` and `runs_total{task}` |
| Lock discipline | "Don't hold a `spin::Mutex` across `yield_now()`" ‚Äî a rule, not enforced. Preemption + a per-task preempt-disable counter is v0.5.5 or rolled into v0.6 prep |

## Why these choices

- **Kernel threads first, userspace later.** The roadmap order is
  load-bearing. Userspace adds two things on top of threading: a
  second CPU mode and a per-process page table. Both are real work,
  neither is conceptually entangled with "I can switch contexts."
  Land threading clean; userspace becomes additive.
- **Cooperative only.** Preemption forces the lock-discipline cliff:
  `spin::Mutex` doesn't disable interrupts, so a timer-driven
  preempt mid-lock would deadlock the next thread that wants the
  same lock. Fix is a per-task preempt-disable counter, bumped on
  every `Mutex::lock()`, checked in the timer IRQ. That's a real
  wiring job touching every lock site. Defer until v0.6's userspace
  prep, where preempt-disable is the forcing function and design
  pressure can shape the lock model cleanly. v0.5 with cooperative
  yields is the right unit of risk.
- **Heap-backed stacks.** The v0.4 heap exists; `Box<[u8; 16384]>`
  works today. (i) static `.bss` stacks would cap thread count at
  link time; (iii) dedicated VA + guard pages is the v0.6+ shape
  and needs another `mmu::map` design pass. Heap stacks are the
  comfortable middle for v0.5.
- **Per-task span stack.** Today's `SPAN_REGISTRY` is global ‚Äî
  implicit "one execution context" assumption. With multiple
  threads, a global stack would have thread A close thread B's
  span. The fix is structural: move the span stack into the `Task`
  struct, swap it on context switch (just a pointer swap; no
  memcpy). Refactor is small because the existing `SpanRegistry`
  already lives in `kernel-core` as testable data.
- **`ContextSwitch` as a wire primitive.** Without it, scheduler
  decisions are invisible ‚Äî you can only infer "something switched"
  from interleaved span streams. With it, the scheduler is a
  first-class traceable entity: count switches, histogram their
  latency, break down by reason. This is the SnitchOS-specific
  unlock. ~10 bytes per switch on the wire; cheap.
- **`task_id` on `SpanStart`.** Lets Tempo render task ownership as
  a span attribute (`thread.name` per OTLP semantic conventions).
  Backwards-compatible: existing scenarios still parse, the field
  reads as 0 (= "task 0 / main") for any frame that pre-dates the
  scheduler.

## Architectural details

### `Task` and `Runqueue` (kernel-core)

```rust
// kernel-core/src/sched.rs
#[derive(Clone, Copy, PartialEq, Eq, Debug)]
pub struct TaskId(pub u32);

#[derive(Clone, Copy, Debug)]
pub enum TaskState {
    Ready,
    Running,
    /// v0.5 placeholder. Won't be reached until v0.5.x adds blocking.
    Blocked,
    Exited,
}

/// Pure data shape of a task. The kernel-side `KernelTask` adds raw
/// state pointers + stack ownership; this struct holds the things
/// the host-side tests need to reason about.
#[derive(Clone, Debug)]
pub struct TaskMeta {
    pub id: TaskId,
    pub name: heapless::String<32>,  // or interned StringId
    pub state: TaskState,
    pub runs: u64,
    pub cpu_time_ticks: u64,
}

pub struct Runqueue {
    ready: alloc::collections::VecDeque<TaskId>,
}

impl Runqueue {
    pub fn new() -> Self { ... }
    pub fn push_back(&mut self, id: TaskId) { ... }
    pub fn pop_front(&mut self) -> Option<TaskId> { ... }
    pub fn len(&self) -> usize { ... }
    pub fn is_empty(&self) -> bool { ... }
}
```

Host tests pin: FIFO order, len/empty, pop-from-empty is None, push-pop round-trip.

The interesting decision: keep raw pointer fields out of `kernel-core` (`forbid(unsafe_code)`). The kernel-side `KernelTask` extends `TaskMeta` with `sp`, `stack`, `span_stack` ‚Äî those live in `kernel/`.

### Per-task span stack (kernel-core refactor)

Current shape:

```rust
// kernel-core/src/span.rs ‚Äî today
static mut SPAN_REGISTRY: SpanRegistry<...> = ...;
```

After:

```rust
pub struct SpanStack { ... }  // what used to live inside SpanRegistry
impl SpanStack {
    pub fn push(&mut self, span: SpanFrame) { ... }
    pub fn pop(&mut self) -> Option<SpanFrame> { ... }
    pub fn current(&self) -> Option<&SpanFrame> { ... }
}

// SpanRegistry now takes &mut SpanStack, doesn't own it
impl SpanRegistry {
    pub fn open(stack: &mut SpanStack, name: StringId, ...) { ... }
    pub fn close(stack: &mut SpanStack, id: SpanId) { ... }
}
```

The kernel binary then owns one `SpanStack` per task, inside the `Task` struct. Context switch swaps "which `SpanStack` the registry uses next" by parameter-passing rather than by mutating a static.

This is a refactor of working code, so it's done in ¬ßstep 2 in isolation, with host tests staying green throughout. Tracing's existing per-test SpanRegistry coverage carries over.

### Wire format additions

```rust
// protocol/src/lib.rs
pub enum Frame<'a> {
    // ...existing variants...
    ThreadRegister { id: u32, name: &'a str },
    ContextSwitch { from: u32, to: u32, t: u64, reason: SwitchReason },
}

#[derive(Clone, Copy, PartialEq, Eq, Debug, serde::Serialize, serde::Deserialize)]
pub enum SwitchReason {
    Yield,
    Preempt,    // unused in v0.5; reserved for v0.5.x
    Blocked,    // placeholder
    Exit,
}

// SpanStart gains a field:
pub enum Frame<'a> {
    SpanStart {
        id: SpanId,
        name_id: StringId,
        parent: Option<SpanId>,
        t: u64,
        task_id: u32,   // NEW
    },
    // ...
}
```

`task_id == 0` is "task 0 / main" ‚Äî gives every existing fixture a sensible default. Postcard's enum encoding is positional within each variant, so we add the new variants at the end of the `Frame` enum (per the CLAUDE.md "don't reorder variants" rule).

Collector: matching arms in `OwnedFrame::from_borrowed`. New OTLP attribute on every span: `thread.name` from the most recent `ThreadRegister` that named the span's `task_id`.

### Context switch (kernel)

```rust
// kernel/src/sched.rs
#[repr(C)]
pub struct TaskContext {
    pub ra: u64,
    pub sp: u64,
    pub s0: u64, pub s1: u64, pub s2: u64, pub s3: u64,
    pub s4: u64, pub s5: u64, pub s6: u64, pub s7: u64,
    pub s8: u64, pub s9: u64, pub s10: u64, pub s11: u64,
}

// kernel/src/sched.S
// switch(from: *mut TaskContext, to: *mut TaskContext)
// a0 = from, a1 = to
.global switch
switch:
    sd ra,  0(a0)
    sd sp,  8(a0)
    sd s0, 16(a0)
    // ... s1..s11 ...
    ld ra,  0(a1)
    ld sp,  8(a1)
    ld s0, 16(a1)
    // ... s1..s11 ...
    ret
```

Smoke test (step 5): hand-allocate a small stack, write `marker_fn` as the bottom of the call chain, rig `to.ra` and `to.sp` so the first switch lands inside `marker_fn`. After the switch, an `AtomicBool` set by `marker_fn` must be true. Calls `switch` once; isolates the asm correctness from scheduler complexity.

### First-time entry

A freshly-spawned task hasn't run yet. Its `TaskContext` is constructed so that the first `switch` into it loads `ra = entry`, `sp = top_of_stack`. When the switch's `ret` executes, control jumps to `entry`. The entry function never returns (signature `fn() -> !`) ‚Äî if it tries, we'll crash, which is the right answer for v0.5 (task exit is v0.5.x).

### `kmain` becoming task 0

```rust
// kmain ‚Äî end of init
sched::init_with_current_as_task_0("main");
sched::spawn("idle", idle_entry);
sched::spawn("task_a", task_a_entry);
sched::spawn("task_b", task_b_entry);

// existing heartbeat loop continues here; it's now task 0's body.
loop {
    while !trap::TICK_PENDING.swap(false, Ordering::Relaxed) {
        sched::yield_now();  // NEW ‚Äî used to be wfi
    }
    span!("kernel.heartbeat");
    // ... emit metrics ...
}
```

`yield_now()` inside the wait loop is the cooperative point. When `TICK_PENDING` is false, task 0 yields, lets other ready tasks run, eventually re-runs and re-checks. (The idle thread is the one that ends up `wfi`-ing when nothing else wants the CPU.)

### Scheduler metrics (deferred-emission, as always)

Atomics on the scheduler module:

```rust
// kernel/src/sched.rs
pub static CONTEXT_SWITCHES: AtomicU64 = AtomicU64::new(0);
pub static LAST_SWITCH_DURATION: AtomicU64 = AtomicU64::new(0);
pub static YIELD_COUNT: AtomicU64 = AtomicU64::new(0);
// ...etc
```

`yield_now()` increments `CONTEXT_SWITCHES` and records `rdtime` delta around the asm into `LAST_SWITCH_DURATION` (heartbeat samples into a histogram). Same pattern as `trap::LAST_IRQ_DURATION`.

`runqueue_depth` and `tasks_total` are read directly under the scheduler lock by the heartbeat ‚Äî cheap, single-thread-safe.

### Per-task metrics

Each `Task` carries `cpu_time_ticks: AtomicU64` and `runs: AtomicU64`. `yield_now()` bumps them via `rdtime` delta around the task's on-CPU window.

Emission: heartbeat iterates the task list (briefly takes the scheduler lock) and emits `snitchos.task.cpu_time_ticks{task="name"}` per task. With 3-5 named threads this is fine.

### Integration scenarios

- **`thread-spawn-and-yield`**: spawns `task_a` and `task_b` with simple loops emitting a counter span per iteration plus a yield. Asserts the wire shows alternating `task_a.tick` and `task_b.tick` spans (interleaved within a tolerance). Proves the scheduler is genuinely round-robining.
- **`context-switches-on-wire`**: asserts at least N `ContextSwitch` frames arrive within T seconds, with at least one switch from `task_a` to `task_b` and vice versa. Proves the wire primitive is emitted at the right moments.
- **`span-survives-yield`**: a task opens a span, yields, gets re-scheduled, ends the span. Asserts the `SpanStart` and `SpanEnd` for that span have the same `task_id` (i.e. no cross-task closure). Proves per-task span stack works.

## What this doesn't do

- **Preemption.** Cooperative only. Threads must yield voluntarily.
  A misbehaving thread hangs the kernel ‚Äî documented as known
  limitation. Preemption is v0.5.x or v0.6 prep.
- **Thread exit / reaping.** `fn() -> !` for entries means tasks
  don't return. v0.5.x or v0.6 will need a proper exit path
  (yield + state=Exited + reaper thread collecting stacks).
- **Blocking primitives.** No condvars, no mutexes-that-sleep. Spin
  locks only; threads that need to wait either spin or yield in a
  loop. Real blocking lands when there's a workload that needs it.
- **Per-process address spaces.** Single shared kernel address
  space; per-process tables are v0.6.
- **Stack guard pages.** Stacks are flat `Box<[u8; 16384]>`; an
  overflow corrupts adjacent heap silently. Guard pages need the
  v0.6 dedicated-stack-VA shape.
- **Priorities or fair-share.** Strict round-robin. Priority comes
  in v0.8.

## Risk and known unknowns

- **Hidden static-vs-per-task state.** `SPAN_REGISTRY` is the
  obvious one but maybe not the only one. Anywhere kernel code
  said "the current thread" implicitly will break. Mitigation: an
  audit pass during step 2 ‚Äî grep for `static mut` and `Mutex<...>`
  that holds anything semantically per-execution-context.
- **First switch into a fresh task.** Rigging the initial stack so
  the asm's `ret` lands in `entry` is the kind of off-by-one bug
  that costs a day. Mitigation: the smoke test in step 5 isolates
  this from scheduler complexity. Get the asm right before the
  scheduler is in the picture.
- **Lock-while-yielding bugs.** Even with cooperative scheduling,
  it's tempting to `yield_now()` while holding a spin lock ‚Äî
  deadlock waits for the next thread that wants the lock. Mitigation:
  document the rule, add a debug-only check (a per-task lock-held
  counter that asserts zero at every `yield_now()`).
- **Stack overflow goes silent.** No guard page yet. Heap-adjacent
  corruption could manifest as bizarre allocator behaviour 100ms
  later. Mitigation: leave a "canary word" at the bottom of each
  stack, checked at every `yield_now()`. Loud death beats silent
  corruption.
- **Wire-format reordering.** Adding `task_id` to `SpanStart` is a
  field addition (postcard handles fine), but reordering existing
  fields silently breaks every old fixture. Mitigation: append
  `task_id` at the end of the struct fields; integration tests
  catch any drift.

## Post angle

"Following a trace across a context switch." Demo screenshots:

- Grafana thread timeline showing three threads sharing CPU.
- Tempo trace view of a multi-task trace where spans from task A
  and task B appear concurrently at their real timestamps.
- A span explicitly carrying `thread.name="task_a"` as an
  attribute, end-to-end.

The argument: most kernels treat threading as "you can run more
than one thing." SnitchOS treats threading as "scheduler decisions
are now first-class observable events." `ContextSwitch` on the
wire is the artifact that makes that real.

## What's next after v0.5

- **v0.5.x (preemption + lock discipline)** if it's worth its own
  cut, otherwise rolled into v0.6 prep. Per-task preempt-disable
  counter, timer IRQ wired to scheduler tick, every `spin::Mutex`
  audited for safe preempt-disable semantics.
- **v0.6 ‚Äî first userspace process** (per roadmap). Per-process
  page tables (uses `mmu::map` from v0.4 step 5), U-mode entry,
  one ambient-authority syscall.
- **Thread exit / reaping**, blocking primitives, stack guards ‚Äî
  these queue up against the workloads that motivate them. None
  are v0.5 blockers.

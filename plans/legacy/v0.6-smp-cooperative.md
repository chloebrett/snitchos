# v0.6 — SMP (cooperative)

Second hart online, per-CPU discipline made real, and a concrete
producer/consumer workload that visibly responds to parallelism.
After this milestone the kernel is *organised around harts*: the
chokepoints from v0.5's prefactor (`kernel::sync`, `PerCpu`) earn
their keep under genuine multi-hart contention, and the wire format
carries `hart_id` so traces can be sliced by CPU.

This is **SMP, cooperative** — not SMP-with-preemption. Each hart
runs its own runqueue; tasks `yield_now()` voluntarily exactly as
they do today. Idle harts `wfi` and wake on an IPI when work
arrives. Preemption is still v0.9 or later.

## Why this step (and why now)

Two arguments, weighted differently:

- **The audit-vs-refactor asymmetry.** Late capabilities are a
  *known, scoped refactor* — v0.6a → v0.6b in the original roadmap
  was literally designed as a "rewrite syscalls to be
  cap-mediated" transition. Late SMP is an *unbounded audit*: every
  global, every lock, every "no one else is here" assumption has to
  be re-examined, and RISC-V's weak memory model means the bugs are
  silent until the second hart wakes up. Refactors have visible
  diffs; audits have invisible bugs. The community consensus
  (seL4, Redox, recent Zephyr) is that retrofitting SMP onto a
  single-core kernel is one of the hardest things you can do —
  which is why new kernels go "SMP from day one" (meaning the
  *mental model* is multi-hart from the first line, not that the
  bring-up code runs on day one).

- **The window is still open.** The v0.5 prefactor (`kernel::sync`,
  `PerCpu`, the `disallowed_types` lint) put the chokepoints in
  place exactly for this. The remaining audit surface is small and
  enumerable: four globals (`CURRENT_TASK`,
  `CURRENT_TASK_ENTRY_TICK`, `CURRENT_SPAN_CURSOR`, the
  cooperative-scheduler `SCHEDULER` mutex), plus a weak-memory pass
  over existing `Atomic*` ordering choices. After capabilities and
  IPC land (concurrency-shaped subsystems by nature), the audit
  surface grows sharply. We are probably one milestone, maybe two,
  away from the cost curve inflecting.

What this milestone deliberately does *not* do: userspace,
syscalls, capabilities, preemption, work stealing across harts,
per-hart heaps. Each is independently valuable and independently
sized; bundling them here would push the milestone past the point
where it can ship as one coherent post arc.

## The post arc (load-bearing for milestone shape)

Three observability-led posts fall out of this milestone, all
using the same producer/consumer histogram workload:

1. **Cooperative single-core baseline.** Producer and consumer
   alternate via `yield_now()` on one hart. Establishes throughput
   baseline, queue-depth oscillation, and the lock-wait-time
   histogram. Runnable from step 1 of the milestone — before any
   SMP work — and shippable on its own.
2. **SMP, `Mutex<VecDeque>` for the queue.** Producer on hart 0,
   consumer on hart 1. Measure. Likely surprising: cache coherence
   on the lock cacheline may produce *less* speedup than expected,
   or even slowdown if the lock is on every sample. The post is
   "here's what I expected, here's what I got, here's how I
   diagnosed it from the spans."
3. **SMP, `heapless::spsc::Queue` for the hot path.** Swap the
   chokepointed queue for a lock-free SPSC. Re-measure. The post is
   "the chokepoint earned its keep by being visible — and here's
   what replacing it bought us."

Promising "watch it get faster" is dishonest. Promising "watch
what happens" is the project's identity. Whichever way the numbers
land, the observability machinery is what makes the story tellable.

## Decisions locked in

| decision | choice |
|---|---|
| Hart count | 2 (QEMU `-smp 2`). Higher counts deferred — 2 is enough to prove discipline; >2 brings work-stealing pressure that doesn't fit v0.6 |
| Scheduling model | **Cooperative**. Per-hart runqueue, per-hart idle task, `yield_now()` voluntarily, idle hart `wfi`s |
| Inter-hart wakeup | IPI on `spawn` to a non-local hart, and on cross-hart wake events. SBI `sbi_send_ipi` from kernel; software-interrupt trap handler dispatches by `ipi_message: PerCpu<AtomicU32>` |
| Hart bring-up | SBI HSM (`sbi_hart_start`) from hart 0 in `kmain`; secondaries park at a known entry, MMU on, then `wfi` until first wake |
| `tp` register convention | `tp` holds per-hart-data pointer post-MMU. Set in the bring-up path. Used by `current_hartid()` and `PerCpu::get` |
| Per-CPU lift | `CURRENT_TASK`, `CURRENT_TASK_ENTRY_TICK`, `CURRENT_SPAN_CURSOR`, runqueue, idle task — all move into `PerCpu<T>` |
| Page-table mutation | `mmu::map`/`unmap` broadcasts TLB shootdown IPI; per-hart ack barrier; initiator waits before returning |
| Task→hart affinity | None (yet). `spawn(name, entry)` keeps "spawn onto current hart". A `spawn_on(hart, name, entry)` variant exists for the workload demos. No migration, no stealing |
| Wire format | `hart_id: u8` field on `SpanStart`, `ContextSwitch`, and new `HartRegister { id, role }`. `SpanStart`/`ContextSwitch` are wire-incompatible changes — done now while the wire is still pre-userspace |
| Workload | Producer/consumer histogram. Producer generates LCG samples in batches of 64. Consumer bins into `[AtomicU64; BUCKETS]`. Queue between them changes per phase (Mutex<VecDeque> then heapless::spsc) |
| Workload telemetry | Per-batch produce/consume spans (with `hart_id`), lock-wait-time histogram, queue-depth gauge, per-bin atomic counters |
| Preemption | **No.** Deferred to v0.9 |
| Userspace | **No.** Deferred to v0.7 |

## Why these choices

- **Cooperative SMP, not preemptive SMP.** Preemption needs full
  trap-frame-from-IRQ context switching (today's switch elides
  caller-saved per the SysV ABI; that breaks under preemption that
  can land mid-instruction). Bundling preemption with SMP doubles
  the complexity and obscures which bugs come from which
  subsystem. Cooperative SMP is a coherent intermediate stop — it
  is also approximately how the Go runtime's M:N scheduler
  presents to the user (voluntary yield points, multi-CPU
  underneath).
- **Two harts, not N.** Two is the smallest interesting number.
  Three+ pulls in work-stealing pressure (idle hart with empty
  runqueue, other hart with three queued tasks — should it steal?
  if so, how?) which is a real design discussion that doesn't
  belong in this milestone. Two harts with affinity-by-spawn
  cleanly demonstrate the discipline without forcing the policy
  question. `MAX_HARTS` becomes 2; the path to N is just bumping
  the constant once policy questions are answered.
- **Per-hart runqueue, no migration.** A single global runqueue
  with `Mutex<Runqueue>` would work but would put the scheduler
  itself on the contention path — exactly the anti-pattern v0.6 is
  supposed to teach against. Per-hart runqueue is what real
  kernels do, costs us nothing now, and the no-migration policy
  defers all the interesting (hard) scheduling questions.
- **`tp` for per-hart pointer.** RISC-V reserves `tp` for this.
  Picking it now is a one-way door of sorts (changes touch asm
  later), so commit. `current_hartid()` becomes a single
  `ld a0, 0(tp); ret`-equivalent — cheap enough to inline
  everywhere `PerCpu::get` is called.
- **Wire-format break now, not later.** Adding `hart_id` to
  `SpanStart` and `ContextSwitch` is a postcard-encoding break
  (positional fields). It is *cheap* to do pre-userspace because
  no host capture is load-bearing yet. It is *expensive* to do
  post-userspace because external tools start consuming the
  format. Decide now.
- **Mutex<VecDeque> before SPSC.** v0.6's thesis is "the
  chokepoints earn their keep." Skipping straight to a lock-free
  queue would mean the hot path never touches `kernel::sync::Mutex`,
  and the chokepoint isn't exercised by the headline workload.
  Starting with `Mutex<VecDeque>` lets the contention *show up* in
  the lock-wait histogram; then swapping to SPSC and watching it
  vanish is the strongest version of the observability post.
- **Producer/consumer, not ping-pong.** Both were on the table.
  Ping-pong is more *visually* obvious in a trace (perfect
  alternation), but producer/consumer has a richer telemetry
  story: queue depth as a gauge, backpressure as a concept,
  histogram convergence as a correctness oracle. Ping-pong is a
  good integration test (`smp-ping-pong-cadence` below), not the
  headline workload.

## Architectural details

### `tp` register and `PerCpu`

`tp` holds a pointer to the per-hart data block. Set after MMU on in the bring-up path (hart 0 and secondaries both):

```rust
// kernel/src/percpu.rs
#[repr(C, align(64))]  // cacheline align to avoid false sharing across harts
pub struct PerHartData {
    pub hart_id: u32,
    pub current_task: AtomicPtr<TaskContext>,
    pub current_task_entry_tick: AtomicU64,
    pub current_span_cursor: AtomicPtr<SpanCursor>,
    pub ipi_pending: AtomicU32,        // bitflags per IpiMessage variant
    pub shootdown_ack: AtomicU32,      // bumped by shootdown handler
    // ...etc
}

pub static PER_HART_DATA: [PerHartData; MAX_HARTS] = ...;

#[inline(always)]
pub fn current_hartid() -> u32 {
    // tp points at PER_HART_DATA[hartid]; offset 0 is hart_id.
    let tp: u64;
    unsafe { core::arch::asm!("mv {}, tp", out(reg) tp); }
    unsafe { (*(tp as *const PerHartData)).hart_id }
}
```

`PerCpu<T>` (the v0.5 stub) is reimplemented over this. The API surface stays stable; the implementation stops being a 1-hart placeholder.

Cacheline alignment matters: `PerHartData` instances must not share cachelines, or every hart's writes will invalidate every other hart's cache. 64-byte alignment is the conservative choice for current RISC-V cores.

### Hart bring-up (SBI HSM)

```rust
// kernel/src/smp.rs
pub fn start_secondaries() {
    for hart_id in 1..MAX_HARTS {
        // entry_pa is the physical address of `_secondary_start` in asm
        let entry_pa = va_to_pa(_secondary_start as usize as u64);
        let opaque = hart_id as u64;
        match sbi::hart_start(hart_id, entry_pa, opaque) {
            Ok(()) => {}
            Err(e) => panic!("hart {hart_id} failed to start: {e:?}"),
        }
    }
}
```

`_secondary_start` (asm) mirrors `_start` for the bring-up sequence: stack setup, SATP load, `tp` setup, then jumps into a Rust `secondary_main(hart_id)` which:

1. Initializes its `PerHartData` slot
2. Emits a `HartRegister { id, role: Worker }` frame (through the same telemetry channel — `kernel::sync::Mutex` makes this safe across harts now)
3. Spawns its idle task onto its own runqueue
4. Enters the scheduler — which immediately picks idle, which `wfi`s

A secondary hart is now a wholly normal scheduler context, just with an empty (idle-only) runqueue until something spawns onto it.

### IPI primitive

Two channels of IPI:

- **Software interrupt** (SBI `send_ipi`) — fires the target hart's software-interrupt trap. Handler reads `PER_HART_DATA[hartid].ipi_pending`, clears it atomically, dispatches per-bit.
- **No data on the wire** — the IPI is just "look at your pending bits." Sender writes the pending bit *before* sending the IPI (release-acquire pair).

```rust
#[repr(u32)]
pub enum IpiMessage {
    Wakeup        = 1 << 0,
    TlbShootdown  = 1 << 1,
    // future: Reschedule, Halt, etc.
}

pub fn send(target_hart: u32, msg: IpiMessage) {
    PER_HART_DATA[target_hart as usize]
        .ipi_pending
        .fetch_or(msg as u32, Ordering::Release);
    sbi::send_ipi(1u64 << target_hart);
}
```

The trap handler reads `ipi_pending` with `Acquire`, swaps to 0, then dispatches. Multiple flags can be set between sends; handler walks all set bits.

### TLB shootdown

The protocol — write down before any PTE-mutating code lands. This is the one piece of v0.6 that becomes load-bearing for every future milestone touching page tables.

**Initiator (hart performing the unmap/remap):**

1. Mutate the PTE locally (already works in `mmu::map`).
2. Local `sfence.vma vaddr` — flushes this hart's TLB.
3. For each other live hart: set the per-hart `shootdown_va` slot,
   send `IpiMessage::TlbShootdown`.
4. Spin-wait on per-hart `shootdown_ack` atomics until each target's
   ack count has advanced past the value observed before sending.
5. Return.

**Target (other harts):**

1. IPI handler reads `shootdown_va`, runs `sfence.vma vaddr`.
2. `fetch_add(1, Release)` on `shootdown_ack`.

Subtleties to document:

- **What if a target is currently in a trap handler?** Software
  interrupts in RISC-V are taken at the next instruction boundary
  when interrupts are enabled. If the target had interrupts
  disabled (in a critical section), the IPI is pending and runs
  the moment they're re-enabled. Initiator's spin-wait absorbs the
  delay; no correctness issue.
- **What if two harts shoot down concurrently?** Each has its own
  `shootdown_va` slot keyed by *initiator*, not target. So the
  per-hart `shootdown_va` is actually `[AtomicU64; MAX_HARTS]`
  indexed by initiator hart. Initiator-side ack tracking is also
  per-(target, initiator). This is more bookkeeping than the
  cheapest possible scheme but eliminates the "two initiators
  collide and one's shootdown is lost" race.
- **Single hart**: shootdown self is a local `sfence.vma` only,
  no IPI, no spin-wait. Common path.

Two-hart scenario for v0.6 means this is not yet exercised under
heavy contention. Document the policy; if v0.6's two harts surface
real bugs, fix them; the design has to survive the eventual
many-hart future without structural change.

### Per-hart runqueue + scheduling

```rust
// kernel-core/src/sched.rs
pub struct Scheduler {
    // PerCpu<Runqueue> in the kernel; kernel-core stays host-testable
    // by parameterizing over hart count.
    runqueues: [Runqueue; MAX_HARTS],
}

impl Scheduler {
    pub fn push_to(&mut self, hart: u32, task: TaskId) { ... }
    pub fn pop_local(&mut self, hart: u32) -> Option<TaskId> { ... }
    pub fn len_for(&self, hart: u32) -> usize { ... }
}
```

`spawn(name, entry)` calls `push_to(current_hartid(), id)` and emits no IPI (the calling hart is already running). `spawn_on(hart, name, entry)` calls `push_to(hart, id)` and sends `IpiMessage::Wakeup` to that hart — needed so an idle hart `wfi`-ing in its idle task wakes up to pick up the new work.

`yield_now()` picks from the *current hart's* runqueue. If empty, picks the local idle task. No stealing.

### Wire format additions

```rust
// protocol/src/lib.rs
pub enum Frame<'a> {
    // ...existing variants...
    HartRegister { id: u8, role: HartRole },  // new
    // SpanStart and ContextSwitch gain hart_id: u8
    SpanStart {
        id: SpanId,
        name_id: StringId,
        parent: Option<SpanId>,
        t: u64,
        task_id: u32,
        hart_id: u8,    // NEW
    },
    ContextSwitch {
        from: u32,
        to: u32,
        t: u64,
        reason: SwitchReason,
        hart_id: u8,    // NEW
    },
}

#[derive(Clone, Copy, PartialEq, Eq, Debug, serde::Serialize, serde::Deserialize)]
pub enum HartRole {
    Boot,
    Worker,
}
```

Postcard's positional encoding means the additions to `SpanStart` and `ContextSwitch` *break* the wire format. This is the right time to do it: pre-userspace, no external consumers. Bump a `PROTOCOL_VERSION` constant; the integration tests are the only existing consumer and they get updated atomically.

Collector: `OwnedFrame::from_borrowed` arms for the new variants/fields; OTLP attribute `host.cpu_id` populated from `hart_id` on every span.

### Producer/consumer workload

```rust
// kernel/src/demo/histogram.rs
const BUCKETS: usize = 64;

pub struct Workload {
    queue: kernel::sync::Mutex<heapless::Deque<u64, 4096>>,
    // Phase 3 swaps this for: queue: heapless::spsc::Queue<u64, 4096>,
    histogram: [AtomicU64; BUCKETS],
    samples_produced: AtomicU64,
    samples_consumed: AtomicU64,
    lock_wait_ticks: AtomicU64,
    queue_depth_sum: AtomicU64,
    queue_depth_samples: AtomicU64,
}

pub extern "C" fn producer_entry() -> ! {
    let workload = &WORKLOAD;
    let mut rng = Lcg::new(0xdead_beef);
    loop {
        let span = tracing::span_start("produce.batch");
        let batch: [u64; 64] = core::array::from_fn(|_| rng.next());

        let t0 = arch::rdtime();
        {
            let mut q = workload.queue.lock();
            for s in batch { let _ = q.push_back(s); }
        }
        let wait = arch::rdtime() - t0;
        workload.lock_wait_ticks.fetch_add(wait, Ordering::Relaxed);
        workload.samples_produced.fetch_add(64, Ordering::Relaxed);

        span.end();
        sched::yield_now();
    }
}

pub extern "C" fn consumer_entry() -> ! {
    let workload = &WORKLOAD;
    loop {
        let span = tracing::span_start("consume.batch");
        let drained = {
            let mut q = workload.queue.lock();
            let n = q.len().min(64);
            let mut buf = [0u64; 64];
            for i in 0..n { buf[i] = q.pop_front().unwrap(); }
            (buf, n)
        };
        for &s in &drained.0[..drained.1] {
            let bin = (s as usize) % BUCKETS;
            workload.histogram[bin].fetch_add(1, Ordering::Relaxed);
        }
        workload.samples_consumed.fetch_add(drained.1 as u64, Ordering::Relaxed);

        span.end();
        sched::yield_now();
    }
}
```

Bin counts are correctness oracles: after N produced samples,
`sum(histogram[*])` must equal `samples_consumed` — never less,
never more. Mutation testing pressure: a broken `pop_front` that
double-counts or drops samples is caught by this invariant.

### Workload telemetry — what makes the post

Heartbeat drains these into Prometheus + Grafana:

- `snitchos.workload.samples_produced_total` / `samples_consumed_total` — throughput
- `snitchos.workload.lock_wait_ticks_total` + Prometheus histogram derived from periodic samples → p50/p95/p99 lock wait time
- `snitchos.workload.queue_depth` — gauge sampled periodically (Σ depth / Σ samples = average depth)
- `snitchos.workload.histogram_bin{bin="0".."63"}` — bin totals, for correctness eyeballing in Grafana
- Span attributes on each `produce.batch` / `consume.batch`: `hart_id`, `batch_size`, `lock_wait_ticks`

The Grafana dashboard is the post. Three side-by-side panels for the three phases would also work as a single capstone graphic.

## Integration scenarios

- **`workload-cooperative-baseline`** — Single-core cooperative
  producer/consumer. Asserts `samples_consumed > 10_000` within
  10s, `Σ histogram_bin == samples_consumed`. Runs in phase 1, before any SMP work — proves the workload itself.
- **`smp-secondary-hart-boots`** — `HartRegister { id: 1 }` frame
  appears within 5s of boot. Proves SBI HSM + the secondary-entry
  path works.
- **`smp-ipi-wakes-idle-hart`** — Hart 1 is idle (`wfi`); hart 0
  calls `spawn_on(1, ...)`; the spawned task's first span appears
  within 100ms. Proves wake-via-IPI.
- **`smp-tlb-shootdown-visible`** — Hart 0 maps a fresh page after
  hart 1 is online; hart 1 reads through that VA without fault.
  Failure mode without shootdown: hart 1's TLB has no entry but
  the page-fault handler resolves it (Sv39 walks the table) — so
  this scenario must explicitly assert a *re*-mapped page is
  visible, not just a newly-mapped one. Concretely: map V→A, hart
  1 reads A; remap V→B, hart 1 reads B and gets B not A. Without
  shootdown hart 1 reads stale A.
- **`smp-spans-carry-hart-id`** — Tasks spawned on different harts
  produce spans whose `hart_id` matches the spawn target. Proves
  the wire-format path end-to-end.
- **`smp-producer-consumer-correctness`** — As `workload-cooperative-baseline` but with producer on hart 0, consumer on hart 1. Asserts `Σ histogram_bin == samples_consumed` (correctness under concurrent access). Runs in phases 2 and 3.
- **`smp-ping-pong-cadence`** — Two tasks on two harts alternate K
  flag-flips via a `[AtomicU64; 2]`. Asserts K alternations
  observed in span order within T seconds. A correctness oracle
  for the IPI/wakeup path independent of the producer/consumer
  workload.

## What this doesn't do

- **Preemption.** Cooperative only. Tasks still yield voluntarily.
  A misbehaving task on hart 0 still hangs hart 0 (but hart 1
  keeps running, which is itself an interesting observation).
  Preemption is v0.9, and lands in the same `kernel::sync`
  chokepoint that absorbs preempt-disable hooks.
- **Userspace.** Still all kernel-mode threads. v0.7.
- **Capabilities.** Still ambient. v0.7b or v0.8.
- **Work stealing.** A hart with an empty runqueue stays idle even
  if its sibling has 3 queued tasks. The migration question is
  deferred along with the priority question — both belong with
  the priority/time-sliced scheduler in v0.9.
- **Per-hart heaps.** The global kernel heap is shared, accessed
  under its existing mutex. The heap is *not* on the producer/consumer
  hot path (the queue is pre-allocated), so this is fine for v0.6
  and the lesson can wait.
- **Hart hotplug / halt.** Secondaries come up at boot and never
  stop. No `sbi_hart_stop`, no power management. v1.x+.
- **`>2` harts.** `MAX_HARTS = 2` for v0.6. Bumping to N is a
  constant change once the migration/stealing/affinity questions
  are answered, which they aren't.
- **virtio MQ.** virtio-console TX still serializes through a
  single descriptor slot under a mutex. Documented in
  `scaling-corners.md` as a known performance issue; correctness
  is fine. Multi-queue virtio is a separate optimization track.

## Open questions to resolve before step 7

- **`PROTOCOL_VERSION` policy.** Add a version constant now; bump
  on every wire break going forward. Worth doing as part of step
  3 so the discipline is in place before userspace introduces
  external consumers.
- **`current_hartid()` cost.** Inline-asm `mv {}, tp` is cheapest
  but means every call site emits the load. If profiling shows
  this in a hot loop, cache in a local. Probably not worth
  worrying about pre-measurement.
- **Should `HartRegister` include `mhartid`?** SBI exposes
  `mhartid`, which is the platform hart id (may not start at 0,
  may have gaps). Our logical `hart_id` is a dense `0..MAX_HARTS`
  index. Include both so post-mortem traces from real hardware
  can be correlated with platform docs. Cheap.
- **Cacheline size for `PerHartData` alignment.** 64 is safe on
  every current RISC-V core. Document as a constant
  (`CACHELINE_BYTES`) rather than a magic number in the
  `#[repr(align(...))]`.

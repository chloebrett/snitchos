# v0.6 step 13 — `smp-tlb-shootdown-visible`

The one v0.6 SMP scenario that asserts shootdown's *purpose*, not its
plumbing. `shootdown-storm` already proves the IPI payload-read path
(counters climb); it explicitly does **not** prove a remapped page is
visible on the other hart — its own doc-comment admits sfence on the
wrong VA still passes. This scenario closes that gap.

## The validity trap (why this is subtle)

A *fresh* map can't prove shootdown works: if hart 1 has no TLB entry
for V, the page-fault walker resolves the new PTE correctly **whether
or not** shootdown ran. The test only has teeth if hart 1 has
**V→A cached in its TLB before the remap**. Under QEMU TCG the softmmu
TLB only flushes on an `sfence.vma` QEMU intercepts, so a missing /
wrong-VA shootdown genuinely leaves a stale entry — *provided* hart 1's
intervening spin doesn't naturally evict V. We accept that residual
risk and **prove teeth with a deliberately-broken counterfactual**
(same discipline as the deflake storms): a wrong-VA shootdown variant
must make the scenario fail.

## Remap primitive (decided: real API, not test scaffolding)

`mmu::map` does a **local sfence only** and `core_mmu::map` errors
`AlreadyMapped` on an existing leaf — so "remap" can't be two `map`s.
We add a real primitive, which also closes the step-9 "mmu mutation
broadcasts shootdown" gap (today only the storm calls `shootdown`):

- **`kernel_core::mmu::remap(root_pa, va, new_pa, perms, mem)`** — pure,
  host-tested. Walks *existing* branches only (never allocates),
  overwrites the existing 4 KiB leaf, returns new
  `MapError::NotMapped` if there is no valid leaf to overwrite (unmapped
  VA, missing intermediate, or a huge leaf in the way).
- **`kernel::mmu::remap(va, new_pa, perms)`** — wrapper over
  `KernelPtMem`; on success calls **`shootdown(va)`** (cross-hart),
  not just a local sfence. This is the only real mmu path that fires
  shootdown.

## Workload `tlb-shootdown` (itest-workloads only)

`WorkloadKind::TlbShootdownVisible`, bootarg `workload=tlb-shootdown`
(host-tested parse). Two actors over one dedicated test VA `V` (a fresh
root slot above the heap, so `map` allocates clean intermediates) and
two pre-filled frames A (sentinel `0xA…`) and B (sentinel `0xB…`),
written via the linear map:

- **Setup:** `map(V→A)`; `expected = sentinel_A`; `round = 0`.
- **hart 0 initiator (task, yields so heartbeat keeps emitting):**
  for `i` in `1..=N`: wait `hart1_acked >= i-1` (hart 1 cached frame
  `i-1`); `remap(V → frame[i%2])` (→ shootdown flushes hart 1's stale
  entry); `expected.store(sentinel[i%2])`; `round.store(i, Release)`.
- **hart 1 reader (task via `spawn_on(1)`):** loop — Acquire-load
  `round`; on a new round read `*V`; if `!= expected` bump
  `stale_reads`; `hart1_acked.store(round, Release)`. The read in round
  `i-1` is what seeds the TLB that round `i`'s remap must invalidate.

Ordering: initiator stores `expected` then `round` (Release); reader
Acquire-loads `round` then reads V then `expected`. The shootdown
completes (hart 1's IPI handler sfenced + acked) inside `remap` before
`round` is published, so a *correct* shootdown means the round-`i` read
re-walks to frame `i`.

Heartbeat emits (always, not one-shot):
- `snitchos.smp.tlb_remap_rounds` — proves it actually ran.
- `snitchos.smp.tlb_stale_reads` — the oracle; must stay 0.

## Scenario

`smp-tlb-shootdown-visible` (cpu_bound): wait until `tlb_remap_rounds
>= N` (ran enough rounds — not a vacuous pass), and assert
`tlb_stale_reads` is never observed `> 0`.

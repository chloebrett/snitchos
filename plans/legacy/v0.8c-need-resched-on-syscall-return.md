# v0.8c ‚Äî `need_resched` on syscall return ‚Äî NOT A REAL GAP (resolved by analysis)

**Status:** **CLOSED ‚Äî investigated 2026-06-14, no fix needed.** Originally raised
as "a syscall-heavy userspace task can dodge timer preemption." On inspection the
escape is already closed by hardware interrupt masking. Recorded here so the
question isn't re-opened.

## The original hypothesis (WRONG)

v0.8 preemption fires only from `handle_timer` gated on `SPP == User`. The worry:
if the timer catches a task **mid-syscall** (`SPP == 1`), the preemption is
cancelled (not deferred), and a task that spends most of its time in S-mode (a
tight `ecall` loop) could dodge preemption disproportionately. Proposed fix: a
`need_resched` flag set in the timer and drained at the kernel‚Üíuser return.

## Why it does not happen

RISC-V clears `sstatus.SIE` to 0 when it takes a trap (and stashes the old value
in `SPIE`). **Nothing in SnitchOS re-enables `SIE` during trap handling** ‚Äî not
`trap_entry` (`kernel/src/trap.S`), not `trap_handler`, not any syscall handler.
The only SIE-set sites are:
- `enable_timer_interrupts` (`trap.rs`) ‚Äî boot-time global enable for normal exec.
- `user::run` (`user.rs`) ‚Äî sets `SPIE` so interrupts are live *in U-mode* after
  `sret`.

Consequences:
1. A whole syscall runs with **interrupts masked**. A timer coming due mid-syscall
   does not trap; it only sets the pending bit `sip.STIP`.
2. At `sret`, `SIE` is restored from `SPIE` (‚Üí1) and we drop to U-mode. The
   still-asserted timer (level-triggered: `STIP` stays set until `stimecmp` is
   advanced) fires **immediately**, now from U-mode ‚Üí `SPP == 0` ‚Üí ordinary
   preemption applies before the task retires meaningful work.
3. Therefore `handle_timer` is reached with `SPP == 1` **only** when the timer
   interrupts a genuine **kernel thread** running with `SIE == 1` (`task_a`,
   `idle`, `heartbeat`). Declining to preempt those is the *intended*
   cooperative-kernel design, not a gap.

A `need_resched` flag would be dead code: it would only ever be set in the
`SPP == 1` case, which corresponds to kernel threads we deliberately don't
preempt ‚Äî never to a deferred-from-syscall userspace task.

## If the interrupt discipline ever changes

This conclusion depends entirely on **"traps run with `SIE == 0`."** If a future
version re-enables interrupts inside long syscalls (e.g. to bound IRQ latency
during a slow IPC/copy path), then a timer *could* land with `SPP == 1` while
servicing a user task, and **this gap re-opens** ‚Äî at which point the
`need_resched` design above (set in timer when `SPP == 1`, drain at the
provably-lock-free kernel‚Üíuser return boundary) is the right fix. Until then: no.

## Guard ‚Äî BUILT 2026-06-14

`workload=syscall-hog` + scenario `syscall-hog-still-preempted` now pin the
"syscall-heavy tasks are still preempted" behaviour.

- **Program:** `user/hello/src/bin/syscall_hog.rs` ‚Äî loops a cheap ambient
  `DebugWrite` with no `yield`, so it sits at ~100% S-mode occupancy (the U-mode
  window per iteration is just the loop branch + `ecall` setup).
- **Layout:** `kmain` spawns `syscall_hog` + a cooperative `worker_a` on hart 1
  (mirrors the `user-hog` fixture).
- **Assertion:** a `ContextSwitch{Preempt}` leaving `syscall_hog` within 30s
  (the hog never yields, so a switch away from it is necessarily a preemption),
  plus a heartbeat-still-alive check.
- **Why it catches the regression:** at ~100% S-mode occupancy nearly every timer
  deadline lands with `SPP == 1`. If a future change enabled interrupts inside
  syscalls and skipped mid-syscall preemption without a `need_resched` drain,
  almost all preemptions would be dropped and the hog would dodge ‚Äî failing the
  assertion. `user-hog` (100% U-mode, always `SPP == 0`) cannot catch this; the
  two fixtures are complements.
- **Status:** passes 10/10 under `--repeat`, max wait ~1s. Host TDD: `bootargs`
  parse test `selects_syscall_hog` (RED‚ÜíGREEN) in `kernel-core`.

# v0.4 step 3 — physical frame allocator

Add a physical frame allocator. After this step, the kernel can ask
for a fresh 4 KiB physical page (via `frame::alloc()`) and give one
back (`frame::free(...)`). All of physical RAM becomes reachable
via a linear map at higher-half VAs.

Read `plans/v0.4-memory-step-3-frame-allocator-concepts.md` first.
That doc has the conceptual landscape and trade-offs; this doc is
the locked-in shape.

## Why this step

Frame allocator is the first thing in v0.4 that adds actual new
capability rather than infrastructure. Everything past here depends
on it:

- **Step 4 (kernel heap)**: heap needs backing pages.
- **Future per-process page tables (v0.6)**: each process's table is
  made from fresh frames.
- **Drivers**: DMA buffers, device-specific mappings.
- **`ioremap`-style** code that maps a new MMIO range later.

Without it, the kernel can only use the statically-reserved memory
that's part of the kernel image at boot. With it, the kernel can
grow.

## Decisions locked in

| decision | choice |
|---|---|
| Tracking strategy | Bitmap (1 bit per 4 KiB frame) |
| Bitmap storage | `static mut` array, cap'd at 4 GiB of RAM (128 KiB `.bss`) |
| Linear map | Yes; one 1 GiB Sv39 huge-page leaf in the root table |
| `LINEAR_OFFSET` | `0xffffffd0_00000000` (so PA `p` → VA `p + LINEAR_OFFSET`) |
| API shape | Free functions backed by `static Mutex<Allocator>` |
| Instrumentation | Atomic counters; drained by heartbeat loop |
| Init order | Before `unmap_identity` (so DTB is still reachable) |
| Reserved at boot | Kernel image (linker bounds), DTB 2 MiB region, the bitmap itself |

## Architectural details

### `LINEAR_OFFSET` and `pa_to_kernel_va`

```rust
// kernel-core/src/mmu.rs
pub const LINEAR_OFFSET: usize = 0xffffffd0_00000000;

/// Convert a physical address to the kernel's linear-map VA. Inverse
/// of `va_to_pa` for the linear-map range.
pub const fn pa_to_kernel_va(pa: usize) -> usize {
    pa + LINEAR_OFFSET
}
```

`pa_to_kernel_va` lives next to `va_to_pa` and `KERNEL_OFFSET`. Two
host tests pin: identity-range PA maps to expected VA; PA at top of
1 GiB linear range still fits in canonical-high.

### Linear-map leaf in `mmu::enable`

Add one more closure parallel to the existing `map_id_kernel`,
`map_higher_kernel`, `map_id_mmio`, `map_higher_mmio`:

```rust
// One 1 GiB huge-page leaf — installs a leaf directly in the root
// table, no mid table needed.
unsafe fn install_linear_map_leaf(perms: PtePerms) {
    let va = mmu::LINEAR_OFFSET + 0x80000000;  // start of RAM
    let pa = 0x80000000;
    let leaf = leaf_pte(pa, perms);
    let root_idx = (va >> 30) & 0x1ff;
    (&mut *(&raw mut BOOT_PT_ROOT)).set_entry(root_idx, leaf);
}
```

For QEMU `virt` with up to 1 GiB of RAM, one leaf covers everything.
If RAM exceeds 1 GiB we'd add more leaves (or a mid table). For now,
hard-cap at 1 GiB by ignoring DTB-reported RAM beyond that — and log
a warning.

### `kernel-core::frame::Bitmap`

Pure data. No allocator state; just the bit-twiddling.

```rust
pub struct Bitmap<'a> {
    bits: &'a mut [u64],
    capacity: usize,  // total frames (may be less than bits.len() * 64)
}

impl<'a> Bitmap<'a> {
    /// Wrap a backing buffer + capacity. All frames start in-use; caller
    /// must explicitly `free` the ones that are actually available.
    pub fn new(bits: &'a mut [u64], capacity: usize) -> Self;

    /// Mark a frame range as free (available for alloc). Used at init.
    pub fn release_range(&mut self, start_frame: usize, count: usize);

    /// Allocate the lowest-indexed free frame. Returns its frame index.
    pub fn alloc(&mut self) -> Option<usize>;

    /// Mark a frame as free.
    pub fn free(&mut self, frame: usize);

    pub fn capacity(&self) -> usize;
    pub fn count_free(&self) -> usize;
    pub fn count_in_use(&self) -> usize { self.capacity() - self.count_free() }
}
```

Host tests cover:
- new bitmap is fully in-use
- release_range marks the right frames free
- alloc returns the first free frame and marks it used
- free marks a used frame as available
- alloc/free round-trip
- alloc returns None when out of frames
- count_free / count_in_use track correctly

`bits` is a slice not an owned buffer — the kernel passes a slice of
its static `[u64; N]`. Kernel-core stays no-alloc.

### `kernel::frame` module

```rust
// kernel/src/frame.rs
use kernel_core::frame::Bitmap;

const FRAME_SIZE: usize = 4096;

// Static backing storage. 4 GiB / 4 KiB / 64 bits per u64 = 16384 u64s
// = 128 KiB. Sized for our cap; init clamps to actual RAM.
const MAX_FRAMES: usize = 4 * 1024 * 1024 * 1024 / FRAME_SIZE;
static mut FRAME_BITS: [u64; MAX_FRAMES / 64] = [0u64; MAX_FRAMES / 64];

// The allocator. Holds a Bitmap with a 'static reference into FRAME_BITS.
static FRAME_ALLOC: spin::Once<spin::Mutex<Bitmap<'static>>> = spin::Once::new();

// Telemetry counters — read by heartbeat, never blocked on alloc.
pub static ALLOC_COUNT: AtomicU64 = AtomicU64::new(0);
pub static FREE_COUNT: AtomicU64 = AtomicU64::new(0);
pub static ALLOC_FAIL_COUNT: AtomicU64 = AtomicU64::new(0);

pub unsafe fn init_from_dtb(dtb: &Fdt) -> Result<(), InitError> {
    // 1. Walk dtb.memory().regions() to find RAM bounds.
    // 2. Compute frame count = min(ram_size / FRAME_SIZE, MAX_FRAMES).
    // 3. Create Bitmap wrapping FRAME_BITS, capacity = frame count.
    //    All frames start in-use.
    // 4. Mark "free" the frames outside reserved regions:
    //    - kernel image (__kernel_start..__kernel_end)
    //    - DTB 2 MiB page (dtb_phys's aligned region)
    //    - frame bits storage (FRAME_BITS region)
    //    - SBI firmware (everything below kernel image start, i.e. 0x80000000..__kernel_start)
    // 5. Install in FRAME_ALLOC.
}

pub fn alloc() -> Option<PhysFrame> {
    let mut bitmap = FRAME_ALLOC.get()?.lock();
    match bitmap.alloc() {
        Some(frame_idx) => {
            ALLOC_COUNT.fetch_add(1, Ordering::Relaxed);
            Some(PhysFrame(frame_idx * FRAME_SIZE + RAM_BASE))
        }
        None => {
            ALLOC_FAIL_COUNT.fetch_add(1, Ordering::Relaxed);
            None
        }
    }
}

pub fn alloc_zeroed() -> Option<PhysFrame> { ... }
pub fn free(frame: PhysFrame) { ... }
pub fn stats() -> Stats { ... }
```

`PhysFrame` is a newtype around `usize` for type safety. It exposes
`addr()` (physical address) and `kernel_va()` (linear-map VA).

### Init order in `kmain`

Move `frame::init_from_dtb` between the existing init bits and
`unmap_identity`:

```rust
unsafe { trap::init_timer(timebase_hz) };
unsafe { frame::init_from_dtb(&dtb).expect("frame allocator init failed") };
let _ = dtb;
unsafe { mmu::unmap_identity() };
```

Then in the heartbeat loop, drain the frame counters into metrics
alongside the existing IRQ duration / intern count / time ticks.

### Telemetry drain

Same deferred pattern as the IRQ handler:

```rust
// In the heartbeat span:
tracing::emit_metric(frames_alloc_count, frame::ALLOC_COUNT.load(...) as i64);
tracing::emit_metric(frames_free_count, frame::FREE_COUNT.load(...) as i64);
tracing::emit_metric(frames_in_use_gauge, frame::stats().in_use as i64);
```

`frame::stats()` does take the lock briefly (to read `count_in_use`)
but it's outside any allocator's critical section, so no recursion
risk.

### Integration scenario

`frame-allocator-stats`: assert that the metric
`snitchos.frames.allocated_total` shows up at least once with value
> 0 (because the allocator will be exercised in step 4's heap or
we can exercise it during init for a smoke test) — or just assert
the gauge `snitchos.frames.in_use` is non-zero (since reserved
frames count as in-use).

Actually nothing in step 3 *allocates*. The bitmap just gets
initialized. So the assertion is on the *available* count — assert
that `snitchos.frames.free` is > 0 (we have RAM available).

Could also alloc one frame at init time as a smoke test and assert
the counter incremented.

## Risks and known weaknesses

- **DTB has zero RAM regions** (impossible on QEMU virt but worth a
  guard). `init_from_dtb` returns `Err(InitError::NoRam)`; kernel
  panics. Documented as "shouldn't happen."
- **Reserved-region overlap math.** If the DTB places things in
  weird orders we could double-reserve or miss a region. Mitigation:
  process reservations in a known order (kernel image, DTB region,
  bitmap region, sub-SBI region), with tests in kernel-core covering
  overlap.
- **`unmap_identity` runs after frame init.** Frame init reads DTB,
  which is in the identity gigapage. Fine because we init *before*
  unmap. If anyone moves init after unmap, panic.
- **1 GiB linear-map cap.** Hardcoded to one 1 GiB leaf. If a future
  platform has >1 GiB of RAM (QEMU virt with `-m 2G`), we'd silently
  ignore the excess. Init logs the cap; future step adds a multi-leaf
  case.
- **`MAX_FRAMES = 4 GiB / 4 KiB = 1M frames = 128 KiB of `.bss`.**
  Real cost. Worth it for the simplicity. If we ever care about
  smaller kernels we can shrink to a 1 GiB cap (32 KiB).
- **Race between alloc and stats.** `stats()` takes the lock briefly.
  Doesn't matter — heartbeat is single-threaded today, and
  Mutex<Bitmap> is single-hart.

## Done state

- `cargo test -p kernel-core` covers the new `Bitmap` + `LINEAR_OFFSET`
  /  `pa_to_kernel_va` tests.
- `cargo build -p kernel --target riscv64gc-unknown-none-elf` clean.
- `cargo xtask test` passes all existing scenarios + new
  `frame-allocator-stats` (5 total).
- Metric `snitchos.frames.free` appears on the wire with a value
  reflecting RAM minus reservations.
- The linear map is installed in the boot table; from any kernel
  code, `pa_to_kernel_va(p)` yields a usable VA for any allocated
  frame.

## What's not in this plan

- **Step 4**: kernel heap on top of frames.
- **Contiguous multi-frame allocation** (for DMA buffers).
- **Frame reference counts**.
- **Per-frame metadata** (Linux's `struct page`).
- **NUMA-aware allocation**.
- **Multi-leaf linear map** for >1 GiB platforms.

## Things to flag during execution

- After step 2, `&raw const __kernel_start as usize` gives the
  *higher-half* VA (PC at higher-half + PC-relative). For frame
  init's reservation pass we need the *physical* range. Compute via
  `va_to_pa(&raw const __kernel_start as usize)`.
- Same for `__kernel_end` and `FRAME_BITS`.
- The `dtb` borrow has to outlive `frame::init_from_dtb` — confirm
  the `let _ = dtb;` lands AFTER frame init.

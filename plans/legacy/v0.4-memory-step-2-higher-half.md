# v0.4 step 2 — higher-half kernel + identity unmap

Move the kernel from running at identity virtual addresses (where
VA = PA) to running at higher-half virtual addresses (Linux-style
PAGE_OFFSET layout). When step 2 lands, `kmain` executes at
`0xffffffff80200000`-ish, `sp` points into the higher-half, and the
identity mapping is gone — any access to a low VA faults.

Read `plans/v0.4-memory-concepts.md` § 3-4 first; `plans/v0.4-memory-step-1-satp-on.md`
for the foundation step 2 builds on.

## Why this step

Step 1 turned paging on but with identity mappings only — no
behavioural change. Step 2 is the behavioural transition: the
kernel image lives at a different virtual address than its physical
address.

Forward-looking benefits (none of these are unlocked *today*, all
gate later milestones):

- **v0.6 userspace** wants the kernel mapped at the same VAs from
  every process's address space. Higher-half is the convention
  every kernel uses for this.
- **Page-fault diagnostics from boot.** A NULL deref now faults with
  scause = "page fault, no mapping at 0x0" instead of touching
  physical 0.
- **Linker layout matches Linux/BSD.** Lessons and references
  translate directly.

We do this now (vs. deferring to v0.6) because the boot table is
small, mostly empty, and the test scaffolding is fresh. Doing it
later means churning more code.

## Decisions (defaults — flag if you want to override)

- **`KERNEL_OFFSET = 0xffffffff_80000000`.** Kernel image at
  `0xffffffff_80200000` (= `KERNEL_OFFSET + 0x80200000`). Matches
  Linux RISC-V's `PAGE_OFFSET`. Alternative: `0xffffffc0_00000000`
  (low end of canonical-high). Either works for Sv39; going with
  Linux convention.
- **Linker ORIGIN moves to higher-half.** Every symbol's VMA is
  higher-half. **LMA stays at `0x80200000`** — OpenSBI still loads
  the kernel at that physical address. Achieved with
  `AT(0x80200000)` on the SECTIONS clauses (or `AT> ram_phys` with
  a second MEMORY region). Decision: use `AT(...)` for simplicity.
- **Dual-map happens by extending the existing `mmu::enable` —
  same boot table, more entries.** Add a third static
  `BOOT_PT_MID_HIGHER` for the higher-half mid table. Root gains
  one more branch entry (root index 511 for VA `0xffffffff_80200000`).
  No fresh page table allocation needed.
- **Trampoline lives in Rust (`mmu.rs`), not in `entry.S`.** A small
  `asm!` block does the absolute jump to higher-half + sp fix-up.
  Restructuring `entry.S` to know about higher-half is a bigger
  change reserved for the "satp in entry.S" step much later.
- **Trampoline mechanism: explicit absolute jump via `li` of the
  full 64-bit higher-half target, then `jr`.** Don't use `la`
  because `la` is PC-relative and would compute an identity-half
  address. Construct the higher-half VA by `add`ing
  `KERNEL_OFFSET` to a PC-relative result, then `jr`.
- **`sp` fix-up: `sp += KERNEL_OFFSET`.** After the trampoline,
  while still at identity-half (the addresses are aliased through
  the dual-mapped table), bump sp once. Stack contents are
  identical because the same physical page is reachable through
  both VAs.
- **Identity unmap: zero the root entries that cover identity
  ranges (`root[0]` for MMIO, `root[2]` for kernel image), then
  `sfence.vma`.** Don't bother zeroing the mid-table entries — the
  root branch is gone, so nothing can walk to them. They're dead
  data inside `BOOT_PT_MID_KERNEL` / `BOOT_PT_MID_MMIO`. The
  higher-half mid table stays live.
- **Wait — MMIO needs higher-half mappings too.** virtio-mmio +
  UART are accessed from the heartbeat loop's frame emit. If we
  unmap their identity mapping, the writes go to higher-half VAs
  for them too. **So the higher-half mid table must contain
  entries for MMIO regions, not just the kernel image.** This is
  a real change from step 1's mapping set.
- **Success signal: span `kernel.mmu.higher_half` around the
  trampoline + new integration scenario.** Tests that the span's
  SpanStart appears (emitted from identity-VA code), the matching
  SpanEnd appears (emitted from higher-half-VA code), and a
  heartbeat follows.

## Architectural details

### The linker change

`linker.ld`:

```ld
MEMORY {
  ram (rwx) : ORIGIN = 0xffffffff80200000, LENGTH = 128M
}
SECTIONS {
  __kernel_start = ORIGIN(ram);
  .text : AT(0x80200000) {
    KEEP(*(.text.boot))
    *(.text .text*)
  } > ram
  .rodata : { ... } > ram
  ...
}
```

`AT(0x80200000)` keeps the LMA at the physical address OpenSBI
expects to load us at, while the VMA (used for symbol values) is
higher-half. The kernel ELF will report headers like "this section
loads at 0x80200000, runs at 0xffffffff80200000."

### What's already running between `_start` and `mmu::enable()`?

This is the subtle part. At `_start`:
- PC = `0x80200000` (physical address, where OpenSBI jumped).
- Symbol values in the binary are higher-half.
- entry.S uses PC-relative addressing for everything (jal, branches,
  auipc + addi). So `la t0, __stack_top` computes the
  *higher-half* address of `__stack_top` because the offset
  encoded in the auipc was generated assuming higher-half link
  addresses.

**That means `sp` immediately gets set to a higher-half VA, but
the MMU is off and that VA is unmapped.** The kernel would fault
on the first stack push.

Solution: entry.S needs to compute the identity-half address of
`__stack_top` by subtracting `KERNEL_OFFSET` from the
linker-provided value. Same trick for any absolute address it
needs.

Concretely entry.S changes from:

```asm
la sp, __stack_top
```

to:

```asm
la sp, __stack_top                     # sp = higher-half VA
li t0, 0xffffffff80000000              # KERNEL_OFFSET
sub sp, sp, t0                         # sp -= KERNEL_OFFSET → identity VA
```

After `mmu::enable()` runs (dual-mapped), both VAs work. After the
trampoline + sp fix-up + unmap, only higher-half works.

### The dual-map

`mmu::enable()` already identity-maps kernel + MMIO. Extend it to
also map the same physical pages at higher-half VAs:

```rust
// Identity (carry over from step 1):
for region in regions { map_2mib(root, mid_identity, region.va_id, region.pa, perms); }

// Higher-half (new in step 2):
for region in regions {
    let va_high = region.pa + KERNEL_OFFSET;
    map_2mib(root, mid_higher, va_high, region.pa, perms);
}
```

Both mid tables are filled; root has entries for both gigapages.
The higher-half kernel gigapage's root index is 511 (for
`KERNEL_OFFSET = 0xffffffff_80000000`). MMIO at PA `0x10000000`
maps to higher-half VA `0xffffffff_90000000`, root index 510... or
do we want MMIO at a different higher-half VA?

Actually — MMIO mapping at `PA + KERNEL_OFFSET` puts UART at
`0xffffffff_90000000`. That works but it's weird to have MMIO
mixed in with kernel image at the same offset. Real kernels often
use a separate "vmalloc" or "ioremap" range for device mappings.
For step 2, **keep it simple: MMIO at `PA + KERNEL_OFFSET`, same
offset as the kernel.** Real ioremap can come later.

### The trampoline

```rust
pub unsafe fn jump_to_higher_half() {
    // Compute the higher-half target VA for the jump.
    // `la` is PC-relative → gives us the *identity-half* address
    // of the resume point. Add KERNEL_OFFSET to get higher-half.
    unsafe {
        asm!(
            // Compute target = (PC-relative addr of 1f) + KERNEL_OFFSET.
            "la   t0, 1f",
            "li   t1, {offset}",
            "add  t0, t0, t1",
            // Fix sp the same way.
            "add  sp, sp, t1",
            // Jump.
            "jr   t0",
            "1:",
            // We're now at higher-half. Fall through to caller.
            offset = const KERNEL_OFFSET,
            options(nostack),
        );
    }
}
```

After `jump_to_higher_half()` returns, the caller's `ra` is... still
identity-half. When `jr ra` happens to return, we'd jump back to
identity, which is still mapped (dual-map). Then we'd unmap and
fault.

**Fix:** the trampoline must not return via the normal `ra`. Either:
- (a) Inline the trampoline directly in `kmain` so there's no
  `ra`-based return; the next instruction after the `asm!` block is
  what we want.
- (b) Make the trampoline a `!` function that calls a continuation.

Option (a) is the simpler thing — `kmain` does the trampoline
inline. Going with (a).

But wait — if we inline the trampoline in `kmain`, then the
function-local `ra` and other saved registers in the prologue
were stored at identity-half VAs. When `kmain` returns (it
doesn't, but for any local function call inside kmain),
saved-register loads use sp, which is now higher-half, and the
physical memory is the same. So the prologue's saved data is
reachable via higher-half VA. Good — works because dual-map is
still in place at this point.

The identity unmap has to happen AFTER any returns/jumps that
depend on identity-half PCs. Since we're inlining and there's no
return, this is fine.

### Identity unmap

```rust
pub unsafe fn unmap_identity() {
    unsafe {
        // Zero the root entries that cover identity-half ranges.
        let root = &mut *(&raw mut BOOT_PT_ROOT);
        root.set_entry(0, 0);  // MMIO identity gigapage
        root.set_entry(2, 0);  // Kernel identity gigapage
        asm!("sfence.vma", options(nostack));
    }
}
```

`PageTable` needs a new `set_entry(idx, value)` method (or
`clear_entry(idx)`). Pure logic, add to kernel-core with a host
test.

### kmain sequence after step 2

```rust
unsafe { trap::init_timer(timebase_hz) };

{
    span!("kernel.mmu.enable");
    unsafe { mmu::enable(&dtb) };  // dual-mapped now
}

{
    span!("kernel.mmu.higher_half");
    unsafe {
        mmu::jump_to_higher_half();  // sp and PC now at higher-half
        mmu::unmap_identity();        // sfence; identity is gone
    }
}

// Heartbeat loop. Runs at higher-half VAs.
loop { ... }
```

Both spans appear on the wire. The `higher_half` SpanEnd is emitted
from code that's running at a higher-half VA — proving the
trampoline worked.

## Risks and known weaknesses

- **`sp` arithmetic wrong direction.** entry.S subtracts
  KERNEL_OFFSET (to get identity VA from higher-half symbol). The
  trampoline adds it (to get higher-half VA from identity PC).
  Easy to swap mentally and end up with bizarre fault addresses.
  Pin both directions in comments.
- **`la t0, 1f` and the `1:` label inside the same `asm!` block.**
  Local labels (numeric) work in Rust `asm!` blocks. Verify with a
  build before relying on it.
- **Stack contents during the trampoline.** `kmain`'s local
  variables in stack frames are reachable via both VAs while
  dual-mapped. After the unmap, identity-half loads of stack would
  fault. The compiler's stack accesses use sp-relative addressing,
  so as long as `sp` is higher-half, loads resolve to higher-half
  VAs. Verified by the fact that `kmain` doesn't take identity-half
  pointers to its own locals.
- **Statics in `.bss` (intern table, span registry, pre-init
  buffer).** Their addresses are higher-half (linker says so).
  Accesses use higher-half VAs. The dual-map covers them; after
  unmap, only higher-half resolves — which is what we want.
- **Function pointers.** The address-of operator on a function
  yields its higher-half VA. Calls through function pointers
  (e.g. the trait-object dispatch in `KernelSink`) go to
  higher-half. Fine post-unmap.
- **`virtio_console::CONSOLE.get()`.** The `Once<spin::Mutex<usize>>`
  holds the MMIO base as a `usize` — a physical address. The mutex
  itself is at a higher-half VA. The value it returns is still the
  physical address `0x10000000`, which we're now using as a
  *virtual* address. Since we mapped `0x10000000` → higher-half VA
  `0xffffffff90000000`, accessing the stored value as a VA goes
  through identity which is unmapped → fault.

  **This is the big landmine.** Either:
  - (a) Update `CONSOLE` to store the higher-half VA after unmap.
  - (b) Keep an MMIO identity mapping forever (don't unmap MMIO
    in identity).
  - (c) Translate `virtio_console::send`'s use of the address —
    add KERNEL_OFFSET when dereferencing.

  Option (a) is correct long-term. Option (b) is a pragmatic
  shortcut. Option (c) leaks the higher-half detail into virtio_console.
  **Recommendation: (a) — update CONSOLE to the higher-half VA as
  part of the trampoline.** Same logic applies to the `UART`
  static in `console.rs` (NS16550A base also stored as physical
  address).

  This is a real chunk of step 2 that's easy to forget. Flagged.

- **`fdt::Fdt`'s borrow.** The kernel's `dtb` variable is a
  `&Fdt` over the DTB bytes at the physical address OpenSBI gave
  us. That borrow is live until `dtb` goes out of scope in kmain.
  After unmap, dereferences through `dtb` go to its physical
  address as a VA — unmapped, faults. We're done using `dtb` by
  the time we trampoline, but the compiler doesn't know. Either
  drop `dtb` explicitly before unmap, or accept the borrow is
  trivially dropped at scope end after the loop.

  Recommendation: explicitly `drop(dtb)` before
  `unmap_identity()`. One line, makes intent clear.

## Decisions to confirm before execution

These are the ones with non-trivial alternatives. Flag now or
during execution:

1. **`KERNEL_OFFSET` value.** Defaulting to `0xffffffff_80000000`
   (Linux RISC-V). Override to `0xffffffc0_00000000` if you want
   the lower end of canonical-high.
2. **MMIO higher-half VA.** Defaulting to `PA + KERNEL_OFFSET`
   (kernel range). Override to a separate ioremap range if you'd
   rather keep MMIO out of the kernel-image range.
3. **`CONSOLE` / `UART` static update.** Mentioned in risks — these
   need to be patched as part of the trampoline (option (a)).
   Confirming this is the chosen approach.
4. **`entry.S` change.** Necessary so `sp` lands at an identity-half
   VA at boot. Confirming we're touching entry.S in this step (vs.
   leaving it for a later step where the whole boot path moves
   into asm).

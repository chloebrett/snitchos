# v0.5 pre-factor — SMP-shaped sync primitives

Land before v0.5 threading. Introduce `kernel::sync::{Mutex, Once}`
as the single chokepoint for kernel-internal locking, and a stub
`kernel::percpu::PerCpu<T>` for per-CPU data. Today everything
behaves exactly like the underlying `spin::Mutex` / `spin::Once` /
`[T; 1]`; the wrappers exist so that when preempt-disable (v0.9)
and SMP (v0.6) land, the change is *one file*, not 20 call sites.

This is purely architectural plumbing. No behaviour change today.

**Status (post-v0.5):** delivered. This prefactor is the load-bearing
prerequisite for v0.6 (SMP cooperative) — `PerCpu<T>` will be
reimplemented over `tp`-based per-hart data and the `kernel::sync`
chokepoint absorbs the multi-hart locking semantics in one place.
Preempt-disable hooks remain no-op until v0.9 (preemption). See
`plans/legacy/v0.6-smp-cooperative.md`.

## Why this step

Today the kernel has six `spin::Mutex`-based statics (INTERN_TABLE,
PRE_INIT_BUFFER, UART, CONSOLE, FRAME_ALLOC, HEAP) and three uses of
`spin::Once`. None are taken from IRQ context, so raw `spin::Mutex`
works correctly. That correctness is incidental — it depends on a
discipline ("don't lock from IRQ handlers") that's currently
enforced by the IRQ handler being a one-liner.

Threading (v0.5) and preemption (v0.5.x) put real pressure on lock
semantics:

- **Don't context-switch while holding a spin lock.** With
  cooperative scheduling, this is a rule; with preemption, it
  needs structural enforcement (preempt-disable counter).
- **IRQ-safe lock acquisition.** The moment any IRQ handler wants
  to touch a kernel data structure (log a tick, signal a worker
  thread, update a metric), we need IRQs disabled during lock
  acquisition or we self-deadlock.
- **Per-CPU state.** SMP-correct kernels keep a lot of state
  per-hart (current task, runqueue, IRQ depth, preempt counter).
  Even single-hart code should be written through a per-CPU
  abstraction so SMP migration is mechanical.

Doing the migration *before* threading lands means v0.5's actual
scheduler work writes against `kernel::sync` from day one. The
wiring of preempt-disable and SMP IRQ-disable happens in *one*
place when each lands.

## Decisions locked in

| decision | choice |
|---|---|
| Lock flavour | **One** flavour (IRQ-safe by default); see "Why" + follow-ups |
| Where the wrapper lives | `kernel/src/sync.rs` — no kernel-core piece |
| Naming | Short names: `kernel::sync::{Mutex, MutexGuard, Once}`. Idiomatic Rust convention (`std::sync::Mutex`, `tokio::sync::Mutex` — every crate ships its own) |
| Raw `spin::Mutex` outside `kernel::sync` | Forbidden via clippy `disallowed_types` lint |
| Migration shape | Single pass, type-driven find-and-replace, one commit per pre-factor step |
| Per-CPU storage | `kernel::percpu::PerCpu<T>` wrapping `[T; MAX_HARTS]`, indexed by `current_hartid()` (returns 0 today) |
| `MAX_HARTS` | 1 for v0.5; bumped when SMP actually lands |
| `current_hartid()` impl | Returns 0 unconditionally for v0.5. Real impl (`sscratch` or `mhartid` route) when SMP arrives |

## Why these choices

- **One flavour now.** The deadlock failure mode of "wrong flavour
  picked for a lock the IRQ handler grabs" is exactly the bug
  category we don't want to debug intermittently. Always-IRQ-safe
  is conservative and costs ~2 CSR instructions per lock — invisible
  at our workload scale. **Reserve the right to split into
  `lock()` + `lock_irqsave()` later** if a perf-critical path
  needs it; the wrapper's internals are the only thing that has
  to change, callers stay the same.
- **Short names + lint enforcement.** Matches the Rust ecosystem
  (`std::sync::Mutex`, `parking_lot::Mutex`, `tokio::sync::Mutex`
  — every crate has its own). The `disallowed_types` clippy lint
  closes the "stray `use spin::Mutex`" hole. Keeps call sites
  clean.
- **No kernel-core piece.** `kernel-core` has `#![forbid(unsafe_code)]`
  and no statics with side effects — locks live where the hardware
  is. Keeps the existing kernel-core / kernel boundary clean.
- **Single-pass migration.** Six locks, three `Once`s, all in
  kernel/. The diff is a mechanical type swap — splitting it across
  commits gains nothing.
- **`PerCpu<T>` stub.** The shape (`PerCpu<RunQueue>`, `PerCpu<TaskPtr>`)
  drives the call-site syntax SMP will need. Implementing the stub
  now means future SMP work doesn't need to retrofit every per-CPU
  read/write through a new abstraction.
- **`current_hartid() = 0`.** Same idea — having callers go through
  the function today means swapping in the real impl is one-line.
  No need to commit to `sscratch`-vs-CSR-route yet; that's a
  decision for the actual SMP work.

## Big-picture step list

| # | step | size |
|---|---|---|
| 1 | `kernel::sync` module with `Mutex<T>` wrapping `spin::Mutex<T>` (currently no-op hooks); `Once<T>` wrapping `spin::Once<T>` | small |
| 2 | `kernel::percpu` module with `PerCpu<T>` + `current_hartid()`; both behave as if single-hart | small |
| 3 | Add `disallowed_types` clippy lint at workspace level blocking raw `spin::Mutex` / `spin::Once` outside `kernel::sync` | small |

## Architectural details

### `kernel::sync`

```rust
// kernel/src/sync.rs
use core::ops::{Deref, DerefMut};

pub struct Mutex<T> {
    inner: spin::Mutex<T>,
}

pub struct MutexGuard<'a, T> {
    inner: spin::MutexGuard<'a, T>,
    // Future fields (v0.5.x / SMP):
    //   prior_irq_enabled: bool,
    //   preempt_token: PreemptToken,
}

impl<T> Mutex<T> {
    pub const fn new(value: T) -> Self {
        Self { inner: spin::Mutex::new(value) }
    }

    pub fn lock(&self) -> MutexGuard<'_, T> {
        // Acquisition hook stub. v0.5.x adds preempt::disable();
        // SMP adds save+disable of sstatus.SIE here.
        MutexGuard { inner: self.inner.lock() }
    }
}

impl<T> Deref for MutexGuard<'_, T> {
    type Target = T;
    fn deref(&self) -> &T { &self.inner }
}

impl<T> DerefMut for MutexGuard<'_, T> {
    fn deref_mut(&mut self) -> &mut T { &mut self.inner }
}

impl<T> Drop for MutexGuard<'_, T> {
    fn drop(&mut self) {
        // Release hook stub. v0.5.x restores preempt; SMP restores
        // sstatus.SIE.
    }
}

pub struct Once<T> {
    inner: spin::Once<T>,
}

impl<T> Once<T> {
    pub const fn new() -> Self {
        Self { inner: spin::Once::new() }
    }

    pub fn call_once<F: FnOnce() -> T>(&self, f: F) -> &T {
        self.inner.call_once(f)
    }

    pub fn get(&self) -> Option<&T> {
        self.inner.get()
    }
}
```

The `MutexGuard` deliberately holds the inner guard, not a reference
to it. That's the seam where preempt-disable / IRQ-disable state will
live (a `bool` for prior SIE, a token for preempt counter). `Drop`
will restore them in reverse order of acquisition.

### `kernel::percpu`

```rust
// kernel/src/percpu.rs
const MAX_HARTS: usize = 1;

pub struct PerCpu<T> {
    cells: [T; MAX_HARTS],
}

impl<T> PerCpu<T> {
    pub const fn new_with(cells: [T; MAX_HARTS]) -> Self {
        Self { cells }
    }

    /// Read this CPU's slot. Single-hart today; SMP later indexes by
    /// `current_hartid()`.
    pub fn this_cpu(&self) -> &T {
        &self.cells[current_hartid()]
    }

    pub fn this_cpu_mut(&mut self) -> &mut T {
        &mut self.cells[current_hartid()]
    }
}

/// Current hart's ID. v0.5: always 0. SMP: read from `sscratch`
/// (populated by trap entry) or `mhartid` (M-mode delegate).
#[inline]
pub fn current_hartid() -> usize {
    0
}
```

No use sites in this prefactor. The type exists so v0.5 scheduler
work can write `static CURRENT_TASK: PerCpu<...>` from day one; SMP
just changes `MAX_HARTS` and `current_hartid()`.

### `disallowed_types` lint

`clippy.toml` at the workspace root:

```toml
disallowed-types = [
    { path = "spin::Mutex", reason = "use kernel::sync::Mutex" },
    { path = "spin::MutexGuard", reason = "use kernel::sync::MutexGuard" },
    { path = "spin::Once", reason = "use kernel::sync::Once" },
]
```

`kernel::sync` itself needs `#[allow(clippy::disallowed_types)]` to
construct the inner types. That's the *only* place the allow is
permitted; every other site goes through the wrappers.

`kernel-core` and `protocol` don't use spin and aren't affected.

### Migration mechanics

Six type swaps:

```rust
- static INTERN_TABLE: spin::Mutex<InternTable> = spin::Mutex::new(...);
+ static INTERN_TABLE: crate::sync::Mutex<InternTable> = crate::sync::Mutex::new(...);
```

The `MutexGuard` lifetime and `Deref` semantics are identical
(wrapper just forwards), so callers don't notice.

For `spin::Once<spin::Mutex<T>>`:

```rust
- static FRAME_ALLOC: spin::Once<spin::Mutex<Allocator>> = spin::Once::new();
+ static FRAME_ALLOC: crate::sync::Once<crate::sync::Mutex<Allocator>> = crate::sync::Once::new();
```

The migration commit should not change any logic. Bisectability matters
if a weird interaction shows up later.

## What this doesn't do

- **No actual preempt-disable.** The hooks in `Mutex::lock()` /
  `Drop` are no-ops. Preempt-disable lands with v0.5.x (or rolled
  into v0.6 prep — whichever sequence fits).
- **No actual IRQ-disable.** Same — the hook exists, the
  implementation is empty. Adding it is a few CSR instructions in
  `Mutex::lock()` + `Drop`, gated on `current_hartid()`-aware
  scheduling discipline.
- **No real `current_hartid()`.** Returns 0. The real impl needs
  trap-entry plumbing to populate `sscratch` (or equivalent). When
  SMP lands, this is the first piece to fix.
- **No `RwLock`, no `BiteMutex`, no fairness/ticket variants.**
  None of our current locks need read-mostly semantics or
  starvation-free guarantees. Add when motivated.
- **No `Lazy` / `OnceCell`-style abstractions.** `Once<T>` covers
  what we use. Future ergonomics adds are easy.
- **No `static mut` audit.** Several kernel statics are accessed
  via raw `&raw const` rather than locks (the page table root,
  the frame bitmap backing storage). They're correct under
  single-hart by construction; SMP work will revisit each.

## Risk and known unknowns

- **The `disallowed_types` lint might miss generic usage.** E.g.
  `Vec<spin::Mutex<X>>` could slip through. Mitigation: after the
  migration commit, `cargo clippy` and `grep -r "spin::"` should
  both come up empty outside `kernel::sync` + Cargo.toml.
- **Future preempt-disable adds latency to every lock acquisition.**
  When v0.5.x lands the real preempt counter, expect a measurable
  bump on heap alloc paths (which take the heap lock per call).
  Probably fine; flag for profiling when the time comes.
- **`Once<Mutex<...>>` is two acquisitions for one logical op.**
  This is true today and continues post-prefactor. Slight
  pessimisation if both ever need preempt-disable. Acceptable.
- **API drift risk.** If `spin` releases a 1.0 with breaking
  changes, our wrapper insulates callers but we still have to
  update the wrapper. Net cost of the prefactor: one update site
  instead of N.

## Follow-ups (deferred, but tracked)

- **Split into `lock()` (non-IRQ) + `lock_irqsave()` (IRQ-safe)**
  when a perf-critical hot path emerges that takes a lock the IRQ
  handler provably cannot touch. The wrapper internals change; the
  signatures stay backward-compatible. Drive by measurement, not
  speculation.
- **Real `current_hartid()`** via `sscratch` — set up in trap entry
  to point at per-hart state, readable via a single load. Lands
  with the first multi-hart bring-up.
- **Real preempt-disable counter** lives in per-task or per-CPU
  state. The `MutexGuard`'s preempt-token field becomes a real
  thing.
- **Real IRQ-disable** in `Mutex::lock()` saves and clears
  `sstatus.SIE`; `Drop` restores. ~6 CSR instructions inline.

## Done means

- `kernel/src/sync.rs` and `kernel/src/percpu.rs` exist.
- All six static `Mutex` sites + three `Once` sites are migrated.
- `clippy.toml` disallowed-types blocks raw spin usage outside
  `kernel::sync`.
- `cargo xtask test` — all 8 integration scenarios still pass.
- `cargo test -p kernel-core --lib` — all 84 host tests still
  pass (no kernel-core changes).
- `cargo clippy --all-targets --all-features` — no new warnings;
  `disallowed_types` fires only on intentional internal usage
  inside `kernel::sync` (with an allow).
- `grep -rn "spin::" kernel/src/` — only matches inside
  `sync.rs`.
- CLAUDE.md notes that all kernel-internal locking goes through
  `kernel::sync`; documents the one-flavour rationale and the
  two-flavour upgrade path.
- README "Working" bullet acknowledges the lock surface and the
  preempt-disable / IRQ-disable hooks (currently no-op).

## How this sets up v0.5 threading

Once this lands, the v0.5 plan's "lock discipline" risk section gets
simpler: every kernel lock already goes through `kernel::sync`. To
add cooperative-yield safety (debug-assert no yield while holding a
lock), or eventually preempt-disable, *the only file that changes
is `kernel::sync`*. v0.5 itself doesn't have to revisit any of the
six lock sites.

`PerCpu<T>` is similar — `static CURRENT_TASK: PerCpu<Option<TaskId>>`
is the v0.5 shape, written from day one. SMP later just changes
`MAX_HARTS` and `current_hartid()`; the call sites are stable.

Pre-factoring the sync layer first turns v0.5 threading from "build
scheduler + audit all locks for safety" into "build scheduler; the
locks are already shaped right."

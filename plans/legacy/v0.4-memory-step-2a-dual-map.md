# v0.4 step 2a — dual-map (kernel runs at identity, higher-half added)

The smallest possible move toward higher-half. **No linker change,
no behavioural change.** Extend the boot page table so the kernel
image is mapped twice: at its identity VA (where the kernel actually
runs) and at its higher-half VA (where step 2c will eventually
relocate the kernel).

Read `plans/v0.4-memory-findings.md` first if any of this looks
arbitrary — that doc records what already went wrong on a more
ambitious attempt.

## Why this scope

Higher-half is at least 3 landings, not 1. This plan is the first
one. After it lands:

- Boot table has both identity and higher-half mappings for the
  kernel image.
- Kernel still runs at identity VAs (no behaviour change).
- `cargo xtask test` is byte-identical to before.

That's it. It's deliberately under-ambitious — the goal is to
prove we can build a dual-mapped table without breaking anything,
in isolation from the trampoline and linker-layout work.

## Non-goals (explicit, to prevent scope creep)

- **No linker change.** `ORIGIN(ram)` stays at `0x80200000`.
- **No `mmu::enable` move.** Stays late in `kmain`, just before
  the heartbeat loop.
- **No trampoline.** No `jump_to_higher_half`. No `sp` fix-up.
- **No identity unmap.** No `unmap_identity`.
- **No `CONSOLE` / `UART` static patching.** Those still hold
  physical addresses; we don't touch them.
- **No DTB-region or MMIO higher-half mapping.** Only kernel image.
  MMIO stays identity-only.

The higher-half mapping is unused after this step. That's fine.

## Concrete changes

Just two:

1. **`kernel/src/mmu.rs`**:
   - Add `pub const KERNEL_OFFSET: usize = 0xffffffff_00000000;`.
   - Add `static mut BOOT_PT_MID_HIGHER_KERNEL: PageTable = PageTable::new();`.
   - In `enable()`, inside the existing kernel-image loop, add a
     second `map_2mib` call that puts a leaf in
     `BOOT_PT_MID_HIGHER_KERNEL` at `addr + KERNEL_OFFSET` →
     `addr`, with the same `perms`.
2. **`kernel-core/src/mmu.rs`**: one host test pinning that a
   higher-half VA's vpn2 differs from identity vpn2 — `(0xffffffff_80200000 >> 30) & 0x1ff` is
   510, kernel identity at `0x80200000` is vpn2 2. Different
   gigapages → different root entries → no aliasing.

That's the whole change. No other file touched.

## Verification

- `cargo test -p kernel-core` — the new test passes, all others
  still pass.
- `cargo xtask test` — all 4 scenarios still pass, no behaviour
  change.
- `cargo build -p kernel --target riscv64gc-unknown-none-elf` —
  builds clean.

No new integration scenario. There's nothing observable to assert
against — the higher-half mapping exists in the table but the
kernel never uses it. We're just proving the table-building code
doesn't break anything.

## Done state

- Three things change: one constant, one static, one duplicated
  call in `enable()`. Plus one kernel-core test.
- Kernel binary builds and boots identically to before.
- All tests green.

## What's not in this plan

Step 2b: dummy read through a higher-half VA after `enable()` to
prove the higher-half mapping actually works at runtime (smoke
test, no linker dependency).

Step 2c: linker change to higher-half ORIGIN + move `mmu::enable`
early. Where the real risk lives. Separate plan.

Step 2d: trampoline + sp fix-up + identity unmap. Where the
kernel transitions from "running at identity, higher-half exists"
to "running at higher-half, identity gone." Separate plan again.

## Things to flag if they happen during execution

- **Build error on `KERNEL_OFFSET` value.** `0xffffffff_00000000`
  fits in `usize` on 64-bit. If clippy complains about the
  signed-vs-unsigned thing, suppress with a justified
  `#[allow]`.
- **Aliasing concerns with `&raw mut BOOT_PT_*`.** Three
  simultaneous mutable references via `&raw mut`. The compiler
  doesn't complain today; if it ever does, take pointers as
  `*mut PageTable` instead of `&mut PageTable`.
- **Test count change.** Should go from 37 to 38 (one new mmu
  test). Anything else means something else broke.

# Plan: v0.6 step 11 — migrate producer/consumer workload to SMP

**Branch**: main (per project rule — all work on main)
**Status**: Active

## Goal

Run the producer on hart 0 and the consumer on hart 1 so the
`Mutex<VecDeque>` queue takes genuine cross-hart contention, and prove
the correctness oracle (`histogram_sum >= samples_consumed`) holds
across the hart boundary under repeated runs.

## Context (where we are)

The SMP substrate is fully in place and tested: two harts boot
(`smp-secondary-hart-boots`), IPI wakeup works (`ipi-self-wakeup`,
`smp-spawn-on-hart-1-runs`), TLB shootdown landed (`mmu.rs:411`),
per-CPU lift is done (`CURRENT_TASK`/`CURRENT_SPAN_CURSOR` are
`PerCpu`), and `spawn_on(hart, …)` runs a task cross-hart.

What is *not* done: the headline workload still runs entirely on hart 0
(`main.rs:309-310` — both `producer_entry` and `consumer_entry` go
through `sched::spawn`, i.e. hart 0). The v0.6 thesis — "the chokepoint
earns its keep under real cross-hart contention" — has therefore never
actually been exercised.

## Key design decision (needs sign-off)

**The SMP workload is a feature-gated kernel variant, not the new
default boot.**

- The default build keeps producer + consumer on hart 0, so
  `workload-cooperative-baseline` (which runs the default build) stays
  green and continues to mean "single-hart baseline."
- A new `smp-workload` cargo feature spawns producer on hart 0 and
  consumer on hart 1, and strips `task_a`/`task_b` for a clean
  measurement surface (mirrors how the `deflake-*` features replace the
  default boot workload).
- The new `smp-producer-consumer-correctness` scenario builds the
  kernel with `--features smp-workload` via
  `Harness::spawn_with_features`, exactly like `heap-oom`/`oom-leak`.

Rationale: smallest known-good increment, keeps both data points (single
vs cross-hart) reproducible on demand, and matches the established
feature-gating convention. The alternative — making cross-hart the
default — silently changes what the baseline scenario measures and
loses the single-hart data point from CI.

*Alternative if you'd rather: make cross-hart the default and rename
the baseline scenario. Say the word and I'll re-plan.*

## The memory-ordering hazard (the real substance of this step)

Today the consumer (`workload.rs:105`) does, all `Relaxed`:

```
for s in batch { HISTOGRAM[bin_of(s)].fetch_add(1, Relaxed) }   // (A)
SAMPLES_CONSUMED.fetch_add(n, Relaxed)                          // (B)
```

and the heartbeat (`heartbeat.rs:316-317`) reads, all `Relaxed`:

```
consumed      = SAMPLES_CONSUMED.load(Relaxed)                  // (C)
histogram_sum = Σ HISTOGRAM[*].load(Relaxed)                    // (D)
```

On one hart this is fine: (A) happens-before (B) in program order, so a
same-hart reader always sees `histogram_sum >= consumed`. **Across
harts with `Relaxed`, (B) can become visible to hart 0 before (A)**, so
the heartbeat can emit `consumed` ahead of the bins → the
`histogram_sum >= consumed` oracle appears violated → a flaky failure
that depends on QEMU `thread=multi` interleaving (the same class of bug
as deflake "bug B").

**Fix:** make (B) the release of the bin writes and (C) the matching
acquire.

- Consumer: `SAMPLES_CONSUMED.fetch_add(n, Release)` (bins stay
  `Relaxed`).
- Heartbeat: read `SAMPLES_CONSUMED.load(Acquire)` **before**
  `histogram_sum()`. Acquire pairs with the consumer's Release, so every
  bin write sequenced before that Release is visible to the subsequent
  `histogram_sum()` read ⇒ `histogram_sum >= consumed` for the value
  just read. (The heartbeat already reads consumed before histogram —
  the order is correct; only the orderings need strengthening.)

This is step-6 (weak-memory audit) reasoning applied exactly where step
11 first makes it load-bearing.

## Acceptance Criteria

- [ ] A `smp-workload` cargo feature exists; under it the kernel spawns
      `workload_producer` on hart 0 and `workload_consumer` on hart 1
      (consumer via `spawn_on(1, …)` after `SECONDARY_READY`), and does
      not spawn `task_a`/`task_b`.
- [ ] The default build is unchanged (both workload tasks on hart 0);
      `workload-cooperative-baseline` still passes.
- [ ] New scenario `smp-producer-consumer-correctness` (feature-gated
      build) asserts `samples_consumed_total` reaches a threshold and
      then `histogram_sum >= consumed` — i.e. no samples lost across the
      cross-hart handoff.
- [ ] The correctness scenario passes under `cargo xtask itest
      --repeat 10` (the commit gate from memory — single-run-green hid
      cross-hart flakes before).
- [ ] Consumer's `SAMPLES_CONSUMED` bump is `Release`; the heartbeat's
      read is `Acquire`; both carry a comment explaining the cross-hart
      pairing.

## Steps

Every step follows RED-GREEN-MUTATE-KILL MUTANTS-REFACTOR.

### Step 11.1: Cross-hart correctness scenario + `smp-workload` wiring

**Acceptance criteria**: With the kernel built `--features smp-workload`,
the producer runs on hart 0 and the consumer on hart 1, and a new
integration scenario `smp-producer-consumer-correctness` observes
`snitchos.workload.samples_consumed_total >= N` and subsequently
`snitchos.workload.histogram_sum >= N`. The scenario passes under
`--repeat 10`. Default build and `workload-cooperative-baseline`
unaffected.

**RED**: Add the `smp-producer-consumer-correctness` scenario (modeled
on `workload_cooperative_baseline`, but via
`Harness::spawn_with_features("smp-workload", &["smp-workload"])`) and
register it in `itest.rs::SCENARIOS`. It fails first because the
`smp-workload` feature and the cross-hart spawn wiring don't exist yet
(consumer never runs on hart 1 → `samples_consumed` reflects nothing /
build gate absent).

**GREEN**: 
- Add `smp-workload = []` to `kernel/Cargo.toml` `[features]` with a
  doc comment.
- In `main.rs`: extend the `cfg(not(any(deflake-…)))` guard on the
  default workload block to also exclude `smp-workload`; under
  `smp-workload`, spawn `workload_producer` on hart 0 and, in the
  post-`SECONDARY_READY` block (next to `hart_1_probe`), `spawn_on(1,
  "workload_consumer", workload::consumer_entry)`.
- Drop the `allow(dead_code)` cfgs on `workload.rs` items so they
  compile under the new feature (or extend them to include it).

**MUTATE**: `cargo xtask mutants` over `kernel-core::workload` (the pure
logic — `bin_of`, `Lcg`) is the host-testable surface; the spawn-site
wiring is covered by the integration scenario, not mutants (bare-metal
`kernel` is excluded per mutation-testing setup). Confirm no new
survivors in `kernel-core::workload`.

**KILL MUTANTS**: Address any survivor in the pure logic; ask if value
is ambiguous.

**REFACTOR**: Assess whether the producer/consumer spawn placement reads
cleanly under the now-three-way cfg (default / deflake / smp-workload).
Only refactor if it reduces the cfg noise without hiding intent.

**Done when**: scenario green under `--repeat 10`, baseline still green,
default build unchanged, mutation report reviewed, human approves commit.

### Step 11.2: Harden the cross-hart ordering of the correctness oracle

**Acceptance criteria**: The `SAMPLES_CONSUMED` increment in the
consumer is `Release` and the heartbeat's `SAMPLES_CONSUMED` read is
`Acquire`, both commented with the pairing rationale; the
`smp-producer-consumer-correctness` scenario remains green under
`--repeat 10` (ideally a larger repeat to raise confidence, given the
flake is interleaving-dependent).

**RED**: This ordering bug is interleaving-probabilistic, so a
deterministic unit RED isn't available on the host (host tests are
single-threaded). The guard is the repeated integration run. Document
in the plan that the "test" is `--repeat N` on
`smp-producer-consumer-correctness`; capture a baseline flake rate (if
any) on the pre-fix build for honesty, then apply the fix.

**GREEN**: 
- `workload.rs` consumer: `SAMPLES_CONSUMED.fetch_add(n, Release)` (bins
  stay `Relaxed`); update the module ordering comment.
- `heartbeat.rs` `emit_workload_metrics`: read `SAMPLES_CONSUMED` with
  `Acquire` before calling `histogram_sum()`; comment the pairing.

**MUTATE**: N/A for ordering (mutants don't model memory ordering).
Note this explicitly.

**KILL MUTANTS**: N/A.

**REFACTOR**: Consider whether the `histogram_sum >= consumed` invariant
and its ordering contract deserve a short note in
`plans/scaling-corners.md` / the workload module doc so the next
counter added outside the lock inherits the discipline.

**Done when**: orderings in place + commented, scenario green under
`--repeat 10` (and a higher repeat if quick), human approves commit.

# v0.5.x: minimal task-exit

Cooperative-v0.5 tasks are `extern "C" fn() -> !`. They cannot return,
cannot terminate, cannot release their stack. The `deflake-spawn-storm`
scenario's design ceiling (N effective trials per boot ≈ 1) is a direct
consequence: once a spawned task exists, hart 1 round-robins through it
forever and never re-enters `wfi`, so subsequent spawns arrive while
hart 1 is busy and never exercise the wfi → IPI → trap → switch trial.

This document plans the *minimal* task-exit support — enough to unblock
the storm scenario, not full lifecycle management. Reaping
(`Box<Task>` + stack drop) is deferred to a follow-up.

## Goal

A task can call `sched::exit_now()` (or be the closure equivalent) and
the scheduler will:

1. Remove it from its hart's runqueue.
2. Mark it `Exited`.
3. Switch into the next ready task without saving the exiting task's
   register state.
4. Never resume the exited task.

The exited `Box<Task>` and its `Box<Stack>` remain allocated in
`sched.tasks` forever. Bounded leak — bounded by per-boot spawn count.
For the storm at N=200, ~3 MiB of leaked stacks. Heap budget: fine.
For long-running general use: not fine — that's the follow-up.

## Non-goals

- Reaping. Exit'd tasks are leaked.
- `join()` / wait-for-completion. The storm signals completion via an
  atomic counter; no synchronisation primitive needed here.
- Cross-hart exit notification. SMP-cooperative only-this-hart
  scheduler today.
- Exit from the panic handler. Panic still terminates the whole
  kernel, not just the current task.
- Closing open spans on exit. The task is expected to balance its own
  spans before calling `exit_now`; if it doesn't, we get a leaked
  cursor entry. Documented constraint, not a hidden bug.

## API

```rust
// kernel/src/sched.rs

/// Terminate the calling task. Marks it `Exited`, removes it from the
/// runqueue, and switches into the next ready task. Never returns.
///
/// The exited task's `Box<Task>` and stack remain allocated until
/// kernel reboot — this is the minimal-scope variant; reaping lands
/// in a follow-up. Open spans on the calling task's cursor are NOT
/// closed; balance before calling.
///
/// Calling `exit_now` when the runqueue is otherwise empty leaves
/// the hart with no ready task. Today that means the current hart's
/// caller — `hart_1_main` for the worker hart, `kmain` for the
/// boot hart — would have to be left on the queue. Storm scenarios
/// ensure this by never exiting `hart_1_main` itself.
pub fn exit_now() -> !;
```

## State machine

```
   spawn       run        exit_now
   ────►Ready─────►Running────────►Exited (terminal)
                   ▲   │
            yield  │   │ yield
                   └───┘
                  Ready
```

`kernel_core::sched::TaskState`: add `Exited`. Existing `Running` ↔
`Ready` transitions are unchanged.

## Mechanism

### Asm: `switch_into(*to: TaskContext)`

A load-only sibling of `switch`. No save side; just loads
callee-saved + `sp` + `ra` from `*to` and `ret`s. Argument in `a0`.

```
.global switch_into
.align 2
switch_into:
    ld ra,    0(a0)
    ld sp,    8(a0)
    ld s0,   16(a0)
    # ... s1..s11 as in switch ...
    ret
```

Same TaskContext layout. Asm lives in `kernel/src/sched.S` adjacent
to `switch`.

### Rust: `exit_now`

```rust
pub fn exit_now() -> ! {
    let current_id = TaskId(CURRENT_TASK.this_cpu().load(Relaxed));
    let me = percpu::current_hartid();

    let next_ctx = {
        let mut sched = SCHEDULER.lock();

        // Mark current Exited. Drop from the task table? No — leaked.
        // Just flip state; future `task_snapshots` skips it.
        for task in &sched.tasks {
            if task.id == current_id {
                task.state.store(TaskState::Exited as u32, Relaxed);
                break;
            }
        }

        // Current isn't on the runqueue (it's running); nothing to
        // remove there. Pick next.
        let next_id = sched.runqueues[me].pop_front()
            .expect("exit_now: runqueue empty, no task to switch into");

        // Resolve next's context pointer + span cursor.
        let mut next_ctx: *mut TaskContext = null_mut();
        let mut next_cursor: *mut SpanCursor = null_mut();
        for task in &sched.tasks {
            if task.id == next_id {
                next_ctx = task.context.get();
                next_cursor = (&task.span_cursor as *const SpanCursor) as *mut SpanCursor;
                task.runs.fetch_add(1, Relaxed);
                break;
            }
        }
        assert!(!next_ctx.is_null());

        CURRENT_TASK.this_cpu().store(next_id.0, Relaxed);
        CURRENT_SPAN_CURSOR.this_cpu().store(next_cursor, Relaxed);
        next_ctx
        // lock dropped
    };

    CONTEXT_SWITCHES.fetch_add(1, Relaxed);
    tracing::emit_context_switch(current_id, /* not coming back */ TaskId(0), SwitchReason::Exit);
    CURRENT_TASK_ENTRY_TICK.this_cpu().store(tracing::timestamp(), Relaxed);

    // SAFETY: next_ctx is from a Box<Task> still in sched.tasks.
    // The Box<Task> for the current (exiting) task is also still
    // there, just marked Exited — we are about to abandon its
    // stack, but the heap allocation lives on (leaked).
    unsafe { switch_into(next_ctx) };
    // unreachable — switch_into never returns to here.
    #[allow(clippy::empty_loop)]
    loop {}
}
```

`SwitchReason::Exit` is a new variant on the wire — see "Wire format
changes" below.

## State + runqueue changes

- `kernel_core::sched::TaskState`: add `Exited`. Stored as `u32` in
  `Task::state` (already `AtomicU32`? — confirm during impl).
- `yield_now` does NOT need changes — exited tasks aren't on the
  runqueue, so `pop_front` never returns one. `task_snapshots`
  should skip `Exited` tasks so heartbeat metrics don't keep
  reporting them.

## Wire format changes

`protocol::SwitchReason` currently has `Yield`. Add `Exit`. Postcard
encodes enums positionally — `Exit` must come AFTER `Yield` to avoid
breaking captures. New variant ⇒ also update
`OwnedFrame::from_borrowed` so host tests compile.

If we want to defer the wire change, exit can re-use `Yield` for v0.5.x
and we add `Exit` when there's a host-side consumer that cares. **Pick
defer-the-wire-change**: simpler, no protocol commit risk.

## Lifetime + safety reasoning

The asm `switch_into` reads from `*to_ctx` then `ret`s. After `ret`,
execution is in the next task's body on the next task's stack. The
exiting task's stack is no longer in use — `sp` was overwritten.

The exiting task's `Box<Task>` (which owns the `Box<Stack>`) lives in
`sched.tasks: Vec<Box<Task>>`. We do not remove it. The `Box<Stack>`
inside is never accessed again. The `TaskContext` inside is never
read again (we won't switch back to a context flagged Exited). So
the leaked Boxes are inert — held by the Vec, dereferenceable, but
nothing reads them.

Edge case: a future iteration of `task_snapshots()` walks all tasks
including Exited. If we don't filter, the heartbeat keeps emitting
metrics for dead tasks. Fix: `task_snapshots()` skips
`state == Exited`. Same for the runqueue-depth / tasks-total gauges.

## Failure modes

1. **Runqueue empty on exit.** The expect-panic above. In practice the
   storm scenario keeps `hart_1_main` on hart 1's queue, so this
   shouldn't fire. Defensive check during dev.
2. **Exiting task held the scheduler lock across the switch.** We
   drop before `switch_into`. Same discipline as `yield_now`.
3. **Span open across exit.** Cursor entry leaked. Bound: one per
   exit. Documented; no fix.
4. **Concurrent yield from same hart.** Cooperative; impossible by
   construction.
5. **Other hart spawns onto a runqueue we're popping from.** Hart 1's
   runqueue is hart-1-only-modified except for cross-hart spawn,
   which takes the SCHEDULER mutex — already serialised.

## Integration: storm body rewire

`kernel/src/deflake.rs::spawn_storm::body`:

```rust
pub extern "C" fn body() -> ! {
    let marker: u64 = 0xdead_beef_cafe_f00d;
    core::hint::black_box(marker);
    ACK_COUNTER.fetch_add(1, Ordering::Release);
    crate::sched::exit_now();
}
```

That's the entire storm-side change. Hart 1's runqueue drains
between iterations: only `hart_1_main` left → it yields, no other
ready task, falls through to `wfi`. Next spawn arrives while hart 1
is in `wfi`. **Each iteration is now a real wfi → IPI → trap →
switch trial.**

## Verification

1. Unit test in `kernel-core` for `TaskState::Exited` round-trip and
   `task_snapshots()` filtering. (kernel-core stays
   host-buildable.)
2. New scenario: `sched-task-exits-cleanly`. Spawn one task that
   bumps an atomic and exits. Assert the atomic == 1 AND the
   `tasks_total` gauge does not climb monotonically across
   heartbeats. (Heartbeat over many ticks; the exited task should
   stop counting.) Catches the "skip Exited in snapshots" bit.
3. Rerun `deflake-spawn-storm --repeat 20` with fix on. Should still
   be 20/20. Confirms exit + the integration didn't regress the
   storm.
4. Rerun `deflake-spawn-storm --repeat 20` with fix off. **Hypothesis:**
   per-trial rate is now meaningfully exposed. Could be still 0%
   (H1/H2 falsified for real) or could be ≥80% (fresh-switch race
   confirmed). Either is a strong signal.

## Open questions to settle during impl

- `Task::state` field — is it currently `AtomicU32`, or `TaskState`
  directly behind a lock? Determines how `exit_now` writes the
  Exited marker. Check during implementation.
- The unreachable `loop {}` after `switch_into`: clippy will warn.
  Use `unreachable!()` instead (still `-> !`) and document the
  asm-never-returns invariant.
- Should `exit_now` decrement any "live tasks" counter? Could be a
  useful metric. Adds one atomic. Cheap. Add.

## What this does NOT unlock

- Long-running boot workloads that need eventual cleanup. They still
  leak.
- A task that wants to terminate another task. We only support
  self-exit. Cross-task termination is more interesting and pretty
  different in shape.
- Reuse of the exited task's id slot. New spawns mint fresh ids
  forever — `alloc_task_id()` is a monotonic counter. Reuse would
  need a freelist and isn't worth it at v0.5.x.

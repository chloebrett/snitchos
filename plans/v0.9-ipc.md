# v0.9 — IPC over capabilities (synchronous endpoints)

**Work lands on:** `main` (no feature branches — see CLAUDE.md)
**Status:** Design complete, not started. Sits on the v0.8 preemptive
scheduler (block/wake is born on the better substrate) and the v0.7b
capability system (`Object` is already "the extension point for the
object zoo (`Endpoint`, …)").

Two userspace processes exchange a message through a kernel-mediated
**synchronous endpoint**, and the trace follows the message across the
process boundary. The headline: **"two processes talking, and watching
every word."**

Full philosophy + locked decisions predating this plan:
[docs/ipc-design.md](../docs/ipc-design.md). This plan is the v0.9
*implementation cut* of that design.

## Scope decision: synchronous `send`/`receive`, kernel-brokered endpoint

The v0.9 *core* is the smallest honest demonstration of the headline.
Everything richer is a numbered follow-on, each its own plan.

| in the v0.9 core | deferred (own follow-on plan) |
|---|---|
| Synchronous endpoint: `send` / `receive`, rendezvous semantics | `call` / `reply` + one-shot reply caps (**v0.9b** — the first real cap-transfer) |
| Blocking (`TaskState::Blocked` grows up) + direct switch | Notifications (payload-free async primitive) (**v0.9c**) |
| Inline message registers (a few machine words) | `MemoryRegion` bulk transfer + the userspace channel library |
| **Trace context kernel-populated** across the boundary | Cap-transfer (caps travelling *in* messages) |
| Kernel-brokered endpoint (boot grants A=`SEND`, B=`RECV`) | Cross-hart wake (IPI to a sleeping hart) |
| Two processes time-sliced on **one** hart | Abortable / timeout sends (deadlock mitigation) |

**Why `send`/`receive` and not `call`/`reply` for the core.** A one-way
`send` already crosses the trace boundary — the sender's current span id
rides in the message (kernel-populated), the receiver opens its handling
span as a *child*. Producer→consumer where the consumer's work-span is a
child of the producer's send-span **is** the headline. `call`/`reply`
buys clean RPC, but its one-shot reply cap means the kernel inserts a
transient cap into the *receiver's* CapTable at rendezvous — the
mutate-the-receiver's-table mechanism we deliberately deferred with the
"kernel-brokered, no cap-transfer" call. So `call`/`reply` becomes
**v0.9b**, the milestone that *earns* cap-transfer honestly ("we have
one-way messages; RPC needs a reply capability — here's the first one").

**Why kernel-brokered.** There is no cap-transfer and no spawn-with-caps
today; each process is bootstrapped independently (`Process::bootstrap`)
with a fixed cap pair. For two processes to share an endpoint *something*
must give them a common cap. The kernel creates one endpoint at boot and
bootstraps process A with a `SEND` cap and process B with a `RECV` cap.
This sidesteps cap-transfer entirely and gets the rendezvous + trace
demo working; passing the endpoint *honestly* (init holds it, hands it
out) arrives with v0.9b's reply caps.

**Why one hart.** Synchronous rendezvous on a single hart needs nothing
cross-hart: A `send`s, blocks, the scheduler switches to B, B `receive`s,
they rendezvous. The two processes are time-sliced by the v0.8
preemptive scheduler. The endpoint *state machine* and its locking are
SMP-safe **by construction** (a `kernel::sync::Mutex`, no
"no-one-else-is-here" assumptions — born multi-hart-correct per the
roadmap), but the *cross-hart wake path* (IPI a `wfi`-sleeping hart when
its task becomes runnable) is genuinely more machinery and is deferred.

## Decisions locked in

| decision | choice |
|---|---|
| Primitives this milestone | **Synchronous endpoint only** (`send` + `receive`). Notifications = v0.9c. |
| Rendezvous semantics | Endpoint is at rest `Idle` / `SendersWaiting` / `ReceiversWaiting`; **never both sides waiting at once**. `send` with a receiver waiting completes without blocking; otherwise the sender blocks parked on the endpoint. `receive` mirrors. |
| Payload | **Inline message registers**, fixed `MSG_WORDS` (start at 4 = `a0..a3`). Bulk via `MemoryRegion` deferred. Caps-in-messages deferred. |
| Trace context | **Kernel-populated, out-of-band** (does not consume a message register). At rendezvous the kernel stamps the receiver task's span context with the sender's current span id; the receiver's next span opens as a child. Userspace cannot opt out or forget. |
| Endpoint as a cap object | New `Object::Endpoint { id: EndpointId }` in `kernel-core::cap`; new `Rights::SEND` / `Rights::RECV` bits (binary literals, no `1<<0` no-ops). `CapObject::Endpoint` **appended** to the wire enum. |
| Endpoint object storage | Kernel static `EndpointTable` behind `kernel::sync::Mutex`, `EndpointId` indexes it. Pure rendezvous logic in `kernel-core::ipc`; the kernel owns the table + per-task parked-message storage. Mirrors `Runqueue` (kernel-core) vs `SCHEDULER` (kernel). |
| Blocking | `TaskState::Blocked` (today a placeholder, `sched.rs:24`) grows up. `block_current()` switches away without re-enqueuing; `wake(id)` sets `Ready` + `push_back`. Sits on the v0.8 preemptive scheduler. |
| Switch mechanism | Block/wake rides the **existing cooperative `switch`** via `yield_now`'s descendants — a blocking `send`/`receive` is a *voluntary* relinquish at a known-safe point (caller-saved regs already dead per SysV ABI). sched.S unchanged. |
| Syscalls | Append `Send` and `Receive` to `abi::Syscall` (positional — **append only**). New `kernel::sched`-mediated trap arms. |
| Refusals | Reuse existing `RefusalReason` (`CapNotFound`/`CapWrongRights`/`CapWrongObject`) for endpoint cap denial — invocation is validated exactly like telemetry/span (`invoke_send`/`invoke_recv` mirror `invoke_telemetry`). |
| Wire frame | Append `Frame::Message { .. }` (append-only); emitted via the **deferred-emission** pattern, never from inside the rendezvous critical section. |
| Telemetry | `snitchos.ipc.messages_total`, `snitchos.ipc.blocks_total`. Deferred-emission (bump atomic at rendezvous/block, drain in heartbeat). |

## Lock discipline (read before Step 4/5)

The rendezvous mutates two tasks' state + the endpoint under the
endpoint `Mutex`, then **must release the guard before switching away**.
This is the v0.5 rule with teeth:

- **Never hold a `kernel::sync::Mutex` across the switch.** `send`/
  `receive` block, which switches. Drop the endpoint guard first — Tasks
  live in `Box<Task>` so raw context pointers stay valid past the drop
  (same pattern as `SCHEDULER`).
- **Bind the guard; don't `let x = *EP.lock();`.** That releases at the
  `;` and any shared state touched afterward is unprotected. This exact
  shape was the ~2% cross-hart wedge in `virtio_console::send` (a dropped
  `MutexGuard`). See [plans/tx-staging-cross-hart-race.md](tx-staging-cross-hart-race.md).
- Never emit a telemetry frame from inside the rendezvous critical
  section — bump an atomic, drain in the heartbeat (same as the alloc
  path and the IRQ handler).

## Steps

Every step is RED-GREEN-MUTATE-KILL MUTANTS-REFACTOR. No production code
without a failing test. *Present acceptance criteria and confirm before
coding each step.* Natural PR boundaries marked.

### Step 1: Endpoint rendezvous state machine (pure host-tested logic)

**Acceptance criteria**: A new `kernel-core::ipc` module with an
`EndpointState` (`Idle` / `SendersWaiting(VecDeque<TaskId>)` /
`ReceiversWaiting(VecDeque<TaskId>)`) and two pure functions
`on_send(state, me) -> (EndpointState, RendezvousAction)` and
`on_receive(...)`, where `RendezvousAction = Block | Rendezvous { peer:
TaskId }`. Holds the invariant **never both senders and receivers
waiting simultaneously**. No kernel behaviour change. Mirrors
`heap::watermark_grow_decision` / `sched::quantum_expired`.
**RED**: host tests for — `send` into `Idle` → `Block`, state becomes
`SendersWaiting([me])`; `receive` into `SendersWaiting([s])` →
`Rendezvous{peer:s}`, state returns toward `Idle`; the mirror pair;
FIFO ordering when several wait; the invariant (a `send` into
`ReceiversWaiting` rendezvouses, never appends a sender alongside a
receiver).
**GREEN**: the match logic.
**MUTATE**: `mutation-testing` — verify the empty/non-empty queue
branches and the which-side-waiting discriminant all have killers.
**KILL MUTANTS**: add cases for survivors.
**REFACTOR**: only if it adds value.
**Done when**: host tests + mutation report green, human approves commit.

> **PR boundary** — pure logic, independently mergeable. This is the
> minimal first boundary the design targets.

### Step 2: `Endpoint` capability object + `SEND`/`RECV` rights

**Acceptance criteria**: `kernel-core::cap` gains `Object::Endpoint { id:
EndpointId }`, `Rights::SEND` (`0b0010`) and `Rights::RECV` (`0b0100`),
and `invoke_send(table, handle) -> Result<EndpointId, Denied>` /
`invoke_recv(...)` mirroring `invoke_telemetry`/`invoke_span` (resolve →
right check → object match). `protocol::CapObject` gains an **appended**
`Endpoint` variant; roundtrips through postcard; `OwnedFrame` arm exists.
*Confirm before coding.*
**RED**: cap unit tests (accept with right, refuse `MissingRight`,
refuse `WrongObject`, refuse unknown handle — the four `invoke_*`
cases); protocol roundtrip for `CapObject::Endpoint`.
**GREEN**: add variants + functions + arm.
**MUTATE**: `kernel-core` + `protocol`.
**KILL MUTANTS**: address survivors (esp. `SEND` vs `RECV` mask).
**REFACTOR**: n/a.
**Done when**: tests + mutation green, human approves commit.

> **PR boundary** — cap-object + wire-enum addition, additive, no
> behaviour depends on it yet.

### Step 3: `Message` wire frame (append-only)

**GREEN**: add the variant + arm.

**KILL MUTANTS**: address survivors.
**REFACTOR**: n/a.
**Done when**: roundtrip + mutation green, human approves commit.

> **PR boundary** — wire-format addition, nothing emits it yet (mirrors

### Step 4: Block/wake scheduler primitive

**Acceptance criteria**: `TaskState::Blocked` grows up. `block_current()`
marks the running task `Blocked` and switches to the next `Ready` task
**without** re-enqueuing it; `wake(id)` sets it `Ready` and
`push_back`s. A `block-wake-smoke` itest proves a task can block on a
dummy condition and be woken by another task and resume. Idle keeps the
hart alive when nothing is `Ready`. *Confirm before coding.*
**RED**: kernel-core unit tests for the `Runqueue`/state transitions
that are pure (block = not re-enqueued; wake = enqueued once, idempotence
guard if a task is woken twice); itest `block-wake-smoke`.
**GREEN**: the transition wiring in `kernel::sched` + any pure helper in
`kernel-core::sched`.
**MUTATE**: pure transition helpers.
**KILL MUTANTS**: survivors.
**REFACTOR**: assess.
**Done when**: host tests + `block-wake-smoke` green under `--repeat 10`
(scheduler-path → commit-gate rule), mutation reviewed, human approves.

> **PR boundary** — blocking primitive, IPC-independent and reusable.

### Step 5: `Send`/`Receive` syscalls + endpoint table + message copy (the milestone heart)

**Acceptance criteria**: Under a new `workload=ipc`, process A holds a
`SEND` cap and process B a `RECV` cap to one kernel-brokered endpoint; A
`send`s an inline message, B `receive`s it, and the payload words arrive
intact in B. The rendezvous blocks the first-arriver and direct-switches.
Bad cap / wrong rights / wrong object are refused via `SyscallRefused`
(reusing existing reasons). *Confirm before coding.*
**RED**: itest `ipc-message-crosses` — assert B observes the exact
payload A sent (a sentinel value on the wire, e.g. via a B-side
telemetry counter set from the received word), and that a
`SyscallRefused` is emitted when a no-`SEND` cap tries to send.
**GREEN**: append `Syscall::{Send,Receive}` (abi) + runtime wrappers
(`user/runtime`) + two user binaries (`user/ipc-sender`,
`user/ipc-receiver`); `EndpointTable` static (`kernel::sync::Mutex`);
trap arms `handle_send`/`handle_receive` that resolve the cap
(`invoke_send`/`invoke_recv`), drive `kernel_core::ipc::on_{send,receive}`,
copy `MSG_WORDS` from sender `TrapFrame` → parked storage → receiver
`TrapFrame`, and block/wake via Step 4 — **dropping the endpoint guard
before the switch**; `WorkloadKind::Ipc` + bootargs parse arm
(host-tested) + spawn layout that bootstraps A/B with the shared
endpoint caps.
**MUTATE**: bootargs parse arm; the cap-resolution call sites.
**KILL MUTANTS**: survivors.
**REFACTOR**: assess (esp. the parked-message storage shape).
**Done when**: `ipc-message-crosses` green under `--repeat 10`, mutation
reviewed, human approves.

> **PR boundary** — IPC works end-to-end (message crosses). A shippable
> v0.9 on its own, minus the trace headline.

### Step 6: Trace context crosses the boundary (the headline)

**Acceptance criteria**: At rendezvous the kernel stamps the receiver
task's span context with the **sender's current span id**; the first
span B opens after `receive` has `parent == A's span id`. An itest
asserts the parent linkage on the wire. *Confirm before coding.*
**RED**: itest `ipc-trace-crosses` — A opens a span, `send`s inside it;
B `receive`s and opens a span; assert B's `SpanStart` carries
`parent == A's open span id` (both ids known from the wire), proving the
trace continued across the process boundary.
**GREEN**: thread the sender's current span id into the parked message
(out-of-band, kernel-side); on rendezvous set the receiver's
`SpanCursor` incoming-parent so its next `span_start` is a child.
**MUTATE**: the parent-stamping site (mutate to "parent = 0" / "parent =
own span" and confirm a test catches it).
**KILL MUTANTS**: survivors.
**REFACTOR**: assess.
**Done when**: `ipc-trace-crosses` green under `--repeat 10`, mutation
reviewed, human approves.

> **PR boundary** — the v0.9 headline. "Watch the trace cross the
> boundary" is now true and asserted.

### Step 7: IPC telemetry

**Acceptance criteria**: `snitchos.ipc.messages_total` increments once
per rendezvous and `snitchos.ipc.blocks_total` once per block, observable
in an itest; `Frame::Message` (Step 3) is emitted per rendezvous. All via
deferred-emission (atomic bumped at the event, drained in heartbeat) —
never from inside the rendezvous critical section. *Confirm before
coding.*
**RED**: itest asserts `messages_total >= 1` and a `Message` frame with
matching `from`/`to`/`endpoint` under `workload=ipc`.
**GREEN**: register the counters; bump atomics at rendezvous/block; drain
+ emit the `Message` frame in the heartbeat loop.
**MUTATE / KILL MUTANTS**: as usual.
**REFACTOR**: assess.
**Done when**: metric + frame observed, `--repeat 10` green, human
approves.

> **PR boundary** — observability for the new behaviour.

## Follow-ons (each its own plan, after the core)

- **v0.9b — `call`/`reply` + reply capabilities.** The RPC fast path
  (direct switch both ways) and the project's **first real cap-transfer**
  (kernel mints a one-shot reply cap into the receiver). Strong essay
  milestone: "RPC needs a reply capability."
- **v0.9c — Notifications.** Payload-free async primitive (per-object bit
  set, signal/wait). Models interrupts + readiness signals.
- **Bulk transfer + userspace channel library.** `MemoryRegion` cap
  transfer; the blessed `mpsc`-shaped bounded channel library over
  region + notification (policy in userspace).
- **Cross-hart wake.** IPI a `wfi`-sleeping hart when its task becomes
  runnable; cross-hart IPC workload. The endpoint state machine is
  already SMP-safe; this adds only the wake-delivery path.

2. Refactoring assessment — run `refactoring` skill.
3. `cargo xtask clippy` (whole workspace, kernel for riscv) clean.

## Open questions

- **`MSG_WORDS` value.** Start at 4 (`a0..a3`); revisit once a real
  workload (v0.11 metrics ingestion) shows the typical message shape.
- **Parked-message storage location.** On the blocked sender's `Task`
  struct, or in the endpoint's queue entry? The sender is parked and not
  running, so either is sound; decide at Step 5 by which keeps the
  rendezvous copy simplest.
- **Deadlock.** Synchronous send/receive can deadlock (A→B, B→A). v0.9
  accepts this (acyclic-call-graph discipline); abortable/timeout sends
  are a deferred follow-on per the design doc.
- **Does the Step 4 `block-wake-smoke` peer need to be userspace?** A
  kernel-thread peer is simpler to assert on; a userspace peer is the
  more honest demo. Decide at Step 4 (same trade-off the v0.8 plan hit).

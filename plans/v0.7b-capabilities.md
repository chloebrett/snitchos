# v0.7b: the capability system (the wrongness, repaid)

Refactor v0.7a's one **ambient** syscall into a **capability
invocation**. This is the milestone where the project's second pillar
lands: *no ambient authority — every userspace resource access goes
through an unforgeable handle, validated against the calling process's
own table.* The kernel surface collapses to one idea: **invoke a
capability.**

> Read first: [docs/capability-system-design.md](../docs/capability-system-design.md)
> (the locked decisions) and [plans/v0.7a-first-userspace.md](v0.7a-first-userspace.md)
> (the substrate this grows onto).

## Goal

The `hello` program stops calling a bare `EmitMetric` syscall and
instead **invokes a `TelemetrySink` capability it was granted at
bootstrap**. The kernel:

1. resolves the handle in *the calling process's* `CapTable`,
2. checks the cap's `EMIT` right,
3. emits the metric — **identity kernel-stamped**, to the counter the
   cap is bound to.

A process that holds **no** such cap, or presents a **bad/stale
handle**, is **denied** — observably (`snitchos.cap.denied_total` + a
traced authority-decision event). That denial is the capability twin of
v0.7a's `faulter` isolation probe, and it is the milestone's payoff.

## The one-line before/after

```
v0.7a (ambient):   ecall(a7=EmitMetric, a0=value)         → kernel emits, no questions
v0.7b (capability): ecall(a7=Invoke, a0=handle, a1=value) → kernel: resolve handle in MY table
                                                              → check EMIT right
                                                              → emit (identity = me, by kernel)
                                                              → else DENY (observable)
```

## Non-goals (deferred, by milestone)

- **Cap transfer / grant between processes** — impossible here: one
  process, no IPC to pass a cap *through*. Lands at **v0.8** with
  `Endpoint`. v0.7b caps are *kernel-granted at bootstrap*, invoked, and
  rights-checked — never moved between processes.
- **Demonstrated attenuation / multiple methods** — `TelemetrySink` is a
  boolean object (one `EMIT` bit, one method). The cap *system* carries
  the machinery; this object does not exercise it. Attenuation gets a
  real demo against `File` (v0.10); transfer against `Endpoint` (v0.8).
  **Do not inflate `TelemetrySink` to make this milestone feel bigger.**
- **`MemoryRegion` / a `map` syscall** — no consumer until memory is
  *shared*; lands at v0.8. (Wholesale vs. the allocator's retail — see
  the design doc.)
- **User-named metrics (`emit(name, value)`)** — needs `SUM` +
  user-pointer validation (deferred past v0.7a). v0.7b is **value-only**:
  the cap is bound at creation to a kernel-registered counter; no string
  crosses the boundary, `SUM` stays `0`.
- **Revocation** — no `revoke` operation ships. But the *handle layout*
  carries a generation tag now (see Step 2) so revocation is a cheap
  additive later.
- **A real `init` / multi-process bootstrap** — "init" in v0.7b is the
  single user process; the kernel grants its root cap directly. The
  delicate grant-graph bootstrap is a v0.8 concern.

## What we already own (reuse, don't rebuild)

- **The process's address space.** `user::run` already builds a
  per-process root (`mmu::new_user_root`) and holds `root_pa`. v0.7b
  grows this into a small **process struct** that *also* owns the
  `CapTable`. (Today there is no process struct — `root_pa` is a local;
  Step 3 gives it a home.)
- **The syscall dispatcher.** `trap.rs::handle_user_ecall` (one `match`
  arm) is the entire refactor surface. It currently emits
  unconditionally; it becomes resolve → check → dispatch.
- **The ABI crate.** `snitchos_abi::Syscall` — `EmitMetric = 0` becomes
  `Invoke = 0` (renumbering the *only* syscall is free; nothing else
  uses the value).
- **Telemetry emission.** `tracing::emit_metric` / `register_counter` —
  the kernel side of `TelemetrySink::invoke`. The metric frame already
  exists on the wire → **no `protocol::Frame` change for the happy
  path** (the denial *event*, if we make it a first-class frame rather
  than a counter, is the one possible wire addition — see Step 5).
- **The denial-as-counter precedent.** `snitchos.user.faults_total`
  (v0.7a) is exactly the shape `snitchos.cap.denied_total` should take —
  a kernel-registered counter bumped from the trap path, drained in the
  heartbeat. Same deferred-emission discipline.

## The capability model, concretely

```rust
// kernel-core/src/cap.rs  — pure, host-tested. No MMIO, no CSRs.

/// Rights are a bitmask. v0.7b defines exactly one bit; the type is the
/// extension point (Endpoint/File add bits later without reshaping).
pub struct Rights(u32);
impl Rights { pub const EMIT: Rights = Rights(1 << 0); /* … */ }

/// What a capability points at. One variant in v0.7b; the enum is the
/// extension point for the object zoo (Endpoint, MemoryRegion, …).
pub enum Object {
    /// Bound at creation to a kernel-registered counter `StringId`.
    TelemetrySink { counter: StringId },
}

pub struct Capability { pub object: Object, pub rights: Rights }

/// A process's capability table. Opaque `u32` handle = index + generation
/// (slotmap-style). `generation` is dead weight in v0.7b (nothing bumps
/// it) but makes revocation a one-liner later and stops a future reused
/// slot from aliasing a stale handle.
pub struct CapTable { /* Vec<Slot> where Slot { generation, Option<Capability> } */ }

impl CapTable {
    pub fn insert(&mut self, cap: Capability) -> Handle;          // bootstrap grant
    pub fn resolve(&self, h: Handle) -> Result<&Capability, CapError>;
    //                                  ^ generation mismatch / empty slot / OOB = CapError
}
```

The handle encode/decode (index ↔ generation packing), `insert`,
`resolve` (with all three rejection modes), and the `Rights` check are
**pure functions, host-tested in `kernel-core`** — exactly the
finding-seams split v0.7a used for the ELF parser. The kernel side only
owns: *where the `CapTable` lives* (the process struct) and *wiring the
trap arm to it*.

## The steps

Ordered so the pure, host-testable core lands first (TDD-friendly), the
ABI rename is mechanical, and the scary part (live syscall refactor) is
last and small.

```
1. CapTable + Rights + resolve (kernel-core, pure, host-tested)
        │
2. Handle = index+generation encoding (kernel-core, pure, host-tested)
        │
3. Process struct owns a CapTable; bootstrap grants the TelemetrySink cap
        │
4. ABI: Syscall::Invoke; user/hello invokes the handle (not bare EmitMetric)
        │
5. trap.rs: handle_user_ecall → resolve → check EMIT → emit | DENY (observable)
```

Steps 1–2 are pure and independent. Step 3 needs 1–2. Step 4 is
mechanical (ABI + user crate). Step 5 needs 3 + 4 and is the live
refactor — but it is *smaller* than v0.7a Step 4 because the privilege
drop already works; we are only changing what one trap arm does.

---

### Step 1 — `CapTable`, `Rights`, `Object`, `resolve` (kernel-core, pure)

`kernel-core/src/cap.rs`. No `unsafe`, no statics. TDD in
`cargo test -p kernel-core`:

- `insert` returns a handle that `resolve`s back to the same cap.
- `resolve` of an empty slot → `CapError::Empty`.
- `resolve` of an out-of-bounds index → `CapError::OutOfBounds`.
- a cap with `EMIT` passes the right check; one without it is denied.
- attenuation shape (even if unused): inserting two caps to the same
  `counter` with different `Rights` yields two independent handles. (Test
  the *mechanism* exists; we don't ship a second-rights demo.)

### Step 2 — generation-tagged handles (kernel-core, pure)

Handle = `index + generation` packed in `u32` (e.g. low 16 / high 16, or
whatever the table-size budget wants — decide in impl, document the
split). TDD:

- decode(encode(index, gen)) round-trips.
- `resolve` with a stale generation (slot reused) → `CapError::Stale`,
  **not** the wrong cap. This is the test that earns the generation
  field its keep, even though nothing bumps generations in v0.7b — write
  it by manually constructing a reused slot.

### Step 3 — the process struct owns a `CapTable`

> **DONE (2026-06-10).** `kernel/src/process.rs`: `Process { root_pa,
> caps: Mutex<CapTable> }`. The grant *policy* is host-tested in
> `kernel_core::cap::CapTable::bootstrap_telemetry` (one `TelemetrySink`,
> `EMIT`, at the well-known handle `0`); `Process::bootstrap` wraps it.
> `user::run` grants at boot and snitches `snitchos.cap.grants_total`
> (registered in `init_metric`). `caps` carries a documented
> `allow(dead_code)` until the Step 5 dispatcher reads it. New scenario
> `userspace-grant-snitched` asserts the grant counter reaches the wire
> (teeth-checked: it reds when the snitch is disabled). Existing userspace
> itests stay green; the syscall path is untouched until Steps 4–5.

Today `user::run` holds `root_pa` as a local. Give the process a home:

```rust
// kernel/src/user.rs (or a new kernel/src/process.rs)
pub struct Process {
    pub root_pa: usize,        // was the local in run()
    pub caps: CapTable,        // NEW
}
```

**Bootstrap grant** — the "root cap to init only" rule, concretely: when
building the user process, register the bound counter once
(`snitchos.user.telemetry_total`, the v0.7a counter — reuse it) and
`insert` a `TelemetrySink { counter }` cap with `Rights::EMIT`. Record
the handle it lands at; the user program needs to know it (see Step 4).

**The grant snitches.** Grants *create* authority, so the kernel snitches
when it hands one out — bump `snitchos.cap.grants_total` (register the
counter at boot, same deferred-emission discipline as `faults_total`).
This is the cheap first form of authority observability; the richer
`CapEvent` frame is the sequenced follow-on (see Wire format). The
kernel snitches its own grants freely — no cap required.

**Where it lives / locking:** the `CapTable` sits behind
`kernel::sync::Mutex` *from day one* even though v0.7b is single-thread
per process — this is the "born multi-hart-correct" rule the SMP
sequencing bought us. `resolve` on the syscall path takes the lock
briefly; **never held across `sret` or `yield_now`** (the v0.5 lock
discipline). One process, so uncontended now; correct when v0.8 adds a
second.

### Step 4 — ABI + user program

- `snitchos_abi`: `Syscall::EmitMetric = 0` → `Syscall::Invoke = 0`.
  Document the register ABI: `a0 = handle`, `a1 = value`, return in `a0`
  (0 ok, nonzero `CapError` discriminant).
- **How does `hello` know its handle?** Bootstrap inserts the cap into a
  fresh table, so the first grant lands at a **well-known handle**
  (index 0, generation 0). Define it as a constant in the ABI crate
  (`abi::TELEMETRY_SINK_HANDLE`) so kernel and program agree without
  magic numbers — the same no-hardcoding discipline as the syscall
  number. (A more general "spawn hands the process its initial handles"
  mechanism is a v0.8 concern; a fixed bootstrap handle is correct for
  one root cap.)
- `user/hello`: `_start` → invoke `(TELEMETRY_SINK_HANDLE, value)` via
  `ecall(a7=Invoke)`. `faulter` is unchanged (it probes the page-table
  firewall, not the cap firewall) — but see Step 5 for the *new* denial
  probe.

### Step 5 — the live refactor + the denial demo

> **DONE (2026-06-10).** Landed as two sub-increments, each suite-green:
> **4+5a (happy path):** ABI `EmitMetric → Invoke = 0` + `TELEMETRY_SINK_HANDLE`;
> `hello`/`faulter` invoke the handle (`a0`=handle, `a1`=value); the trap
> dispatcher reaches the running process via a per-CPU
> `process::CURRENT_PROCESS` pointer (mirrors `CURRENT_SPAN_CURSOR`) and
> calls the pure, host-tested `kernel_core::cap::invoke_telemetry`
> (resolve → `EMIT` → bound counter), dropping the caps lock before
> emitting. `userspace-emits-telemetry` stays green — same telemetry,
> now capability-mediated. **4+5b (denial payoff):** `hello` then invokes
> handle 1 (ungranted); the deny arm snitches `snitchos.cap.denied_total`.
> New scenario `userspace-cap-denied` (scenario-first; red before the
> counter existed). Chose to reuse `hello`+`workload=userspace` over a
> separate `denier` program — one program shows authority granted *and*
> refused, no new workload/fixture plumbing. All four userspace scenarios
> green at `--repeat 5`; `invoke_telemetry` host-tested (granted / unknown
> / stale / missing-right).

`trap.rs::handle_user_ecall` becomes:

```rust
fn handle_user_ecall(frame: &mut TrapFrame, proc: &Process) {
    match Syscall::from_usize(frame.a7 as usize) {
        Some(Syscall::Invoke) => {
            let handle = Handle::from_raw(frame.a0);
            frame.a0 = match proc.caps.lock().resolve(handle) {
                Ok(cap) if cap.rights.contains(Rights::EMIT) => {
                    if let Object::TelemetrySink { counter } = cap.object {
                        // identity is implicit/kernel-stamped: we emit as
                        // THIS process; the program never names itself.
                        tracing::emit_metric(counter, frame.a1 as i64);
                    }
                    0
                }
                Ok(_) => { count_denied(); CapError::NoRight as u64 }   // held, but no EMIT
                Err(e) => { count_denied(); e as u64 }                  // bad/stale/empty handle
            };
        }
        None => frame.a0 = u64::MAX,
    }
    frame.sepc = frame.sepc.wrapping_add(4); // unchanged: step past ecall
}
```

**Wiring `proc` into the trap handler** is the one genuinely fiddly bit:
the handler currently takes only the `TrapFrame`. The running process's
`CapTable` has to be reachable from trap context. Options (settle in
impl): a per-CPU "current process" pointer (`kernel::percpu`, mirrors
`CURRENT_TASK`), or hang it off the existing current-task state. Lean
per-CPU current-process pointer — it generalizes to v0.8.

**The denial scenario — `userspace-cap-denied`** (the payoff): a second
program (mirror the `hello`/`faulter` split) that invokes a handle it was
**not** granted — e.g. `Invoke(handle=1, …)` when only handle 0 exists,
or a stale-generation handle. Assert:

- `snitchos.cap.denied_total ≥ 1` appears within N seconds (the kernel
  refused an unauthorized invocation),
- the program gets a nonzero error back in `a0` (we can't see this
  directly — surface it: have the program emit nothing / spin, and rely
  on the counter + the *absence* of its metric),
- the kernel keeps heartbeating (a denied cap is a clean refusal, not a
  wedge).

And the happy-path scenario **`userspace-cap-invoke`** (rename/replace
v0.7a's `userspace-emits-telemetry`): `hello` invokes its granted cap →
`snitchos.user.telemetry_total` appears → heartbeat survives. Same
assertion as v0.7a; the *mechanism underneath* changed from ambient to
mediated.

Both `--repeat 10`.

## Wire format changes — counters first, then a `CapEvent` frame

Two phases, both inside v0.7b, sequenced so the cheap form lands first:

**Phase 1 — counters, no wire change.** `snitchos.cap.grants_total`
(Step 3) and `snitchos.cap.denied_total` (Step 5), both reusing the
`faults_total` precedent. The happy-path invoke still emits the existing
metric frame. This is the milestone's *required* observability; it ships
with the steps above and needs no `protocol::Frame` change.

**Phase 2 — the `CapEvent` frame.** Once the counters are green, add a
first-class authority event — a counter can't carry a grant's fields
(granter, grantee, object, rights). Add `CapEvent` **after** existing
variants (postcard is positional) + the matching
`OwnedFrame::from_borrowed` arm. Kinds: `Granted | Invoked | Denied`
now; `Revoked | Transferred` reserved for later. **Design it with the
fields the host-reconstructed tree needs from day one**, even though
v0.7b can't exercise them:

- a **global cap id** (monotonic `u64`, minted like a span id) — *not*
  the per-process `Handle`, which is local and ambiguous across
  processes;
- a **parent-cap-id** field (always "root" at v0.7b — no derivation yet,
  but cheap to include and expensive to retrofit, same logic as the
  generation tag);
- granter / grantee identity, object type, rights.

The kernel only *emits* `CapEvent`s; it stores no tree. The collector
reconstructs the derivation tree from the stream (see the design doc's
"authority as a host-reconstructed tree"). The **visualiser is v0.8+** —
at v0.7b the tree is a depth-1 star (kernel → bootstrap caps), not worth
a Grafana node-graph yet; the frame design is what makes that v0.8 view
possible without a wire-format change later.

## Where the handle comes from — DONE (register-passed), with BootInfo deferred

> **DONE.** The hardcoded `abi::TELEMETRY_SINK_HANDLE` is **gone**. The kernel
> now **hands** the process its capability: `enter` sets `a0` to the granted
> handle before `sret`, `crt0` forwards `a0` into `rust_main(startup: Startup)`,
> and the program calls `startup.telemetry().emit(...)`. Neither side hardcodes
> a slot — grant the cap anywhere and the program uses what it was told.
> Teeth-checked: feeding `enter` the wrong handle makes the telemetry vanish
> (the program genuinely reads `a0`, it isn't a hidden constant).

The earlier cheat: the program named its capability with a constant
(`TELEMETRY_SINK_HANDLE = 0`), relying on an *assumption about deterministic
insert order* rather than a message. Enforcement (resolve + rights-check) was
always real; only the **handing-over** was faked — and it leaked
kernel-internal allocation order into the program. Register-passing fixes the
provenance: the kernel knows the handle (`Process::bootstrap` returns it) and
passes it; the program receives it.

**Why a register, and what's still deferred.** One register carries one handle
— exactly right for v0.7b's single cap. The program-facing contract is the
`Startup` type (`repr(transparent)` over `usize`), surfaced by `crt0` to
`rust_main`. When caps multiply (v0.8 IPC), `a0` becomes a **pointer to an
in-memory `BootInfo` page** listing many handles + what each is — but the
`Startup` interface stays put, so programs don't change and the migration is
contained to `enter` + `crt0`. This is seL4's `seL4_BootInfo`, structurally
**Linux's `auxv`** (the kernel writes startup facts at entry and forgets them;
the runtime captures them — env vars are the same shape one layer up). So:
*register-passed handle now (done), memory-mapped `BootInfo` at v0.8 when there
is more than one cap to hand over.* Transfer-over-IPC (`Endpoint`, v0.8) is how
caps flow *after* startup; `BootInfo` is how the *initial* set arrives.

## Key gotchas (collected)

1. **Lock discipline.** `CapTable` behind `kernel::sync::Mutex`; resolve
   takes it briefly; **never** hold across `sret`/`yield_now`. Same rule
   that bit v0.5/v0.6 (`virtio_console::send` MutexGuard).
2. **`resolve` is a trust boundary.** A bad handle from U-mode must
   return `CapError`, never index out of bounds or alias a stale slot —
   that's the whole point of validating against the caller's table.
   Generation tag (Step 2) is what makes "stale slot" a *rejection*
   rather than a *wrong cap*.
3. **Identity stays kernel-stamped.** The program passes `value` only,
   never an identity — that's what makes forgery structurally
   impossible. Don't add an identity arg "for flexibility."
4. **No string crosses the boundary** (SUM=0 preserved). The cap names
   the counter; the user names nothing. If a future syscall reads a user
   buffer, set `SUM` only around that access (flagged in v0.7a).
5. **`sepc += 4`** still required — unchanged from v0.7a, but the refactor
   must not drop it.
6. **Denied-counter emission from trap context** follows the deferred-
   emission rule: bump an atomic / register the counter at boot, never
   intern a *new* string inside the trap handler.
7. **Renumbering `Syscall`** is safe here because `Invoke` *replaces* the
   sole `EmitMetric` (value 0 reused). It is a register ABI, not a
   postcard wire format — no positional-compat concern. Do **not**
   carry both; the milestone's whole point is that the ambient one is
   *gone*.

## Open questions to settle during impl

- **Current-process reachability from trap context:** per-CPU
  `CURRENT_PROCESS` pointer (lean — generalizes to v0.8) vs. hanging it
  off `CURRENT_TASK` state.
- **Handle bit split:** index/generation widths in the `u32` (table is
  tiny in v0.7b — generous generation bits are cheap).
- **`Process` home:** keep it in `kernel/src/user.rs` or split a
  `kernel/src/process.rs` now (lean: split now — v0.8 grows it).
- **Denial signal:** counter only (lean) vs. a `CapDecision` `Frame`
  variant for the per-decision trace demo (revisit for the post).
- **Does the kernel adopt caps internally anywhere in v0.7b?** Design doc
  leaves this open. Lean **no** for the milestone — userspace boundary
  only; internal adoption is a later, separate judgement.

## The post angle (v0.7b)

"Why I rewrote the syscall layer: ambient authority vs. capabilities."
The before/after is a single trap arm, but the *idea* is the project's
identity crystallizing — the kernel surface becomes "invoke a capability
you were granted," and a process reaching for authority it doesn't hold
is **refused and snitched on in the same breath**. The two pillars
(observability + capabilities) touch exactly at
`snitchos.cap.denied_total`: every authority decision is observable.
v0.7a built the *before* on purpose; this is where it gets paid off.

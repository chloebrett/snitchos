# v0.4 step 4 — kernel heap

Add a `#[global_allocator]` backed by frames. After this step, the
kernel can use `Box`, `Vec`, `String`, `BTreeMap`, and anything else
in `alloc::`. Sub-page allocations stop requiring fixed-size arrays.

Conceptual walkthrough was done interactively (no separate concepts
doc); the recap is in §"Why these choices" below. This doc is the
locked-in shape.

## Why this step

Step 3 gave us 4 KiB pages. Most kernel data isn't 4 KiB:

- Telemetry data structures (spans, intern entries) want
  `Vec`/`BTreeMap`, not fixed arrays.
- Future per-process state (`TaskControlBlock`, FD tables) is a
  natural fit for `Box<T>`.
- String formatting (`format!`, `to_string`) needs a heap.

Without a sub-page allocator we either pre-declare every collection
size at link time or build ad-hoc carve-up logic at every call site.
With it, the `alloc` crate gives us the standard collections for
free.

## Decisions locked in

| decision | choice |
|---|---|
| Allocator crate | `linked_list_allocator` (first-fit, single region) |
| Heap region strategy | (a) reuse linear map — one contiguous frame run at boot |
| Bootstrap stage | None. Frame allocator is already live before heap init |
| Heap size | 4 MiB (1024 frames) at boot, fixed for v0.4 |
| Heap VA | `pa_to_kernel_va(first_frame_pa)` — i.e. inside the existing linear-map window |
| Init order | After `frame::init_from_dtb`, before `unmap_identity` |
| `alloc_error_handler` | Default panic for now (revisit when something real depends on it) |
| Instrumentation | Atomic counters; drained by heartbeat loop (same pattern as frames / IRQs) |
| Contiguous-frames API | Explicit `Bitmap::alloc_contiguous(n)` rather than relying on `alloc()` ordering |

## Why these choices

- **`linked_list_allocator`** — ~300 LOC, widely used in `no_std`
  kernels, simple to wire. Fragmentation is not a real workload concern
  yet. `talc` is the modern alternative; can swap later without
  changing call sites since both go through `GlobalAlloc`.
- **Region strategy (a)** — incremental and low-risk. We already
  have the linear map; one contiguous frame run gives us a contiguous
  VA range with zero new page-table code. Strategy (b) — a dedicated
  VA range with per-frame PTE install — is the right long-term answer
  (it lets the heap grow without finding a new contiguous PA run) but
  it needs a PTE-mutation API we don't have yet. Defer to a
  fast-follow when step 5+ forces it.
- **No bootstrap stage** — the concepts doc anticipated needing a
  `static mut [u8; 65536]` because frame allocator init might want
  `Vec`. The actual step 3 init doesn't allocate — it's a per-frame
  loop against three reservation ranges. So heap init can simply call
  `frame::alloc_contiguous(1024)` and hand the result to
  `linked_list_allocator::init`. One stage, no leak.
- **4 MiB** — comfortable for v0.4. 128 MiB total RAM, ~70 KiB kernel
  + 144 KiB tables/bitmap, leaves ~127 MiB. 4 MiB is generous for
  what the kernel allocates today (probably KiB-scale) and easy to
  bump if a workload needs more.
- **Explicit `alloc_contiguous`** — `Bitmap::alloc` currently returns
  the lowest free frame, so calling it N times sequentially at boot
  returns N contiguous PAs by luck of layout (free run starts at
  `__kernel_end` and runs to the DTB frame). Relying on that as an
  implicit contract is brittle; making it explicit pins the
  guarantee in tests and at the type level.

## Architectural details

### `Bitmap::alloc_contiguous`

```rust
// kernel-core/src/frame.rs
impl<'a> Bitmap<'a> {
    /// Allocate `n` consecutive free frames. Returns the starting frame
    /// index, or `None` if no run of `n` exists. Marks all `n` bits used
    /// and decrements `frames_free` by `n` on success.
    pub fn alloc_contiguous(&mut self, n: usize) -> Option<usize> { ... }
}
```

Implementation: scan bits for the first run of `n` consecutive 1s. A
naive linear scan is fine (O(capacity) worst case, called once at
boot). Optimisation can wait.

Host tests:
- `alloc_contiguous_returns_run_start`
- `alloc_contiguous_spans_word_boundary` — run straddles a `u64` word
- `alloc_contiguous_returns_none_when_no_run_fits`
- `alloc_contiguous_skips_used_bits_in_middle`
- `alloc_contiguous_decrements_frames_free_by_n`
- `alloc_contiguous_zero_is_noop` (or rejected — pick a contract)

### `kernel::frame::alloc_contiguous`

```rust
// kernel/src/frame.rs
pub fn alloc_contiguous(n: usize) -> Option<PhysFrame> {
    let bm = ALLOCATOR.get()?;
    let start_idx = bm.lock().alloc_contiguous(n)?;
    let pa = ram_base() + start_idx * FRAME_SIZE;
    // also bump frames.allocated_total by n
    Some(PhysFrame::from_pa(pa))
}
```

Returns the first frame; the caller knows the count (it asked for
`n`). Counter bookkeeping bumps by `n`, not 1.

### `kernel::heap`

```rust
// kernel/src/heap.rs
use linked_list_allocator::LockedHeap;

#[global_allocator]
static HEAP: LockedHeap = LockedHeap::empty();

const HEAP_FRAMES: usize = 1024;  // 4 MiB

pub fn init() {
    let first = frame::alloc_contiguous(HEAP_FRAMES)
        .expect("heap init: no contiguous run of 1024 frames");
    let va = mmu::pa_to_kernel_va(first.pa()) as *mut u8;
    let size = HEAP_FRAMES * FRAME_SIZE;
    // SAFETY: `va..va+size` is a contiguous, writable, kernel-only
    // region backed by frames just reserved by us. Nothing else can
    // alias it because the frame allocator just handed it to us.
    unsafe { HEAP.lock().init(va, size); }
}
```

`LockedHeap` is the `linked_list_allocator` type that implements
`GlobalAlloc` over a `spin::Mutex<Heap>`. Same lock discipline as
the existing kernel statics.

### Metrics emission, deferred-style

```rust
// kernel-core/src/heap_stats.rs
pub static ALLOC_COUNT: AtomicU64 = AtomicU64::new(0);
pub static DEALLOC_COUNT: AtomicU64 = AtomicU64::new(0);
pub static ALLOC_FAILED: AtomicU64 = AtomicU64::new(0);
pub static BYTES_USED: AtomicUsize = AtomicUsize::new(0);
```

`alloc`/`dealloc` cannot emit frames themselves — the virtio TX path
takes a `Mutex` that, on first use, registers strings through the
intern table which also locks. Anything reachable from inside
`alloc` taking a lock that, on a different code path, allocates →
deadlock.

So: `GlobalAlloc::alloc` does a `linked_list_allocator` call, then
bumps the atomics with relaxed ordering. The heartbeat thread reads
them and emits `Metric` frames. Same shape as
`snitchos.frames.allocated_total` from step 3.

Metric names:

- `snitchos.heap.alloc_total` — counter
- `snitchos.heap.dealloc_total` — counter
- `snitchos.heap.alloc_failed_total` — counter
- `snitchos.heap.bytes_used` — gauge

`bytes_used` increments by `layout.size()` on success, decrements
on `dealloc`. Approximate (`linked_list_allocator` has internal
overhead per allocation), but the curve is what Grafana needs.

### Init order in `kmain`

```rust
mmu::enable(...);                 // step 2
frame::init_from_dtb(...);        // step 3
heap::init();                     // step 4 — NEW
mmu::unmap_identity();            // step 2's tail
// ... heartbeat loop ...
```

Heap init lives in the same window as frame init: paging on, linear
map live, identity map still around (irrelevant — we don't touch
it). `unmap_identity` happens after.

### `alloc_error_handler`

Stable Rust 1.68+ has a default — panic with a generic message. We
take the default for now. If a smoke ever runs the heap dry and the
panic message isn't informative enough on the wire, we can install a
custom handler that emits a frame first.

## Integration scenario: `kernel-heap`

`xtask/src/itest/scenarios.rs`:

1. Boot the kernel.
2. Within 5 s, observe a `snitchos.heap.bytes_used` metric ≥ some
   threshold (set by a heartbeat-driven smoke `Box::new([0u8; 256])`
   that intentionally leaks for the scenario, or by feature-gating
   an explicit alloc).
3. Observe `snitchos.heap.alloc_total > 0`.
4. Observe at least one heartbeat after the smoke runs (assert the
   heap doesn't break the boot path).

Same harness pattern as `frame-allocator-metrics`. No new feature
flag needed for the basic case — a small `Box::new` in the
heartbeat smoke is harmless. If we want to demonstrate
`alloc_failed_total > 0` like the OOM frame scenario, that's a
separate `heap-oom` scenario behind a feature, follow-up.

## What this doesn't do

- **Growable heap.** Fixed at 4 MiB. If anything needs more, bump
  `HEAP_FRAMES` and recompile. Real growth requires strategy (b) —
  a dedicated VA range with per-frame PTE install — and that needs a
  page-table mutation API.
- **Multiple heaps / per-cpu heaps.** v0.4 is single-core; revisit
  when SMP arrives.
- **`alloc_zeroed` optimisation.** `linked_list_allocator` provides a
  default that calls `alloc + memset`. Fine.
- **Slab caches.** If kernel-heavy workloads later show fragmentation
  from same-sized allocations, drop in a `slab_allocator` layer above
  `linked_list_allocator`. Not now.
- **Frame strategy (b) — dedicated VA range with PTE-install.** Punted
  to a future step. The PTE-install API will be needed for v0.5+
  anyway; the heap will be its first customer once it lands.
- **Allocator A/B comparison.** Fast-follow: swap in `talc` behind the
  same `#[global_allocator]` and compare against `linked_list_allocator`
  under representative workloads (steady-state churn, fragmenting
  churn, OOM). For that comparison to be meaningful, the
  instrumentation in this step needs to be rich enough to see the
  difference — i.e. don't just track `bytes_used`; also expose enough
  signal to detect fragmentation behaviour and tail latencies. See the
  "Instrumentation rigour" note below.

### Instrumentation rigour (forward-looking)

The four metrics above are the floor. To make a later
`linked_list_allocator` vs `talc` comparison fair, consider adding —
either now or in a follow-up — at least:

- `snitchos.heap.bytes_capacity` — gauge, the total heap region size
  (so `bytes_used / bytes_capacity` is a meaningful ratio).
- `snitchos.heap.largest_free_block` — gauge, fragmentation indicator.
  `linked_list_allocator` exposes this; `talc` should too.
- `snitchos.heap.free_block_count` — gauge, count of free-list nodes.
  Rising-while-`bytes_used`-flat is the classic fragmentation
  signature.
- Per-call-size histogram or at minimum size buckets — `talc`'s
  advantage shows up under specific size mixes; without size signal
  we can't attribute differences.

For this step we ship the four basic counters; the richer
fragmentation/size signals are the precondition for the A/B
comparison and can land alongside the `talc` evaluation.

## Risk and known unknowns

- **`linked_list_allocator` interior locks.** `LockedHeap` uses a
  `spin::Mutex`. If anything reachable from inside `alloc` tries to
  allocate again, deadlock. The deferred-emission pattern is
  precisely the guardrail; do not be tempted to emit a frame from
  inside `GlobalAlloc::alloc`.
- **First `alloc` after enabling.** Will the very first allocation
  trigger anything weird? Most likely no — the heap is initialised
  with a known region; first `alloc` is just a list walk. Worth
  watching the boot trace in QEMU for an unexpected fault.
- **`format!` from inside a panic handler.** Already a risk; the
  heap doesn't make it worse, but be aware that a panic emitting a
  formatted message now goes through the heap. If the heap is the
  thing that panicked, we re-enter. Mitigation: keep `panic_handler`
  using `core::fmt::Write` into a stack buffer if it currently does;
  if not, this is a follow-up.
